Migration: Host Snapshots to Deterministic Lockstep

This is the plan to delete the authoritative host, snapshot wire, client prediction, and reconciliation model. The destination is the one defined in design_philosophy.html: every peer runs the same deterministic simulation, and only player command bundles cross the wire.

Working title: Budget Annihilation. Repo: rts-prototype. Reference feel: Beyond All Reason. Reference networking shape: classic deterministic RTS lockstep, not server-authoritative shooter snapshots.

How To Work This Plan

This document is live migration state, not archival notes. A partially finished chunk is rewritten to describe only the remaining work. A wrong chunk is corrected before or while the code changes land. Newly discovered work is added as a new chunk with dependencies and an exit check.

  2. Take one actionable chunk at a time. If the chunk is too broad to finish cleanly, split it into smaller chunks first and commit that doc update with the code or as its own preparation commit.
  3. When code or docs change, update this file in the same commit: delete completed chunks, rewrite changed chunks, and add follow-up chunks created by discoveries made during the work.
  4. Do not leave stale checklists, checked-off bullets, or completed items in this plan. The source of truth is the remaining actionable work, not a history of what used to be true.
  5. Each migration change ends with the relevant verification, git status, a focused git commit, and git push. If the branch has no upstream, push with git push -u origin <branch>.
Delete host truth There is no final architecture where GameServer owns the world and remote clients hydrate it from snapshots. A relay may order commands, but it must not own truth.

Current Reality

The existing codebase is a successful host-snapshot prototype. It is not a lockstep engine with a networking swap left to do. The migration must be honest about these facts:

  • src/game/server/GameServer.ts owns the headless world, runs a host tick with ServerTickLoop, and emits snapshots through ServerSnapshotPublisher.
  • The current tick is driven by main-thread setInterval and variable deltaMs. Lockstep needs a fixed simulation step gated by command availability.
  • The simulation is not yet "one Rust crate." rts-sim-wasm/src/lib.rs is about 19k lines of dense kernels, slabs, and snapshot encoding, while much of the gameplay loop still lives in TypeScript under src/game/sim/.
  • Command.tick already exists, but commands enter through a host authority path. Several server-control commands apply immediately instead of going through a deterministic per-tick command stream.
  • NetworkCommandTransport.ts is client-to-host. It sends individual commands, not per-peer per-tick bundles, and it relies on NetworkRole = 'host' | 'client'.
  • NetworkSnapshotTransport.ts, stateSerializer*.ts, snapshot*.ts, and the Rust snapshot encoders are a large state-replication system. Lockstep deletes that surface.
  • ClientViewState, ClientPredictionStepper, ClientUnitPrediction, and ClientProjectilePrediction are prediction/reconciliation machinery. In lockstep, the local sim is the state.
  • Fog of war is currently partly a per-recipient snapshot filter. In lockstep, every peer has full state; fog is a local render/query filter over deterministic visibility aggregates.

Do not build "lockstep" on top of the current GameServer mental model and call the job done. A host that still simulates truth, filters visibility, and streams view state is the old architecture with smaller packets.

Target Contract

Area Target rule
Simulation owner Every peer owns a local instance of the same Rust/WASM simulation. TypeScript coordinates boot, IO, UI, and rendering. It does not own per-unit or per-tick gameplay loops.
Tick model Fixed 60 Hz simulation ticks. Time enters as tick count, not wall-clock delta. The sim advances tick N only after the lockstep barrier has every peer's bundle for N.
Wire model Per-peer command bundles: { target_tick, peer_id, seq, commands }. Empty bundles are mandatory. Hash diagnostics are allowed. World state is not.
Command order Commands execute in a stable order, initially (target_tick, peer_id, seq, command_index). All peers derive the same order from bytes, not arrival timing.
Initial state Game settings, map seed, player slots, blueprint versions, and initial RNG seed are a signed starting manifest. A replay is that manifest plus the ordered command log.
Determinism Every future-affecting field is covered by world_hash(). Peers exchange hashes every K ticks in dev and test. Divergence pauses the match and writes artifacts for diffing.
Renderer The renderer samples local sim output and keeps purely visual state local: command hints, camera, selection, locomotion cosmetics, audio scheduling, and fog presentation.
Cheat model Every peer has full state. Maphacks are accepted for this architecture. Ranked/untrusted play requires a future observer/relay architecture outside this migration.

Migration Rules

  1. No new snapshot features. Snapshot work is limited to bug fixes needed to keep the prototype usable while the replacement lands.
  2. No new simulation hot paths in TypeScript. New gameplay, physics, targeting, visibility, command validation, or per-unit loops belong in Rust/WASM unless they are temporary scaffolding with a deletion step.
  3. One behavior, one implementation. When a Rust/WASM path replaces a TypeScript path, the TypeScript path is deleted in the same migration chunk.
  4. Feature flags expire quickly. A flag may bridge a risky merge, but the next chunk must either delete the old path or delete the new experiment.
  5. Every chunk has a determinism check. If the chunk changes sim state, it adds or updates a replay/hash test before it is considered complete.
  6. Every command is data. Commands use stable numeric IDs, append-only enums, explicit nullability, and schema-validated payloads. No deep JSON object trees for per-tick wire traffic.
  7. Silence is a packet. Idle peers send empty bundles. A missing bundle means "cannot advance," not "nothing happened."

Actionable Chunks

These chunks are ordered by dependency, but they are not permanent milestones. Work the first chunk that is both useful and unblocked. If a chunk becomes outdated, rewrite it before using it as instructions.

LS-05: Replace Sim-Facing Variable Delta With Fixed Ticks determinism

Depends: none. Touch: ServerTickLoop.ts, GameServer.tick, Simulation.update, PhysicsEngine3D.step, wind/economy/building systems that consume dtMs.

  • Make sim-facing update code receive integer tick plus constant SIM_STEP_MS. Wall-clock time may decide catch-up count, but not sim math.
  • Remove MAX_TICK_DT_MS as gameplay behavior. If catch-up needs a cap, it pauses or reports load instead of stretching the tick.
  • Update wind, economy, construction, physics, and force systems to use the fixed step consistently.

Exit: local runs at different render frame rates produce the same tick count and same sim-facing timestep sequence.

LS-06: Add World Hash V1 desync detection

Depends: LS-05. Touch: WorldState.ts, Simulation.ts, physics/projectile pools, economy, visibility, command queues, rts-sim-wasm/src/lib.rs.

  • Add world_hash() over all fields that affect future ticks: entities, physics bodies, projectiles, economy, build queues, targeting state, visibility aggregates, RNG, game phase, and pending commands.
  • Use stable ordering by IDs or sorted keys. Do not hash object iteration order from Map, Set, HashMap, or HashSet unless that order is made deterministic first.
  • Expose hash history in a dev-readable form so future chunks can compare peers and replay runs.

Exit: the same local scenario run repeatedly emits identical per-tick hashes, and a deliberate future-affecting mutation changes the hash.

LS-07: Build A Headless Replay Harness test foundation

Depends: LS-06. Touch: scripts/, npm scripts, manifest and command-bundle codec, sim bootstrap paths.

  • Create a headless runner that loads a manifest plus command bundles, advances fixed ticks, and writes per-tick hashes.
  • Keep the harness independent of DOM, Vue, Three.js, PeerJS, and render code.
  • Add at least one small deterministic fixture and one longer smoke fixture that exercises movement, construction, projectiles, economy, and fog-relevant visibility.

Exit: a repo command can replay a command log headlessly and fail if any expected hash differs.

LS-08: Audit And Fence Nondeterminism correctness

Depends: LS-07. Touch: rts-sim-wasm/src/lib.rs, src/game/sim/**, src/game/server/**, targeted static checks.

  • Classify every sim-affecting use of Math.random, Date.now, performance.now, sin, cos, pow, HashMap, and HashSet.
  • Move gameplay randomness to seeded in-state RNG. Keep cosmetic randomness in render/audio code only, with comments where confusion is likely.
  • Route sim-affecting transcendentals through deterministic helpers or authored lookup tables. Replace outcome-affecting unordered iteration with stable structures or sorted traversal.

Exit: grep output has an allow-list for remaining nondeterministic APIs, and the headless replay fixtures remain hash-stable.

LS-09: Create The Rust Sim Runtime API WASM boundary

Depends: LS-08. Touch: rts-sim-wasm/src/, src/game/sim-wasm/init.ts, generated wasm bindings.

  • Add a narrow runtime surface: initialize from manifest, enqueue command bundles, advance one tick, read render packets, read diagnostics, and compute world_hash().
  • Keep TypeScript as the caller and owner of browser APIs only. Do not add new gameplay behavior to TypeScript while creating this boundary.
  • Split Rust modules out of lib.rs when it helps isolate runtime ownership, but avoid refactors not needed by the chunk.

Exit: TypeScript can instantiate a Rust runtime object from a manifest and advance a no-op or minimal scenario with a stable hash.

LS-10: Move World Bootstrap And Storage Behind The Runtime ownership

Depends: LS-09. Touch: ServerBootstrap.ts, WorldState.ts, spawn/player layout, terrain/deposit generation, Rust world storage.

  • Make manifest-driven world creation happen behind the Rust runtime for the active path.
  • Port entity IDs, transforms, ownership, teams, health, build state, and initial terrain/deposit facts into runtime-owned storage.
  • Delete or shrink TypeScript bootstrap pieces as Rust ownership takes over. Do not leave a second gameplay world as a supported fallback.

Exit: a local battle can create its initial world from the Rust runtime, and TypeScript reads it only through runtime projections or diagnostics.

LS-11: Move Command Validation And Application Into Rust commands

Depends: LS-10. Touch: commandSanitizer.ts, commandAuthority.ts, commandExecution.ts, Rust command modules.

  • Port command payload validation, ownership checks, and gameplay application into the runtime.
  • Convert server-control toggles that affect gameplay, such as fog, mirrors, force fields, tick rate, and converter tax, into manifest fields or scheduled commands.
  • Keep selection and camera local. Commands that act on selected units must carry explicit IDs.

Exit: command bundles applied by Rust produce the same gameplay effects as the old TypeScript path for covered command fixtures, and the old execution path is removed for those commands.

LS-12: Port Economy, Construction, And Factory Queues sim systems

Depends: LS-11. Touch: economy.ts, construction*.ts, factoryProduction.ts, buildingActiveState.ts, Rust runtime modules.

  • Move income, spending, converter tax, construction progress, factory queue advancement, and building active/open state into Rust-owned tick logic.
  • Hash every future-affecting queue, stockpile, progress, and active-state field.
  • Delete TypeScript implementations for systems fully replaced in the active path.

Exit: build/economy replay fixtures run through Rust-owned systems and match expected hashes without TypeScript gameplay fallback.

LS-13: Port Physics Body Ownership And Unit Motion physics

Depends: LS-12. Touch: PhysicsEngine3D.ts, UnitForceSystem.ts, unit motion helpers, Rust body pools.

  • Make Rust own body creation, force accumulation, gravity, terrain contact, collision, integration, and entity/body sync.
  • Remove the TypeScript server-side physics ownership path as each body category moves.
  • Preserve the design philosophy: mass, force, acceleration, velocity, position, and collision are the source of gameplay motion.

Exit: movement and collision replay fixtures advance through Rust-owned physics and produce stable hashes.

LS-14: Port Projectiles, Damage, Targeting, And Visibility combat

Depends: LS-13. Touch: src/game/sim/combat/**, DamageSystem.ts, visibility/sensor systems, Rust combat modules.

  • Move projectile stepping, collision, damage application, turret targeting, line of sight, radar/sight aggregates, cloak/detection, and scan pulses into Rust-owned tick logic.
  • Make fog of war a local render/query filter over full deterministic state. Do not preserve per-recipient snapshot filtering as gameplay architecture.
  • Hash targeting state, projectile state, pending events, visibility aggregates, and RNG used by combat outcomes.

Exit: combat and fog fixtures run from command bundles through Rust-owned systems with stable hashes and no state-wire dependency.

LS-15: Define Local Render Packets render boundary

Depends: LS-14. Touch: Rust runtime render projections, ClientViewState.ts, snapshot intake helpers, renderer scene binding.

  • Define process-local render packets emitted by the sim runtime for the renderer. They may contain positions, events, visibility tiers, economy display data, and health bars because they never cross the network.
  • Separate render packets from network snapshots in names, types, and module paths.
  • Keep camera, selection, pending command hints, audio spread, and purely visual interpolation outside deterministic state unless they affect gameplay.

Exit: the renderer can consume local render packets for a local game path without asking ServerSnapshotPublisher for world state.

LS-16: Move The Sim Runtime Into A Worker threading

Depends: LS-15. Touch: new src/game/sim-wasm/simWorker.ts, worker message types, game canvas lifecycle, runtime initialization.

  • Load and own the Rust runtime inside a Web Worker. The main thread sends command intents and receives render packets and diagnostics.
  • Implement a fixed-step worker tick pump that can catch up without using main-thread setInterval as sim time.
  • Add lifecycle handling for start, pause, stop, teardown, and manifest mismatch errors.

Exit: a local worker-backed battle runs for 30 seconds, can be backgrounded and foregrounded, and continues without main-thread timing becoming simulation time.

LS-17: Build Single-Peer Lockstep Locally wire before wire

Depends: LS-16. Touch: local session orchestration, input helpers, command-bundle scheduler, game canvas handoff.

  • Make local play send its own commands through the same command-bundle scheduler the network will use.
  • Apply input delay in local play. Render pending command hints immediately, but keep them out of sim state.
  • Emit empty bundles for idle ticks so the barrier semantics exist before WebRTC is involved.

Exit: single-player/offline play advances only from manifest plus command bundles, and headless replay of the same bundles matches local hashes.

LS-18: Save And Load Command-Log Replays replay

Depends: LS-17. Touch: replay file types, persistence, headless replay harness, optional UI affordance.

  • Write replay files containing manifest, sim/command schema versions, ordered command bundles, and optional hash checkpoints.
  • Load replay files by rebuilding the sim from the manifest and replaying command bundles. Do not add snapshot replay truth.
  • Reject incompatible replays explicitly when manifest, blueprint, command, or sim versions do not match.

Exit: any local command-log replay can be loaded headlessly and produces its recorded hash checkpoints.

LS-19: Reshape WebRTC Command Transport network

Depends: LS-18. Touch: NetworkCommandTransport.ts, NetworkManager.ts, NetworkTypes.ts, data-channel monitor/heartbeat integration.

  • Replace client-to-host individual command messages with peer-symmetric command-bundle messages.
  • Keep PeerJS signaling, room codes, lobby roster, data-channel health, and heartbeat. Remove gameplay authority from NetworkRole = 'host' | 'client' paths.
  • Send one bundle per peer per target tick, including empty bundles. Start with two peers and no relay election.

Exit: two connected peers can exchange validated command bundles without sending gameplay state snapshots.

LS-20: Add Lockstep Barrier And Hash Diagnostics Over WebRTC network correctness

Depends: LS-19. Touch: worker lockstep scheduler, network diagnostics, hash history, desync artifact writer.

  • Release tick N only after every peer's validated bundle for tick N is present.
  • Broadcast world_hash() every 60 ticks in dev/test. On mismatch, pause, name the tick, and write manifest, command log, hash history, and compact state fingerprints.
  • Expose barrier wait in UI/diagnostics so a missing peer is visible instead of feeling like silent local lag.

Exit: artificial late, duplicate, and missing bundles produce deterministic barrier behavior and clear diagnostics on both peers.

LS-21: Prove Two-Peer Lockstep Gameplay first playable

Exit: completed matches and live spectator sessions both run from manifest plus command bundles, with explicit version compatibility checks.

Tests And Gates

  • Fixed-step replay: one command log, one manifest, identical per-tick hashes across repeated headless runs.
  • Browser determinism: the same 10-minute replay in Chrome, Firefox, and Safari via Playwright, 20 times back-to-back.
  • Hash coverage regression: adding a future-affecting field without hashing it fails a targeted test or linter check.
  • Transport invariant: command bundle tests prove empty bundles are emitted, late bundles stall, duplicate bundles are rejected idempotently, and arrival order cannot affect execution order.
  • No-state-wire regression: once LS-22 and LS-23 land, tests or static checks fail if gameplay network messages contain entity, projectile, economy, minimap, or visibility state.
  • Backgrounding: a worker sim survives tab backgrounding without main-thread timing becoming simulation time.
  • Replay compatibility: replay loader validates manifest version, blueprint version, command schema version, and sim version before playback.

Do Not Build

  • Do not add rollback netcode. Predict-and-rewind is a different architecture with a different state-save cost model.
  • Do not build host migration for the old snapshot server. The host goes away; that work dies with it.
  • Do not add server-filtered fog to lockstep. Full state exists on every peer by design.
  • Do not keep "prediction mode" as a player-facing option after lockstep. Pending command hints are visual feedback, not gameplay prediction.
  • Do not use periodic state snapshots as replay truth. Optional checkpoints may accelerate seeking, but command logs remain authoritative.
  • Do not make PeerJS roles mean gameplay authority. At most, a peer can be a signaling owner or command relay.

What Done Looks Like

An 8-player game runs with every peer simulating the same Rust/WASM world from the same ordered command stream. Bandwidth does not scale with unit count. Replays are small command logs. Spectators run the sim. Fog is local presentation over full state. A desync is detected quickly, paused clearly, and debugged from artifacts. The host/client snapshot mental model is gone from gameplay code, not merely hidden behind a new filename.

Migration rule of thumb: if world state still crosses the network, if a host still owns truth, or if TypeScript still owns the sim hot path, the migration is still in progress.
