Migration: Host Snapshots to Deterministic Lockstep

This is the plan to delete the authoritative host, snapshot wire, client prediction, and reconciliation model. The destination is the one defined in design_philosophy.html: every peer runs the same deterministic simulation, and only player command bundles cross the wire.

Working title: Budget Annihilation. Repo: rts-prototype. Reference feel: Beyond All Reason. Reference networking shape: classic deterministic RTS lockstep, not server-authoritative shooter snapshots.

Delete host truth There is no final architecture where GameServer owns the world and remote clients hydrate it from snapshots. A relay may order commands, but it must not own truth.
Delete state on the wire The wire format is scheduled player commands plus hash diagnostics. Entity rows, projectile rows, visibility payloads, minimap rows, and delta baselines are migration debt.
Delete the old path A phase is not complete while the replacement and predecessor both compile as supported gameplay paths. Temporary flags are allowed only with a named removal phase.

Current Reality

The existing codebase is a successful host-snapshot prototype. It is not a lockstep engine with a networking swap left to do. The migration must be honest about these facts:

  • src/game/server/GameServer.ts owns the headless world, runs a host tick with ServerTickLoop, and emits snapshots through ServerSnapshotPublisher.
  • The current tick is driven by main-thread setInterval and variable deltaMs. Lockstep needs a fixed simulation step gated by command availability.
  • The simulation is not yet "one Rust crate." rts-sim-wasm/src/lib.rs is about 19k lines of dense kernels, slabs, and snapshot encoding, while much of the gameplay loop still lives in TypeScript under src/game/sim/.
  • Command.tick already exists, but commands enter through a host authority path. Several server-control commands apply immediately instead of going through a deterministic per-tick command stream.
  • NetworkCommandTransport.ts is client-to-host. It sends individual commands, not per-peer per-tick bundles, and it relies on NetworkRole = 'host' | 'client'.
  • NetworkSnapshotTransport.ts, stateSerializer*.ts, snapshot*.ts, and the Rust snapshot encoders are a large state-replication system. Lockstep deletes that surface.
  • ClientViewState, ClientPredictionStepper, ClientUnitPrediction, and ClientProjectilePrediction are prediction/reconciliation machinery. In lockstep, the local sim is the state.
  • Fog of war is currently partly a per-recipient snapshot filter. In lockstep, every peer has full state; fog is a local render/query filter over deterministic visibility aggregates.

Do not build "lockstep" on top of the current GameServer mental model and call the job done. A host that still simulates truth, filters visibility, and streams view state is the old architecture with smaller packets.

Target Contract

Area Target rule
Simulation owner Every peer owns a local instance of the same Rust/WASM simulation. TypeScript coordinates boot, IO, UI, and rendering. It does not own per-unit or per-tick gameplay loops.
Tick model Fixed 60 Hz simulation ticks. Time enters as tick count, not wall-clock delta. The sim advances tick N only after the lockstep barrier has every peer's bundle for N.
Wire model Per-peer command bundles: { target_tick, peer_id, seq, commands }. Empty bundles are mandatory. Hash diagnostics are allowed. World state is not.
Command order Commands execute in a stable order, initially (target_tick, peer_id, seq, command_index). All peers derive the same order from bytes, not arrival timing.
Initial state Game settings, map seed, player slots, blueprint versions, and initial RNG seed are a signed starting manifest. A replay is that manifest plus the ordered command log.
Determinism Every future-affecting field is covered by world_hash(). Peers exchange hashes every K ticks in dev and test. Divergence pauses the match and writes artifacts for diffing.
Renderer The renderer samples local sim output and keeps purely visual state local: command hints, camera, selection, locomotion cosmetics, audio scheduling, and fog presentation.
Cheat model Every peer has full state. Maphacks are accepted for this architecture. Ranked/untrusted play requires a future observer/relay architecture outside this migration.

Migration Rules

  1. No new snapshot features. Snapshot work is limited to bug fixes needed to keep the prototype usable while the replacement lands.
  2. No new simulation hot paths in TypeScript. New gameplay, physics, targeting, visibility, command validation, or per-unit loops belong in Rust/WASM unless they are temporary scaffolding with a deletion step.
  3. One behavior, one implementation. When a Rust/WASM path replaces a TypeScript path, the TypeScript path is deleted in the same migration slice.
  4. Feature flags expire quickly. A flag may bridge a risky merge, but the next phase must either delete the old path or delete the new experiment.
  5. Every phase has a determinism check. If the phase changes sim state, it adds or updates a replay/hash test before it is considered complete.
  6. Every command is data. Commands use stable numeric IDs, append-only enums, explicit nullability, and schema-validated payloads. No deep JSON object trees for per-tick wire traffic.
  7. Silence is a packet. Idle peers send empty bundles. A missing bundle means "cannot advance," not "nothing happened."

Phase Plan

Phase 0: Freeze The Old Architecture documentation + guardrails

Make the destination impossible to misunderstand before code starts moving. The current snapshot stack can keep the prototype playable, but it stops being a place where new architecture lands.

  • Rename references from the deleted long-form migration filename to migration.html.
  • Update stale comments that describe Rust/WASM as serving "authoritative server tick and client prediction" once those areas are touched.
  • Mark NetworkSnapshotTransport, snapshot serializers, client prediction, and server snapshot publishers as migration debt in comments near their entry points.
  • Stop adding snapshot DTO fields unless they unblock active gameplay during the migration.

Exit: docs and comments no longer present host authority as the target architecture.

Phase 1: Deterministic Fixed-Step Core before networking

Lockstep cannot start from a variable-delta host tick. First, make the local simulation step deterministic and hashable, even while some of the code still lives in TypeScript.

  • Replace the sim-facing variable deltaMs contract with fixed tick duration. Wall-clock time may decide how many ticks to run, but the sim receives an integer tick and a constant step.
  • Add world_hash() coverage for all future-affecting state: entities, physics bodies, projectiles, economy, build queues, command queues, targeting state, visibility aggregates, RNG state, and game phase.
  • Add a replay-from-command-log harness that can run headlessly and emit per-tick hashes.
  • Audit rts-sim-wasm/src/lib.rs and TypeScript sim paths for sim-affecting transcendentals, unordered iteration, wall-clock reads, ambient random, and parallel reductions.
  • Current Rust audit items already visible by grep: HashMap/HashSet and sin/cos/pow occur in rts-sim-wasm/src/lib.rs. Delete snapshot-only uses with the snapshot path; route sim-affecting math through deterministic helpers.
  • Make CommandQueue execute exact ticks, not "all commands with tick <= current tick," once delayed scheduling owns missed/late command policy.

Exit: the same 10-minute command log produces identical hashes 20 times in a row in local headless runs.

Phase 2: Rust/WASM Owns The Sim engine ownership

The design philosophy's target is one Rust simulation, not a TypeScript sim with Rust helper kernels. Move ownership behind a Rust runtime boundary before treating network lockstep as solved.

  • Create a narrow sim runtime API around Rust/WASM: initialize from manifest, enqueue scheduled command bundles, advance one tick, read render packets, read diagnostics, compute world_hash().
  • Port TypeScript-owned sim systems into Rust in dependency order: world storage, command application, physics body ownership, economy, projectiles, damage, targeting, visibility, construction, pathing, and factory queues.
  • Keep configuration as JSON data validated by shared schema. Rust reads the same blueprints that TypeScript currently imports.
  • Delete TypeScript implementations as their Rust replacements become live. Do not leave "fallback" gameplay code in the tree.
  • Split rts-sim-wasm/src/lib.rs into owned modules as needed; the important part is ownership and determinism, not preserving the monolith.
  • Remove Rust snapshot-baseline and snapshot-encoder exports once render packets replace network snapshots.

Exit: a local game can run from a Rust/WASM sim runtime without GameServer owning gameplay truth.

Phase 3: Worker Runtime And Render Sampling threading

The sim cannot live on the main thread. Backgrounded tabs, rendering stalls, and UI work would otherwise stall the lockstep barrier for every peer.

  • Add a sim worker, likely under src/game/sim-wasm/simWorker.ts, that loads the Rust/WASM runtime and owns tick advancement.
  • Main thread sends input-derived command intents and receives local render packets. These packets are process-local, not network snapshots.
  • Replace ServerTickLoop with a worker tick pump that can run catch-up ticks using fixed step semantics.
  • Renderer interpolation samples adjacent local sim frames or sim-authored render events. It never asks the network for positions.
  • Keep camera, selection, pending command hints, audio spread, and fog presentation outside the deterministic sim unless they affect gameplay.

Exit: background a tab for 30 seconds, foreground it, and the worker catches up without main-thread starvation or divergent hashes.

Phase 4: Commands-Only Local Loop And Replay wire before wire

Build the lockstep protocol locally before using WebRTC. Single-player should already behave as "one peer sending command bundles through the same scheduler the network will use."

  • Define command bundle bytes with stable command IDs, append-only enums, peer ID, sequence number, target tick, and explicit empty-bundle representation.
  • All gameplay-affecting changes, including current server-control toggles such as fog, mirrors, force fields, tick rate, and converter tax, become scheduled commands or initial manifest fields.
  • Selection and camera remain local. If a command needs selected IDs, the command payload carries those IDs explicitly.
  • Input delay applies even in local play. Pending command hints render immediately but remain non-sim state.
  • Replay save/load uses the exact starting manifest plus ordered command bundles. No snapshot replay format is introduced.

Exit: local play, replay playback, and headless replay all consume the same command-bundle stream and produce matching hashes.

Phase 5: Two-Peer Lockstep Over WebRTC first network proof

Replace client-to-host commands and host-to-client snapshots with peer-symmetric command exchange for the smallest real case.

  • Reshape NetworkCommandTransport.ts into a peer command-bundle channel. It must not ask whether the local process is gameplay host before accepting a command bundle.
  • Keep NetworkManager for PeerJS signaling, room codes, lobby roster, data channel health, and heartbeat, but separate those concerns from gameplay authority.
  • Every peer sends a bundle for every target tick to every other peer, including empty bundles. Start with two peers before relay election.
  • The worker lockstep barrier releases tick N only when every peer's bundle for N is present and validated.
  • Broadcast world_hash() every 60 ticks in dev/test. On mismatch, pause both peers, dump manifest, command log, local hash history, and compact sim state fingerprints.
  • Do not stream snapshots as a rescue path. A desync is a correctness failure, not an invitation to reconcile.

Exit: a 1v1 LAN game runs for 20 minutes with no desyncs, no network state snapshots, and replayable command logs.

Phase 6: Delete Snapshot And Prediction Gameplay Paths no dual stack

Once two-peer lockstep can play, remove the old host-snapshot architecture before scaling to more players. This is where the migration becomes real.

  • Delete GameServer.ts as gameplay authority. Any reusable bootstrap logic moves behind the Rust sim runtime or manifest builder first.
  • Delete ServerTickLoop.ts, ServerSnapshotPublisher.ts, ServerSnapshotMetaBuilder.ts, and server debug snapshot plumbing.
  • Delete NetworkSnapshotTransport.ts, stateSerializer*.ts, snapshot*.ts, snapshot pool/baseline code, and Rust snapshot encoder exports.
  • Delete ClientPrediction*, ClientUnitPrediction, ClientProjectilePrediction, ClientSnapshotApplier, and reconciliation diagnostics.
  • Replace GameConnection, LocalGameConnection, and RemoteGameConnection with a peer/session abstraction that owns command transport and local sim runtime only.
  • Make ClientViewState either disappear or become a read-only render projection fed by the local sim worker. It must no longer imply "client copy of remote truth."

Exit: grepping for snapshot/prediction authority names finds no supported gameplay path, and the app still plays a 1v1 lockstep match.

Phase 7: N-Peer, Relay Election, And Input Delay scale

Scale only after the old path is gone. More peers amplify every hidden ambiguity in bundle order, timeout policy, and command validation.

  • Start with full mesh up to four players. Every peer still validates every peer's command bundles and hash stream.
  • Add command-relay election for larger rooms. The relay aggregates and rebroadcasts bundles but does not simulate authoritative truth.
  • Measure RTT continuously through the existing heartbeat layer. Set input delay to ceil(max_rtt / tick_ms) + slack and renegotiate on large swings.
  • Show a waiting overlay when the barrier stalls on a peer. Name the peer and tick window; do not hide the stall as local lag.
  • Handle disconnects deterministically: pause for a drop decision, then continue with a new peer manifest or end the match. Do not let each browser improvise.

Exit: 4v4 plays on residential connections with adaptive input delay, no snapshot fallback, and stable replay output.

Phase 8: Spectate, Replays, And Tooling free wins

After the command log is the source of truth, spectators and replays are no longer special state streams. They are late or offline sim runners.

  • Replay files contain manifest, version stamp, command bundles, optional hash checkpoints, and optional state checkpoints for fast seek.
  • Live spectators join by fetching the manifest and command history, replaying forward, then tailing new bundles.
  • Analysis tools read the same command log and can run headless without DOM, Vue, Three.js, or PeerJS.
  • Version compatibility is explicit. If a sim change breaks replay determinism, the replay loader says so instead of pretending.

Exit: any completed match can be loaded and watched from commands alone; live spectate uses replay-then-tail.

Tests And Gates

  • Fixed-step replay: one command log, one manifest, identical per-tick hashes across repeated headless runs.
  • Browser determinism: the same 10-minute replay in Chrome, Firefox, and Safari via Playwright, 20 times back-to-back.
  • Hash coverage regression: adding a future-affecting field without hashing it fails a targeted test or linter check.
  • Transport invariant: command bundle tests prove empty bundles are emitted, late bundles stall, duplicate bundles are rejected idempotently, and arrival order cannot affect execution order.
  • No-state-wire regression: once Phase 6 lands, tests or static checks fail if gameplay network messages contain entity, projectile, economy, minimap, or visibility state.
  • Backgrounding: a worker sim survives tab backgrounding without main-thread timing becoming simulation time.
  • Replay compatibility: replay loader validates manifest version, blueprint version, command schema version, and sim version before playback.

Do Not Build

  • Do not add rollback netcode. Predict-and-rewind is a different architecture with a different state-save cost model.
  • Do not build host migration for the old snapshot server. The host goes away; that work dies with it.
  • Do not add server-filtered fog to lockstep. Full state exists on every peer by design.
  • Do not keep "prediction mode" as a player-facing option after lockstep. Pending command hints are visual feedback, not gameplay prediction.
  • Do not use periodic state snapshots as replay truth. Optional checkpoints may accelerate seeking, but command logs remain authoritative.
  • Do not make PeerJS roles mean gameplay authority. At most, a peer can be a signaling owner or command relay.

What Done Looks Like

An 8-player game runs with every peer simulating the same Rust/WASM world from the same ordered command stream. Bandwidth does not scale with unit count. Replays are small command logs. Spectators run the sim. Fog is local presentation over full state. A desync is detected quickly, paused clearly, and debugged from artifacts. The host/client snapshot mental model is gone from gameplay code, not merely hidden behind a new filename.

Migration rule of thumb: if world state still crosses the network, if a host still owns truth, or if TypeScript still owns the sim hot path, the migration is still in progress.
