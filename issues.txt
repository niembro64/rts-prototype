==============================================================================
 OPTIMIZATION AUDIT — 2026-05-13
==============================================================================

Findings from a five-part read-only audit (sim hot path, network /
serialization, spatial grid, Three.js rendering, Vue UI). Each item has a
file:line citation, a one-line observation of the cost, and a concrete fix.

Impact scale:
  LARGE   ≈ ms+ tick/frame win, or material bandwidth / GC reduction
  MEDIUM  ≈ sub-ms but reproducible; matters under load
  SMALL   ≈ microseconds saved, cleanliness, or correctness guard

Items are grouped by subsystem and ordered LARGE → MEDIUM → SMALL within each.
Some items are marked [VERIFY] — the cost claim wasn't 100% confirmed at audit
time and should be measured (or read more code) before touching.

 ISSUES:

------------------------------------------------------------------------------
 SPATIAL GRID
------------------------------------------------------------------------------

[MEDIUM] GridCell objects never pooled / pruned
  src/game/sim/SpatialGrid.ts:156-161
  Every getOrCreateCell allocates a fresh object; cells whose contents
  drained to empty stay in `this.cells`. Long sessions grow the map.
  Fix: pool GridCell on first emptying (move to a free list) and reclaim
  in subsequent allocations.

[MEDIUM] Line-query overflow defeats partitioning entirely
  src/game/sim/SpatialGrid.ts:1070-1072
  When a line query would span > 4096 cells, the fallback dumps EVERY
  occupied cell into nearbyCells. Safety valve, but turns one pathological
  beam into an O(world) scan.
  Fix: at minimum, fast-reject cells whose AABB doesn't intersect the
  segment (cheap) before adding. Or cap the result count and let the
  caller's per-entity filter do the work.

[SMALL] Capture vote removal is linear scan of cell players
  src/game/sim/SpatialGrid.ts:196-208
  removeCaptureVote walks entry.players backwards every time a unit moves
  land cells. If many units share a cell it's O(k).
  Fix: store players as Map<playerId, refCount> instead of array.

[SMALL] _dedup Set has implicit "consume result first" invariant
  src/game/sim/SpatialGrid.ts:548-549, 605, 926, 976, 1097, 1126, 1172
  Multiple query methods clear _dedup at entry; if a caller starts a new
  query while still iterating an earlier result, it silently breaks.
  Fix: document the invariant loudly at the top of the file, or assert in
  dev builds that nearby results aren't held across calls.

------------------------------------------------------------------------------
 SIM HOT PATH
------------------------------------------------------------------------------

[MEDIUM] LOS checked on every passing-rank candidate in dense crowds
  src/game/sim/combat/targetingSystem.ts:372-382
  chooseBestTargetCandidate runs hasWeaponLineOfSight for every candidate
  whose rank gate passes — that's hundreds of segment-vs-terrain walks
  per tick on a busy map. The eventual best target wins, but LOS for the
  losers is wasted.
  Fix: rank candidates by distance/rank first, then LOS-check only the
  top-K (3-5). The current best survives until a closer, LOS-clear one
  appears. The existing density-stride helps but doesn't change shape.

[MEDIUM] Force-field clearance per candidate
  src/game/sim/combat/targetingSystem.ts:384-388
  hasWeaponForceFieldClearance iterates the active-fields list for every
  candidate. Most candidates have nothing to do with any field, but the
  inner loop still runs.
  Fix: per-tick, build a cheap "fields-near-this-unit" sub-list once,
  pass that to chooseBestTargetCandidate. Falls back to full list when
  empty. Pairs with the Z-band fix above.

[SMALL] currentFireTargetRankSq computed twice on engaged weapons
  src/game/sim/combat/targetingSystem.ts:861, 908
  Pre-scan computes the rank to decide if a candidate query is needed
  (gated on `state==='engaged' && ranges.fire.min`); Pass 2 then
  recomputes both rank AND distSq (gated on `state==='engaged'`). For
  the overlap subset (engaged + has fire.min), both calls do identical
  spatial work. The gating mismatch makes a direct cache awkward; a
  cleanest fix is a per-weapon `{rank, distSq}` slot that pre-scan
  populates and Pass 2 reads-or-recomputes.

------------------------------------------------------------------------------
 NETWORK / SERIALIZATION
------------------------------------------------------------------------------

[LARGE] AoI filter is O(entities × recipients) AABB scan
  src/game/network/stateSerializer.ts:208-229
  isEntityInsideAoi runs per-entity per-recipient with no spatial index.
  10k entities × 10 listeners = 100k AABB tests per delta.
  Fix: build a per-emit spatial coarse index (the existing spatial grid
  is overkill; a 4-level quadtree or 64-unit cell grid works) and query
  per-listener AoI rect for the small set of entities inside it.

[MEDIUM] Projectile snapshot re-serialized per recipient
  src/game/network/stateSerializerProjectiles.ts (called per listener)
  Projectile visibility is barely filtered (no FOW hides in-flight
  projectiles), so the same payload gets serialized N times.
  Fix: cache the projectile snapshot per emit at publisher level (same
  pattern audio/spray uses under FOW-OPT-20).

[MEDIUM] prevEntityIds walked three times per delta
  src/game/network/stateSerializer.ts:289-348, 408-418
  AoI cleanup, visibility cleanup, and ghost cleanup are three separate
  walks of the same set.
  Fix: merge into one classifying loop that branches per entity (AoI-out
  vs. vis-hidden vs. ghost-eligible).

[MEDIUM] Client over-predicts entities at 60fps for 10Hz snapshots [VERIFY]
  src/game/network/ClientPredictionStepper.ts:165-200
  ClientUnitPrediction.ts:199-250
  Every render frame steps every active-prediction entity forward, even
  though the next server snapshot will overwrite. ~5 wasted prediction
  steps between snaps in steady state.
  Fix: short-circuit the step when (velocity, acceleration) variance is
  below a threshold, OR cap prediction to expected-server-arrival time
  (RTT-aware).
  Caveat: prediction is what hides snap jitter — measure before touching.

[MEDIUM] Spray visibility is double point-check per recipient
  src/game/network/stateSerializerSpray.ts:38-46
  For each spray, isPointVisible is called twice (source, target) per
  recipient. Each call walks the vision source hash.
  Fix: early-out — if the source point fails, skip the target check.
  Better: pre-filter sprays per team once (same FOW-OPT-20 cache shape).

[MEDIUM] capturedSnapshotEntityStates cache misses on visibility re-entry
  src/game/network/stateSerializerEntityDelta.ts:395-405
  src/game/network/stateSerializer.ts:382-400
  When an entity becomes visible again, getNextEntityState falls back to
  nextStateScratch and re-captures, even though the per-emit cache could
  have served it.
  Fix: in the visibility-walk loop, populate the cache for newly visible
  entities so the entity-delta encoder doesn't recompute.

[MEDIUM] Beam path points reallocated per prediction step
  src/game/network/ClientPredictionStepper.ts:64-68
  projPts.length resized to match tgtPts every frame; trailing point
  objects dropped & re-created.
  Fix: pool beam-point objects on a free list (same pattern as the
  existing _beamPointPool elsewhere); reuse instead of recreate.

[SMALL] serverMeta clone spreads 6 sub-objects per recipient per keyframe
  src/game/network/snapshotClone.ts:374-394
  Metadata barely changes tick-to-tick but the clone does 6 spread copies.
  Fix: pool the serverMeta object and copy scalars by name; only allocate
  on actual change.


------------------------------------------------------------------------------
 THREE.JS RENDERING
------------------------------------------------------------------------------

[MEDIUM] SelectionOverlayRenderer3D and ProjectileRangeEnvelope3D
        allocate Mesh per ring [VERIFY]
  src/game/render3d/SelectionOverlayRenderer3D.ts:88, 348
  src/game/render3d/ProjectileRangeEnvelope3D.ts:211
  `new THREE.Mesh(...)` per selection ring / range overlay = one draw
  call per ring. With many selected units or many ranges visible, this
  scales draw count linearly.
  Fix: use InstancedMesh with a shared ring geometry; encode color/scale
  per instance. Verify how often these are created — if they're pooled
  across selection lifetimes this is a non-issue.

[MEDIUM] UnitDetailInstanceRenderer3D clones geometries / materials unnecessarily
  src/game/render3d/UnitDetailInstanceRenderer3D.ts:83, 90-91, 97, 435-436
  Constructor `.clone()`s the input turretHeadGeom / barrelGeom /
  barrelMat / mirrorGeom; createPool clones again per poly variant.
  Duplicates GPU memory and breaks sharing across pools.
  Fix: drop the .clone() calls unless a per-pool variant is actually
  required; share a global poly-geometry cache keyed by shape ID.

[MEDIUM] Lazy per-player material allocation can hit render path
  src/game/render3d/Render3DEntities.ts:315-322
  getPrimaryMat instantiates MeshLambertMaterial on first use per team;
  if it fires mid-render (new spectator, late replay swap), it triggers
  shader compile (1-10ms stall).
  Fix: pre-allocate materials for max-supported team count (16) during
  init. Cheap upfront, no mid-frame compiles.

[MEDIUM] Leg instance count not trimmed to in-scope units
  src/game/render3d/LegInstancedRenderer.ts:149, 228, 291
  frustumCulled=false everywhere; instance count is total-units-with-legs
  rather than on-screen units with legs.
  Fix: write leg matrices only for in-scope units (Render3DEntities
  already has the scope set) and shrink instanceCount accordingly.

[MEDIUM] Render targets / shadow maps not verified [VERIFY]
  src/game/render3d/ThreeApp.ts
  An RTS with real-time shadow maps is usually a frame-time disaster.
  Confirm shadows are off (or limited to 1-2 cheap lights at low res) and
  that any post-fx render targets are allocated once per resolution
  change, not per frame.

[SMALL] CanvasSpritePool flags every slot dirty on use
  src/game/render3d/CanvasSpritePool.ts:49
  slot.texture.needsUpdate=true every draw, even if the canvas didn't
  change.
  Fix: track a dirty flag on the slot; mark needsUpdate only when the
  canvas was actually redrawn.

[SMALL] Various `frustumCulled = false` without intent comments
  Multiple files in src/game/render3d/
  Setting frustumCulled=false on world-spanning effect meshes is correct;
  on per-entity meshes it's wrong. Most are correct here, but the
  pattern is uncommented so future edits are easy to get wrong.
  Fix: one-line comment on each frustumCulled=false explaining why
  (world-spanning vs. known-coverage).

 VUE UI
------------------------------------------------------------------------------

[LARGE] reactive() wrapping high-frequency snapshot data
  src/components/gameCanvasSceneUi.ts:50-79, 92
  economyInfo, minimapData, selectionInfo are reactive objects mutated
  via Object.assign() on every render-frame snapshot apply. Every nested
  property read in templates pays proxy overhead, and every assignment
  triggers Vue's tracker. TopBar reads ~12 nested paths per render.
  Fix: replace reactive() with individual refs for the scalars TopBar /
  Minimap / SelectionPanel actually read (currEnergy, totalIncome,
  manaCurr, manaMax, ...). Set them only when the value changes.

[MEDIUM] Control-bar models rebuilt as fresh objects on every dep change
  src/components/GameCanvas.vue:707-738 (battleControlBarModel)
  src/components/GameCanvas.vue:740-771 (serverControlBarModel)
  computed<...>(() => ({ ... 30+ fields ... })) returns a new object
  identity any time any dependency changes. Every child BarButton /
  BarLabel re-renders since they can't memoize on prop identity.
  Fix: use shallowRef for the model; manually assign only properties
  whose source values changed. Or split into smaller per-section
  computeds so unrelated changes don't cross-pollinate.

[MEDIUM] Minimap composes the full canvas every camera-quad change (60Hz)
  src/components/Minimap.vue:458-461
  watch on props.data.cameraQuad fires every render frame; compose()
  redraws the minimap overlay each time.
  Fix: split the watch — camera-quad-only updates use a 30 Hz RAF
  throttle and redraw just the camera-box layer; entity layer keeps its
  existing 20 Hz throttle.

[MEDIUM] :style="{...}" allocates an object literal per render
  src/components/TopBar.vue:85, 104, 148, 179, 210
  `:style="{ width: energyPct + '%' }"` etc. Each render allocates the
  object even when the value didn't change.
  Fix: lift each into a computed (`energyBarStyle = computed(...)`) so
  Vue caches the object identity until the value changes.

[SMALL] Format functions called inline in templates
  src/components/TopBar.vue:142, 153, 157, 173, 184, 188, 204, 215, 219
  `{{ fmtStock(economy.stockpile.curr) }}` — pure function but a new
  string allocates every render.
  Fix: wrap each in a computed; the cached string survives across
  unrelated re-renders.

[SMALL] WorldDirectionHud watches 5 wind/camera sub-fields
  src/components/WorldDirectionHud.vue:400-409
  Watch array depends on cameraYaw + wind.{x,y,speed} + compact; fires
  at sim-tick frequency, then internally throttles to RENDER_INTERVAL_MS.
  Fix: replace the watch with watchThrottled at 33ms (matches the
  internal throttle); avoids the queued RAF storm in the first place.

[SMALL] Inline `:active` expressions inside v-for buttons
  src/components/GameCanvasBattleControlBar.vue:50-56 (and similar)
  `:active="model.currentAllowedUnits.includes(ut)"` runs on every parent
  re-render; combined with model-identity churn above, every button
  recomputes its active state every model rebuild.
  Fix: hoist the .includes check into a computed Set lookup, or memoize
  BarButton's active prop with a shallow-equal comparator.
