SNAPSHOT WIRE SIZE PLAN (SNAP-WIRE-01 family)

==============================================================================

------------------------------------------------------------------------------
SNAP-WIRE-01 - parent: snapshot payloads exceed the current per-client
               internet budget at small battle sizes.
------------------------------------------------------------------------------

Evidence:
- With roughly 200 on-screen units, PLAYER CLIENT telemetry showed
  DIFFSNAP avg around 150 KiB, about 130% over the current reasonable
  64 KiB DIFFSNAP target.
- FULLSNAP avg showed about 1.2 MiB, about 17% over the current
  reasonable 1 MiB FULLSNAP target.
- At 30 DIFFSNAPs/sec plus 1 FULLSNAP/sec, those numbers imply roughly
  45 Mbps per remote client before WebRTC overhead.

Why it matters:
Snapshot bandwidth is per remote client and burns host upstream first.
A 4-client game at this rate can require well over 150 Mbps of host
upstream just for state snapshots, before commands, WebRTC framing,
retransmits, congestion behavior, and other traffic.

Fix direction:
Do the child items below as separate, measurable PRs. Each item must
report before/after DS SIZE and FS SIZE numbers from the PLAYER CLIENT
bar, preferably with the same unit count and snapshot cadence.

------------------------------------------------------------------------------
SNAP-WIRE-02 - beam and rocket turrets render head-only and skip aim
               motion on the wire (DONE on landing of this change).
------------------------------------------------------------------------------

What changed:
- Added a `headOnly: boolean` flag on TurretBlueprint, set true for
  the beam family (beamTurret, miniBeam, megaBeamTurret, towerBeamTurret)
  and the rocket family (salvoRocketTurret, fastRocketTurret).
- 3D renderer skips barrel mesh construction and the per-tick
  applyTurretAimPose3D call for these turrets. The head sphere is the
  whole visual; it shows the unit's player color when idle / tracking
  and flips to white when turret.state === 'engaged'.
- Snapshot serializer pins rot / vel / pitch / pitchVel to 0 for
  headOnly turrets on the wire. captureEntityState + the Rust turret
  pool sync mirror this, so the JS / Rust diff masks both see 0-vs-0
  on aim motion and no longer dirty the host entity from rotation
  drift. Target / engaged-state / forceField-range transitions still
  dirty the entity normally.
- Sim still tracks the live rotation / pitch internally for beam
  direction and rocket launch &mdash; only the wire and the renderer
  drop the values.

Why it matters:
A tracking turret used to push its host entity into ENTITY_CHANGED_TURRETS
every frame from yaw/pitch threshold breaches. The wire payload remained
fixed-stride (10 floats per turret), so reducing dirty frequency was the
bandwidth lever rather than the per-row width.

Follow-up worth measuring:
- Compare DIFFSNAP / FULLSNAP sizes on a battle dominated by beam +
  rocket units (e.g. the `units.json` mix that bundles miniBeam on
  smaller chassis, plus a salvoRocketTurret line).
- If the wire savings warrant it, consider a follow-up that shrinks
  the wire stride per turret to skip the four angular slots entirely
  when headOnly is set &mdash; would require coordinated changes in the
  Rust wire encoder/decoder and would be the natural sequel.
