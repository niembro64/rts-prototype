==============================================================================
 BIG PICTURE — re-port performance-sensitive sim hot paths to Rust/WASM
==============================================================================

A previous WASM physics port (commit cf6fb480, Mar 18 2026) wrapped the
2D PhysicsEngine.ts. When the M-series rewrote physics to true-3D
(commits 6b5d19e0 through 001cb75c, Apr 24 2026) the WASM stack was
deleted along with the 2D engine it wrapped — the Rust crate at
rts-physics-wasm/, the PhysicsEngineWasm.ts wrapper, IPhysicsEngine.ts,
WasmBatch.ts, the built physics-wasm/pkg/ artifacts, the build:wasm
npm script, and the FORCE_WASM_PHYSICS toggle all went away. The
docs/wasm-migration-principle.md doc (commit 00ae8d28, May 1) codified
the go-forward policy: "prefer WASM for new or modified hot-path
systems when the work can be expressed as large batched numeric
operations over typed arrays."

The current PhysicsEngine3D + every adjacent sim hot path (projectile
motion, turret rotation, targeting / LOS, spatial grid, terrain
sampling, pathfinding, snapshot serialization) all run as pure
TypeScript on the JS thread. We're paying interpreter + V8 overhead
on every tick where a Rust SoA inner loop would be 5-20× faster on
the same workload, and we're paying GC pressure on every per-tick
allocation that a Rust `&mut [f32]` would avoid entirely.

This block enumerates the work to restore high-performance hot paths.
Each phase is independently shippable — Phase 0 first, then any of
1-8 in any order, with the easy + high-payoff ones (1, 2, 3) front-
loaded. Pre-phase profiling (Phase 0) tells us which order to actually
execute the rest in.

------------------------------------------------------------------------------
 Phase 0 — Measure. (NON-OPTIONAL.)
------------------------------------------------------------------------------
Before sinking weeks into a Rust port, instrument the current
TypeScript hot paths so we know what to port FIRST. The original
2D-to-WASM port was driven by tick-time profiling; the same
discipline applies here. Deliverables:

  - Per-system tick-time breakdown captured under a representative
    workload (target: 500 units, peak combat). Numbers wanted for
    UnitForceSystem, PhysicsEngine3D, projectileSystem, turretSystem,
    targetingSystem, SpatialGrid, stateSerializer, Pathfinder, and
    "everything else". Existing tickMs probes capture the wall-clock
    of the full tick; we need sub-system breakdown.
  - GC pause profile (devtools allocation timeline) for a 30-second
    representative session.
  - Reproducible benchmark scenario saved to a config so we can
    re-run after each migration step and compare.
  - PRIORITY TABLE: ordered list of subsystems by (cost_ms_per_tick ×
    portability_to_WASM) — this drives the order of Phase 1-8.

This phase ships only TypeScript instrumentation. No Rust yet. It's
the gate that decides whether the migration plan below is worth its
cost, and which items to do FIRST.

------------------------------------------------------------------------------
 Phase 2 — PhysicsEngine3D → Rust/WASM. (THE BIG ONE.)
------------------------------------------------------------------------------
Port the 926-line PhysicsEngine3D.ts to Rust. This is the
biggest single chunk of CPU per tick and the most natural fit for
WASM (tight numeric inner loops over a fixed-size Body3D pool).

Scope:
  - SoA Body3D representation (separate Vec<f32> for x, y, z, vx,
    vy, vz, ax, ay, az, mass, invMass, radius, groundOffset, plus
    bitfields for sleeping / isStatic / shape). Index = entity slot.
  - Explicit Euler integrator with the exact integration order
    documented at the top of PhysicsEngine3D.ts (forces+gravity →
    terrain spring → air drag → ground tangent damp → position).
  - Sphere-vs-cuboid resolver (unit ↔ building).
  - Sphere-vs-sphere resolver, with the existing spatial-hash
    broadphase reused or ported.
  - Ground spring + terrain normal sampling — terrain heightmap
    needs to be available on the WASM side too (see Phase 6).
  - Sleep state machine + sleepTicks accounting; wakeBody() called
    from JS-side via an applyForce-style API.
  - applyForce, applyImpulse, hasUpwardSurfaceContact, the awake
    body iterator, etc. Match the existing IPhysicsEngine surface
    so the swap is genuinely drop-in.
  - **NEW since the last port — must be in the Rust version:**
      * Hover lift integrator: F_up = K/altitude − c·vz with
        K = m·g·hoverHeight (UnitForceSystem hover branch).
      * Quaternion orientation triad (orientation/angularVel3/
        angularAccel3) + damped-spring law over axis-angle of Δq.
      * Bank-into-turn target-quat construction from body-frame
        velocity components.
  - JS shim PhysicsEngineWasm.ts re-implementing the public surface
    by marshalling through SoA typed arrays exposed from WASM as
    JsValue / Uint8Array views (zero-copy where possible).

Acceptance: GameServer can run a 60Hz sim with PhysicsEngine3D
swapped for the WASM port, and the Phase 0 benchmark scenario shows
a measured perf win.

------------------------------------------------------------------------------
 Phase 3 — projectileSystem → Rust/WASM.
------------------------------------------------------------------------------
projectileSystem.ts is 1221 lines and updates every live projectile
every tick — ballistic gravity, homing steering, terrain-follow,
beam-path point integration, plus the collision query against units +
buildings + terrain. Ballistic integration is the cleanest WASM
candidate (pure numeric over a typed pool); the collision-query side
needs the spatial grid (Phase 5) ported too to stay in WASM.

Sub-items:
  - Projectile SoA pool (id, pos, vel, accel, gravity flag, lifespan,
    homingTarget, homingTurnRate, sourceEntity, etc.).
  - Ballistic integration kernel (vz -= g·dt; pos += vel·dt).
  - Homing intercept solver (solveKinematicIntercept) — already
    pure numeric; ports cleanly. The "Phase 4: WASM batch turret
    rotation + projectile integration" commit (f3c3eac4) shows the
    earlier batched approach.
  - Terrain-follow path (clamp Z to surface + groundOffset).
  - Beam-path point integration (the per-point ax/ay/az chain).
  - Collision queries — needs Phase 5 (spatial grid in WASM) to
    stay fully on the Rust side without crossing the boundary per
    projectile per tick.

------------------------------------------------------------------------------
 Phase 4 — turretSystem + targetingSystem → Rust/WASM.
------------------------------------------------------------------------------
turretSystem.ts (245 lines) runs the damped-spring per turret per
tick. targetingSystem.ts (1104 lines) is the heavier piece —
candidate scan, distance/rank ordering, LOS top-K check, force-field
clearance, mirror priority. Both are hot paths on busy maps.

Sub-items:
  - Damped-spring integrator (integrateDampedRotation) — the new
    angularAcceleration output is part of this; trivial port.
  - chooseBestTargetCandidate: top-K insertion + LOS check
    structure can be a Rust kernel returning an entity id (or -1).
  - lineOfSight.ts (244 lines): segment-vs-terrain walk. Pure
    numeric, accesses heightmap + entity AABBs — both ported.
  - aimSolver.ts (648 lines): ballistic / kinematic solvers. Pure
    numeric, easy port.
  - Spatial-grid query of enemy candidates (depends on Phase 5).

------------------------------------------------------------------------------
 Phase 5 — SpatialGrid → Rust/WASM.
------------------------------------------------------------------------------
SpatialGrid.ts is 1438 lines — the 3D voxel hash that
EVERY other hot path consults (physics broadphase, targeting,
projectile collision, render scope, AoI). Migrating it second-or-
third is the natural force multiplier — once it's in Rust, every
caller below avoids the WASM↔JS round-trip per query.

Sub-items:
  - Cell hash + 48-bit packed key (already documented in current code).
  - insert/remove/move APIs keyed by entity id.
  - The query suite: in-radius, in-rect, enemy-of-player, etc.
  - Result buffer: callers iterate a Rust-owned slice; expose a
    zero-copy view to JS for the rare in-JS consumer.

------------------------------------------------------------------------------
 Phase 6 — Terrain sampling → Rust/WASM.
------------------------------------------------------------------------------
Terrain.ts is small (66 lines top-level) but getSurfaceHeight /
getSurfaceNormal are called from EVERY hot path (physics ground
spring, projectile terrain-follow, hover lift, targeting LOS, render
chassis tilt). The heightmap itself is large; living in WASM-side
linear memory eliminates per-sample JS array index cost.

Sub-items:
  - Move the heightmap Vec<f32> + sample(x, y) interpolation into
    Rust.
  - Move getSurfaceNormal (gradient sample with the water-exclusion
    rule documented in the current code).
  - Expose the LAND_CELL_SIZE constants on the Rust side too.
  - The map-building side (terrain/ generators) can stay in TS;
    only the per-tick sample path moves.

------------------------------------------------------------------------------
 Phase 7 — Pathfinder → Rust/WASM.
------------------------------------------------------------------------------
Pathfinder.ts is 851 lines. A* over the build/walk grid, called
asynchronously when a unit issues a move. Less time-critical than
the per-tick stuff above but ports cleanly (open/closed set, heap,
g/f-cost arrays — all numeric). The win shows up most on big maps
with hundreds of unit waypoint updates per second.

------------------------------------------------------------------------------
 Phase 8 — Snapshot serializer hot loops → Rust/WASM.
------------------------------------------------------------------------------
stateSerializer*.ts is several files totaling ~3000 lines. The
per-entity inner loop (qPos / qVel / qRot quantization, delta
encoding, FOW visibility check, AoI gate) IS hot under load —
every server tick that ships a snapshot pays it for every visible
entity per recipient.

Boundary cost is the concern here: a per-entity callback from Rust
to JS to write into a MessagePack DTO would dominate. Right approach
is probably: build the snapshot as a Rust-owned bytes buffer, then
hand it to JS once per recipient for transmission. Means moving the
MessagePack encoder to Rust too (or picking a Rust crate like
`rmp-serde`).

Lower priority than 2-6 — measure first. May not be worth the
boundary-marshalling complexity.

------------------------------------------------------------------------------
 Cross-cutting — concerns that affect every phase
------------------------------------------------------------------------------
  - **Boundary cost.** Every JS↔WASM call has overhead (~10-100 ns
    per call). A naive port that calls into WASM once per entity per
    tick can be SLOWER than pure TS. Design every kernel as a single
    batched call per tick that operates over a Rust-owned SoA.
  - **Memory model.** Body / entity state lives in linear memory on
    the WASM side. JS reads the same state via DataView / Uint8Array
    views (zero-copy). Writes from JS use explicit setter methods so
    Rust can validate / wake bodies / mark dirty.
  - **Determinism.** Server sim must be bit-identical across hosts
    for any future deterministic-rollback netcode. JS `Math.sin` !=
    Rust `f32::sin` in the last bit. If we ever want determinism,
    pick one math library (likely a Rust soft-math impl) and use it
    on both sides. Defer until determinism becomes a requirement.
  - **Debugging.** WASM-side panics are harder to chase than TS
    errors. Use `console_error_panic_hook` and a clear "(rust)"
    prefix on every log. Keep TS scaffolding (entity ID lookups,
    blueprint config) JS-side so the WASM kernel only sees primitive
    SoA data; that minimizes the surface area to debug.
  - **Hot-reload dev workflow.** wasm-pack rebuild on every Rust
    edit is slow; pair with cargo watch + a JS-side polyfill flag
    that lets you swap back to a pure-TS implementation per-system
    during local debugging. Cost: a small TS shadow of each Rust
    kernel — worth it for productivity.
  - **Single-threaded vs multi-threaded.** WASM threads via
    SharedArrayBuffer + web workers are real on modern browsers but
    add COOP/COEP headers and complicate WebRTC. Start single-
    threaded; revisit only if Phase 0 profiling shows we'd benefit
    AND the current sim is genuinely CPU-bound on one core.
  - **Compile size.** wasm-pack builds with `--release` produce
    much smaller bundles. Strip symbols. Verify the final .wasm
    stays under ~200KB for fast initial load.
