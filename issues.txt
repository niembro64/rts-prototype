 RTS PROFESSIONAL AUDIT - 2026-05-11
------------------------------------------------------------------------------

1. P1 - Group movement scales as one A* path request per selected unit.

   Evidence:
   - src/game/sim/commandExecution.ts:99-130 creates path actions separately for
     each selected unit, including formation moves.
   - src/game/sim/Pathfinder.ts:746-760 expands each path action by calling
     findPath, so a large selection can trigger hundreds or thousands of searches
     from a single command.

   Impact:
   Large armies will spike the sim thread on initial move/attack commands. This
   is especially risky for an RTS where 500-1000 unit selections should remain
   responsive.

   Fix direction:
   Add shared group pathing, leader paths with local steering, flow fields, or
   hierarchical pathing. At minimum, coalesce same-target path requests and fan
   out formation offsets after a shared route is found.

2. P1 - No fog-of-war or hidden information boundary is present in network state.

   Evidence:
   - src/game/network/stateSerializer.ts:95-118 filters entity state by AOI only,
     not by player vision.
   - src/game/network/stateSerializer.ts:265 always includes a minimap snapshot
     when AOI is enabled.
   - src/game/network/stateSerializerMinimap.ts:34-54 serializes all units and
     buildings from the world for the minimap, with no recipient/team visibility
     filter.

   Impact:
   Enemy unit and building positions are exposed to clients regardless of vision.
   If the game is meant to have normal RTS scouting, ambushes, radar, or hidden
   tech, this leaks core strategic information.

   Fix direction:
   Implement authoritative per-player visibility and last-seen state. Filter
   snapshots, minimap data, selection/target details, and combat events by the
   recipient's visibility, with explicit spectator/debug bypasses.

3. P2 - Core RTS command ergonomics are still missing.

   Evidence:
   - docs/command-ux-direction.md describes expected RTS command parity:
     formation move, fight/attack area, guard, patrol, reclaim, repair area,
     stop, wait, repeat, stances, pings, and more.
   - src/types/commands.ts currently exposes only the narrower set: select,
     move, clearSelection, startBuild, queue/cancel unit, rally/factory
     waypoints, fireDGun, jump, repair, attack, and server/AOI settings.

   Impact:
   The game will feel hard to control at RTS scale even if the sim works. Missing
   stop/hold-position, guard/assist, reclaim, repair-area, attack-move/attack-
   ground, control groups, queue editing, and stance controls block serious
   player evaluation.

   Fix direction:
   Prioritize a small competitive command set before adding more unit variety:
   stop/hold, guard/assist, reclaim/repair area, attack-move or attack-area,
   control groups, visible/editable command queues, and factory repeat/loop UI.

4. P2 - Submunition spread uses Math.random() inside sim projectile creation.

   Evidence:
   - src/game/sim/combat/ProjectileCollisionHandler.ts:241-244 comments that
     submunition spread is cosmetic and does not feed back into deterministic sim
     state.
   - src/game/sim/combat/ProjectileCollisionHandler.ts:259-300 uses Math.random()
     to set child projectile velocities; those projectiles are real sim entities
     that can later collide and deal damage.

   Impact:
   Live server authority may hide this in normal multiplayer, but replay,
   deterministic test, lockstep, save/load, or host migration paths will diverge.

   Fix direction:
   Use the world's deterministic RNG, or pass a deterministic RNG into
   spawnSubmunitions. Add a seeded replay/test case that confirms identical
   projectile outcomes across runs.

5. P2 - D-gun spends energy before validating a usable target direction.

   Evidence:
   - src/game/sim/commandExecution.ts:249-268 spends the D-gun energy cost before
     checking whether the target point is exactly on the commander/mount.
   - The zero-distance check returns without firing after the resource spend.

   Impact:
   A misclick directly on the firing unit consumes energy and produces no shot.
   This is small but feels unfair because the player receives no effect for a
   high-value tactical resource spend.

   Fix direction:
   Validate non-zero target distance before spendInstant, or refund the cost if
   the command cannot emit a projectile.

6. The "tick" unit legs used to be stretched downwards while in the air, which made the landing seem logical, normally when animals land they land with extended legs that can compress over time, absorbing the impact. The current tick legs are pulled up to their body while in the air. Look up how animals actually have their legs positioned when jumping, while in the air, and then when landing, and adjust the tick legs to match that. This will make the jump and landing look more natural. We may need a more advanced leg extension system to handle how animals actually do it, where the amount the legs are stretched or pulled up depends on distance from the ground?
