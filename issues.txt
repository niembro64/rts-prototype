[14] Two flagship Rust sim modules are 10k+ LOC flat files, violating the file-size
     SSoT discipline (boilerplate is mechanically extractable without touching the
     wire-format parity contract)
    Category: ssot / file-size discipline | Severity: MEDIUM
    Evidence: rts-sim-wasm/src/snapshot.rs = 11,317 LOC (flat `mod snapshot` at
      lib.rs:343), with ~31 mechanically-identical scratch-pool ptr/ensure pairs
      (e.g. turret pool :44-80 is structurally byte-for-byte identical to proj_vel
      pool :1205-1241, differing only by name/elem-type/stride).
      rts-sim-wasm/src/combat_targeting.rs = 10,734 LOC (215 fns, `mod` at
      lib.rs:340) mixing SoA storage (:356), ballistic aim (:1591), observation
      masks (:2252/:2285), and 8 per-tick FSM batch processors (:4100-:5267).
    Design: Single Source Of Truth — file-size discipline (html:2955-2974, :3152):
      "split cohesive behavior into small helper files."
    DONE (snapshot.rs dedup): the 15 uniform (count*stride) scratch pools are now
      generated by one snapshot_scratch_pool! macro (SSoT); snapshot.rs 11271 ->
      10992 LOC, replay byte-identical. The remaining pools are genuinely distinct
      (2-ensure string table, stride-1 u8/u32 buffers, packed encoders) and were
      left hand-written.
    REMAINING (file-size, intra-crate extraction only - no cross-language merge, do
      NOT alter the encoder<->test parity contract):
    - snapshot.rs: move the macro + the now-compact pool invocations into a
      snapshot/scratch.rs submodule (mod scratch; pub(crate) use scratch::*) to get
      the file under 10k. The invocations are scattered between encoder fns, so this
      is a careful cut/paste, not a contiguous move.
    - combat_targeting.rs (10,734 LOC): split the cohesive groups into child
      submodules - combat_targeting/fsm.rs (the 8 per-tick FSM batch processors
      ~4100-5267), observation.rs (mask build/marking ~2000-2330), ballistic.rs
      (aim ~1591) - leaving the SoA pool struct + accessors in the parent. Child
      modules can read the parent's private items, so `use super::*` should suffice;
      validate with build:wasm + cargo test + replay (must be identical - pure code
      motion). Higher risk: a single missed visibility/private ref breaks the build.

    {force, traction, friction} + a lift family (gravityCounterUpwardForceRatio,
    hoverHeightUpwardForce, swimHeightUpwardForce) - so any unit can mix walk/fly/
    swim, zeros meaning "cannot act in that medium". Ground under water counts as
    ground; air vs water are mutually exclusive by the water line.

    DONE: the design is fully specified in budget_design_philosophy.html (new
    Physics-Model subsection "Locomotion is one fully-abstracted force profile across
    ground, air, and water" + the summary-list bullet), including the medium-sampling
    rule, the swim-lift = water-line analogue of hover lift, and the behaviour-
    preserving migration.

    REMAINING (implementation; behaviour-preserving + opt-in so it validates against
    identical replay checksums):
    - Type (src/types/locomotionTypes.ts) + schema (blueprintSchema.json locomotion
      variants) + codegen: add the per-medium force/traction/friction fields + swim
      lift (+ randomisation/EMA siblings), all defaulting to 0/inert. The existing
      single traction/driveForce become the ground terms.
    - Rust force kernel (unit_kinetics.rs): add UF_ROW_* for the new terms; the
      kernel already carries UF_FLAG_IN_WATER and water probes. Apply water force/
      friction/swim-lift ONLY when submerged and the term is non-zero, leaving the
      existing ground/air (hover/flying) paths byte-identical so current units are

    - UnitForceSystem.ts: marshal the new values + the above/below-water medium flag.

FULLY-ABSTRACTED TURRET SYSTEM: SPAWN + RESOURCE-PYLON TURRETS (2026-06-25, user request)
--------------------------------------------------------------------------------
Replaces the single "construction turret" with a uniform turret taxonomy. Three
emitter families: COMBAT (emits shot/ray, existing); SPAWN turrets (create an
entity + fire an init laser); RESOURCE-PYLON turrets (move one resource in one
role). New turret kinds:
  turretSpawnBuildingsAndTowers, turretSpawnUnits,
  turretResourcePylonConstructionMetal, turretResourcePylonConstructionEnergy,
  turretResourcePylonExtractionMetal,   turretResourcePylonExtractionEnergy
Re-equip: builders (commander / construction drone / construction hovercraft /
fabricator) = a spawn turret + construction metal+energy pylons; queens =
turretSpawnUnits (instant + cooldown + launch); metal extractor = extraction-metal
pylon; solar/wind/geo = extraction-energy pylon. Construction pylons spin and
spray resource balls at the build target (auto-aimed, so they point down for
flying builders). Determinism: economy stays in energyDistribution; run
deterministic:audit each phase; keep encoder<->contract-test mirrors in sync.
