ClientViewState prepares body HUD, body-name, shield, contact-shadow,
ground-print, unit render-row, and building render-row packets. Unit/building
rows carry render keys, entity/turret row references, and dirty/lifecycle flags,
and the renderers consume those rows without steady-state per-row Entity map
lookups. Unit base-pose chain generation (tilt quaternion, inverse tilt,
parent quaternion, lifted position, and parent matrix) and unit chassis
part-matrix generation now run as batched WASM scratch-buffer kernels.
Instanced shield-panel slab matrices also run through a WASM scratch-buffer
kernel, and instanced unit turret barrel/head matrices plus turret mount-center
positions now batch through WASM. Unit turret aim/root-pitch math also batches
through WASM, and shield-panel support-root aim math reuses that same batch.
Building turret aim/root-pitch math now uses that shared batch too. Building
group and primary-body transform matrices now batch through a render-pose
WASM scratch-buffer kernel. TypeScript still owns construction-emitter timing,
which semantic aim source to use, dirty-row selection, material and visibility
orchestration, and mesh lifecycle. The remaining hot work is still TypeScript
pose/lifecycle generation and Three.js scenegraph churn: mirror support-root
attachment/visibility work, locomotion, fade, overlays, and cache pruning; plus
per-building turret attachment/visibility/material/spin, animation, overlay,
and cache-pruning work. This is likely to dominate at the "thousands of units"
target.

Required fix:
Move the remaining mirror support-root attachment/visibility work, locomotion,
fade, overlay, building turret attachment/material/spin, animation, and cache
pruning into WASM-owned slabs, GPU-updated buffers, or explicit dirty queues as
practical, then restrict Three.js work to buffer uploads and true lifecycle
changes.
