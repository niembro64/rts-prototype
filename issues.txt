----- Shot/turret emission abstraction + addressable ids -----

A18. [MED] Force-field is the one emission configured inline on the turret, not a shot referenced by id.
    LOCATION: src/types/blueprints.ts:137-142 (TurretBlueprint.forceField +
      forceFieldPanels, inline); src/types/shotTypes.ts:177-180 (ShotBlueprint =
      projectile|beam|laser, NO force variant); src/game/sim/blueprints/index.ts:
      533-558 (three-way resolve: forceField inline / projectileId lookup /
      constructionEmitter).
    RULE: "Atomic Gameplay Pieces" — shot is one umbrella with variants {projectile,
      beam, laser, forceField}; a turret names exactly one shot BY ID and carries no
      inline emission payload, so it stays swappable between hosts.
    PROBLEM: beam/laser/plasma/rocket resolve through SHOT_BLUEPRINTS[projectileId]
      (turret -> ShotId), but force-field is built inline from
      TurretBlueprint.forceField{angle,transitionTime,barrier} + forceFieldPanels.
      The runtime ForceShot exists (shotTypes.ts:271) yet there is no
      ForceFieldShotBlueprint and no ShotId path for it, so force-field turrets are a
      special case that breaks the "turret = standardized swappable emitter" contract.
    FIX: add a ForceFieldShotBlueprint variant to the ShotBlueprint union; move the
      barrier/material config onto it; reference it via the turret's shot id; collapse
      the three-way branch into one uniform SHOT_BLUEPRINTS[shotId] lookup. Keep the
      projected geometry (panel array vs sphere) on the mount per "Materials Are
      Independent Of Shape".

A19. [LOW] The turret's single emission reference is named projectileId but resolves beams/lasers too.
    LOCATION: src/types/blueprints.ts:119 (projectileId: ShotId | null);
      src/game/sim/blueprints/index.ts:546-553 (comment "Projectile or beam turret");
      src/game/sim/blueprints/turrets.ts:21 ('projectileId' in field list).
    RULE: "Atomic Gameplay Pieces" + fully-abstractable names — a turret names the one
      shot it emits; the field name must not bias toward a single variant.
    PROBLEM: the lone emission reference is projectileId, but
      SHOT_BLUEPRINTS[projectileId] returns ProjectileShotBlueprint | BeamShotBlueprint
      | LaserShotBlueprint — the name already lies for beam/laser turrets, and will lie
      harder once force-field becomes a shot (A18).
    FIX: rename projectileId -> shotId (or emitsShotId) across blueprints.ts, the
      loader (blueprints/index.ts, turrets.ts), and the turret JSON.

A20. [MED] No first-class build-site entity carrying the product's id from frame one.
    LOCATION: src/game/sim/commandExecution.ts:460-463 (only positional "build site"
      pathing refs); no buildShell/buildSite entity in src/types/sim.ts (grep clean).
    RULE: "Universal Runtime Entity IDs" / "A build site is the product, carrying its
      own id from frame one" — the under-construction unit/building IS the real entity
      from spawn, with one continuous EntityId it keeps through completion; no
      shell -> product re-id handoff.
    PROBLEM: there is no build-shell/build-site entity in the TS sim. Construction
      appears to reference a site positionally / by command, not as a first-class
      entity the construction turret locks onto by id, the player selects, and that
      survives completion unchanged. Without it, selection + queued orders + lock-on
      cannot persist across the finish, and provenance cannot name the in-progress
      target.
    FIX: model the under-construction unit/building as the real entity from spawn
      (nanoframe state: HP + completion-fraction ramp), allocate its final EntityId at
      construction start, and target/select/queue/lock against that id through
      completion. NB: construction path audited THIN, not confirmed-absent — re-verify
      before editing.

C16. [HIGH] Remaining terrain mesh baking and repair should move to Rust/WASM.
    LOCATION: src/game/sim/terrain/*.ts.
    RULE: "TypeScript coordinates, WASM computes" and "shared deterministic kernels".
    PROBLEM: terrain mesh baking and repair are CPU-heavy, deterministic, and
      sim-critical. Keeping them in TS leaves startup cost and client/server drift
      risk in the orchestration layer.
      Terrain read-side sampling/LOS/pathfinder masks and metal-deposit connected
      resource footprint growth, ring placement, snapped grid placement, and explicit
      dTerrain height derivation are now Rust/WASM-owned. Null-height metal-deposit
      terrain anchoring and static terrain buildability grid baking are also
      Rust/WASM-owned, with TS kept as object assembly/fallback. Mesh cell-triangle
      indexing, triangle neighbor metadata, final vertex-height smoothing, and
      neighbor-discrepancy marking for crack repair are now Rust/WASM-owned.
    FIX: port the remaining adaptive mesh topology generation and crack-repair
      passes to Rust/WASM in stages, with TS kept as orchestration only until each
      old path can be deleted.

================================================================================
VERIFIED-CLEAN (audited, no action needed — recorded so we don't re-investigate)
================================================================================
- No acceleration on the wire; PREDICT mode is POS/VEL only; no ACC option.
- The four EMA channels (movement pos/vel, rotation pos/vel) are each applied to
  ALL their documented fields (unit transform/velocity, beam point pos/vel, yaw
  scalar + orientation quat + turret yaw/pitch, angularVelocity3 + turret rates,
  force-field range visualizer).
- Travelling projectiles snap on velocityUpdate events (no EMA target populated).
- Delta thresholds: position uses absolute error, velocity uses relative; the
  zero-velocity edge is sent; baseline = last value actually sent per recipient.
- EntityType has Tower as a true peer of Unit/Building; fabricator is a tower, not a
  building; pure-infra buildings carry no turrets/lock-on.
- Lock-on exclusion lists are honored subtractively in targeting for turrets and
  host-level unit/tower lock-on; candidate enumeration covers the
  unit/tower/building/turret-mount families.
- Host-directed vs fully-autonomous turret mounts are fully wired: the tag lives
  on the unit/tower MOUNT entry (TurretMount.hostDirected / BuildingTurretMount.
  hostDirected), NOT on the turret blueprint; the loader throws on !=1 host-directed
  per turret kind; a host lock-on overwrites host-directed turrets while
  fully-autonomous turrets run independently; host-directed mounts fall back to
  autonomous when the host has no lock; and the fight-stop ratio counts only
  host-directed mounts.
- Airborne mount invariant (mount.y===0 && mount.z===0, no zResolver) is enforced and
  throws at startup.
- Rockets inherit the launch target and never re-acquire; lost target → ballistic.
- Sensors are capability sources with fixed radii (not derived from weapon range).
- Resource movement: every stockpile mutation funnels through ResourceMovementSystem
  credit/debit; renderer drives one shared pylon rig for all resource visuals.
- Selection panels are gated on entity TYPE capability flags, not per-blueprint buttons.
- No banned libs (Phaser/Pixi/Matter/cannon/ammo) are imported anywhere in src/.
- No legacy/old/_unused dirs or .old/.bak files in src/ or rts-sim-wasm/.
- Sim-relevant top-level config exports are JSON-backed; remaining TS files are loaders
  or derived display code, not authoritative tuning tables.
- Client unit prediction motion uses a batched Rust/WASM kernel for target extrapolation
  and rendered entity motion; the old TS single-body wrapper path is gone.
