VERIFIED-CLEAN (audited, no action needed — recorded so we don't re-investigate)
================================================================================
- No acceleration on the wire; PREDICT mode is POS/VEL only; no ACC option.
- The four EMA channels (movement pos/vel, rotation pos/vel) are each applied to
  ALL their documented fields (unit transform/velocity, beam point pos/vel, yaw
  scalar + orientation quat + turret yaw/pitch, angularVelocity3 + turret rates,
  force-field range visualizer). NB: body angularVelocity3 is EMA-corrected but
  NOT integrated into orientation — see A34 (the channel binding exists; the
  integration step does not).
- Travelling projectiles are event-driven (spawn / velocityUpdate / despawn) and
  dead-reckoned — they do NOT ride the movement-position/velocity EMA channels.
  (Snap-on-event behavior itself has a gap — see A25.)
- Delta thresholds: position uses absolute error, velocity uses relative; the
  zero-velocity edge is sent; baseline = last value actually sent per recipient.
- Server-side backpressure exists: NetworkSnapshotTransport gates on
  dataChannel.bufferedAmount and skips sends when the buffer is backed up.
- Weapon range acquisition uses true 3D distance (dx,dy,dz) against the envelope;
  the ground range ring uses sqrt(R²−h²) for display only.
- Forces integrate via a=F/m → velocity → position; no direct position teleport
  for movement. Gravity is applied in unit force integration and projectile
  stepping (the only gravity exemption is the rocket multiplier — see A27).
- EntityId is an opaque generation-checked handle; player/team/host/blueprint/
  storage live in metadata rows, not packed into the number.
- Build sites and factory unit shells are the real product entities from spawn:
  one continuous EntityId from nanoframe to finished structure; no re-id handoff
  at completion.
- EntityType has Tower as a true peer of Unit/Building; the fabricator is in the
  tower catalog (EntityType Tower), not a building; pure-infra buildings carry no
  turrets/lock-on.
- Turrets name one neutral `shotId`; force-field spheres use a `forceField`
  ShotBlueprint (`forceFieldBarrierShot`) instead of inline turret emission data,
  while panel/sphere projection geometry stays on the mount side.
- Lock-on exclusion lists are honored subtractively at LEVEL 0 (friend/enemy +
  entity family) for turrets and host-level unit/tower lock-on; candidate
  enumeration covers the unit/tower/building/turret-mount families. (Level-1 named
  exclusions are NOT applied — see A22.)
- Host-directed vs fully-autonomous turret mounts are wired: the tag lives on the
  unit/tower MOUNT entry (TurretMount.hostDirected / BuildingTurretMount.
  hostDirected), NOT on the turret blueprint; a host lock-on overwrites host-
  directed turrets while fully-autonomous turrets run independently; host-directed
  mounts fall back to autonomous when the host has no lock; and the fight-stop
  ratio counts only host-directed mounts. (The one-per-kind STARTUP THROW is NOT
  enforced — see A21.)
- Rockets inherit the launch target and never re-acquire; lost target → ballistic.
- Airborne mount invariant (mount.y===0 && mount.z===0, no zResolver) is enforced
  and throws at startup for hover/flying turrets.
- Locomotion visual state (wheel/tread/leg/fan) is computed client-side in the rig
  and does NOT cross the wire.
- Camera anchoring is two orthogonal config axes (anchorPointCameraScreen +
  anchorPointCameraTerrain) resolved by one shared resolver; all six combinations
  are reachable from config, not a special-case branch.
- Sensors are capability sources with fixed radii (NOT derived from WEAPON range).
  (Builder sight IS partly derived from build-reach — a separate, milder coupling;
  see A33.)
- Resource movement: every per-tick stockpile delta funnels through the
  ResourceMovementSystem credit/debit path; no side-channel mutates the player pool.
  (Pylon COVERAGE for every resource-affecting host is incomplete — see A37.)
- Selection panels are gated on entity TYPE capability flags, not per-blueprint
  buttons; mixed selections reduce to the intersection of per-type action sets.
- Blueprints are pure data (discriminated unions by string tag); no closures or
  functions smuggled into blueprint JSON.
- Steady-state tick paths use counted loops and reused buffers; the map/filter
  instances in the codebase are in setup/non-hot paths, not per-tick hot loops.
- No banned libs (Phaser/Pixi/Matter/cannon/ammo/rapier) are imported anywhere in
  src/; the only @dimforge entry is a local no-op stub (see D1).
- No legacy/old/_unused dirs or .old/.bak files in src/ or rts-sim-wasm/.
- Top-level *Config.json exports are JSON-backed. (Remaining authoritative tuning
  inside src/game/sim/ TS modules is NOT yet JSON — see A32.)
- Client unit prediction motion uses a batched Rust/WASM kernel for target
  extrapolation and rendered entity motion; the old TS single-body wrapper path is
  gone.
- Adaptive terrain mesh build is fully Rust/WASM-owned (C16 closed). One kernel
  `terrain_build_adaptive_mesh` runs the whole host bake — hierarchy refine, 2:1
  balance, conforming T-junction topology, crack-repair loop, vertex-height
  smoothing, triangle neighbor metadata, and the cell->triangle index — sampling the
  same analytic height generator the metal-deposit kernels use. TypeScript is pure
  object assembly (config slice in, TerrainTileMap out); the old TS topology/repair
  code and the granular neighbor/discrepancy/smoothing WASM bridge fallbacks were
  deleted (Delete The Old Path). The 22-value generation-config + 5-stride flat-zone
  packers now live once in terrainGenerationConfig.ts, shared by the mesh baker and
  metalDepositConfig.ts. Cell-index WASM exports stay (client wire-unpack reuses them).
