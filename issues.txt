----- Shots carry immutable source provenance -----

A10. [MED] Authoritative projectile state lives in TS, not Rust — provenance has
     nowhere to migrate.
    LOCATION: rts-sim-wasm/src/lib.rs:1825-1833 (ProjectilePool = pos/vel/
      time_alive only); snapshot path lib.rs:20107-20156.
    RULE: "Shots carry immutable source provenance" + "TypeScript Coordinates,
      WASM Computes".
    PROBLEM: the Rust ProjectilePool holds only kinematics; all shot identity and
      provenance live in the TS Projectile entity. The stated migration direction
      (projectile sim -> Rust) has no struct field to carry the source record, and
      the spawn-snapshot wire ships only turretId/sourceTurretId/playerId/
      sourceEntityId — not root/team/blueprint/spawn_tick/parent.
    FIX: when projectile motion moves to Rust, add a ShotSource SoA block to
      ProjectilePool and extend the spawn wire format.

A11. [LOW] source_team_id / EntityMeta.team_id cannot be satisfied — no team model.
    LOCATION: rts-sim-wasm/src/lib.rs:12261-12334 (relationship = player-id
      compare); src/game/sim/combat/damageHelpers.ts:342.
    RULE: "Shots carry immutable source provenance" / "Universal Runtime Entity
      IDs" (team_id field).
    PROBLEM: there is no team/alliance concept anywhere; all friend/enemy and
      friendly-fire checks are pure player-id comparisons, so source_team_id and
      EntityMeta.team_id have no value to store.
    FIX: either introduce a team model and capture a frozen source_team_id at
      spawn, or explicitly scope team_id out of the design until teams exist.

----- Minor / boundary-enforcement / test gaps (areas that audited mostly clean) -----

A17. [LOW] No separate tower blueprint catalog; tower-vs-building is a runtime
     spawn decision.
    LOCATION: src/game/sim/blueprints/buildings.json (single catalog, no
      towers.json); src/types/buildingTypes.ts:25-33 (TOWER_BUILDING_TYPES);
      src/game/sim/spawn.ts:280-285 (isTowerBuildingType); lib.rs:8482-8483
      ("a tower's blueprint name is a BuildingType").
    RULE: "Towers Are Static Hosts" — "The blueprint goes in the tower catalog
      (e.g. towers.json), not the building catalog."
    PROBLEM: towers (factory, megaBeamTower, cannonTower) live in buildings.json
      and are split out at spawn time by a hardcoded TOWER_BUILDING_TYPES set;
      there is no towers.json, and a tower's blueprint name is still a BuildingType.
    FIX: optional structural alignment — extract a towers.json catalog so the
      tower/building boundary is authored in data, not a runtime type set.
      (Behavior is already correct; this is doc-alignment, not a bug.)

SECTION C — PERFORMANCE / SCALE (target: 5,000 units)
================================================================================

C16. [HIGH] Remaining terrain mesh baking and repair should move to Rust/WASM.
    LOCATION: src/game/sim/terrain/*.ts.
    RULE: "TypeScript coordinates, WASM computes" and "shared deterministic kernels".
    PROBLEM: terrain mesh baking and repair are CPU-heavy, deterministic, and
      sim-critical. Keeping them in TS leaves startup cost and client/server drift
      risk in the orchestration layer.
      Terrain read-side sampling/LOS/pathfinder masks and metal-deposit connected
      resource footprint growth, ring placement, snapped grid placement, and explicit
      dTerrain height derivation are now Rust/WASM-owned. Null-height metal-deposit
      terrain anchoring and static terrain buildability grid baking are also
      Rust/WASM-owned, with TS kept as object assembly/fallback. Mesh cell-triangle
      indexing and final vertex-height smoothing are now Rust/WASM-owned.
    FIX: port the remaining adaptive mesh topology generation and crack-repair
      passes to Rust/WASM in stages, with TS kept as orchestration only until each
      old path can be deleted.

VERIFIED-CLEAN (audited, no action needed — recorded so we don't re-investigate)
================================================================================
- No acceleration on the wire; PREDICT mode is POS/VEL only; no ACC option.
- The four EMA channels (movement pos/vel, rotation pos/vel) are each applied to
  ALL their documented fields (unit transform/velocity, beam point pos/vel, yaw
  scalar + orientation quat + turret yaw/pitch, angularVelocity3 + turret rates,
  force-field range visualizer).
- Travelling projectiles snap on velocityUpdate events (no EMA target populated).
- Delta thresholds: position uses absolute error, velocity uses relative; the
  zero-velocity edge is sent; baseline = last value actually sent per recipient.
- EntityType has Tower as a true peer of Unit/Building; fabricator is a tower, not a
  building; pure-infra buildings carry no turrets/lock-on.
- Lock-on exclusion lists are honored subtractively in targeting for turrets and
  host-level unit/tower lock-on; candidate enumeration covers the
  unit/tower/building/turret-mount families.
- Host-directed vs fully-autonomous turret mounts are fully wired: the tag lives
  on the unit/tower MOUNT entry (TurretMount.hostDirected / BuildingTurretMount.
  hostDirected), NOT on the turret blueprint; the loader throws on !=1 host-directed
  per turret kind; a host lock-on overwrites host-directed turrets while
  fully-autonomous turrets run independently; host-directed mounts fall back to
  autonomous when the host has no lock; and the fight-stop ratio counts only
  host-directed mounts.
- Airborne mount invariant (mount.y===0 && mount.z===0, no zResolver) is enforced and
  throws at startup.
- Rockets inherit the launch target and never re-acquire; lost target → ballistic.
- Sensors are capability sources with fixed radii (not derived from weapon range).
- Resource movement: every stockpile mutation funnels through ResourceMovementSystem
  credit/debit; renderer drives one shared pylon rig for all resource visuals.
- Selection panels are gated on entity TYPE capability flags, not per-blueprint buttons.
- No banned libs (Phaser/Pixi/Matter/cannon/ammo) are imported anywhere in src/.
- No legacy/old/_unused dirs or .old/.bak files in src/ or rts-sim-wasm/.
- Sim-relevant top-level config exports are JSON-backed; remaining TS files are loaders
  or derived display code, not authoritative tuning tables.
- Client unit prediction motion uses a batched Rust/WASM kernel for target extrapolation
  and rendered entity motion; the old TS single-body wrapper path is gone.
