AIMING / TARGETING SYSTEM ISSUES

AIM-05 - Homing guidance is kinematic velocity steering, not bounded force/thrust physics.

Evidence:
- design_philosophy.html:429-454 says guided projectiles should counter gravity with thrust inside the steering update, and that counter-gravity should be bounded by available thrust.
- src/game/math/HomingSteering.ts:1-12 and 31-45 rotate the velocity vector toward the target and preserve speed.
- src/game/sim/combat/projectileSystem.ts:784-890 applies gravity, computes an intercept, then rotates velocity with applyHomingSteering.

Why it matters:
The current homing path effectively lets steering erase gravity's directional effect by rotating post-gravity velocity back toward the target ray. Projectile mass and available thrust do not bound the correction. A weak rocket cannot sag naturally under insufficient lift, which conflicts with the physics model.

Fix direction:
Represent homing as an acceleration/thrust budget: lateral guidance plus counter-gravity in one bounded vector, integrated through the same projectile physics path. Keep the current Rodrigues function only as a temporary visual approximation if needed, with a removal target.

AIM-06 - Ballistic lead uses partial acceleration as if it were physical acceleration.

Evidence:
- src/types/sim.ts:139-153 says velocity is authoritative and thrustDir is the desired drive input.
- src/game/sim/combat/combatUtils.ts:281-300 builds acceleration from unit.movementAccel plus optional gravity only.
- src/game/sim/combat/aimSolver.ts:442-520 feeds that acceleration into the constant-acceleration ballistic solver.
- src/game/sim/combat/projectileSystem.ts:824-860 repeats the same acceleration input for homing intercept prediction.

Why it matters:
movementAccel excludes important real forces such as terrain spring, damping, jump actuation, collision correction, recoil, and external impulses. The solver can therefore produce "exact" intercepts against an acceleration state that is not actually the derivative of authoritative velocity.

Fix direction:
Either feed the solver an actual authoritative body acceleration from the physics step, or make the lead model explicitly velocity-only for units unless a real acceleration source is available. Do not mix partial gameplay intent acceleration into a physical constant-acceleration solver without naming that approximation.

AIM-07 - Force-field targeting clearance for arcs uses a straight segment, not the shot trajectory.

Evidence:
- src/game/sim/combat/targetingSystem.ts:332-365 applies force-field clearance to all weapons, including high arcs.
- src/game/sim/combat/lineOfSight.ts:202-237 checks only whether the straight source-to-target segment crosses a field boundary.

Why it matters:
High-arc ballistic shots and vertical launchers do not travel along the straight source-to-target chord. Targeting can reject an arc that would physically clear a shield, or allow an arc whose actual projectile path would hit the shield. Projectile collision later uses the moving projectile segment, so lock-on rules and impact rules can disagree.

Fix direction:
Use a trajectory-aware shield query for projectile weapons, conservative enough for performance but based on the same ballistic path envelope the projectile will actually follow. At minimum, document straight-segment clearance as an intentional conservative targeting rule.

AIM-08 - Aiming/targeting hot paths remain TypeScript object loops instead of WASM/SoA batches.

Evidence:
- design_philosophy.html:520-570 and 614-619 call out targeting batches, per-unit loops, projectile stepping, and spatial queries as migration targets for Rust/WASM over typed memory.
- src/game/sim/combat/targetingSystem.ts:405-545 and 609-1122 still run the targeting FSM, candidate ranking, LOS gating, and ballistic gating over Entity/Turret objects in TypeScript.

Why it matters:
The current code has good local pooling and some WASM helpers, but the main targeting loop still scales as JS object orchestration over armed entities and candidate arrays. That is acceptable for prototype correctness, but it is not aligned with the stated 5,000-unit architecture.

Fix direction:
Keep the TypeScript behavior as the reference while designing a data-oriented targeting batch: flat turret/entity inputs, candidate slots, visibility/LOS results, target output IDs/states, and parity checks during migration. Delete the old path when parity is proven.

AIM-09 - Combat/targeting state still relies on optional fields in hot domain objects.

Evidence:
- design_philosophy.html:827-859 says simulation, prediction, networking, and entity-state modules should prefer explicit nulls or numeric sentinels over optional properties.
- src/types/sim.ts:390-445 still has optional turret fields such as worldPos, worldVelocity, aimTargetYaw, aimErrorYaw, ballisticAimInRange, forceField, and losBlockedTicks.
- Combat state also uses optional priorityTargetId, priorityTargetPoint, and nextCombatProbeTick.

Why it matters:
These fields sit directly in aiming, targeting, snapshot, and prediction paths. Optional shape changes and optional chaining make absence easy to miss and work against the intended SoA/WASM migration.

Fix direction:
When touching these files, migrate hot combat fields toward explicit nulls or numeric sentinels. For example, targetId can become -1 in packed/hot representations, cached world mount fields can always exist with worldPosTick = -1, and aim-error fields can be initialized to 0 with an explicit hasAimTarget flag if needed.
