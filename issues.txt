[049][ARCH] src/game/sim/combat/projectileSystem.ts (~1800 lines) is no
longer the integration/guidance math owner — those kernels run in Rust
(pool_step_packed_projectiles_batch, projectile_integrate_step_batch,
projectile_homing_guidance_apply_batch in rts-sim-wasm/src/projectile.rs) —
but TS still runs per-projectile loops around them: packed-pool
gather/scatter passes (slot map + swap-remove live in JS), traveling-
projectile policy packing (homing target validation/aim points, D-gun
terrain-follow ground sampling — one JS->WASM scalar call per projectile),
arming, and snapshot velocity-delta events. Beam stepping (per-tick re-trace
incl. DamageSystem.findBeamPath) is still fully TS, and collision
orchestration (src/game/sim/combat/ProjectileCollisionHandler.ts, ~1789
lines) is a TS per-projectile loop around six WASM kernels, several invoked
count=1 per projectile (hitbox sweep, terminal classify/plan). Rust already
has resident terrain samplers and entity slabs, so nothing blocks the port
on data residency. Dependency order: (1) generalize ProjectilePool to all
traveling projectiles with the missing columns (prev pos, hp, arming,
homing/policy, gravity multiplier, profile constants) plus a Rust slot
allocator and spawn/despawn API — reflection-response must write the pool,
not JS mirrors, to close the Pass-1 re-gather backdoor; (2) fold
integrate/arming/gravity/D-gun terrain-follow/homing into one pool pass
(keep single-row exports — client prediction uses them); (3) update Rust
spatial-grid projectile slots in-crate post-integration, delete the JS
restamp (SimulationCombatController.ts:184); (4) port the collision loop as
one pool pass with true batches (note: today's loop detonates mid-iteration,
so batching changes observable ordering — needs explicit parity reasoning);
(5) port beam tracing last (needs Rust-resident polyline + prev-trace
caches); (6) shrink TS to spawn/despawn orchestration consuming Rust diffs.

[06][USER_REQUESTS] Regarding our "submunitions" in our "mortar" shot types - I think we can clean them up a bit. If the host "mortar" shot exploded because it was shot down - its submunitions should have a bias to continue in that same direction more or less to show that there is a momentum associated with the movement. But if the host "mortar" shot exploded because it hit something the submunitions need to be reflected off of the surface to show some kind of extra "particle" explosion. We are mostly there but there are still some inconsistencies that I see about directions of where the submunitions fly - maybe we can formalize and elegantly code this better.
