 Quaternion adoption for hover units — DONE
==============================================================================
The full quaternion + ω-vector + α-vector representation is now wired
end-to-end for the `hovercraft` unit (and any future hover-class
content). Ground units / turrets stay on their existing scalar
yaw/pitch model per the earlier analysis — the new triad is gated on
`locomotion.type === 'hover'`. Summary of what landed:

  Sim state. Unit gains optional `orientation: Quat`,
  `angularVelocity3: Vec3`, `angularAcceleration3: Vec3`. Identity
  quat on spawn for hover units; undefined for ground units.

  Math helper. src/game/math/Quaternion.ts — Quat ops (mul, conjugate,
  normalize, integrate, axis-angle of shortest-path Δq) plus the
  one-law damped-spring step: α = k · (axis·angle) − c · ω. No
  preferred axis, no gimbal lock — the elegant alternative the
  earlier note described.

  Sim physics. UnitForceSystem hover branch builds a target quat from
  (target yaw from thrust dir, target pitch ∝ −forward velocity,
  target roll ∝ +lateral velocity, both clamped) and runs the
  quat-spring at critical damping. Yaw is no longer instantaneously
  snapped for hover units; it evolves coordinately with pitch and
  roll. transform.rotation is kept in sync to the quat's yaw
  extraction so every existing 2D-yaw consumer (turret world mount
  math, camera, AI, network code) stays correct.

  Wire format. NetworkServerSnapshotEntity.unit gains optional
  `orientation` / `angularVelocity3` / `angularAcceleration3`.
  Serializer ships them only when the entity has one (ground units
  pay zero overhead — MessagePack drops undefined fields). PREDICT
  serializer gate already covers them: POS drops omega+alpha, VEL
  drops alpha, ACC ships all three.

  Client. ServerTarget carries the triad; applyClientUnitVisualPredic
  tion drifts the entity's orientation quat toward target via
  componentwise lerp + renormalize each frame.

  Renderer. yawGroup uses quaternion.set when orientation is present;
  falls back to the existing `rotation.set(0, -yaw, 0)` Euler write
  otherwise. World→Three.js basis swap is
  `q_three = (-q_world.x, -q_world.z, q_world.y, q_world.w)`
  (verified manually against yaw/pitch/roll basis mappings).

The hovercraft now visibly banks INTO turns and noses down when
moving forward — a real continuous rotation, not snap-yaw. The
machinery is in place to give roll / arbitrary orientation to any
future entity (spinning projectiles, ragdoll debris, ships listing,
etc.) — just declare locomotion type 'hover' (or extend the gating
predicate) and the quat triad and bank physics come for free.

==============================================================================
 PREDICT serializer — fully-omit-via-optional-wire-fields (deferred)
==============================================================================
The PREDICT POS/VEL/ACC scheme is fully wired end-to-end. The local +
remote bandwidth gate zeros velocity / acceleration fields on the
wire for recipients in POS / VEL mode (zero encodes in 1-byte
MessagePack ints vs ~5-byte quantized floats, so this is a real
wire-byte saving even without making the fields optional).

One follow-up remains: make velocity / accel wire fields OPTIONAL so
MessagePack drops them entirely instead of shipping zeros. Bigger
saving, much wider blast radius (every reader path needs `?? 0`
defaults). Defer until profiling shows the current zero-write payload
isn't already meeting bandwidth targets.
