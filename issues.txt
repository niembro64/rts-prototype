MEMORY LEAKS

[ISSUE_MEMORY_LEAK_01][HIGH] Lockstep outbound command-frame buffer grows for the entire match with no ack-based eviction
  Where: src/game/network/NetworkLockstepTransport.ts:50,100,171-198,239-246; src/components/gameCanvasRealBattleStartup.ts:804-820,1040; src/types/network.ts:346-352
  What: `outboundCommandFrames` Map is `.set()` once per coordinated frame (~60Hz) storing the full command payload, and is never deleted — the only removal is the wholesale `clear()` at :244, invoked solely from NetworkManager.beginNetworkSetup() session reset. resendCommandFramesAfter() only reads. The coordinator already computes each peer's `latestAck.ackFrame` but no prune path exists. With allowLateJoin:false, acked frames are never replayed.
  Why it matters: Steady, match-length-proportional heap climb on the live multiplayer authority host (which also runs the authoritative sim), reaching tens of thousands of pinned command objects over a long browser match — exactly the unbounded growth the design budget cannot absorb.
  Fix: Add `pruneAcknowledgedFrames(minAckedFrameAcrossAllPeers)` that deletes every key <= the minimum ackFrame across connected peers (from latestAckForPlayer()), called each frame after resendCommandFramesToLaggingPeers().

[ISSUE_MEMORY_LEAK_02][HIGH] Projectile velocity-update decode allocates a fresh Array plus two nested {x,y,z} DTOs per projectile per snapshot (no pool)
  Where: LIVE offender is src/game/network/snapshotProjectileWirePack.ts:525-562 (unpackProjectileVelocityUpdatesV2: `new Array(total)` at :529, per-row `{id, pos:{x,y,z}, velocity:{x,y,z}, clearHomingTarget}` at :540-553); same pattern in V2 spawns (:346-350) and beam updates (:648+). (NOTE: the originally-cited V1 functions at :823-847 are dead code — packProjectilesForWire always emits V3.) Consumer: src/game/network/ClientViewState.ts:888-916.
  What: The live decoder builds three object literals per projectile every decode; the consumer only reads scalars into locals and discards each vu, so every update object plus its two nested vectors is immediate garbage. This file has no rent-pool, unlike the entity decoder's rentDecodedVec3/Quat (snapshotEntityWirePack.ts:124-185).
  Why it matters: At hundreds-to-thousands of in-flight shots/beams, thousands of short-lived allocations per second on the client decode path during heavy combat — sustained GC pressure on the render thread exactly when frame budget is tightest; direct violation of "no allocations in per-projectile loops."
  Fix: Mirror the entity-decode rent-pool, or decode straight into ClientProjectileStore's owned per-projectile fields without materializing DTOs (consumer only wants scalars). Apply the same to V2 beam update/point and spawn/despawn decoders.

[ISSUE_MEMORY_LEAK_03][MEDIUM] FogOfWarFog3D allocates a {x,y,radius} object per vision source every fog-update frame
  Where: src/game/render3d/FogOfWarFog3D.ts:256 (sources.length=0), :267,:275 (push literals), :111 (VisionSource[] decl), :401-409 (isInVision); ClientViewState.ts:1065-1068,1534-1536; gated by RtsScene3DRenderPhase.ts:446-451.
  What: collectSources() resets via `.length=0` (discarding the objects) then `this.sources.push({x,y,radius})` for every owned unit/building across all vision player ids — and these getters return the full un-culled roster (the sibling SightBoundaryRenderer3D scope-culls; fog does not). The objects exist only to carry three numbers. (This is per-frame GC churn, not retained growth, but is on a default-on hot path.)
  Why it matters: At thousands of units, thousands of short-lived objects per effect-frame scanning the full roster -> sustained GC pressure and frame-time spikes during large engagements; fog is on by default so this is always live.
  Fix: Convert `sources` to SoA parallel typed arrays (Float32Array x/y/radius + count) or a pooled-struct array that reuses objects (write fields, track live count, reset count=0). Index by count in isInVision/estimateTargetActiveCount.

[ISSUE_MEMORY_LEAK_04][MEDIUM] Lockstep dedup Sets (seenCommandKeys / seenCommandFrameKeys) accumulate one entry per command/frame for the whole match
  Where: src/game/network/NetworkLockstepTransport.ts:46-47,294,303,308,311,339-342; only clear() at :240-241 (called from NetworkManager.ts:471 session reset).
  What: acceptCommand()/acceptCommandFrame() add unique string keys per command (`gameId:executeFrame:playerId:playerSequence:commandIndex`) and per frame (`frame:frameSequence`); neither Set is ever pruned (no `.delete(` anywhere). Both grow unbounded with frame count for the whole match on every lockstep participant. (Entries are small ~40-60-byte strings; modest absolute footprint but genuinely unbounded.)
  Why it matters: Continuous heap climb on host and clients over long matches; same unbounded-growth class as the outbound frame buffer and a clear violation of the no-per-frame-accumulation netcode rule.
  Fix: Bound both Sets by a frame horizon sharing the same min-acked-frame watermark proposed for outboundCommandFrames: bucket keys per executeFrame so a single watermark advance drops a whole bucket once a frame is fully acked / older than the input-delay + resend window.

[ISSUE_MEMORY_LEAK_05][LOW] SmokeTrail3D.spawnPuff allocates a fresh Puff object (and an RGB object) on every spawn, even when reusing a pooled slot
  Where: src/game/render3d/SmokeTrail3D.ts:613 (hexToRgb01 object alloc), :623-640 (fresh Puff literal), :646-652 (steady-state eviction reassigns pool.active[i]=puff); contrast in-place mutation at :442-465; colorUtils.ts:53-61 (unused allocation-free writeHexToRgb01Array).

  Where: src/game/network/snapshotEntityWirePack.ts:565-668 (unpackUnit: literals at :572,:578,:587,:599,:607,:613,:661-665), 670-712 (unpackBuilding: :673,:680,:685,:690-694,:700); pools exist+used only on hot path at :354,:449,:465,:468,:476,:481,:488. [ISSUE_MEMORY_LEAK_08][LOW] A* path reconstruction allocates a fresh Vec on every call instead of reusing pooled scratch
  Where: rts-sim-wasm/src/pathfinder.rs:835 (`let mut path: Vec<u32> = Vec::new()`), grown at :838, returned at :854-855; consumed-and-dropped in-call at :1156; existing pooled scratch at :73,:100; call site src/game/sim/Pathfinder.ts:199.
  What: Path-reconstruction allocates a fresh Vec that grows from empty (several reallocs for long paths) and hands ownership out via AStarResult.cells, even though PathfinderState already pools waypoint_scratch and other buffers. (Not a leak — fully consumed and dropped in-call; per-replan, not per-tick.)

  Fix: Reconstruct into a persistent pooled `path_scratch: Vec<u32>` in PathfinderState (clear()+reuse), exposing slice/length to the caller.

PERFORMANCE & OPTIMIZATION

[ISSUE_PERFORMANCE_01][HIGH] EntityCacheManager does a full O(n log n) sort + ~25-list re-bucket of EVERY entity multiple times per tick because projectile spawn/despawn shares one dirty flag with units/buildings
  Where: src/game/sim/EntityCacheManager.ts:79 (single dirty bool), :81-83 (invalidate, no type check), :116-118 (sortedEntities push+sort), :120-257 (~25-list rebuild); WorldState.ts:351,390 (addEntity/removeEntity invalidate for all types incl 'shot'), :495-560 (getters call rebuild); SimulationCombatController.ts:133,150,175,190,206; ServerSimulationCore.ts:61-72 + GameServer.ts:425 (TS Simulation is live per-tick authority).
  What: addEntity/removeEntity set a single shared dirty flag for ANY entity type including projectiles. The next getter calls rebuildIfNeeded, which copies all entities, runs `sortedEntities.sort((a,b)=>a.id-b.id)`, then rebuilds ~25 sublists with per-entity branching. Because projectiles spawn and despawn every tick during a firefight and the combat controller interleaves add/remove between cache reads, this full-world sort+rebuild runs 2-3+ times per 60Hz tick — driven entirely by projectile churn even though unit/building lists are unchanged.
  Why it matters: At thousands of units (cap options up to 2187) with hundreds of in-flight projectiles, this is the single largest avoidable per-tick host cost, re-triggered by the most frequent mutation in the game; pure TS orchestration safe to fix without Rust work.
  Fix: Split the dirty flag by category. Maintain projectile lists incrementally (push on add, swap-remove on remove) without invalidating unit/building/armed/combat-target buckets; only set the heavy dirty flag when a unit/building/tower changes. Drop the per-rebuild full sort — keep entities in id order incrementally, or sort only the small projectile delta.

[ISSUE_PERFORMANCE_02][HIGH] A* resets the entire walk grid (4 full passes over grid_w*grid_h) on every pathfinding call
  Where: rts-sim-wasm/src/pathfinder.rs:751-763 (g_score=INF, f_score=INF, parent=-1, closed=0); grid sized to full map at :148,:175-182; reuse-pattern precedent at body_pool.rs:1247.
  What: Each A* call re-initializes four full Vecs over all grid_w*grid_h cells before searching, regardless of how few cells the search expands. On the default map this is ~281k cells (~1.1M writes/call); on the largest map ~1.4M cells (~5.7M writes/call). The cost is paid on every call (per unit per move/replan).
  Why it matters: A short path touching a few dozen cells still pays an O(grid_cells) clear four times; when many units replan in one tick (combat, blocked paths, formation moves), this full-grid memset dominates pathfinding and produces tick spikes when the game is busiest — the classic A* scalability trap on the deterministic-lockstep authority.
  Fix: Use a per-cell generation/epoch stamp (`Vec<u32> visited_gen` + `current_gen` bumped each call): g/f/parent/closed are valid only when `visited_gen[cell] == current_gen`, making reset O(1). Mirror the existing generation pattern in body_pool.rs:1247; only expanded cells need stamping.

[ISSUE_PERFORMANCE_03][MEDIUM] All per-tick spatial HashMaps use std default SipHash instead of a fast integer hasher
  Where: rts-sim-wasm/src/lib.rs:21 (std HashMap = SipHash); spatial_grid.rs:69,100,109,130 with per-cell get at :1386,:1421,:1456; combat_targeting.rs:185,215,387,399 with KxK lookups at :2085,:2106 and entity_slot_by_id.get at :3322,:3873; body_pool.rs:1198,1216; Cargo.toml (no fast-hash dep); per-tick wiring at sim-wasm/init.ts:4100 and targetingInputStamping.ts:762.
  What: Every spatial structure on the per-tick hot path keys a `HashMap<u64,_>`/`HashSet<u32>` built with the std default SipHash-1-3 RandomState (a HashDoS-resistant cryptographic hash, irrelevant for internal integer cell keys). The contact solver, broadphase queries, observation/sensor index, and targeting all perform millions of integer hash lookups per tick. (Constant-factor swap, not an algorithmic regression.)
  Why it matters: SipHash is many times slower than a multiply-shift hash; at thousands of units these lookups are pure overhead on the dominant inner loops and directly cap unit count in the browser.
  Fix: Add `rustc-hash` (FxHashMap) or a small inline multiply hasher and switch every per-tick HashMap/HashSet (cells, observation_cells, entity_slot_by_id, sphere contact cells, query dedup) to it; keys are already packed integers so FxBuildHasher is a near drop-in (~3-10x faster/lookup). Leave setup-only maps (terrain bake) on the default.

[ISSUE_PERFORMANCE_04][MEDIUM] Targeting validation paths hash entity-id -> slot via SipHash map per turret/candidate instead of using already-known dense slots (unverified)
  Where: rts-sim-wasm/src/combat_targeting.rs:3322 (combat_targeting_entity_slot_for_id), called from :2575,:3335,:4429,:4550,:4599,:4849,:6161,:6896,:7125; incremental map maintained at :921/:928.
  What: combat_targeting_entity_slot_for_id does a SipHash lookup and is invoked from per-turret existing-lock validation, priority-target resolution, and candidate-slot fallback fill. The fast candidate path already avoids this via precomputed_candidate_slots, but the existing-lock/validation and priority paths re-hash an i32 id to a slot every tick per armed turret.
  Why it matters: Every armed turret with a live lock re-hashes its target id each tick to revalidate range/LOS — thousands of SipHash lookups per tick at scale, on top of the candidate scan; compounds the default-hasher finding.
  Fix: After switching to a fast hasher, cache the resolved dense target slot on the turret/lock state when a lock is established and revalidate cheaply (`pool.entity_id[slot] == target_id`), falling back to the map only on a miss. The map is maintained incrementally so a cached slot stays valid until despawn.

[ISSUE_PERFORMANCE_05][MEDIUM] Release wasm is optimized for size (-Oz) and the optimization pass is silently skipped if wasm-opt is absent (unverified)
  Where: package.json:12 (build:wasm); rts-sim-wasm/Cargo.toml ([package.metadata.wasm-pack.profile.release] wasm-opt = false).
  What: Cargo.toml disables wasm-pack's bundled wasm-opt; package.json runs a manual `wasm-opt -Oz` gated on `command -v wasm-opt` — so on any machine/CI without binaryen the shipped wasm gets only rustc opt-level=3 with no wasm-opt pass and the build still succeeds silently. When it does run, `-Oz` optimizes for size, not speed.
  Why it matters: For a compute-bound sim targeting thousands of units, `-Oz` trades runtime speed for binary size, and the silent-skip path means a contributor/CI lacking binaryen ships a slower module without warning — both reduce unit-count headroom.
  Fix: Use `wasm-opt -O3` (with the same `--enable-*` flags) and make the optimization pass mandatory in the release build (fail if wasm-opt is missing, or vendor/install binaryen in CI). Keep `-Oz` only for a separate explicit size build.

MISSING BAR FUNCTIONALITY / UX

[ISSUE_MISSING_BAR_01][HIGH] No build-power assist: fabricators cannot guard/assist another builder or a factory to pool build power
  Where: src/game/sim/commandExecution.ts:2015-2020,2131-2149 (executeGuardCommand/enqueueGuardAction); Simulation.ts:873-914 (guard = follow only); energyDistribution.ts:340-348 (pooling keyed strictly on builder.currentBuildTarget); construction.ts:199,387 (only writers of currentBuildTarget); factoryProduction.ts:320-336 (factory guard only stamps produced-unit rally); design intent at budget_design_philosophy.html:3055,1910.
  What: Guard exists only as follow/defend behavior — it never adds the guard's constructionRate to the guarded entity's build target. The energy build-consumer aggregation only sums builders whose currentBuildTarget equals the target; a guarding fabricator has no such target set, and factory guard only redirects the produced unit's rally. No code makes a con assist a factory's production or another con's build job. The design doc explicitly names "assist" as an intended intent, so this is a real gap, not deferred.
  Why it matters: Assist (guarding a builder/factory so idle cons dump build power into whatever it builds) is one of BAR's defining macro mechanics — how players accelerate factories and share build power. Without it, every fabricator must be micromanaged onto each job, factory output can't be sped up, and "guard a factory" silently does nothing economically.
  Fix: When a builder's active action is guard on a friendly builder/factory, mirror the guardee's current build/repair target into the guard's currentBuildTarget (or register it as an extra construction source for that target id) so it contributes constructionRate through the existing energyDistribution pooling; route a guarding con's build power into the factory's currentShellId consumer.

[ISSUE_MISSING_BAR_02][HIGH] No armor/damage-class system: every weapon deals identical damage to every unit category
  Where: rts-sim-wasm/src/damage.rs:1432,1476-1481 (damage_apply_batch takes a single scalar + only fortified-building multiplier); src/game/sim/damage/DamageSystem.ts:2290,2308-2309 (flat damage + only BUILDING_CLOSED_DAMAGE_MULTIPLIER); shots.json/turrets.json/units.json/blueprintSchema.json define no armor/damageClass/category/vs fields.
  What: There is no per-target-category damage table anywhere; the only conditional multipliers are off-building (0.1x) and fortified-building. A weapon deals the same damage to every unit. Not on the design doc's deferred list, so a genuine gap.
  Why it matters: BAR's combat identity is built on armordefs/per-target damage tables (e.g. 4x vs VTOL, 0.3x vs heavy armor). Without damage classes, every weapon is a flat DPS stick and unit counters collapse to raw HP/DPS math — one of the largest combat-feel gaps vs BAR.
  Fix: Add an armor-class tag per unit/building blueprint and a per-shot damage table keyed by armor class. Resolve the multiplier at queue time (u8 class index -> f32 lookup) so the hot damage_apply_batch stays a scalar multiply. Start with a small class set (raider/assault/structure/air).

[ISSUE_MISSING_BAR_03][MEDIUM] Attack command on empty ground is a no-op instead of attack-move
  Where: src/game/render3d/Input3DModeClickController.ts:892-908 (handleAttackClick, `if (!attackCmd) return;` at :905); RightClickCommands.ts:62-63,80-93 (buildAttackCommandAt returns null when no enemy under point); contrast right-click fall-through at Input3DRightDragController.ts:186-208; fight path at Input3DKeyboardController.ts:455-457.
