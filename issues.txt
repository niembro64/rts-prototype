RENDERING PIPELINE AUDIT - 2026-06-04

1. The renderer is still centered on Three.js instead of the target raw
   WebGPU/WGSL architecture.

Evidence:
- src/game/render3d/ThreeApp.ts owns the main renderer and the render tree is
  composed from many Three.js-specific renderers, meshes, materials, and
  instanced helpers.
- budget_design_philosophy.html says Three.js is legacy scaffolding and the
  long-term renderer should use owned buffers, explicit WGSL, and predictable
  batched data flow.

Problem:
The current architecture makes it easy to add more object-oriented Three.js
render paths, but the project target is data-oriented rendering. Each new
Three.js path increases migration cost and can hide JS allocation or buffer
upload costs behind library abstractions.

Required fix:
Stop adding new high-count visuals as bespoke Three.js object graphs. Identify
and migrate the highest-count paths first, such as unit detail instances, legs,
projectiles, terrain/fog/shield overlays, HUD/nameplate packets, and support
surface debug visualization. Replacement paths should use owned typed buffers,
explicit update ranges, and delete the old Three.js path once the new path is
active.
