PERF-1. Death scan walks every unit + every building every tick
  Sites:
    - src/game/sim/Simulation.ts:585-589 (units hp<=0 scan)
    - src/game/sim/Simulation.ts:592-596 (buildings hp<=0 scan)
  Cost: O(N_total) every sim tick. For a base of ~200 units + ~50
    buildings that's 250 hp reads × 60 Hz = 15k/sec. Cheap individually,
    but it's a full walk of stable entities every tick when only the
    handful that took damage this tick can possibly have died.
  Proposed: Maintain a `pendingDeathCheckIds: Set<EntityId>` on
    WorldState (or Simulation) populated by the damage system every
    time it applies a damage event. Each tick: drain the set, do hp<=0
    checks only on the damaged entities. The damage path already calls
    markSnapshotDirty(... ENTITY_CHANGED_HP) — the death-set add can
    sit right next to it.
  Risk: Low. New Set + drain helper; the trigger is a single line in
    the damage application path. Existing CacheManager already
    invalidates on entity removal so dead-entity bookkeeping is intact.

PERF-2. Action hash recomputed for every dirty entity every tick
  Sites:
    - src/game/network/stateSerializer.ts:518-533 (action hash loop in
      captureEntityState)
    - Action mutation sites:
      src/game/sim/factoryProduction.ts:172-176 (initial spawn queue)
      src/game/sim/commandExecution.ts:457,461 (set / append)
      src/game/sim/commanderAbilities.ts:103, energyDistribution.ts:155
  Cost: captureEntityState runs once per tick per dirty entity (we
    already hoisted the per-recipient redundancy out). For an entity
    with N actions in the queue, the per-tick cost is N*4 hash steps
    + N*4 multiplications + N field reads. Most entities sit on a
    stable action queue (move-to-X then idle, or stable patrol cycle)
    yet pay the full hash every tick they appear in the dirty set —
    and any unit whose turret is actively rotating IS dirty (the
    TURRETS bit fires).
  Proposed: Cache `actionHash: number` on `unit` (alongside the action
    array). Invalidate at every mutation site (push, replace, clear).
    captureEntityState reads the cached value; the comparison vs prev
    still works because the cache stays in lockstep with the array.
    To minimize per-mutation-site instrumentation, encapsulate behind
    a small helper (e.g. `setUnitActions(unit, actions)` /
    `pushUnitAction(unit, action)`) that owns the rehash, and route
    all mutations through it.
  Risk: Medium. Touches every action-mutation site; new helpers reduce
    the per-site burden but the audit must be careful (any direct
    `.push()` left unwrapped silently breaks dirty-detection). Add a
    dev-mode hash-vs-cached invariant check to catch drift.

PERF-3. shouldUpdateRichUnitDetails re-walks turrets every frame
  Sites:
    - src/game/render3d/Render3DEntities.ts:415-418 (turrets?.some)
  Cost: Per visible unit per frame: an Array.some over all turrets
    asking "is any turret tracking, engaged, or has a target?" For
    ~200 visible units × ~3 turrets average = 600 checks per frame
    just to decide whether to refresh detail. Short-circuits early in
    the common case but still pays the some() closure-allocation cost
    plus the field reads. The answer is already known to the sim —
    targetingSystem and turretSystem flip exactly the state these
    booleans track.
  Proposed: Add `combat.hasActiveCombat: boolean` (or a packed
    `combat.activeMask: number`) updated where state transitions
    happen in targetingSystem / turretSystem. The renderer reads the
    flag directly. Same pattern as the cached cos/sin and selection
    sets we've already moved sim → render.
  Risk: Low. Setter sites are localized to two files; renderer change
    is one line.

PERF-4. Turret head InstancedMesh write composes with identity quat
  Sites:
    - src/game/render3d/Render3DEntities.ts:1186-1192
  Cost: Per turret-with-head per frame: build a Matrix4 by composing
    (translation, identity-quat, uniform-scale). Matrix4.compose with
    identity quaternion still pays for the quaternion-to-matrix
    conversion that writes the identity rotation into the matrix
    elements. ~10 ops wasted per turret per frame.
  Proposed: Replace the compose with a direct
    `_smoothPartMat.makeScale(headRadius, headRadius, headRadius)`
    + `_smoothPartMat.setPosition(liftedPos.x, liftedPos.y, liftedPos.z)`
    pair. Head meshes have no rotation so the quat path is dead
    weight. (Don't apply this to the barrel path at line 1218+ — those
    DO use non-identity rotations.)
  Risk: Low. One-call swap, semantically identical for identity-quat
    case.

PERF-5. Targeting Pass-2 fires a per-weapon spatial query when no
  weapon needed acquisition
  Sites:
    - src/game/sim/combat/targetingSystem.ts:618-624 (batched query
      gated by `needsAnyQuery`)
    - src/game/sim/combat/targetingSystem.ts:651-653 (per-weapon
      fallback when batched is null)
  Cost: When all weapons on a unit are currently engaged with a target
    (no acquisition needed), `needsAnyQuery` is false and `batchedEnemies`
    stays null. Pass 2 still re-evaluates each weapon's tracking lock,
    and at line 653 each weapon falls back to its own
    `queryEnemyEntitiesInCircle2D` at the WEAPON's position with the
    weapon's own outermostAcquireDistance. For a unit with K weapons
    that's K spatial queries per tick when 1 wider query at the unit's
    position would cover them all (the per-weapon distance check
    downstream still filters precisely).
  Proposed: When entering Pass 2 with `needsAnyQuery=false`, do ONE
    spatial query at the unit's position with `maxAcquireRange +
    maxWeaponOffset` (the same radius the acquisition path uses). All
    weapons share that candidate list. The over-fetch is identical to
    what the acquisition path already accepts and the per-weapon
    distance check is unchanged.
  Risk: Low-Medium. The semantics of "outermostAcquireDistance per
    weapon" is preserved by the downstream rank/distance check; only
    the broad-phase query gets unified. Verify no weapon's
    outermostAcquireDistance exceeds maxAcquireRange (it shouldn't
    by definition — `maxAcquireRange` is the max across all weapons).
