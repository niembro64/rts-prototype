 PREDICT (POS / VEL / ACC) — implementation status, 2026-05-13
==============================================================================

A new three-way toggle "PREDICT: POS / VEL / ACC" lives in the PLAYER
CLIENT bar (label PREDICT, group of three buttons). It selects how
aggressively the client extrapolates physics between server snapshots:

  POS — no integration. Render position drifts toward the snapshot
        position via the existing lerp; no velocity / acceleration
        integration runs on the client this frame.
  VEL — integrate position from server-reported velocity each frame;
        ignore acceleration (no gravity, no movement-input thrust,
        no a*dt² term).
  ACC — full F=ma extrapolation (default; matches old behaviour).

Stored in localStorage under `player-client-prediction-mode`; default
is `acc` so behaviour for existing users is unchanged.

WIRED (reads getPredictionMode() and gates accordingly):
  - src/game/network/ClientUnitPrediction.ts
      advanceSharedUnitMotionPrediction — gates the entire
      advanceUnitMotionPhysicsMutable call. POS skips; VEL passes
      zero accel + zero gravity to the integrator; ACC unchanged.
      Affects both the entity body AND the snapshot target it drifts
      toward (advanceTargetExtrapolation calls the same helper).
      applyClientCombatExpensivePrediction — gates the per-turret
      `weapon.rotation += weapon.angularVelocity * dt` step
      (POS skips; VEL/ACC behave the same since the server doesn't
      report angular acceleration here).
  - src/game/network/ClientPredictionStepper.ts
      applyBeamPathPrediction — beam-path point integration
      `tp.x += vx*dt + ax*halfDt²` is gated. POS skips; VEL drops the
      ax*halfDt² and v += a*dt updates; ACC unchanged.
  - src/game/network/ClientProjectilePrediction.ts
      Both the snapshot-target integration (lines ~218-229) and the
      live projectile dead-reckon (lines ~248-258) are gated by mode.
      Gravity is skipped in POS and VEL; position integration is
      skipped in POS only.

NOT YET WIRED (future work — not gated; runs at full ACC always):
  1. ClientProjectilePrediction.ts → applyClientProjectileHoming.
     The intercept solver (`solveKinematicIntercept`) seeds the target
     state with the homing target's reported acceleration. In VEL mode
     we should pass accel=0 to the solver; in POS mode we should
     probably skip homing steering entirely so the projectile flies
     ballistically.
  2. unitSuspension / chassis tilt EMA. `advanceUnitSuspension` runs
     each frame regardless of PREDICT mode — it's a visual smoothing
     effect, not an authoritative integration, so leaving it alone is
     defensible, but a strict reading of "POS = no client integration"
     would freeze it too.
  3. Server-side integrators that COULD be gated by snapshot
     emission rate (i.e. only send velocity/acceleration when needed).
     stateSerializer.ts always writes velocity/acceleration into every
     snapshot today; a future per-recipient PREDICT-aware serializer
     could omit fields when the recipient is in POS/VEL mode and save
     bandwidth. Not done — would require client→server PREDICT mode
     announcement.
  4. Pure client-side effects that are NOT server-predicted and so
     have no "snapshot to snap to" (deliberately out of scope):
       - src/game/render3d/Debris3D.ts (gravity-driven debris particles)
       - src/game/render3d/ProjectileRangeEnvelope3D.ts (range preview)
     POS mode would freeze them in mid-air; VEL would strip gravity
     and they'd float away. These are authoritatively-client effects,
     not predictions of server state, so the PREDICT toggle leaves
     them at their full F=ma integration. Document this distinction in
     any future "purely client physics" toggle if one is wanted.
  5. Audit pass for any sim-side movement that bypasses the shared
     advanceUnitMotionPhysicsMutable helper. The audit found unit
     bodies, ballistic projectiles, and beam-path points all flow
     through the shared integrators, but a follow-up grep for `+= .* dt`
     across src/game/sim could surface stragglers (jump arcs, knockback
     impulses, force-field push, etc.) that the PREDICT mode should
     also gate when added.
