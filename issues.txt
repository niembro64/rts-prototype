SNAPSHOT WIRE SIZE PLAN (SNAP-WIRE-01 family)

==============================================================================

------------------------------------------------------------------------------
SNAP-WIRE-01 - parent: snapshot payloads exceed the current per-client
               internet budget at small battle sizes.
------------------------------------------------------------------------------

Evidence:
- The latest 2026-05-22 fixed-point capture in
  docs/dp-02-snapshot-wire-instrumentation.md shows the roughly
  200-unit DIFFSNAP stream back inside budget: 52,554 B avg and
  114,475 B hi, against the 64 KiB avg / 128 KiB hi targets.
- The same capture still shows FULLSNAP bootstrap at 1,076,601 B,
  about 28 KiB over the 1 MiB avg target, though still inside the
  1.25 MiB hi budget.
- The remaining large DIFFSNAP sections were audio events (~26.5 KiB),
  entities (~18.6 KiB), minimap entities (~4.0 KiB), and projectiles
  (~2.8 KiB). Further wins need structural row packing and
  section-specific budgets, not generic transport compression.
- At 30 DIFFSNAPs/sec plus 1 FULLSNAP/sec, those current average
  numbers imply roughly 21.2 Mbps per remote client before WebRTC
  overhead.

Why it matters:
Snapshot bandwidth is per remote client and burns host upstream first.
Even when the 200-unit delta stream is under target, every remote client
multiplies host upstream, FULLSNAP is still slightly over target, and
the 1,000 / 5,000-unit rows still need to be proven against the same
network budgets.

Fix direction:
Do child items as separate, measurable PRs. Prioritize structural
compression in the largest remaining sections: audio events, entity
rows, minimap entity rows, projectile rows, and FULLSNAP keyframe/static
sections. Each item must report before/after DS SIZE and FS SIZE numbers
from the PLAYER CLIENT bar, preferably with the same unit count and
snapshot cadence.
