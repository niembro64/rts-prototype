PERF-4. Turret head InstancedMesh write composes with identity quat
  Sites:
    - src/game/render3d/Render3DEntities.ts:1186-1192
  Cost: Per turret-with-head per frame: build a Matrix4 by composing
    (translation, identity-quat, uniform-scale). Matrix4.compose with
    identity quaternion still pays for the quaternion-to-matrix
    conversion that writes the identity rotation into the matrix
    elements. ~10 ops wasted per turret per frame.
  Proposed: Replace the compose with a direct
    `_smoothPartMat.makeScale(headRadius, headRadius, headRadius)`
    + `_smoothPartMat.setPosition(liftedPos.x, liftedPos.y, liftedPos.z)`
    pair. Head meshes have no rotation so the quat path is dead
    weight. (Don't apply this to the barrel path at line 1218+ — those
    DO use non-identity rotations.)
  Risk: Low. One-call swap, semantically identical for identity-quat
    case.

PERF-5. Targeting Pass-2 fires a per-weapon spatial query when no
  weapon needed acquisition
  Sites:
    - src/game/sim/combat/targetingSystem.ts:618-624 (batched query
      gated by `needsAnyQuery`)
    - src/game/sim/combat/targetingSystem.ts:651-653 (per-weapon
      fallback when batched is null)
  Cost: When all weapons on a unit are currently engaged with a target
    (no acquisition needed), `needsAnyQuery` is false and `batchedEnemies`
    stays null. Pass 2 still re-evaluates each weapon's tracking lock,
    and at line 653 each weapon falls back to its own
    `queryEnemyEntitiesInCircle2D` at the WEAPON's position with the
    weapon's own outermostAcquireDistance. For a unit with K weapons
    that's K spatial queries per tick when 1 wider query at the unit's
    position would cover them all (the per-weapon distance check
    downstream still filters precisely).
  Proposed: When entering Pass 2 with `needsAnyQuery=false`, do ONE
    spatial query at the unit's position with `maxAcquireRange +
    maxWeaponOffset` (the same radius the acquisition path uses). All
    weapons share that candidate list. The over-fetch is identical to
    what the acquisition path already accepts and the per-weapon
    distance check is unchanged.
  Risk: Low-Medium. The semantics of "outermostAcquireDistance per
    weapon" is preserved by the downstream rank/distance check; only
    the broad-phase query gets unified. Verify no weapon's
    outermostAcquireDistance exceeds maxAcquireRange (it shouldn't
    by definition — `maxAcquireRange` is the max across all weapons).
