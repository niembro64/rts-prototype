RENDERING PIPELINE AUDIT - 2026-06-04

1. Scoped render phase still performs whole-world projectile/effect scans.

Evidence:
- src/game/scenes/helpers/RtsScene3DRenderPhase.ts still calls
  clientViewState.getProjectiles() for scoped shot name packets and filters each
  projectile with renderScope.inScope().
- The same render phase collects line, smoke-trail, and burn-mark projectiles
  from client-wide projectile caches before downstream renderers apply their own
  scope checks.
- Render3DEntities.collectBeamTurretAim() still scans all line projectiles each
  frame to refresh beam-directed turret aim.

Problem:
The unit/building packet is now spatially scoped, but projectile and effect
packet population still does JavaScript full-list iteration each frame. Scoped
render modes reduce draw work after those scans rather than avoiding the scans.

Required fix:
Back scoped projectile/effect packets with spatially culled render lists or
typed-buffer query ranges. Keep the existing packet objects and scratch arrays,
but fill projectile, beam, smoke, burn-mark, HUD shot-name, and beam-aim inputs
from visible-cell/projectile indices plus explicit exceptions for recently
relevant projectiles or emitters.

2. The renderer is still centered on Three.js instead of the target raw
   WebGPU/WGSL architecture.

Evidence:
- src/game/render3d/ThreeApp.ts owns the main renderer and the render tree is
  composed from many Three.js-specific renderers, meshes, materials, and
  instanced helpers.
- budget_design_philosophy.html says Three.js is legacy scaffolding and the
  long-term renderer should use owned buffers, explicit WGSL, and predictable
  batched data flow.

Problem:
The current architecture makes it easy to add more object-oriented Three.js
render paths, but the project target is data-oriented rendering. Each new
Three.js path increases migration cost and can hide JS allocation or buffer
upload costs behind library abstractions.

Required fix:
Stop adding new high-count visuals as bespoke Three.js object graphs. Identify
and migrate the highest-count paths first, such as unit detail instances, legs,
projectiles, terrain/fog/shield overlays, HUD/nameplate packets, and support
surface debug visualization. Replacement paths should use owned typed buffers,
explicit update ranges, and delete the old Three.js path once the new path is
active.

3. Auxiliary WebGL renderers are outside the main renderer singleton budget.

Evidence:
- src/game/lifecycle/sessionSingleton.ts limits active main GameInstance and
  renderer ownership.
- Loading previews, commander/avatar previews, and the world-direction HUD create
  their own WebGLRenderer instances outside that singleton.

Problem:
These contexts appear to have teardown paths, so this is not currently a proven
leak. It is still an unbudgeted GPU-memory/context-count risk on low-end
devices, especially during loading or UI-heavy screens where preview renderers
and the main renderer may overlap.

Required fix:
Add renderer/context-count telemetry and an explicit auxiliary-renderer budget.
Prefer shared rendering, canvas-2D, or a disabled/low-memory fallback for UI
previews and HUDs when the main game renderer is active.
