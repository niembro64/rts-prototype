Audit findings - 2026-05-07

2. FULL SNAP is not self-contained for live projectiles.
   - Full keyframes serialize units and buildings, but exclude `shot` entities
     (`src/game/network/stateSerializer.ts`).
   - Projectile creation is sent as an event/side channel from `ServerSnapshotPublisher`, and whole state
     messages can be dropped under backpressure in `NetworkManager`.
   - If a client misses a projectile spawn, a later full snapshot recovers units/buildings but cannot recover
     the active projectile or beam. Beam updates also cannot create a missing beam projectile
     (`src/game/network/ClientProjectileStore.ts`).
   - Fix direction: include live projectile state in full snapshots, or make projectile spawn events reliable
     until acknowledged.

3. Debug grid snapshots retain pooled/reused server references.
   - `ServerDebugGridPublisher.refresh` returns internal cached arrays.
   - `SnapshotBuffer` stores `state.grid` by reference and can attach that same grid object to a later pending
     snapshot.
   - `ClientViewState` also stores the grid arrays by reference.
   - On the local host path, later server debug-grid refreshes can mutate data that the client already holds,
     causing debug overlays to change outside normal snapshot application.
   - Fix direction: copy debug grid rows/cells into the buffer/client, or make each emitted debug grid payload
     immutable.

4. Projectile/event DTO copy logic is duplicated in multiple places.
   - Projectile spawn, velocity, beam, and event copy helpers exist independently in:
     `src/game/scenes/helpers/SnapshotBuffer.ts`,
     `src/game/network/ProjectileSpawnQueue.ts`, and
     `src/game/network/snapshotClone.ts`.
   - This is a schema-drift risk: adding a field to a projectile spawn or beam update requires remembering
     every duplicate copy helper.
   - Fix direction: centralize these DTO copy helpers and reuse them from the buffer, queue, and snapshot
     cloner.

5. Dead or legacy protocol state should be removed or documented.
   - `DeltaTrackingState.protocolSeeded` is allocated, tracked, cleared, and added to on full records, but no
     longer appears to gate serialization behavior (`src/game/network/stateSerializer.ts`).
   - `NetworkManager.broadcastState` appears unused and duplicates much of `sendStateTo`
     (`src/game/network/NetworkManager.ts`).
   - `NetworkEntityFactory.createProjectileFromNetwork` supports snapshot `shot` entities, but the current
     serializer excludes shots from full/diff entity snapshots. Projectile creation now appears to use
     `ClientProjectileStore` spawn events instead.
   - Fix direction: delete the dead paths, or document the intended future protocol use so they do not drift
     silently.

6. `NetworkManager.disconnect` leaves callbacks and connection timers behind.
   - `disconnect` clears many callbacks but omits `onLobbySettings`, `getLobbySettings`, and
     `onPlayerInfoUpdate`, which are assigned closures from `GameCanvas.vue`.
   - `hostGame` and `joinGame` create 10-second setup timeouts that are not stored or canceled on disconnect.
     A stale timeout can later call `this.peer?.destroy()` after the user has canceled/retried and created a
     newer peer.
   - Fix direction: clear all callback fields on disconnect, store timeout IDs, cancel them on disconnect, and
     guard timeout callbacks by peer identity or session generation.

7. HOST SERVER Tilt EMA setting is not included in snapshot meta.
   - The host UI sends tilt EMA changes to the server, and `GameServer` applies them.
   - `ServerSnapshotMetaBuilder.build` includes other host-controlled settings such as sim LOD and wind, but
     does not include tilt EMA mode.
   - Remote clients render HOST SERVER controls from snapshot meta, so their read-only tilt EMA value can show
     a stale local persisted setting instead of the host's active setting.
   - Fix direction: add tilt EMA mode to `NetworkServerSnapshotMeta` and reconcile/render it like the other
     host server settings.

8. Surface normal / Tilt EMA updates only piggyback on position snapshots.
   - Serialization sends `unit.surfaceNormal` only on full snapshots or position-dirty updates.
   - `updateUnitTilt` can change surface normals every simulation tick without marking the entity position
     dirty.
   - Stationary units whose normals are still EMA-settling, or units affected by a host tilt-mode change, may
     not send updated normals until they move or until the next full snapshot.
   - Fix direction: add a dedicated surface-normal dirty bit, mark relevant units dirty when tilt mode changes,
     or send normals on a controlled low-rate cadence.
