A4. [HIGH] Force-field panel and sphere are NOT one material differing only in
    geometry — they are two parallel subsystems plus two different shaders.
    LOCATION:
      - hit-test/occlusion: src/game/sim/combat/ForceFieldPanelHit.ts (ray-vs-rect)
        vs forceFieldTurret.ts (ray-vs-sphere); lineOfSight.ts:79-104
        (hasForceFieldClearance vs hasForceFieldPanelClearance, branched at :111-131).
      - Rust: lib.rs:13716-13744 force_field_panel_projectile_intersection vs
        force_field_projectile_intersection; separate pools (forceFieldPanelPool vs
        forceFieldPool); separate REFLECTOR_HIT_KIND_FORCE_FIELD_PANEL/_SPHERE.
      - render: ForceFieldReflectorVisual3D.ts:22-30 (panel = MeshBasicMaterial)
        vs ForceFieldRenderer3D.ts:75-93 (sphere = custom ShaderMaterial).
    RULE: "Materials Are Independent Of Shape" — the panel's flat panels and the
      sphere are THE EXACT SAME force-field material (same reflectivity, same LOS
      occlusion rule, same projectile reflection, same shader); they differ ONLY in
      geometry. No code path may branch material behavior on which piece owns the
      surface.
    PROBLEM: the entire intersection + occlusion + shading stack is duplicated and
      branched on panel-vs-sphere. (Mitigation: the post-hit reflection physics
      reflectVelocityPreserveSpeed and the impact visual ARE shared and shape-
      agnostic, so the divergence is in geometry-intersection/occlusion plumbing and
      rendering, not the response.)
    FIX: unify into one material/policy that takes a shape (rect | sphere) for the
      intersection + normal, with one shared shader; collapse the two clearance
      functions and the two Rust pools.

A5. [MED] Snapshot entity encoding has two implementations, and the TS one runs on
    the live Rust path (double entity walk).
    LOCATION: snapshotEntityWirePack.ts (TS per-entity encoder) vs
      snapshotRustWireEncoder.ts (WASM per-entity encoder); wired in
      snapshotWireCodec.ts:11-21. packEntitiesForWire is called unconditionally at
      snapshotWireCodec.ts:232; the Rust encoder then re-emits the TS-built blob raw
      (snapshotRustWireEncoder.ts:1944-1946), bypassing its own
      encodeEntityUnit/Building kernels for entities.
    RULE: "Delete The Old Path" / "TypeScript Coordinates, WASM Computes" (no large
      serialization loops in TS).
    PROBLEM: a full per-entity TS object-building loop runs every snapshot even when
      the Rust encoder is active, and a second full DTO→wire implementation is
      maintained in lockstep.
    FIX: make either the TS pack path or the Rust per-entity encoder the single
      owner of entity bytes; stop running packEntitiesForWire on the Rust path.

A6. [MED] Wire-code / state enums are hand-duplicated in TS and Rust.
    LOCATION: src/types/network.ts (ENTITY_CHANGED_* bits :685+, TURRET_STATE_* :18-46,
      ACTION_TYPE_*, PROJECTILE_TYPE_*, RESOURCE_KIND_*, RESOURCE_FLOW_*,
      REFLECTOR_HIT_KIND_*) vs lib.rs (e.g. :6743 CT_TURRET_STATE_ENGAGED,
      :14258-14266 changed-field bits).
    RULE: "Config Is Data, Not Code" — wire codes / enum value-name maps both TS and
      Rust must agree on belong in ONE file, two consumers, zero drift.
    PROBLEM: these numeric codes must match byte-for-byte across the boundary but are
      independently hand-written in both languages. (Evidence of the cost: recent
      commit "Rename Rust beam-encoder reflectorKind strings to match TS".)
    FIX: route them through the existing codegen — build.rs:22-79 already generates
      Rust constants from src/sharedSimConstants.json; add the wire/state enums there
      and generate both the TS and Rust sides.

A7. [MED] Producer ON/OFF is carried by two flags (open + producing) instead of one.
    LOCATION: src/game/sim/buildingActiveState.ts:52-59 (state has both open and
      producing; producing slaved to open each tick via setBuildingProducing).
      Damage reads open (isBuildingActiveStateFortified :237-243); solar/extractor
      income reads producing (economy.ts:316); wind income + renderer pose + converter
      read open (economy.ts:325, BuildingAnimationController3D.ts:215,240,286).
    RULE: "Producer Buildings Are ON/OFF" — the ON/OFF state must be the SINGLE
      authoritative input for BOTH the renderer's open/closed pose AND production
      accounting; they must never disagree; "not two separate flags."
    PROBLEM: this is exactly the two-flags-that-could-disagree shape the rule forbids;
      they only agree because producing is mechanically slaved to open every tick.
      (The 0.1 OFF damage multiplier is correct: BUILDING_CLOSED_DAMAGE_MULTIPLIER,
      DamageSystem.ts:1115-1117.)
    FIX: make all production gate on open (delete producing), so there is one flag.

A9. [LOW] Sim-relevant tuning still lives in TS modules.
    LOCATION: src/config.ts (~115 exported const tuning objects; some are sim-relevant,
      many are render-only), src/game/sim/blueprints/displayRosters.ts (roster ordering).
    RULE: "Config Is Data, Not Code" — Rust cannot import .ts; tuning the sim needs
      should be JSON.
    PROBLEM: TS-module tuning tables are a wall for the TS→Rust migration. (Mostly
      resolved already — unit/turret/shot/locomotion/building blueprints are now JSON
      consumed by Rust via build.rs embedding; the .ts files in blueprints/ are
      loaders, not data.)
    FIX: migrate sim-relevant tables out of config.ts into JSON validated by a shared
      schema; leave render-only constants for later.

A10. [LOW] Optional-chaining that can silently no-op inside sim logic.
    LOCATION: src/game/sim/commandExecution.ts:729
      `(target.unit?.hp ?? target.building?.hp ?? 0) > 0`.
    RULE: "Explicit Absence" — no ?. in our sim/server/prediction/networking code; use
      T | null with explicit guards.
    PROBLEM: this is essentially the ONLY risky case left — the rule is otherwise very
      well honored (only ~2 ?. total across sim/server/network, ~6 field?: at
      serialization/settings boundaries; core entity-state types have zero). Worth
      tidying so absence is an explicit decision.
    FIX: replace with explicit discriminated handling of unit vs building vs neither.

C2. [MED] Client snapshot DECODE allocates fresh per-entity DTOs + nested literals
    every snapshot.
    LOCATION: snapshotEntityWirePack.ts:252,270 (new Array), :901-932 + :1010-1041
      (createEmptyUnitSub + {x,y,z}/{x,y,z,w} literals per unit); entered from
      snapshotWireCodec.ts:295.
    RULE: "Pool snapshot DTOs"; "do not allocate in per-unit loops".
    PROBLEM: the ENCODE side is exemplary (pooled PooledEntry + typed-array wire rows),
      but the remote-client DECODE side allocates a fresh entity array plus per-unit
      sub-object + velocity/orientation/angularVelocity3 literals — ~50k+ short-lived
      objects/sec at 5000 units @10Hz, discarded after ClientSnapshotApplier consumes
      them. (Host/local play skips this.)
    FIX: pool the decode DTOs like the encode side.

C3. [MED] unitRenderMode is a static config value ('rich') and never escalates to the
    instanced path at scale.
    LOCATION: src/playerClientGraphicsConfig.json:3 ("unitRenderMode":"rich");
      consumed Render3DEntities.ts:420-443. The fully-instanced 'mass' path
      (UnitMassInstanceRenderer3D) exists but is only used when config says so.
    RULE: "Prefer instancing / LOD grids over per-unit scene-graph complexity."
    PROBLEM: in 'rich' mode every unit gets a per-unit scene-graph group and
      updateUnitMeshes does per-unit Map.get/set, quaternion composition, and
      material-reassignment loops (:553-569) for all units every frame. (Chassis
      spheres / turret heads / barrels ARE instanced, so this is CPU-per-unit
      scene-graph + Map churn, not 5000 draw calls.) Nothing switches to 'mass'/
      'hybrid' as count climbs toward 5000.
    FIX: add automatic count-based render-mode escalation.

C4. [MED] SpatialGrid readback allocates a new Uint32Array view per query.
    LOCATION: src/game/sim/SpatialGrid.ts:244-247 (readScratch returns
      `new Uint32Array(sim.memory.buffer, ...)`).
    RULE: "Pool temp buffers"; "do not allocate in per-tick loops."
    PROBLEM: every query (queryUnitsInRadius / queryEnemy* / queryEntitiesAlongLine,
      used by projectile collision broadphase DamageSystem.ts:174,512,632,713,824,949,
      targeting, capture) allocates a view wrapper — dozens-to-hundreds/tick. Views are
      cheap (no copy) but still garbage.
    FIX: cache the view and rebuild it only when sim.memory.buffer changes (the same
      growth event PhysicsEngine3D already tracks for refreshViews).

VERIFIED-CLEAN (audited, no action needed — recorded so we don't re-investigate)
================================================================================
- No acceleration on the wire; PREDICT mode is POS/VEL only; no ACC option.
- The four EMA channels (movement pos/vel, rotation pos/vel) are each applied to
  ALL their documented fields (unit transform/velocity, beam point pos/vel, yaw
  scalar + orientation quat + turret yaw/pitch, angularVelocity3 + turret rates,
  force-field range visualizer).
- Travelling projectiles snap on velocityUpdate events (no EMA target populated).
- Delta thresholds: position uses absolute error, velocity uses relative; the
  zero-velocity edge is sent; baseline = last value actually sent per recipient.
- EntityType has Tower as a true peer of Unit/Building; fabricator is a tower, not a
  building; pure-infra buildings carry no turrets/lock-on.
- Lock-on exclusion lists use the exact documented field names and are honored
  subtractively in targeting.
- Airborne mount invariant (mount.y===0 && mount.z===0, no zResolver) is enforced and
  throws at startup.
- Rockets inherit the launch target and never re-acquire; lost target → ballistic.
- Sensors are capability sources with fixed radii (not derived from weapon range).
- Resource movement: every stockpile mutation funnels through ResourceMovementSystem
  credit/debit; renderer drives one shared pylon rig for all resource visuals.
- Selection panels are gated on entity TYPE capability flags, not per-blueprint buttons.
- No banned libs (Phaser/Pixi/Matter/cannon/ammo) are imported anywhere in src/.
- No legacy/old/_unused dirs or .old/.bak files in src/ or rts-sim-wasm/.
