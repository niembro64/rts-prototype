[049][ARCH] projectileSystem.ts (~1800 lines) is no longer the
integration/guidance math owner — those kernels run in Rust
(pool_step_packed_projectiles_batch, projectile_integrate_step_batch,
projectile_homing_guidance_apply_batch in rts-sim-wasm/src/projectile.rs) —
but TS still runs per-projectile loops around them: packed-pool
gather/scatter passes, traveling-projectile policy packing (homing target
validation/aim points, D-gun terrain-follow ground sampling), arming, and
snapshot velocity-delta events. Beam stepping (per-tick re-trace incl.
DamageSystem.findBeamPath) is still fully TS, and collision orchestration
(ProjectileCollisionHandler.ts, ~1789 lines) is TS gather/scatter around six
WASM batch kernels. Move per-projectile state residency and policy into the
Rust pool so TS shrinks to spawn/despawn orchestration; port beam tracing and
collision orchestration to Rust; delete the TS loops in the same change.

[051][ARCH] targetingInputStamping.ts (~965 lines) is the TS->Rust
input-stamping shim for the targeting FSM. The readback half is done:
snapshots, rendering, prediction, and combat consumers read the Rust slabs
directly, and FSM state/target/cooldowns/losBlockedTicks are slab-owned. What
remains is the input direction: stampCombatTargetingPool clears and fully
rebuilds every entity/turret slab row from JS Entity/Turret objects each tick
(SimulationCombatController.ts:90), and stampShieldSurfacePool stamps shield
surfaces twice per tick (:86, :163). Make Rust persistently own the
entity/turret input state (or feed deltas) so the per-tick full restamp
disappears, retire the transitional Turret.target/state JS fields
(src/types/sim.ts), then delete the shim. (The shim's header comment claiming
it "copies them back to JS Turret objects" is stale — the copy-back is gone.)

[052][ARCH] Damage classification and HP batching now run in Rust (damage.rs
segment/sphere/area kernels + damage_apply_batch; TS throws without sim-wasm,
TS geometry mirrors are DEV-only asserts). But DamageSystem.ts (~2343 lines)
still materializes resolution side effects in TS after the Rust batches:
knockback force vectors, death contexts (penetration dir/attacker vel),
killer bookkeeping, and HP/killed write-back onto JS entity objects;
damageHelpers.ts (~481 lines) still assembles all death/hit/attackAlert
SimEvents from TS entities. Move those resolution outputs into Rust-owned
event/result buffers so TS receives typed events/knockbacks only, then delete
the remaining TS resolution math/helpers (projectile-row + ground live
geometry in findBeamSegmentHit is the documented post-integration exception —
fold in or re-justify).
