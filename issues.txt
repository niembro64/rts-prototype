A28. [MEDIUM] src/game/sim/blueprints/turrets.json (turretForceFieldPanel: shotId
    points at a beam shot; force-field behavior carried by a turret-side
    forceFieldPanels/panel config) + src/game/sim/combat/forceFieldTurret.ts,
    ForceFieldPanelHit.ts —
    RULE: Atomic Gameplay Pieces (philosophy lines 1296, 1508-1516): shot is the single
    umbrella (projectile/beam/laser/forceField); "forceField is a shot variant like any
    other rather than a special case bolted onto the turret"; a turret names exactly one
    shot by id and carries no inline emission payload. —
    PROBLEM: the force-field PANEL barrier is modeled as turret-side configuration (the
    turret names a beam shot and carries panel/force-field data) instead of the turret
    naming a single forceField shot variant that owns the barrier — the "special case
    bolted onto the turret" the doc calls out. The force-field SPHERE path DOES use a
    forceField shot (forceFieldBarrierShot), so the two force-field turrets are modeled
    inconsistently. —
    FIX: model the panel force-field as a forceField shot variant (like the sphere), have
    turretForceFieldPanel name it by id, and move the barrier material/reflection data off
    the turret onto the shot; keep only panel geometry on the mount per "Materials Are
    Independent Of Shape".

A29. [LOW] src/types/shotTypes.ts (forceField shot "material" = color/alpha/visual only)
    and the reflection logic in src/game/sim/combat/ (reflection governed by world/global
    settings rather than a per-material contract) —
    RULE: "Materials Are Independent Of Shape" (philosophy lines 1824-1860) + shot variant
    owns "barrier material and reflection behavior" (line 1296): a material is a physical
    + visual contract (reflectivity, occlusion, projectile response, hit reaction) defined
    once and shared across shapes. —
    PROBLEM: the force-field "material" on the shot is purely visual (color/alpha); the
    actual barrier behavior (reflection) is driven from world/global settings, and
    absorption / damping / pass-through / per-shot-type response are unmodeled — so a
    material is not the single shape-independent physical+visual contract the doc
    describes. —
    FIX: promote the force-field material to a real contract object (reflectivity,
    occlusion, per-shot-type response, hit reaction, plus visual params) defined once and
    referenced by both panel and sphere shots; drive reflection from that material instead
    of world settings, so panel and sphere behave identically by construction.

A30. [LOW] rts-sim-wasm + the EntityMeta construction/registry (generation set to 0 at
    creation and never incremented on slot reuse; alive set true at spawn, never set false
    on death; death removes the row) —
    RULE: "Opaque handle, explicit metadata" (philosophy lines 1519-1544): EntityId is
    "preferably generation-checked"; EntityMeta carries generation + alive, and lock-on
    refs are "resolved through the registry and generation-checked before use" (lines
    1625-1634). —
    PROBLEM: generation is always 0 and never incremented, and alive is never set false
    (death deletes the row), so the generation-checked stale-handle protection the doc
    describes is not actually functioning — a reused id/slot cannot be distinguished from
    the original via generation. In practice stale refs are caught by row-absence, not by
    the generation check. —
    FIX: either implement generation bumping on slot reuse + set alive=false on death and
    have ref resolution generation-check (matching the doc), or, if row-absence is deemed
    sufficient, update the doc to drop the generation-checked guarantee. The field exists
    but is currently inert.

A31. [WATCH] src/game/sim/factoryProduction.ts (multi-item queue array + dequeue/splice
    by index) and the factory production command/wire path —
    RULE: "Producer Buildings Are ON/OFF" (philosophy lines 1805-1820) + Selection menus
    (lines 1790-1794): producer buildings carry a single binary ON/OFF; the building
    selection panel is ON/OFF + self-destruct only — no per-item queue UI, no batching, no
    queued unit list. —
    PROBLEM: the factory production model still carries a multi-item build QUEUE (an array
    with index-based cancel/splice) in the sim/command/wire layer — the per-item production
    queue the ON/OFF producer rule is meant to retire. (The UI may already be ON/OFF-only,
    but the underlying data/command model still encodes a queue.) WATCH because it may be
    an intentional in-progress migration. —
    FIX: decide the producer model: if ON/OFF is the target, reduce factory production to
    the binary consume→produce→repeat loop with a single rally setting and remove the
    multi-item queue from the data/command/wire model; if a queue is intentionally retained
    for factories, update the doc to carve factories out of the strict ON/OFF rule.

A32. [LOW] src/game/sim/factoryProduction.ts (rally modeled as an appendable list of
    waypoints) and the rally command path —
    RULE: "Producer Buildings Are ON/OFF" (philosophy lines 1805-1820): rally is a separate
    STATIC setting, not a queue entry; producers are a single binary toggle with no waypoint
    list / shift-click batching. —
    PROBLEM: factory rally is modeled as a growable multi-waypoint queued list (shift-append)
    rather than the single static rally point the ON/OFF model implies, extending the
    producer beyond the binary toggle + single rally. —
    FIX: reduce factory rally to a single static rally target (latest set wins), or if
    multi-waypoint rally is intended, document it as an explicit exception. (Pairs with A31.)

A36. [LOW] src/types/blueprints.ts + src/types/shotTypes.ts (hand-written TS types) and
    the parallel hand-written Rust structs in rts-sim-wasm/src/lib.rs —
    RULE: "Schema is the contract" (philosophy lines 2034-2041): "TypeScript types and Rust
    structs are generated from the schema. They are not hand-maintained in two languages —
    that would just relocate the duplication problem from data to types." —
    PROBLEM: blueprint/shot types are hand-maintained separately in TypeScript
    (src/types/*.ts) and in Rust (lib.rs structs) rather than generated from one canonical
    schema, so the "generated from the schema, not hand-maintained in two languages"
    contract is not met and TS/Rust can drift. —
    FIX: introduce a single canonical schema (e.g. JSON Schema) for blueprint/shot pieces and
    generate both the TS types and Rust structs from it, replacing the hand-written
    definitions, so the two languages cannot drift. (Related to C1.)

    FIX: continue migrating per-tick/per-unit authoritative control flow into the Rust sim

    ledger of which systems are Rust-authoritative vs TS-orchestrated so there are never two
    sources of truth. Large multi-step effort; lower urgency than the SECTION A items.

VERIFIED-CLEAN (audited, no action needed — recorded so we don't re-investigate)
================================================================================
(2026-05-30 audit: 21 candidate findings were refuted by independent verifiers
that re-read the live code. The notable ones — things one might assume are gaps
but are not — are folded into the list below.)

- No acceleration on the wire; PREDICT mode is POS/VEL only; no ACC option.
- The four EMA channels (movement pos/vel, rotation pos/vel) are each applied to
  their documented fields. NB: body angularVelocity3 is EMA-corrected but NOT
  integrated into orientation — see A23 (the channel binding exists; the
  integration step does not).
- Travelling projectiles are event-driven (spawn / velocityUpdate / despawn) and
  dead-reckoned — they do NOT ride the movement-position/velocity EMA channels.
- Delta thresholds: position uses absolute error, velocity uses relative; the
  zero-velocity edge is sent; baseline = last value actually sent per recipient.
- Command-upward / snapshot-downward network architecture audited COMPLIANT:
  commands go through the host (authority/sanitize via commandAuthority.ts +
  commandSanitizer.ts), selection is local, snapshots are corrections. (The
  snapshot RATE default is out of band — see A21; and the rate dropdown also
  exposes 32/64/128 options up to the tick rate, refuted as a config-option
  range, not a shipped default.)
- Locomotion arrival control for GROUND, HOVER, and FLYING units correctly uses
  position error + velocity error with thrust saturation for final waypoints
  (compute_arrival_control_thrust in lib.rs ~681-808); final vs intermediate
  waypoints are distinguished (ARRIVAL_FLAG_LAST_ACTION); waypoints are desired
  states, not magnets.
- Gravity integration is universal in the body integrator (lib.rs passes
  authored_az − GRAVITY with no per-body skip branch); homing counter-thrust,
  D-gun terrain-follow, and hover lift are CLEAN — hover fights g·m with a lift
  force, it does NOT skip gravity. Projectile shots now validate a positive
  gravityForceMultiplier; line weapons still use 0 because they are line traces,
  not ballistic bodies.
- Hover lift is an anti-gravity force, not a gravity-skip conditional (verifier
  re-read confirmed; the earlier "hover is a PD pull, not anti-gravity" candidate
  was refuted).
- The host-directed one-per-kind STARTUP THROW IS enforced:
  validateHostDirectedMounts (src/game/sim/blueprints/index.ts:717-733) groups a
  host's mounts by turret kind and requires exactly one hostDirected per present
  kind (zero = orders can't land; ≥2 = ambiguous primary), skipping buildings.
  (The earlier "loader does not enforce" candidate was a misread and is refuted.)
- Level-1 named lock-on exclusions ARE applied in the targeting kernel:
  combat_targeting_level1_mask_excludes (rts-sim-wasm/src/lib.rs ~13000, used at
  13069/13084/13091/13098) filters individual unit/tower/building/turret blueprint
  names AFTER the level-0 family pass, with unit tests
  (level1_body_exclusions_reject_only_matching_blueprints :25975,
  level1_turret_exclusions_filter_individual_mounted_turrets :26046). Level-0
  (friend/enemy + entity family) is also applied. (The earlier "level-1 never
  applied" candidate is refuted.)
- ShotSource provenance is complete and propagated: createTurretShotSource
  (projectileSystem.ts:359-377) populates the full record incl.
  sourceTurretBlueprintId; the "blueprint id can be null" candidate was refuted on
  the merits.
- EntityId is an opaque handle resolved through metadata rows; player/team/host/
  blueprint/storage are NOT packed into the number. (The generation/alive fields
  exist but are currently inert — see A30.)
- Build sites and factory unit shells are the real product entities from spawn:
  one continuous EntityId from nanoframe to finished structure; no re-id handoff.
- EntityType has Tower as a true peer of Unit/Building; the fabricator is in the
  tower catalog, not a building; pure-infra buildings carry no turrets/lock-on.
- Turrets name one neutral shotId; the force-field SPHERE uses a forceField shot
  (forceFieldBarrierShot). (The force-field PANEL is the inconsistent one — it
  names a beam shot with turret-side panel config; see A28/A29.)
- Host-directed vs fully-autonomous turret mounts are wired correctly: the tag
  lives on the unit/tower MOUNT entry (not the turret blueprint); host lock-on
  overwrites host-directed turrets, fully-autonomous run independently;
  host-directed mounts fall back to autonomous when the host has no lock; the
  fight-stop ratio counts only host-directed mounts.
- Rockets inherit the launch target and never re-acquire; lost target → ballistic.
- Airborne mount invariant (mount.y===0 && mount.z===0, no zResolver) is enforced
  and throws at startup for hover/flying turrets.
- Locomotion visual state (wheel/tread/leg/fan) is computed client-side in the rig
  and does NOT cross the wire.
- Camera anchoring is two orthogonal config axes (anchorPointCameraScreen +
  anchorPointCameraTerrain) resolved by one shared resolver for mouse and touch
  rotate/zoom/pan gestures; all six combinations are reachable from config.
- Sensors are capability sources with fixed radii (NOT derived from WEAPON range).
  (Builder sight IS partly derived from build-reach — a separate, milder coupling;
  see A22.)
- Resource movement: every per-tick stockpile delta funnels through the single
  ResourceMovementSystem credit/debit path (creditStockpile resourceMovement.ts:38,
  debitStockpile :53 are the only mutators of stockpile.curr); one shared render
  rig draws all streams. No side-channel mutates the player pool. (The earlier
  side-channel-bypass candidate was refuted; a producer-income-via-config-scalars
  candidate was also refuted — the doc does not require pylon endpoints authored on
  the host blueprint.)
- Selection panels are gated on entity TYPE capability flags, not per-blueprint
  buttons; mixed-type selections show the action-set intersection plus explicit
  select-only controls for narrowing the active type.
- Blueprints are pure data: JSON files imported by thin TS loaders (units.ts imports
  units.json, turrets.ts imports turrets.json, etc.); no closures/functions in
  blueprint JSON; config.ts holds zero authoritative tuning literals. Load-time
  validation THROWS with no silent defaults (jsonValidation.ts assertExplicitFields);
  Explicit Absence is honored (required null fields). Single Source Of Truth holds
  (audio refs, JSON-import derivation, cross-config consistency throws). (Config-Is-
  Data audited CLEAN — the only residual is that TS types and Rust structs are
  hand-written rather than schema-generated; see A36.)
- Steady-state tick + renderer per-frame paths use counted loops and reused scratch/
  Sets; no per-frame map/filter/Object.entries/new-Set in hot paths.
- Collision/splash broadphase and spatial partitioning are fully WASM-owned
  (SpatialGrid is a thin wrapper); no full-world steady-state scans in projectile
  collision or area damage. PhysicsEngine3D is LIVE authoritative physics but a
  correct thin WASM orchestrator (NOT a Rust-sim duplicate, NOT dead).
- No banned libs (Phaser/Pixi/Matter/cannon/ammo/rapier) are imported anywhere in
  src/; the only @dimforge entry is a local no-op stub (see D1).
- No legacy/old/_unused dirs or .old/.bak files in src/ or rts-sim-wasm/.
- Client unit prediction motion uses a batched Rust/WASM kernel for target
  extrapolation and rendered entity motion; the old TS single-body wrapper path is
  gone.
- Adaptive terrain mesh build is fully Rust/WASM-owned (C16 closed). One kernel
  `terrain_build_adaptive_mesh` runs the whole host bake — hierarchy refine, 2:1
  balance, conforming T-junction topology, crack-repair loop, vertex-height
  smoothing, triangle neighbor metadata, and the cell->triangle index — sampling the
  same analytic height generator the metal-deposit kernels use. TypeScript is pure
  object assembly (config slice in, TerrainTileMap out); the old TS topology/repair
  code and the granular neighbor/discrepancy/smoothing WASM bridge fallbacks were
  deleted (Delete The Old Path). The 22-value generation-config + 5-stride flat-zone
  packers now live once in terrainGenerationConfig.ts, shared by the mesh baker and
  metalDepositConfig.ts. Cell-index WASM exports stay (client wire-unpack reuses them).
