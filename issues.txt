AIMING / TARGETING SYSTEM ISSUES

AIM-08 - Aiming/targeting hot paths remain TypeScript object loops instead of WASM/SoA batches.

Evidence:
- design_philosophy.html:520-570 and 614-619 call out targeting batches, per-unit loops, projectile stepping, and spatial queries as migration targets for Rust/WASM over typed memory.
- src/game/sim/combat/targetingSystem.ts:405-545 and 609-1122 still run the targeting FSM, candidate ranking, LOS gating, and ballistic gating over Entity/Turret objects in TypeScript.

Why it matters:
The current code has good local pooling and some WASM helpers, but the main targeting loop still scales as JS object orchestration over armed entities and candidate arrays. That is acceptable for prototype correctness, but it is not aligned with the stated 5,000-unit architecture.

Fix direction:
Keep the TypeScript behavior as the reference while designing a data-oriented targeting batch: flat turret/entity inputs, candidate slots, visibility/LOS results, target output IDs/states, and parity checks during migration. Delete the old path when parity is proven.

MIGRATION PLAN (each phase = its own PR; each lands one kernel and deletes
the TS path it replaces, never both alongside — see "Delete The Old Path").

AIM-08.2 — Port FF / arc-FF gates to Rust kernels [LANDED, partial]
  The force-field clearance gates now dispatch to Rust:
    force_field_clearance_segment(sx,sy,sz, tx,ty,tz, excludeOwner,
                                  maxCrossings) -> 0|1
    force_field_clearance_arc(launchXYZ, launchVxVyVz, flightTime,
                              excludeOwner, maxCrossings)     -> 0|1
  Both read the FF pool slab stamped pre-FSM by stampForceFieldPool
  (split out of stampTargetingInputSlabs so the slab is current-tick
  when the FSM enters). The JS hasForceFieldClearance /
  hasArcForceFieldClearance bodies in lineOfSight.ts are now thin WASM
  dispatchers; the segmentCrossesForceFieldBoundary helper,
  FORCE_FIELD_GRAZE_EPS, and ARC_FF_CLEARANCE_SAMPLES JS constants are
  gone. The per-unit _unitNearForceFields pre-filter is also gone —
  the Rust loop is fast enough that pre-filtering would just trade one
  cache-friendly Rust pass for one cache-friendly Rust pass plus a JS
  fan-out.
  The fast-path lives at the FSM level: forceFieldsActive (boolean,
  computed once per tick from world.forceFieldsBlockTargeting &&
  getActiveForceFields().length > 0) propagates through the gate
  callers and short-circuits before any aim-point resolve or kernel
  dispatch.

  Deferred to AIM-08.LOS below: hasCombatLineOfSight still composes a
  WASM terrain LOS dispatch with a JS-side entity-LOS spatial-grid
  walk. Porting the entity-LOS part needs a slab of all units +
  buildings (positions, radii, IDs) so the Rust kernel can iterate
  blockers in linear memory; that's a chunkier change than the FF
  port and is more naturally bundled with the FSM port in .5 (which
  needs candidate / entity slabs anyway). The phase as originally
  written conflated three gates that have very different data
  dependencies — splitting it lets us land the trivially-portable
  ones now and pay for the entity slab when other phases also need
  it.

AIM-08.LOS — Port hasCombatLineOfSight to Rust (entity-blocker slab)
  Add an entity-blocker slab (or extend an existing one) that carries
  the data hasEntityLineOfSight currently reads from JS: every live
  unit's center + push radius and every live building's center +
  half-extents, plus stable IDs for the source/target exclusion.
  Replace hasCombatLineOfSight's body with a single Rust dispatch
  that does terrain LOS (existing terrain_has_line_of_sight) plus an
  entity LOS scan over the slab (line-vs-sphere for units, line-vs-
  AABB for buildings, source/target excluded). Delete
  hasEntityLineOfSight and the terrain-LOS thin wrapper from
  lineOfSight.ts; the JS file shrinks to nothing more than the
  weapon-shape predicate (weaponNeedsLineOfSight) and the constants
  the FSM still needs (LOS_DROP_GRACE_TICKS).

  Done when: lineOfSight.ts has zero per-call gate math in JS; the
  WASM kernel is the only LOS implementation; parity shim is clean.

AIM-08.3 — Port candidate scoring + ranking
  Move scoreAndFilterCandidate, fireTargetPreferenceRankSq,
  acquisitionTargetPreferenceRankSq, isBetterTargetCandidate,
  isBetterMirrorTargetCandidate into a Rust kernel that takes the
  per-turret candidate slice and writes (bestCandidateIdx, rank,
  distSq, mirrorScore) per turret. The top-K + fallback budget bubble
  sort moves with it. Delete the JS ranking from targetingSystem.ts.

  Done when: the chooseBestTargetCandidate function in JS is a single
  dispatch to a Rust kernel, returning the chosen idx; old ranking
  helpers are gone from the tree; parity shim is clean.

AIM-08.4 — Port ballistic solve to SoA
  solveTurretAim / solveTurretAimAtGroundPoint /
  hasWeaponBallisticSolution become a Rust kernel that takes turret
  slab indices + target positions and writes (hasSolution, flightTime,
  launchVx/Vy/Vz, yaw, pitch). aimSolver.ts shrinks to a thin JS
  bridge. The arc-FF check from .2 then reads the same kernel's
  flightTime / launchVelocity instead of a JS scratch.

  Done when: aimSolver.ts no longer owns ballistic math; the
  ProjectileTurretAim scratch is replaced by slab outputs; parity
  shim is clean for ballisticArcLow + ballisticArcHigh + VLS shapes.

AIM-08.5 — Port the targeting FSM
  The per-armed-entity loop (Pass 0 mount kinematics, Pass 1 existing-
  lock re-validation, Pass 2 candidate scan, fire-gate dispatch, state
  transitions, losBlockedTicks decay) becomes the top-level Rust
  kernel that consumes all of .1-.4 and writes new turret targets +
  states back into the slab. JS dispatch becomes a single
  rustTargetingTick(...) call. Delete targetingSystem.ts FSM logic.

  Done when: targetingSystem.ts only contains the slab-rebuild
  stamping pass (or has moved to its own file); parity shim runs
  end-to-end clean across the supported scenarios.

AIM-08.6 — Port the writeback + tear down the parity harness
  After .5, snapshot serialization and rendering still read JS
  Entity.combat.turrets[]. Drop a thin writeback pass that copies
  slab → JS Turret each tick (for snapshot encoder consumption), and
  remove the parity harness from .0. The slab is now the source of
  truth.

  Done when: the TS targeting pass is gone from the working tree, the
  combat slabs are the per-tick authority, and the parity harness has
  been deleted in the same PR as its last consumer.
