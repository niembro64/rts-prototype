DESIGN PHILOSOPHY GAP PLAN

Source of goal: design_philosophy.html
Current baseline checked against: the current TypeScript/Vue/Three.js/Rust-WASM codebase.

These items are ordered so each step creates a better foundation for the next one.
Delete each whole item when the current implementation fully matches the goal.

DP-01 - Make the normal network cadence match the design target.

Goal:
The design philosophy says normal authoritative snapshots should be 5 to 10 per
second, with server tick rate and snapshot rate independent. Higher snapshot
rates should be diagnostic or low-unit-count luxuries.

Current:
src/serverBarConfig.ts defaults HOST SERVER snapshots to 64 SPS and offers
1/4/8/16/32/64/128. src/config.ts uses high-fidelity delta thresholds
(0.1 wu position and velocity), which fits a higher-cadence stream more than
the stated normal 5-10 SPS target.

Steps:
1. Add explicit 5 and 10 SPS options, then make the default a normal-play value
   in the 5-10 SPS range.
2. Keep high rates available but label and treat them as diagnostics.
3. Tune delta thresholds and projectile update stride for that lower cadence.
4. Verify local host and remote WebRTC play at 5, 8, and 10 SPS with 60-ish TPS.
5. Add a regression scenario that reports correction distance, snapshot bytes,
   encode time, decode/apply time, render FPS, and command response feel.

DP-02 - Turn snapshots into an owned binary wire path instead of JS DTOs plus
library MessagePack on the hot path.

Goal:
Snapshots should be compact binary payloads. The hot path should not rely on
deep JSON-like object trees, generic encoders, or avoidable per-snapshot object
walks.

Current:
src/game/network/stateSerializer.ts builds pooled JavaScript snapshot DTOs.
src/game/network/NetworkSnapshotTransport.ts then encodes them with
@msgpack/msgpack. The Rust crate already has snapshot_encode_* and envelope
encoder functions, and src/game/sim-wasm/snapshotEncoderTest.ts checks
byte-equality fixtures, but production still sends the JS-encoded DTO.

Steps:
1. Instrument current encode time and byte size per listener at 5-10 SPS,
   1k units, and 5k units.
2. Add a feature flag that encodes the existing snapshot DTO through the Rust
   encoder and compares bytes with @msgpack/msgpack in development.
3. Switch production snapshot sending to the Rust encoder once parity is stable.
4. Remove DTO construction from the hottest entity/projectile/economy paths by
   writing directly from SoA/WASM-side pools into the binary envelope.
5. Keep the old object decoder only as a temporary compatibility/debug path, then
   delete it once no sender emits object snapshots.

DP-03 - Make client prediction prove itself at low snapshot cadence.

Goal:
The client sim should stay smooth at render cadence whether snapshots arrive at
60 Hz, 10 Hz, 5 Hz, or briefly not at all. Commands should feel immediate through
local intent feedback, not by giving the client authority.

Current:
ClientViewState, ClientPredictionStepper, ClientUnitPrediction, projectile spawn
queues, beam path targets, drift modes, and prediction LOD already implement a
real prediction/correction path. But the high 64 SPS default can hide weak spots,
and local command feedback is still mostly visual-only previews: selection,
build ghost, line-drag targets, and waypoint drawing. Most gameplay-visible
state waits for the authoritative snapshot.

Steps:
1. Run and record correction behavior with the DP-01 5-10 SPS default.
2. Add prediction diagnostics for per-frame prediction cost, correction distance,
   correction velocity, and target age.
3. Add local intent previews for move, attack, guard, build, repair, reclaim,
   scan, production, and factory waypoints without mutating authoritative state.
4. Confirm projectiles, beams, force fields, jump arcs, hover lift, and turret
   rotation can coast through at least 500-1000 ms of missing snapshots.
5. Add a deliberately delayed/dropped snapshot test mode and keep it passing.

DP-04 - Harden the command authority boundary.

Goal:
Client commands go directly to the authoritative host. The host validates player
authority, filters illegal unit IDs, clamps unsafe parameters, and enqueues only
safe gameplay commands.

Current:
GameServer.receiveCommand already keeps server-control commands host-only for
network players and filters many gameplay commands by owned entity/factory.
However, fromPlayerId === undefined is an admin override, server-control commands
apply immediately outside the normal command queue, and several command payloads
still need a single sanitization layer for finite values, map bounds, radii,
queue indexes, unit/building IDs, and enum strings.

Steps:
1. Replace the implicit undefined-player override with explicit authority modes:
   host-admin, local-offline, player, spectator.
2. Create a single sanitize/authorize command module used by local, host, and
   remote command paths.
3. Clamp target coordinates to the map, reject non-finite values, cap radii and
   waypoint counts, validate queue indexes, and validate unit/building/shot enum
   values before enqueue.
4. Decide which server-control commands are truly immediate and which should be
   queued or snapshot-versioned.
5. Add hostile-client tests for wrong-owner unit IDs, mixed-owner selections,
   bad coordinates, invalid queue indexes, invalid unit IDs, and spectator input.

DP-05 - Make AOI and fog filtering the default remote snapshot path.

Goal:
Area-of-interest, fog filtering, minimap filtering, removals, shroud state, and
recipient-specific deltas are server responsibilities. Full-world snapshots are
for global observers, recovery, or debugging, not normal remote play.

Current:
ServerSnapshotPublisher, stateSerializerVisibility, shroud bitmaps, AOI listener
bounds, team caches, recipient delta tracking, and static-terrain suppression
already point in the right direction. The remaining gap is proving that every
normal remote listener always has the correct recipient filter and AOI behavior,
and that fallback keyframes/debug paths do not leak unnecessary full-world state.

Steps:
1. Audit every addSnapshotListener call and mark whether it is player-filtered,
   spectator/global, lobby preview, or debug.
2. Ensure remote real-battle clients start with a player-filtered listener before
   any gameplay snapshot can be sent.
3. Make camera AOI updates robust when the client is loading, tab-throttled, or
   has not sent a camera footprint yet.
4. Add snapshot-size and entity-count telemetry split by full, delta, AOI, and
   visibility-filtered output.
5. Add tests for hidden enemies, out-of-AOI enemies, destroyed out-of-vision
   buildings, shroud keyframes, scan pulses, and spectator/global views.

DP-06 - Move remaining high-count simulation hot paths toward SoA and WASM.

Goal:
Dense simulation work should use stable IDs, packed numeric fields, typed arrays,
struct-of-arrays storage, and WASM kernels where batching beats JS object loops.
TypeScript should orchestrate, not own high-count numeric loops.

Current:
Several important systems already moved in this direction: Body3D pool,
projectile pool, spatial grid, pathfinder, terrain mesh sampling, hover
orientation batch, snapshot baseline diff, and parts of snapshot encoding.
But authoritative world state is still a Map<EntityId, Entity>, many systems
iterate cached Entity[] lists, and hot paths such as targeting, construction,
energy, capture, cloak/detection, combat activity, factory production, and some
projectile/collision branches still operate through object graphs.

Steps:
1. Produce a per-system tick-cost profile at 1k, 3k, and 5k active units.
2. Rank systems by cost and allocation rate, not by code size.
3. Convert the top offenders to typed-array/SoA storage behind the existing
   public APIs first.
4. Batch numeric kernels into Rust/WASM only after the data is already contiguous.
5. Keep TypeScript as orchestration and remove old object-loop paths after parity
   tests pass.

DP-07 - Complete the mass/force contract for battlefield interactions.

Goal:
Movement, recoil, collision, hover, jump, beams, blast pressure, force fields,
and terrain contact should reduce to mass, force, acceleration, velocity,
position, collision, and correction wherever they physically move things.

Current:
The project already has real 3D state, Body3D, terrain contact, dynamic z,
unit-vs-unit/building collision, force accumulation, propulsion, recoil,
explosion knockback, beam force, hover lift, jump actuation, and shared motion
math. Gaps remain where force handling is still partly 2D, where some gameplay
effects mutate state discretely without a physical effect, and where heavy/light
mass response is not continuously tested across every force producer.

Steps:
1. Audit every call that moves, pushes, clamps, teleports, reflects, explodes,
   hovers, jumps, or corrects a unit/projectile.
2. Convert force producers to explicit 3D force/impulse fields where vertical
   behavior matters.
3. Add tests that compare light/medium/heavy units under the same recoil,
   explosion, beam, collision, force-field, hover, and jump inputs.
4. Keep discrete systems discrete, but ensure their battlefield output enters as
   bodies, forces, energy, mass, constraints, or collisions.
5. Document any intentional non-physical shortcuts as temporary with a scale or
   gameplay reason.

DP-08 - Tighten atomic gameplay blueprint boundaries.

Goal:
locomotion, unit, turret, shot, and building definitions should each own one
clear job and compose through explicit references. A unit should be able to swap
locomotion profiles, turret sets, shot profiles, or factories without hidden
coupling.

Current:
The blueprint files are already split into units, locomotion, turrets, shots,
buildings, and display rosters. The remaining coupling is still visible:
UnitBlueprint embeds locomotion objects and many visual/HUD/audio details;
TurretBlueprint mixes projectile reference, launch force, aiming, range, barrel
visuals, audio, mirror panels, force fields, and construction emitters; Shot
Blueprint mixes physics/damage with smoke/audio/render shape details.

Steps:
1. Define narrow runtime contracts for locomotion, unit, turret, shot, building,
   visual, audio, and UI metadata.
2. Change unit definitions to reference locomotion IDs and turret-set IDs rather
   than embedding mutable object-shaped configs.
3. Move visual-only barrel/body/smoke/HUD/audio data out of authoritative sim
   blueprints or into clearly named visual subcontracts.
4. Make turret-to-shot and building-to-turret links explicit IDs with validation
   tests.
5. Add a recombination test that creates a unit with swapped locomotion, swapped
   turret set, swapped shot profile, and a different factory without touching
   unrelated systems.

DP-09 - Move high-count rendering away from framework-owned scene graph paths.

Goal:
Three.js can remain a productive prototype shell, but bottlenecked renderer paths
should move to owned buffers, owned shaders, direct instancing, and eventually
raw WebGL2/WebGPU where needed.

Current:
The renderer is built around Three.js with substantial custom instancing, object
LOD, render LOD grids, shared geometry/materials, and GPU timing. That matches
the prototype phase but not the long-term raw-rendering goal for thousands of
units and effects.

Steps:
1. Measure draw calls, buffer uploads, material switches, CPU render prep, GPU
   frame time, and per-LOD object counts at 1k, 3k, and 5k units.
2. Identify the first renderer path where Three.js overhead blocks scale.
3. Move that path to owned instanced buffers and custom shaders while keeping the
   same visual contract.
4. Repeat for units, projectiles, beams, terrain overlays, labels, debris, and
   high-count effects in cost order.
5. Keep Three.js only where it is not measurable bottleneck or where replacement
   would not improve scale.

DP-10 - Add scale-target benchmarks and budgets.

Goal:
Every major implementation decision should be evaluated against thousands of
active units in a browser, with a path toward larger scenes. A feature that works
at 50 or 200 units is not done unless it has a scale story.

Current:
The app exposes useful live telemetry: server TPS/CPU, render TPS, frame/render/
logic/GPU timing, long task timing, snapshot rates, unit count, LOD states, and
debug overlays. There is no obvious automated benchmark gate that runs standard
1k/3k/5k scenarios and records pass/fail budgets.

Steps:
1. Add deterministic benchmark scenarios for idle armies, moving armies, heavy
   targeting, projectile-heavy combat, fog/AOI, construction, and mixed battles.
2. Record server tick cost, render cost, prediction cost, snapshot encode/decode
   cost, bytes/sec, allocations, and long tasks.
3. Define budget thresholds for 1k, 3k, and 5k active units.
4. Make benchmark output easy to compare before/after each performance change.
5. Use those budgets to decide when a JS object-loop or Three.js path must move
   to SoA/WASM or owned GPU buffers.
