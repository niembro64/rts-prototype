------------------------------------------------------------------------------
 FOG OF WAR — design vs. current implementation
------------------------------------------------------------------------------

How fog of war should work in an RTS (the canonical model, à la Total Annihilation /
Supreme Commander, Starcraft, Age of Empires):

Every map cell, from a given player's POV, exists in one of THREE states:

  1. UNEXPLORED. Never seen by any of that player's vision sources.
     Rendered as solid black/heavy fog. Terrain unknown, no entities shown.

  2. EXPLORED-BUT-DARK ("shroud" or "soft fog"). Seen at some point, but not
     currently in vision. Rendered as a dimmed/desaturated overlay.
       - Static features (terrain, neutral resources, ENEMY BUILDINGS): shown as
         their LAST-SEEN snapshot — the building stays as a "ghost" at the
         position the player last observed it. If it has since been destroyed
         or moved, the player only finds out by re-visiting.
       - Mobile units (enemy units): NOT shown. They could have moved, so
         showing a ghost would be a lie. Once a player loses LOS on an enemy
         unit, it disappears.

  3. CURRENTLY VISIBLE. At least one of the player's units/buildings has line
     of sight to this tile right now. Real-time information; all entities and
     their state are shown accurately.

Unit sight (line of sight, LOS):
  - Each entity has a sight radius — usually a circle around the entity.
  - Scout-type units typically have larger radii than line units.
  - Buildings have a smaller "passive" radius.
  - Vision is BLOCKED by terrain features in many RTS games (high ground,
     forests, structures). A unit on the far side of a mountain shouldn't
     be visible just because it's inside the circle.
  - Day/night cycles or weather can reduce sight radius (some games).
  - Vision is shared between allies (team play).

Radar (and the related "sonar" / "scanner"):
  - Larger radius than unit sight, but provides STRICTLY LESS information
     than direct vision: it shows enemies as blips on the minimap (and in
     some games, as generic dots in the world) without identifying the unit
     type, owner-specific state, HP, orders, etc.
  - Radar does NOT lift the fog-of-war shroud — visited tiles stay
     "explored-but-dark," radar just adds a contemporaneous blip overlay.
  - Stealth/cloaked units do not appear on radar. Radar jamming hides
     friendly units from enemy radar. (Standard counters.)
  - Air units may need air radar (separate channel) in some games.

Detection (counter to stealth):
  - Cloaked/stealth units are invisible to normal vision.
  - Specific detector units/buildings reveal cloak within their detect
     radius. Without a detector, cloaked enemies are blind spots — even
     while standing in your camp.

Sound / audio reveal:
  - Many RTS games play unit sounds even from out-of-vision positions when
     close enough, hinting at enemy presence without revealing position.

Initial reveal:
  - At game start, the area around each player's starting base is already
     in CURRENTLY VISIBLE. The rest of the map is UNEXPLORED.

What our project currently has:

  - One boolean: world.fogOfWarEnabled. When on, the per-recipient snapshot
     is filtered through SnapshotVisibility (point-in-circle, with a spatial
     hash grid).
  - Vision sources: a player's own UNITS (1200), COMMANDERS (1600), buildings
     (1000), RADAR building (4200), with extra range pads for turret reach
     and builder reach. All sources contribute to the same merged vision
     mask — radar is just "a building with a bigger circle."
  - Entity-private fields (HP, action state, orders) stripped for enemies.
  - When an enemy entity leaves the player's vision, the server emits an
     explicit removal so the client DELETES the entity. Re-entering vision
     re-sends it as a full snapshot.
  - Audio events and projectile spawns are gated by isPointVisible(spawn pos).
  - The dark canvas-alpha overlay that used to dim explored areas was
     removed when "keep enemies as ghosts" was scrapped.

Mismatches with the canonical model (each a discrete issue below):

[FOW-06] No team / shared vision. SnapshotVisibility.forRecipient takes a
  single PlayerId. If allies are added, each ally needs its own filter
  pass OR the visibility mask needs to union all allied players. The
  current code is structurally per-player only.

[FOW-09] Audio reveal is binary on vision (PREREQ: 3D audio not wired
  yet). serializeAudioEvents already filters by full vision, but the
  3D scene's handleSimEvent3D currently only fires visuals — none of
  the AudioManager's playWeaponFire / playWeaponHit / playDeath calls
  are reached from gameplay, so even in-vision events produce no
  audio. Attenuated out-of-vision "I hear distant gunfire" hints
  can't ship until audio playback is hooked into the 3D path; once
  it is, the wider earshot-pad serializer filter + an audioOnly
  flag (so the client suppresses visuals but plays the sound) is
  the right shape.

[FOW-11] Vision history is client-only. FogOfWarShroudRenderer3D
  already tracks a per-tile revealed bitmap for the local player and
  renders three-state shroud (unexplored / explored-dark / currently
  visible), so the canonical RTS model is in place visually. The
  authoritative side, however, has no record — mid-game joins, replays,
  and reconnects all start with a blank shroud, and a tampered client
  could pretend to have explored more than it actually did. Future
  enhancement: mirror the per-player revealed bitmap on the server and
  ship the keyframe diff so reconnects restore their map state.

[FOW-14] No "scan" / temporary reveal API. There is no command type that
  reveals an arbitrary area for a fixed duration (Starcraft scanner
  sweep, Supreme Commander recon drone). All vision today is tied to
  a long-lived entity.
