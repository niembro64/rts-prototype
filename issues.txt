2.0 [CRITICAL] LegInstancedRenderer leaks every game restart.

src/game/scenes/RtsScene3D.ts:396 constructs `new LegInstancedRenderer(...)`,
which allocates 8 instanced leg/joint/pad pools (InstancedBufferGeometry +
SphereGeometry + materials + 4 InstancedBufferAttribute typed arrays each),

src/game/render3d/LegInstancedRenderer.ts:803 (disposes all 8 sub-pools) but has
ZERO call sites. Render3DEntities.destroy() (Render3DEntities.ts:1052) disposes
every other sub-renderer (building, projectile, construction, unitDetail, etc.)
but never calls this.legRenderer.destroy(). Restart reuses the same ThreeApp /
WebGLRenderer / app.world, so the old leg meshes stay attached and a fresh set is
created on every restart -> leaks all leg GPU buffers per session.
FIX: call this.legRenderer.destroy() inside Render3DEntities.destroy() (alongside
the other sub-renderer .destroy() calls at lines 1067-1074).

5.1 [MAJOR] Tick loop drifts; no fixed-timestep accumulator/catch-up; no budget
saturation metric.

src/game/server/ServerTickLoop.ts:13 is a bare setInterval(cb, 1000/hz). GameServer.tick
(:455) clamps per-tick dt to MAX_TICK_DT_MS (good vs spiral-of-death) but there is no
accumulator and no catch-up: a 200ms hitch advances sim by one clamped step, so the
authoritative clock silently slips behind real time and never recovers. TPS/CPU are
measured, but tick-work-ms vs interval-ms saturation is not, so a saturated host can't
tell it's falling behind. Violates "Measure TPS, CPU budget, ... backpressure
continuously."
FIX: drive the loop with a fixed-dt accumulator + bounded catch-up (max N steps/frame),
and surface a tickMs/intervalMs saturation metric into serverMeta.

5.2 [MINOR] First CPU sample is never used as an EMA seed.

src/game/server/GameServer.ts:149 — tickMsInitialized defaults to true (vs tpsInitialized
which is false), and the constructor (:209-210) resets tickMsAvg/Hi to 0 but leaves the
flag true. So the first recordTickWork (:368) runs 0.99*0 + 0.01*workMs instead of seeding
tickMsAvg = workMs; reported CPU load starts artificially near zero and takes ~100 ticks
to converge.
FIX: initialize tickMsInitialized = false.

5.3 [MINOR] WASM shield/targeting/projectile pools not explicitly cleared on stop().

GameServer.stop() (:383-419) clears the spatial grid (+ entity-meta + turret pools) but
not sim.combatTargeting, sim.shieldSurfacePool, or sim.projectilePool. NOT an actual leak
today — all three are count-based and fully re-stamped on the first tick of a new session
— but it relies on "first tick re-stamps everything," which is fragile if any stamp ever
becomes conditional.
FIX: clear them in stop() for defense-in-depth and consistency.

5.4 [MINOR] Sim-slot guard ("a second sim is a hard error") only fires at tick start.

acquireSimSlot is called in GameServer.start() (:310), but create() (:169) and the
PhysicsEngine3D ctor (:419) already allocate the WASM engine handle and spawn bodies into

caller would corrupt the shared pool instead of getting the intended immediate throw.
FIX: acquire the sim slot in create()/the PhysicsEngine3D ctor so allocation into the
shared pool is what throws; release on stop()/dispose().

6.0 [CRITICAL] Backpressure-dropped delta still advanced the per-recipient baseline
-> permanent desync.

The per-recipient delta baseline is committed inside serializeGameState
(copySentPrevState + captureToRustBaseline, stateSerializer.ts:430/472) which runs in
ServerSnapshotPublisher.emit (:256) BEFORE the bytes reach the transport. The transport
then drops the snapshot for backpressure (NetworkSnapshotTransport.ts:64/200/220/244 via
shouldDropForBackpressure checking dc.bufferedAmount, :308) or for pending FULLSNAP
compression (:325) — but nothing rolls back the baseline or forces a keyframe. The channel
is reliable, so the dropped delta is gone, yet the next delta is diffed against state the
client never received. That recipient's view stays corrupt until an unrelated keyframe
happens to repaint the affected fields (fields that stop changing never repaint). Violates
"DIFFSNAP baseline must advance per-field from the last value actually SENT to that
recipient."
FIX: on any backpressure/pending-compression drop, mark that recipient for a forced
keyframe (snapshotPublisher.forceNextKeyframe for that listener), or defer the baseline
commit until the send is confirmed.

6.1 [HIGH] cloneNetworkSnapshot() allocates a fresh cloner per call on the receive
hot path.

src/game/network/snapshotClone.ts:548 — the free function does
`new ReusableNetworkSnapshotCloner().clone(state)`. That class is designed to be held and
reused (its sub-DTO arrays/objects pool across clones), and good callers hold it
(SnapshotBuffer.ts:36, LocalGameConnection.ts:28). But NetworkSnapshotTransport.storePendingState
(:160) and RemoteGameConnection.ts:39 call the free function, so every buffered client
snapshot (5-10Hz + impairment path) reallocates the cloner and all its internal arrays,
defeating the pool. Violates "Pool snapshot DTOs."
FIX: give NetworkSnapshotTransport and RemoteGameConnection a long-lived
ReusableNetworkSnapshotCloner field and call .clone() on it.

6.2 [MEDIUM] Per-unit build DTO allocated in the serialize loop (unit path only).

src/game/network/stateSerializerEntities.ts:719-728 — when ENTITY_CHANGED_BUILDING is set
for a unit under construction, it allocates u.build = { complete, interrupted, paid:{...} }
(two objects) per unit per snapshot. The building path right below (:800-813) correctly
reuses the pooled poolEntry.buildingBuild.
FIX: add a pooled unitBuild to PooledEntry (mirroring buildingBuild) and write into it.

6.3 [MEDIUM] Client receive-coalescing silently drops a delta when one is already
pending.

src/game/network/NetworkSnapshotTransport.ts:156-162 — storePendingState only stores when
there is no pending state (or when upgrading delta->full); a second delta arriving before
the first is consumed is dropped, not coalesced. Deltas are non-idempotent (each carries
only fields changed since last sent), so dropping one loses those corrections with no
resync trigger.
FIX: apply/merge the pending delta before overwriting, or at minimum request a keyframe
when a delta is dropped here.

6.4 [MINOR] Per-emit allocations break the otherwise-strict pooling discipline.

ServerSnapshotPublisher.ts:191-193 allocates three Maps (teamAudioCache/teamSprayCache/
teamMinimapCache) + a visibility cache (:183) every emit (~10Hz); snapshotRustWireEncoder.ts:2177
allocates `const rawTopLevelKeys: string[] = []` per encode. Once-per-snapshot, not
per-entity, so small.
FIX: hoist to instance fields and .clear()/.length=0 per emit.
