SECTION A — DESIGN-PHILOSOPHY INCONSISTENCIES
================================================================================

A10. [LOW] Beam reflection point still branches its WIRE label + visual on
    force-field shape (carved out of A4, which unified the surfaces themselves).
    LOCATION: BeamReflectorKind = 'forceFieldPanel' | 'forceFieldSphere'
      (src/types/shotTypes.ts:288); set in DamageSystem.ts:576/625; rides a beam-
      point flag bit (snapshotProjectileWirePack.ts ~575/664, stateSerializerProjectiles.ts
      ~397), mirrored in the Rust beam encoder, and read by BeamRenderer3D.ts.
    RULE: "Materials Are Independent Of Shape" — a beam reflecting off the force-field
      material should look identical regardless of which shape it bounced off.
    PROBLEM: the two-value enum makes the beam reflection visual + wire bit shape-
      dependent. The A6 enum-codegen pass landed src/wireEnums.json (single source,
      TS imports it + Rust generates via build.rs), so a single reflectorKind code
      now has a home — but collapsing 2 values to 1 and dropping the wire bit is a
      wire-VALUE change (like A5), verifiable only at runtime, not just a dedup.
    FIX: add the force-field reflector code to src/wireEnums.json, collapse
      BeamReflectorKind to a single value, drop the panel-vs-sphere wire bit, and
      verify with the runtime snapshot compare mode before trusting it in multiplayer.

A5. [MED] Snapshot entity encoding has two implementations, and the TS one runs on
    the live Rust path (double entity walk).
    LOCATION: snapshotEntityWirePack.ts (TS per-entity encoder) vs
      snapshotRustWireEncoder.ts (WASM per-entity encoder); wired in
      snapshotWireCodec.ts:11-21. packEntitiesForWire is called unconditionally at
      snapshotWireCodec.ts:232; the Rust encoder then re-emits the TS-built blob raw
      (snapshotRustWireEncoder.ts:1944-1946), bypassing its own
      encodeEntityUnit/Building kernels for entities.
    RULE: "Delete The Old Path" / "TypeScript Coordinates, WASM Computes" (no large
      serialization loops in TS).
    PROBLEM: a full per-entity TS object-building loop runs every snapshot even when
      the Rust encoder is active, and a second full DTO→wire implementation is
      maintained in lockstep.
    FIX: make either the TS pack path or the Rust per-entity encoder the single
      owner of entity bytes; stop running packEntitiesForWire on the Rust path.

A9. [LOW] Sim-relevant tuning still lives in TS modules.
    LOCATION: src/config.ts (~115 exported const tuning objects; some are sim-relevant,
      many are render-only), src/game/sim/blueprints/displayRosters.ts (roster ordering).
    RULE: "Config Is Data, Not Code" — Rust cannot import .ts; tuning the sim needs
      should be JSON.
    PROBLEM: TS-module tuning tables are a wall for the TS→Rust migration. (Mostly
      resolved already — unit/turret/shot/locomotion/building blueprints are now JSON
      consumed by Rust via build.rs embedding; the .ts files in blueprints/ are
      loaders, not data.)
    FIX: migrate sim-relevant tables out of config.ts into JSON validated by a shared
      schema; leave render-only constants for later.

SECTION C — PERFORMANCE / SCALE (target: 5,000 units)
================================================================================

C2. [MED] Client snapshot DECODE allocates fresh per-entity DTOs + nested literals
    every snapshot.
    LOCATION: snapshotEntityWirePack.ts:252,270 (new Array), :901-932 + :1010-1041
      (createEmptyUnitSub + {x,y,z}/{x,y,z,w} literals per unit); entered from
      snapshotWireCodec.ts:295.
    RULE: "Pool snapshot DTOs"; "do not allocate in per-unit loops".
    PROBLEM: the ENCODE side is exemplary (pooled PooledEntry + typed-array wire rows),
      but the remote-client DECODE side allocates a fresh entity array plus per-unit
      sub-object + velocity/orientation/angularVelocity3 literals — ~50k+ short-lived
      objects/sec at 5000 units @10Hz, discarded after ClientSnapshotApplier consumes
      them. (Host/local play skips this.)
    FIX: pool the decode DTOs like the encode side.

C5. [LOW] Per-unit JS↔WASM boundary crossing in the client prediction loop.
    LOCATION: ClientPredictionStepper.ts:262-296 → ClientUnitPrediction.ts:186 →
      unitMotionIntegration.ts:89 (sim.stepUnitMotion, one call per unit per frame
      with a 6-element scratch marshal).
    RULE: "TypeScript Coordinates, WASM Computes" — avoid per-unit boundary loops.
    PROBLEM: the server batches integration via one poolStepIntegrate call, but the
      client crosses the boundary once per predicted unit per frame (N crossings).
    FIX: add a batched client prediction step kernel like the server's.

C6. [LOW] Per-body terrain sampling crosses the WASM boundary once per body per tick.
    LOCATION: PhysicsEngine3D.ts:765-805 (getGroundZ/getGroundNormal per body) →
      terrainSurface.ts:91-94 (terrainGetSurfaceHeight/Normal per call).
    RULE: "TypeScript Coordinates, WASM Computes" — batch boundary work.
    PROBLEM: the integrate kernel is batched but ground sampling is N separate boundary
      crossings per tick. The integrate kernel already takes pre-sampled arrays, so the
      batch surface exists.
    FIX: add a "sample ground for these slots" batch kernel.

C8. [LOW] Per-death debris template array + closure allocation.
    LOCATION: src/game/render3d/Debris3D.ts:446 → buildTemplates :526-536 (fresh array,
      one object literal per fragment, resolveColor closure per call).
    PROBLEM: event-driven (per death), normally fine, but a mass-death moment spawns many
      at once.
    FIX: cache pose-independent templates per unitType; apply only live yaw/pitch.

C9. [MED] Need an output-stability harness before deeper terrain-generation rewrites.
    LOCATION: terrain/deposit generation pipeline:
      src/metalDepositConfig.ts, src/game/sim/terrain/terrainTileMap.ts,
      src/game/sim/terrain/terrainCellTriangleIndex.ts.
    RULE: generation optimizations must not change final terrain visuals, sim mesh,
      deposit positions, deposit heights, resource cells, buildability, or metal-pad
      interactions unless explicitly intended.
    PROBLEM: the next optimization tier touches triangle-collapse decisions, edge
      metadata, repair passes, and cell indexing. These are deterministic but visually
      sensitive; relying on screenshots or "looks right" will miss small logical drift.
    FIX: add a harness that generates deposits + terrain for several map sizes,
      player counts, terrainDetail values, terrain map shapes, and deposit-step/plateau
      configs, then hashes/compares meshVertexCoords, meshVertexHeights,
      meshTriangleIndices, meshTriangleLevels, meshTriangleNeighborIndices/Levels,
      meshCellTriangleOffsets/Indices, metal deposit x/y/height/cells/bounds, and
      buildability grid flags/levels.

C10. [MED] Terrain triangle collapse check still does avoidable per-sample work.
    LOCATION: src/game/sim/terrain/terrainTileMap.ts:
      canCollapseTerrainTriangle / forEachFinePointInTriangle / terrainBarycentricAt.
    RULE: "TypeScript Coordinates, WASM Computes" / avoid allocation and repeated math
      in generation hot loops.
    PROBLEM: collapse testing recomputes world positions, barycentric weights, and
      source normals across many fine lattice points. Some object-return helpers and
      generic barycentric math are still in the innermost subdivision decision path.
    FIX: specialize up/down equilateral triangle lattice scans so interpolation weights
      come directly from integer offsets; avoid per-sample barycentric objects; precompute
      repeated edge/plane terms per candidate triangle.

C11. [MED] Terrain mesh edge metadata still allocates string keys in hot generation paths.
    LOCATION: src/game/sim/terrain/terrainTileMap.ts:
      collectTriangleEdgeOwners, meshEdgeKey, lineKey creation, edge span indexes,
      resolveMeshTriangleEdgeSplits, buildTriangleNeighborMetadata.
    PROBLEM: edge repair/neighbor generation builds many Map entries keyed by strings
      such as "a:b", line keys, and "triangle:edge". We reduced duplicate edge scans,
      but the remaining scans still allocate heavily at high terrainDetail.
    FIX: replace edge and triangle-edge keys with numeric packed keys where safe, and
      keep line classification numeric. Verify with the C9 output harness before/after.

C12. [MED] Terrain leaf balancing rebuilds unit-edge segment maps each pass.
    LOCATION: src/game/sim/terrain/terrainTileMap.ts:
      balanceTerrainTriangleLeaves, forEachTriangleUnitEdgeSegment, latticeSegmentKey.
    PROBLEM: each balance pass walks every leaf's unit edge segments and keys them with
      strings. This can dominate startup on high-detail maps, especially around deposit
      pads and hard terrain transitions.
    FIX: use numeric lattice segment keys and reusable owner buffers, or cache segment
      ownership between balance passes when only a known subset of leaves split.

C13. [LOW] Terrain leaf generation creates many short-lived child arrays/objects.
    LOCATION: src/game/sim/terrain/terrainTileMap.ts:
      buildTerrainTriangleLeaves, terrainTriangleChildren, splitTerrainTriangleLeaves.
    PROBLEM: recursive subdivision allocates arrays and small triangle objects repeatedly.
      This is startup-only, but high detail settings amplify GC pressure.
    FIX: convert subdivision to stack-based traversal with flat triangle records or
      push children directly into caller-owned output buffers.

C14. [MED] Buildability-grid generation repeats neighboring terrain samples.
    LOCATION: src/game/sim/terrain/terrainBuildability.ts:
      buildTerrainBuildabilityGrid -> evaluateBuildabilityFootprint -> isWaterAt /
      getTerrainMeshNormal / getTerrainPlateauLevelAt.
    PROBLEM: every build-grid cell samples nine footprint points, and adjacent cells
      re-query many of the same height/normal/plateau/water results. This happens right
      after terrain generation and scales with map area.
    FIX: cache mesh height/normal/water/plateau samples per build-grid sample point for
      the duration of buildTerrainBuildabilityGrid, then evaluate footprints from that
      cache.

C15. [LOW] Metal-deposit resource-footprint growth still scans the full frontier every
    selected cell.
    LOCATION: src/metalDepositConfig.ts: growMetalDepositResourceCells.
    PROBLEM: current footprints are small, so this is not urgent. If resourceCells or
      resourceRadiusCells grow, each selected cell rebuilds a weighted frontier list and
      recomputes neighbor counts, distances, and weights for every frontier cell.
    FIX: precompute candidate offsets/distances for each radius; keep selected/frontier
      membership numeric; update neighbor counts incrementally. Preserve Map/frontier
      iteration order or verify with the C9 output harness.

C16. [HIGH] Long-term terrain/deposit baking should move to Rust/WASM.
    LOCATION: src/game/sim/terrain/*.ts and src/metalDepositConfig.ts.
    RULE: "TypeScript coordinates, WASM computes" and "shared deterministic kernels".
    PROBLEM: terrain mesh baking, repair, buildability, and deposit layout are CPU-heavy,
      deterministic, and sim-critical. Keeping them in TS leaves startup cost and
      client/server drift risk in the orchestration layer.
    FIX: after the C9 harness exists, port terrain height sampling, mesh baking,
      buildability baking, and deposit generation to Rust/WASM in stages, with TS kept
      as compatibility/diagnostic glue until the old path can be deleted.

VERIFIED-CLEAN (audited, no action needed — recorded so we don't re-investigate)
================================================================================
- No acceleration on the wire; PREDICT mode is POS/VEL only; no ACC option.
- The four EMA channels (movement pos/vel, rotation pos/vel) are each applied to
  ALL their documented fields (unit transform/velocity, beam point pos/vel, yaw
  scalar + orientation quat + turret yaw/pitch, angularVelocity3 + turret rates,
  force-field range visualizer).
- Travelling projectiles snap on velocityUpdate events (no EMA target populated).
- Delta thresholds: position uses absolute error, velocity uses relative; the
  zero-velocity edge is sent; baseline = last value actually sent per recipient.
- EntityType has Tower as a true peer of Unit/Building; fabricator is a tower, not a
  building; pure-infra buildings carry no turrets/lock-on.
- Lock-on exclusion lists use the exact documented field names and are honored
  subtractively in targeting.
- Airborne mount invariant (mount.y===0 && mount.z===0, no zResolver) is enforced and
  throws at startup.
- Rockets inherit the launch target and never re-acquire; lost target → ballistic.
- Sensors are capability sources with fixed radii (not derived from weapon range).
- Resource movement: every stockpile mutation funnels through ResourceMovementSystem
  credit/debit; renderer drives one shared pylon rig for all resource visuals.
- Selection panels are gated on entity TYPE capability flags, not per-blueprint buttons.
- No banned libs (Phaser/Pixi/Matter/cannon/ammo) are imported anywhere in src/.
- No legacy/old/_unused dirs or .old/.bak files in src/ or rts-sim-wasm/.
