TYPESCRIPT IN THE HOT PATH (TS-WASM-01 family)

The "TypeScript Coordinates, WASM Computes" section of
design_philosophy.html is explicit: per-unit, per-projectile, and
per-frame loops must live in WebAssembly (compiled from Rust), not in
TypeScript. Several per-unit loops still run in TS today. None are
catastrophic at 200 units, but they all compound at the 5,000-unit
design target.

==============================================================================

------------------------------------------------------------------------------
TS-WASM-01 - parent: per-entity hot loops still execute in
             TypeScript when their kernels should live in Rust.
------------------------------------------------------------------------------

Evidence:
- src/game/sim/Simulation.ts (1,657 LOC) is still the top-level tick
  driver. Although most heavy kernels delegate to WASM, several
  per-unit loops are TypeScript:
    - UnitForceSystem.applyForces (src/game/server/UnitForceSystem.ts)
      — per-unit force gathering, called every tick from
      GameServer.tick() (GameServer.ts:392 region).
- src/game/server/PhysicsEngine3D.ts (915 LOC) and
  src/game/server/UnitForceSystem.ts (699 LOC) are still instantiated
  in GameServer.ts:192 region and called every tick. The integration
  itself now runs in Rust kernels, but the per-unit force-selection
  loop and physics-body bookkeeping still live in TS.
- Client prediction has a similar shape: ClientPredictionStepper,
  ClientUnitPrediction, and ClientProjectilePrediction iterate over
  predicted entities per render frame in TS, even though they call
  into shared kernels (advanceUnitMotionPhysicsMutable,
  solveKinematicIntercept, computeHomingThrust).
- Snapshot wire packing runs per-snapshot loops in TS
  (snapshotEntityWirePack.ts, snapshotProjectileWirePack.ts,
  snapshotMinimapWirePack.ts, stateSerializer*.ts). At 5,000 entities
  per 10 SPS snapshot, this is on the edge of the budget the
  philosophy reserves for "large serialization loops".

Why it matters:
The philosophy lists per-unit update loops, physics integration and
collision detection, and large serialization loops explicitly in the
"What TypeScript must never own in hot paths" section. The current
split keeps the kernels in Rust but the per-unit iteration in JS.
At 5,000 active units, an N-per-tick TS loop is exactly the scaling
shape the design rules out.

Fix direction:
Do child items as separate, measurable PRs. Continue with the remaining
JS gather/scatter paths, using the arrival-control pattern where it
fits.

------------------------------------------------------------------------------
TS-WASM-01D - child: move snapshot wire-pack and serializer loops
              into Rust now that the encoded shape is stable.
------------------------------------------------------------------------------

Evidence:
- src/game/network/snapshotEntityWirePack.ts (1,100+ LOC),
  snapshotProjectileWirePack.ts, snapshotMinimapWirePack.ts,
  snapshotAudioWirePack.ts, snapshotStaticWirePack.ts, and the
  stateSerializer*.ts files all run per-snapshot loops over the live
  entity/projectile/minimap/audio sets in TS.
- snapshotRustWireEncoder.ts already wraps a Rust encoder for the
  outer frame but still reads from the TS serializer functions, so
  Rust never owns the inner per-entity loop.
- The 2026-05-22 entity packed-row work shows the wire shape is
  stable enough to land in Rust now (V1-V6 versions are append-only
  and the encoder is bit-deterministic).

Fix direction:
Migrate the per-entity / per-projectile / per-minimap / per-audio
emit loops into the Rust encoder. Keep the JS wrappers only as
thin call-into-WASM shells. This also cuts the JS/WASM boundary
crossings the philosophy warns about: today every snapshot copies
entity slab values from WASM into JS arrays just to be re-packed
into a wire buffer.

------------------------------------------------------------------------------
TS-WASM-01E - child: client prediction step body is still a per-unit
              JS loop; move the inner step into WASM and let the JS
              shell just drive cadence.
------------------------------------------------------------------------------

Evidence:
- src/game/network/ClientPredictionStepper.ts,
  ClientUnitPrediction.ts, ClientProjectilePrediction.ts all iterate
  over predicted entities per render frame and call shared kernels
  for the math.
- The philosophy explicitly assigns "client prediction integrator"
  to the WASM layer in the "What TypeScript must never own in hot
  paths" list. The shared kernels are already there; only the
  iteration is still in JS.

Fix direction:
Add a Rust `client_predict_step(dt)` entry point that walks the
predicted-entity slab in linear memory and applies the same EMA /
integration the JS does today. Delete the JS per-entity loop; the
JS layer becomes "set dt, call WASM, sample for render."

property declarations and `?.` access inside our own simulation,
server, prediction, networking, and entity-state modules. Optional
property declarations still appear in core types, and optional
chaining still needs a separate cleanup pass across the in-scope tree.

CONFIG IS DATA, NOT CODE (BLUEPRINT-01 family)

The "Config Is Data, Not Code" section of design_philosophy.html
requires blueprints, tuning tables, and authored content in JSON
validated against a shared schema; the TS-to-Rust migration depends
on Rust being able to read the data directly. Today only one constants
file crosses that bridge.

==============================================================================

------------------------------------------------------------------------------
BLUEPRINT-01 - parent: the Rust sim cannot consume the blueprint
               JSON files; types are hand-maintained on both sides.
------------------------------------------------------------------------------

Evidence:
- rts-sim-wasm/build.rs only consumes
  src/sharedSimConstants.json (universal tuning constants such as
  GRAVITY). It generates a single Rust constants file.
- The blueprint JSON files (src/game/sim/blueprints/units.json,
  buildings.json, turrets.json, shots.json, locomotion.json,
  unitRoster.json, fallbacks.json) are not consumed by Rust. There
  are no include_str! or serde_json calls in
  rts-sim-wasm/src/lib.rs that load these.
- Type definitions are hand-maintained in TypeScript only
  (src/types/blueprints.ts, ~503 LOC). Rust types are defined
  separately inside lib.rs without a generated bridge.

Why it matters:
The TS-to-Rust migration cannot finish while blueprints live as
TypeScript-readable-only data. Hand-maintained parallel types are
the canonical drift surface: a field added to TS today silently
disagrees with the Rust struct tomorrow.

Fix direction:
Address as child PRs. Order: schema first, then Rust consumption,
then code-generation of types from the schema.

------------------------------------------------------------------------------
BLUEPRINT-01A - child: write a single schema (JSON Schema or a
                custom validated format) for each blueprint kind and
                land a build-time generator for both TS types and
                Rust structs.
------------------------------------------------------------------------------

Evidence:
- src/types/blueprints.ts is hand-edited. Rust equivalents (where
  they exist) are hand-edited separately. There is no `schemas/`
  directory at the repo root.

Fix direction:
Add `schemas/blueprints/*.schema.json` (or equivalent). Wire a
build step that generates `src/types/blueprints.generated.ts` and
the Rust counterpart from each schema. Existing hand-edited types
are deleted in the same change once parity is verified
(Delete The Old Path).

------------------------------------------------------------------------------
BLUEPRINT-01B - child: have rts-sim-wasm consume the blueprint JSON
                files directly (at build time via include_str! or
                at runtime via a single dump-from-host frame).
------------------------------------------------------------------------------

Evidence:
- Rust today knows nothing about units, turrets, shots, buildings,
  or locomotion; all of that data lives in TS data structures
  derived from JSON.
- rts-sim-wasm/build.rs already demonstrates the include-and-generate
  pattern for sharedSimConstants.json.

Fix direction:
Extend build.rs to include the blueprint JSON files and emit
generated Rust constants for each blueprint table, keyed by the
generated structs from BLUEPRINT-01A. Migrate hot-path lookups in
lib.rs to read from those tables instead of receiving blueprint
parameters by argument from JS.

WIRE-PATH HOT ALLOCATIONS (WIRE-PERF-01 family)

The "Browser Performance Rules" and "Hot paths are data-oriented"
sections require "Reuse arrays by setting length = 0" and "Pool
snapshot DTOs, event objects, scratch arrays, and temporary buffers."
Several snapshot pack/unpack call sites still allocate fresh arrays
per snapshot.

==============================================================================

------------------------------------------------------------------------------
WIRE-PERF-01 - parent: snapshot pack/unpack helpers allocate fresh
               arrays per snapshot in TS rather than reusing pools.
------------------------------------------------------------------------------

Evidence:
- The encode / DP-02 measurement path in
  src/game/network/snapshotProjectileWirePack.ts now leases pooled
  numeric arrays for the packed projectile spawns, despawns, velocity
  updates, beam updates, and beam-point slabs via
  src/game/network/snapshotPooledScratch.ts. Remaining projectile
  decode arrays are retained snapshot DTO outputs and should not be
  scratch-pooled until snapshot ownership/lifetime is changed.
- The 2026-05-23 entity scratch pass reuses pack-side movement/turret
  group/key scratch, reuses per-group binary writers, avoids allocating
  a detail-row array when the delta is fully binary slabs, and removes
  the transient per-group chunk arrays from the entity movement/turret
  slab finish path. Remaining entity decode / detail DTO arrays are
  retained snapshot outputs and should not be scratch-pooled until
  snapshot ownership / lifetime is changed.
- src/game/network/snapshotMinimapWirePack.ts V2 removed the
  six-number packed row array from the pack path. The 2026-05-23
  minimap scratch pass also reuses pack-side group/key scratch,
  reuses per-group binary writers, and removes the transient chunk
  array. Remaining minimap decode DTO arrays are retained snapshot
  outputs and should not be scratch-pooled until snapshot ownership /
  lifetime is changed.
- These run on the host per snapshot (5-10 Hz) per remote client,
  so the allocation pressure scales with N_entities × N_clients ×
  SPS. At 5,000 entities the allocator gets a steady drumbeat of
  short-lived arrays.

Why it matters:
GC pressure on the host directly translates into snapshot jitter
and tick-budget variance. The philosophy is explicit: "Do not
allocate in per-unit, per-projectile, per-turret, per-cell, or
per-frame loops unless the allocation is proven irrelevant." 5-10
Hz over 5,000 entities is not "proven irrelevant."

Fix direction:
The pack-side scratch paths are covered for projectile, minimap, and
entity slabs. The remaining allocation work is tied to snapshot output
ownership: retained decode DTO arrays cannot be reused safely until the
client snapshot lifetime model changes or TS-WASM-01D moves these loops
behind the Rust encoder.

MISCELLANEOUS WIRE / RENDERER / DIAGNOSTIC ITEMS

==============================================================================

------------------------------------------------------------------------------
RENDER-WEBGPU-01 - Three.js → raw WebGPU migration is unstarted.
------------------------------------------------------------------------------

Evidence:
- package.json still depends on `three` 0.184.0.
- src/game/render3d/ has 86 files, all built around THREE.* APIs.
- A repo-wide grep for `WebGPU`, `WGSL`, `GPUDevice`, or
  `gpu.requestAdapter()` returns zero matches. The migration that
  the philosophy frames as "the largest example" of replacing
  scaffolding has not begun.
- The "Engine Ownership: No Libraries, In-House Rust" section
  treats Three.js as legacy scaffolding and reserves the end-state
  for raw WebGPU + WGSL.

Why it matters:
This is the biggest outstanding migration in the project, and
keeping it visible as a tracked issue ensures it doesn't get
forgotten beneath smaller items. Replacing Three.js is not
something to start with a single PR; it is a multi-month effort
that should start with a bounded slice (e.g. ground plane,
instanced units, terrain) and migrate piecemeal — but in the
philosophy's own framing, until the old path is gone, the
migration is not done.

Fix direction:
Out of scope for any single child here. The right first step is
to land a WebGPU initialization layer in src/game/render3d/ that
coexists with Three.js, port the simplest renderer (likely
TerrainTileRenderer3D or a primitive ground pass) to it, and
delete the Three.js path for that one renderer in the same
change. Iterate. Don't start a parallel renderer tree without a
deletion plan; the philosophy is explicit that two renderer
shapes interfere even when only one is supposed to be active.

------------------------------------------------------------------------------
RENDER-PERUNIT-01 - per-entity render-list updates in TS at
                    per-frame cadence.
------------------------------------------------------------------------------

Evidence:
- src/game/render3d/Render3DEntities.ts iterates over visible
  entities every frame (updateUnits / updateUnitMeshes around
  lines 426+/439+) and updates per-unit pose, turret pose, barrel
  spin, and material color in JS.
- This is the "render-list generation for thousands of objects"
  shape called out in the TS-must-never-own-this list.

Why it matters:
The Three.js scene-graph traversal already has its own per-object
cost; adding a per-unit JS update on top of it compounds the
language-overhead penalty the philosophy is trying to avoid at
scale.

Fix direction:
Tightly coupled to RENDER-WEBGPU-01. Migrating to instanced
WebGPU rendering replaces per-unit JS pose writes with batched
GPU buffer uploads sourced directly from WASM linear memory. Until
then, the best intermediate is to ensure the per-frame JS pass is
strictly batched (no per-unit allocation, no Object.assign, no
.map / .filter / .forEach in the inner loop).

FOG OF WAR CONCEPT REVIEW

==============================================================================

FOW-CONCEPT-02 - visual shroud reveal semantics do not match
                 authoritative full-vision semantics.
------------------------------------------------------------------------------

Evidence:
- src/game/network/stateSerializerVisibility.ts separates full vision
  from radar: full vision sends entities in the main snapshot, while
  radar only reveals minimap blips.
- Full entity visibility in SnapshotVisibility.isEntityVisible() checks
  terrain line of sight through hasTerrainLineOfSight().
- The 2026-05-23 optical-source cleanup removed the legacy combined
  full-or-radar helper and switched server shroud history,
  FogOfWarShroudRenderer3D, FogOfWarFog3D, and SightBoundaryRenderer3D
  to canEntityProvideFullVision()/getEntityFullVisionRadius(). Radar
  no longer clears terrain shroud/fog or draws as optical sight.
- src/game/sim/shroudBitmap.ts still uses a radius-only markCircle()
  pass. It does not apply the same terrain line-of-sight mask as
  SnapshotVisibility.isEntityVisible().
- src/game/render3d/FogOfWarShroudRenderer3D.ts and
  src/game/render3d/FogOfWarFog3D.ts still re-derive local current
  vision as 2D radius coverage, so terrain behind ridges can visually
  clear even when the host would hide enemy entity detail there.

Why it matters:
The player can be shown terrain as currently visible or explored even
where the host would still hide enemy entity detail behind a ridge, or
where the only source is radar that should communicate "sensor contact"
rather than true sight. That weakens the fog concept because the visual
language and the authoritative information contract disagree.

Fix direction:
Define separate visibility layers and render them separately:
full optical vision should use the same host-side source classification
and line-of-sight mask as entity visibility; radar should produce a
distinct sensor/minimap treatment; exploration history should be clear
about whether it means "physically seen" or "sensor-covered." The
server should ship or expose compact masks/source packets for those
layers instead of the client re-deriving a different approximation.

------------------------------------------------------------------------------
FOW-CONCEPT-05 - fog-of-war calculation still lives in TypeScript hot
                 paths instead of the Rust/WASM engine.
------------------------------------------------------------------------------

Evidence:
- design_philosophy.html assigns visibility and fog to the Rust/WASM
  engine, and explicitly lists pathfinding and fog-of-war calculation
  among the hot paths TypeScript must not own.
- SnapshotVisibility in src/game/network/stateSerializerVisibility.ts
  builds per-recipient source pools and spatial hashes in TypeScript.
- updateShroudBitmaps() in src/game/sim/shroudBitmap.ts walks unit and
  building source lists and marks bitmap circles in TypeScript.
- The active visual fog paths are Three.js/canvas systems in
  FogOfWarShroudRenderer3D.ts and FogOfWarFog3D.ts.

Why it matters:
The present shape is acceptable for prototype validation, but it is not
the end-state architecture described by the philosophy. At 5,000 units,
recipient-specific visibility, shroud masks, minimap contacts, and
projectile/audio gates are exactly the kind of repeated numeric work
that should be batched in WASM and handed to TS as compact results.

Fix direction:
Create a FOW-specific child of the TS-WASM migration: move source
classification, team mask merge, visibility masks, shroud bitmap update,
and recipient query kernels into Rust/WASM over packed entity/source
slabs. TypeScript should orchestrate snapshot routing and rendering, not
own the visibility inner loops.
