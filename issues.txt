[BA-PERF-002] Remote snapshot publishing still materializes DTO snapshots before wire-first output

Severity: High

Evidence:
- src/game/server/ServerSnapshotPublisher.ts still serializes a NetworkServerSnapshot object per player listener via serializeGameState() before building a pre-encoded wire payload.
- src/game/network/stateSerializer.ts still walks WorldState and fills DTO arrays/maps for entities, projectiles, minimap, economy, audio, spray, metadata, removals, and visibility flags.
- The current preencoded payload can report whether the Rust snapshot encoder was used and how many entities/top-level sections fell back to raw handling, but the server path still reaches that encoder only after DTO materialization.
- Entity, projectile, minimap, and economy serializers already maintain typed row sources consumed by snapshotRustWireEncoder; the remaining gap is bypassing the DTO arrays for remote wire-first listeners.

Risk:
Remote snapshot publishing still pays DTO graph population before the Rust/WASM or typed-row encoder can emit bytes. Existing pooling and row sources reduce allocation and packing cost, but the hot path remains listener-oriented TypeScript orchestration instead of a true wire-first writer. budget_design_philosophy.html says snapshots should move toward compact binary/typed-buffer output and that TypeScript should orchestrate while Rust/WASM computes hot loops.

Required fix:
Implement a remote preencode path that emits snapshot rows and metadata directly from WorldState/visibility/baseline decisions into Rust/WASM or owned typed buffers before NetworkServerSnapshot DTO arrays are populated. Keep serializeGameState() as the local/in-memory and parity fallback until the direct writer covers full snapshots, deltas, visibility filtering, projectile events, minimap, economy, audio/spray events, static terrain/buildability, removals, game state, and server metadata. Use the payload encoderKind/rawEntityCount/rawTopLevelKeys diagnostics to retire raw fallback sections before deleting the DTO path.

[BA-ARCH-001] Large hot-path files make lifecycle, pooling, and Rust/WASM migration hard to audit

Severity: Medium

Evidence:
- src/game/sim/Simulation.ts is 2535 lines.
- src/game/render3d/Input3DManager.ts is 2073 lines.
- src/game/scenes/RtsScene3D.ts is 1644 lines.
- src/game/render3d/Render3DEntities.ts is 1431 lines.
- src/game/render3d/BuildingAnimationController3D.ts is 1356 lines.
- src/game/sim/WorldState.ts is 1344 lines.
- src/game/server/GameServer.ts is 1081 lines.
- budget_design_philosophy.html explicitly says large files should be split into coherent helpers when touched.

Risk:
Cleanup ownership, retained pools, hot loops, and authoritative/client boundaries are spread across large files. That raises the chance of missed teardown, duplicate runtime paths, accidental per-frame allocation, and stalled migration of TypeScript hot loops into Rust/WASM.

Required fix:
When any of these files are touched for the issues above, split along existing ownership boundaries instead of adding more logic in place. Prioritize lifecycle/teardown helpers, snapshot intake/publisher modules, unit mesh lifecycle vs pose/locomotion updates, input command builders, and simulation subsystems that are candidates for Rust/WASM.
