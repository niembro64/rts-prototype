5.0 [MAJOR] Per-body force application + idle/hover/boundary math still runs as
per-tick TypeScript loops with per-unit allocation.

src/game/server/UnitForceSystem.ts:89-581 (applyForces) is a per-entity body loop
over dynamicBodies each tick that calls getLocomotionForceProfile(...) per unit per
tick (allocates a profile object — :292,:401). src/game/server/PhysicsEngine3D.ts
does three separate full dynamicBodies scans per tick (clearDynamicSurfaceContacts:757,
the integrate gather, accumulator clear :740) plus collectAwakeEntityIds:607 full-scans
every dynamic body; applyMapBoundaryForces:974 is another per-body spring loop. Body
integration/contacts are already Rust-owned, but force application is not. Violates
"TypeScript must never own per-unit update loops / physics integration" and "do not
allocate in per-unit loops."
FIX: migrate force-application math into a packed WASM kernel (the hover-orientation
batch at PhysicsEngine3D :532 is the model). Meanwhile hoist getLocomotionForceProfile
to write into a reused struct, and fold the three step() scans into one pass.

5.1 [MAJOR] Tick loop drifts; no fixed-timestep accumulator/catch-up; no budget
saturation metric.

src/game/server/ServerTickLoop.ts:13 is a bare setInterval(cb, 1000/hz). GameServer.tick
(:455) clamps per-tick dt to MAX_TICK_DT_MS (good vs spiral-of-death) but there is no
accumulator and no catch-up: a 200ms hitch advances sim by one clamped step, so the
authoritative clock silently slips behind real time and never recovers. TPS/CPU are
measured, but tick-work-ms vs interval-ms saturation is not, so a saturated host can't
tell it's falling behind. Violates "Measure TPS, CPU budget, ... backpressure
continuously."
FIX: drive the loop with a fixed-dt accumulator + bounded catch-up (max N steps/frame),
and surface a tickMs/intervalMs saturation metric into serverMeta.

NETWORKING / SNAPSHOTS

6.1 [MEDIUM] Snapshot impairment delay path still allocates fresh cloners.

src/game/network/snapshotClone.ts:548 — the free function does
`new ReusableNetworkSnapshotCloner().clone(state)`. Local and remote pending client
buffers hold long-lived cloners, but LocalGameConnection.subscribeSnapshots and
RemoteGameConnection.stateHandler still pass cloneNetworkSnapshot into
SnapshotImpairmentQueue.schedule. When snapshot impairment/delay is enabled, every delayed
snapshot allocates a fresh cloner and its internal DTO arrays/objects.
FIX: make SnapshotImpairmentQueue own a lifetime-safe delayed snapshot clone pool, or
change the queue contract so callers can return/reuse ReusableNetworkSnapshotCloner
instances only after delivered snapshots can no longer be referenced.

6.3 [MEDIUM] Client receive-coalescing silently drops a delta when one is already
pending.

src/game/network/NetworkSnapshotTransport.ts:156-162 — storePendingState only stores when
there is no pending state (or when upgrading delta->full); a second delta arriving before
the first is consumed is dropped, not coalesced. Deltas are non-idempotent (each carries
only fields changed since last sent), so dropping one loses those corrections with no
resync trigger.
FIX: apply/merge the pending delta before overwriting, or at minimum request a keyframe
when a delta is dropped here.

7.0 [MAJOR, migration surface] Projectile + damage/death authority is still
TypeScript-orchestrated.

Per AUTHORITY_LEDGER.md (TypeScript-Orchestrated rows), the entire
checkProjectileCollisions / _updateTravelingProjectilesJS path
(combat/ProjectileCollisionHandler.ts, combat/projectileSystem.ts:916) and the
damage/death loop (damage/DamageSystem.ts, Simulation.ts:580 updateCombat / :771
cleanupDeadEntities / :897 detonateEntityDeathExplosions) run per-projectile /
per-hit-entity in TypeScript. Position integration is batched into Rust, but collision
detection, reflection, submunition spawn, termination, detonation, and HP write-back are
not. Homing also makes one solveKinematicIntercept + one computeHomingThrust WASM call
per homing projectile per tick rather than a batch (the TS fns are thin Rust wrappers —
NOT duplicated math, so no Delete-The-Old-Path violation, just un-batched). Violates
"TypeScript Coordinates, WASM Computes."
FIX (C1 migration): move the swept-collision + termination loop into a Rust projectile
kernel; batch the homing intercept/thrust like the existing projectileIntegrateStepBatch.
Combine with 3.0.

7.5 [LOW / design decision] ray/shield exist as blueprint families but not runtime
EntityType kinds.

blueprints/rays.json (beam) and blueprints/shields.json are distinct families and shields
get a turret-mounted runtime, but ray/shield are not entries in the Rust EntityType/KIND
enum (only unit/tower/building/shot/turret). This is consistent with the philosophy (rays
and shields are emitter-bound sustained emissions, NOT bodies, and attribute through their
turret rather than minting an id) — so likely correct as-is. Flag only if a future design
needs to point at an individual ray/shield instance by id.
