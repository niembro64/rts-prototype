AIMING / TARGETING SYSTEM ISSUES

AIM-08 - Aiming/targeting hot paths remain TypeScript object loops instead of WASM/SoA batches.

Evidence:
- design_philosophy.html:520-570 and 614-619 call out targeting batches, per-unit loops, projectile stepping, and spatial queries as migration targets for Rust/WASM over typed memory.
- src/game/sim/combat/targetingSystem.ts:405-545 and 609-1122 still run the targeting FSM, candidate ranking, LOS gating, and ballistic gating over Entity/Turret objects in TypeScript.

Why it matters:
The current code has good local pooling and some WASM helpers, but the main targeting loop still scales as JS object orchestration over armed entities and candidate arrays. That is acceptable for prototype correctness, but it is not aligned with the stated 5,000-unit architecture.

Fix direction:
Keep the TypeScript behavior as the reference while designing a data-oriented targeting batch: flat turret/entity inputs, candidate slots, visibility/LOS results, target output IDs/states, and parity checks during migration. Delete the old path when parity is proven.
