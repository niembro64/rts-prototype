ARCHITECTURE / PHILOSOPHY MIGRATION (budget_design_philosophy.html)
================================================================================

7.0 [MAJOR, migration surface — ECS-gated] The last of the projectile/damage/
death migration: move line/swept/area candidate PACKING and death-explosion
CHAINING into Rust. Blocked on the ECS slab work.

Verified against code 2026-06-02 — all the pure-computation kernels already live
in Rust/WASM: projectile integration, homing/terrain-follow thrust, reflector
intersections + reflected response, swept body/building/projectile/turret hit
selection (projectile_hitbox_sweep_batch), terminal consequence classification
(projectile_terminal_consequence_batch), terminal effect planning
(projectile_terminal_effect_plan_batch), submunition launch-velocity scatter
(projectile_submunition_launch_velocity_batch), line/swept segment hits
(damage_segment_hits_batch), splash sphere/AABB overlap + slice-cone + knockback
direction (damage_area_overlap_batch), direct-hit/splash HP/kill output math
(damage_apply_batch), and death-cleanup HP/materialization classification +
compact removal diffs (death_cleanup_diff_batch). All damage HP write-back
(beam/line, swept, area, death-explosion) shares the one batched
damage_apply_batch path (queueDamageToEntityBatch + flushDamageBatch) — there is
no second single-entity apply path to migrate.

What remains in TypeScript is mostly DIFF APPLICATION, which the C1 contract
keeps in TS on purpose: applying terminal-effect flags (despawn / hasExploded /
audio events), materializing submunitions via world.createProjectile, writing
returned HP onto JS entities, emitting death/audio events, and removing JS
entities from WorldState.

REQUIRED FIX (the only Rust work left): pull line/swept/area candidate PACKING
and the death-explosion CHAINING loop into Rust so the segment/area kernels
iterate the spatial-grid candidate slot list internally (killing the
per-candidate TS packing loops) and the chaining loop accumulates kills without
round-tripping HP to JS. Both need a Rust-authoritative per-entity combat slab:
position, hitbox + collision radius, kind, team/owner, HP, turret mount poses,
exclude flags. Building that slab BEFORE the ECS migration would duplicate
authoritative HP/state (a second source of truth), which the design philosophy
forbids — so do this as part of the ECS migration that makes Rust the entity
store, not as a parallel mirror now. Until then this item stays blocked; the TS
that remains is the diff-application layer the contract intends, not new
migration debt.
