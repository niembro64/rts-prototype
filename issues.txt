==============================================================================
 BIG PICTURE — bespoke Rust/WASM physics, shared by client + server
==============================================================================

User directive (2026-05-14): "we need to write our own physics engine
for our needs uniquely. And we need it to be in the fastest languages
for that we can use in a browser. This should be for both frontend
and backend."

One Rust crate at `rts-sim-wasm/`, compiled to WebAssembly via
wasm-pack, loaded by BOTH the authoritative server tick AND the
client prediction stepper. Same integrator, same numerical
behavior, same code path — client prediction is bit-identical to
server authoritative motion.

Bespoke, NOT a generic engine (no Rapier, no Ammo). The existing
PhysicsEngine3D makes RTS-specific shortcuts a generic engine
couldn't (implicit infinite ground plane, sphere bodies for units +
axis-aligned cuboids for buildings, locomotion-aware ground spring,
sleep heuristics tuned for "few hundred mostly-static units",
hover lift + quaternion bank physics in the same pipeline). The
port preserves those shortcuts.

 Phase 10 — Snapshot serializer hot loops — IN PROGRESS
==============================================================================

stateSerializer*.ts is ~1.8k LOC across 13 files (~3k LOC counting
NetworkSnapshotTransport.ts + snapshotDtoCopy.ts + snapshotClone.ts).
Per-entity inner loop (qPos / qVel / qRot quantization, delta
encoding, FOW visibility, AoI gate) is hot under load — every
server tick that ships a snapshot pays it per visible entity per
recipient.

D.2 (MessagePack encoder) shipped: hand-rolled subset matching
@msgpack/msgpack's `ignoreUndefined: true` output on 21 fixture
cases. Self-test runs at module load and aborts initSimWasm if
any case fails. Verified byte-equal with the JS lib on 23 inputs.

Remaining sub-commits:

  D.1 — Entity-meta SoA pool. DONE for fixed-scalar fields.
        Rust-side `EntityMetaPool` covers HP curr/max, owner playerId,
        entity-type tag (unit/building), combat mode, is_commander
        flag, build state (complete + paid energy/mana/metal +
        target id), suspension spring offset/velocity, jump airborne/
        timer, factory state (producing + queue len + progress),
        solar open, build progress. Setters: entity_meta_set_unit /
        entity_meta_set_building (one bulk call per dirty entity per
        tick — amortises boundary overhead). Per-field raw pointers
        exposed for typed-array view reads.

        Slot space is shared with SpatialGrid — the JS-side
        slotByEntityId map in SpatialGrid.ts is the canonical
        mapping. JS-side per-tick population lands with D.3.

        D.1b — Turret sub-pool DONE. Up to 8 turrets per entity at
        fixed offset `entity_slot * MAX + turret_idx` in a flat SoA.
        Per-entity count gates which indices are live. Fields cover
        the snapshot turret DTO (rotation, angular vel/accel, pitch,
        pitch vel/accel, forceFieldRange, target_id). JS-side
        population lands with D.3.

        D.1c — Action sub-pool. Still pending. Actions are variable-
        length per entity with a JS-string ActionType enum that
        needs a stable u8 mapping before porting. Defer until D.3
        forces the schema decision.

  D.3 — Per-entity quantize + delta-encode kernel. Split into a/b/c/d/e.

        D.3a DONE — captureSnapshotEntityStates in
        stateSerializerEntityDelta.ts now syncs each captured entity
        into the entity-meta + turret pools via SpatialGrid.getSlot.
        Both delta and keyframe branches mirror the JS-side fields.
        Guarded on getSimWasm() !== undefined (boot window) and
        spatial slot != -1 (entity not yet in grid).

        D.3b DONE — per-recipient SnapshotBaseline registry in Rust.
        Free-list-managed pool of per-slot SoA arrays mirroring
        stateSerializerEntityDelta.ts:PrevEntityState (transform,
        velocity, hp, weapon counts + turret state, normal, build/
        factory/solar/jump/suspension). Each network listener
        registers a u32 handle via snapshot_baseline_create() at
        session start. Float storage — quantization happens at emit
        time so threshold compares stay byte-equal with the JS path.

        D.3c DONE — snapshot_baseline_capture_unit_slot /
        _capture_building_slot. One WASM call per emitted entity per
        recipient. Pulls HP + build/factory/solar/suspension/jump
        from the entity-meta pool, turret state from the turret pool;
        takes transform / velocity / movement accel / normal /
        action hash / engagement bits as parameters. No JS consumer
        wires these calls yet — D.3e will land that.

        D.3d pending — diff/emit kernel. Reads current pool +
        baseline, applies quantization (qPos×100, qVel×10,
        qRot×1000, qNormal, qSuspension), threshold-compares,
        returns a CHANGED-FIELDS mask + emits MessagePack delta
        bytes via the D.2 encoder.

        D.3e pending — JS-side wiring. serializeGameState's
        per-listener loop calls D.3d to compute mask + bytes, then
        D.3c to update the baseline on successful emit.

  D.4 — `snapshot_build_for_recipient`. One WASM call per recipient
        per snapshot tick. Reads SpatialGrid (Phase 7) for AoI
        candidates, applies FOW visibility (port from
        stateSerializerVisibility.ts), wraps in envelope (tick,
        version, audio events, shroud bitmap), outputs MessagePack
        bytes ready for WebRTC send.

  D.5 — JS-side migration. `stateSerializer.ts` becomes a thin
        orchestrator that owns per-listener visibility precomputation,
        calls snapshot_build_for_recipient for each, hands the bytes
        to the WebRTC transport. The 13 serializer files collapse
        to ~3.

Critical risk: byte-equality requirement. Snapshot bytes must be
byte-equal pre/post for the client decode path to keep working
without changes. Strategy for the next session: capture a 60-tick
golden snapshot from the current JS path BEFORE starting D.3, then
assert byte-equality on every D.3+ commit.

Phase 10 is sized for multi-session work — D.1 alone is one
focused session, D.3-D.5 each warrant their own.

==============================================================================
 Deferred items — re-evaluate when conditions change
==============================================================================

Phase 5d — Beam-path point integration. The PREDICT-gated per-point
ax/ay/az step lives in ClientPredictionStepper.applyBeamSnapshot
Prediction. At a realistic busy-scene load (10 simultaneous beams ×
~5 points each = 50 point steps per FRAME, 7-9 float ops per step)
total per-frame work is ~350 ops. Data lives in JS BeamPoint
object arrays, so a WASM call would have to pack/scatter
Float64Array(50 × 6) every frame just to integrate ~350 ops — net
regression. Revisit only if a stress test ever shows the beam
point loop on the profiler.

Phase 5e — Traveling (non-packed) projectile path. Per-projectile
inner math is trivial scalar work (vz -= GRAVITY*dt, then position
+= vel*dt). Terrain-follow's only non-trivial bit is
world.getGroundZ, which already dispatches to WASM via Phase 8.
A batched per-tick call would marshal ~12 KB per side at busy
scales (50-100 active projectiles × 11 f64 each) to save ~5 µs/
tick. Net loss.

Phase 6b — chooseBestTargetCandidate. The rank formula's distSq +
radius classification could run in Rust against the SpatialGrid
slot SoA. BUT the mirror-target score for passive weapons is
JS-side metadata with no clean WASM schema, requiring a per-
candidate JS pre-pass before any WASM call. Net win at typical
busy candidate counts (~10-30 µs/pick) doesn't justify the new
boundary surface. Re-evaluate if profiling ever shows targeting
hot enough.

Tier E SIMD dual-build — current build enables +simd128 globally;
browsers without SIMD support (iOS Safari < 16.4) fail at WASM
compile time. The right fallback is a dual-build (simd + non-simd
.wasm with runtime feature detection). Deferred until profiling
or analytics show older iOS users hitting the gate.

==============================================================================
 Cross-cutting concerns — apply to every phase
==============================================================================

  - **Boundary cost discipline.** Every JS↔WASM call has overhead
    (~10-100 ns per call). A naive port that calls into WASM once
    per entity per tick can be SLOWER than pure TS. Design every
    kernel as a single batched call per tick that operates over a
    Rust-owned SoA. The fast pattern: "JS pushes intents in →
    Rust runs the whole tick → JS reads results out".

  - **Memory model.** Body / projectile / spatial / terrain state
    lives in linear memory on the WASM side. JS reads via
    DataView / typed-array views (zero-copy). Writes from JS use
    explicit setter methods so Rust can validate / wake bodies /
    mark dirty. EntityId ↔ slot mapping is JS-side (Map<EntityId,
    u32>); slots are stable for the entity's lifetime so JS can
    cache them.

  - **Shared client + server semantics.** The whole point of this
    migration is that client prediction and server authority run
    the SAME integrator. That means every kernel must be a pure
    function of its inputs — no globals, no per-instance hidden
    state, no JS-side intrusion mid-tick. Identical numerical
    behavior is the contract.

  - **Determinism (deferred, but architected for).** Strict bit-
    identical determinism across machines isn't a requirement yet,
    but the structure should NOT block it. No f64 → f32 narrowing
    in unexpected places. Stable iteration order (index-stable
    Vec-based pools). Same WASM module on client + server so
    `std::f64::sin` etc. comes for free under the shared-load
    pattern.

  - **Debugging conventions.** WASM panics are harder to chase
    than TS errors — console_error_panic_hook is wired in
    `#[wasm_bindgen(start)]`. Keep entity ID lookups + blueprint
    config on the JS side so the WASM kernel only sees primitive
    SoA data; that minimizes the surface area to debug.

  - **Hot-reload dev workflow.** `cargo watch -s 'cd .. && npm run
    build:wasm'` recipe is in rts-sim-wasm/README.md. Post-Phase-3
    the Body3D + projectile pools live in WASM linear memory —
    they're not optional. TS fallbacks now run only during the
    brief boot window before initSimWasm resolves, and as
    structural reference impls for bit-identity audits.

  - **Threading: defer.** WASM threads via SharedArrayBuffer + web
    workers are real but require COOP/COEP headers and complicate
    WebRTC. Start single-threaded; revisit only if we hit a hard
    single-core ceiling.

  - **Bundle size budget.** Final .wasm under 250 KB. Currently
    ~180 KB after Phase 7 + 9 + Phase 10 D.2 + SIMD. wasm-opt -Oz
    runs as a build post-pass (gated on local binaryen).

  - **CI builds the .wasm.** `npm run build:wasm` runs before any
    test that loads the module. README documents the prerequisites
    (cargo + wasm-pack on PATH, binaryen optional).
