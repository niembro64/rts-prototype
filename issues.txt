BAR COMMAND AND UI PARITY GAPS

Baseline: budget_design_philosophy.html says player-facing behavior defaults to
Beyond-All-Reason unless explicitly overridden. BAR references for this backlog:
https://www.beyondallreason.info/commands-20 and
https://github.com/beyond-all-reason/Beyond-All-Reason/tree/master/luaui/configs/hotkeys.

Issue: the shared command-hotkey registry and in-game preset switcher only cover
commands that already exist in rts-prototype, with prototype, BAR-grid, and
BAR-legacy presets for that subset, plus runtime multi-key sequence dispatch.
Finish the BAR-compatible keymap and command UI layer by importing the complete
BAR grid/legacy bindings and adding command buttons for missing BAR commands so
visible hotkeys match the active keymap across the whole command surface.

Issue: movement and formation commands are incomplete. rts-prototype has
standard group-spread move, right-drag line formation targets with 3D preview,
and Alt+right-click preserve-current-offset move. Add the remaining BAR Move
and assume formation / Move in formation command surface, slow/fast movement
modifiers where applicable, and complete visual formation previews. Translate
these to the 3D physics model so formation targets respect mass, collision,
terrain height, slopes, water, and pathability.

Issue: construction commands are missing most BAR build ergonomics. Implement
the BAR grid build menu categories, grid-menu cell hotkeys, construction
shortcuts, build grid, build border, build split, build spacing
increase/decrease, build facing/rotation controls, upgrade T1 mex area,
upgrade T1 -> T2, and visual previews for every build shape. rts-prototype has
an allowed-building strip, number keys, single-placement ghost, Shift queueing,
and left-drag build line placement with a 3D line preview through existing
build validation; these do not satisfy BAR fallback behavior.

Issue: queue editing and command insertion are incomplete. BAR supports queue to
next, commandinsert/prepend-between, and queued variants of several orders.
rts-prototype exposes Shift append-queue,
Shift+Ctrl/Cmd add-in-front queueing for queueable orders,
skip-current-order, remove-last-queued-order, clear-queued-orders,
stop-production, wait queueing, basic factory rally, basic factory guard, and
selected-unit current-order/queue-depth feedback. Implement the missing queue
semantics with server authorization and matching visual feedback.

Issue: behavior/state toggles are incomplete. BAR exposes cloak/want-cloak,
gather wait, on/off, stateful on/off, repeat, stateful repeat, stance/fire
state, positioning/move state, trajectory toggle, stop, wait, factory guard,
and related stateful command buttons. rts-prototype currently has only a simple
fire enabled toggle, producer-building on/off, wait, stop, self-destruct, and
basic factory guard, with selection-detail indicators for fire, wait, and
producer on/off state.
Implement the missing states in simulation, snapshots, command authorization,
selection UI, hotkeys, and visible state indicators.

Issue: special abilities are incomplete. Add BAR Capture, Load Transport,
Unload Transport, Resurrect, Resurrect Area, manual launch, and any
unit-specific special buttons beyond commander DGun. rts-prototype has
commander DGun and self-destruct, but no capture, resurrection, transports,
load/unload logic, manual launch weapons, or unit-specific ability command-card
support.

Issue: factory and production UI are incomplete. BAR has factory production
controls beyond selecting one repeat-build unit: repeat/queue state, factory
queue mode/quotas, production queue editing, and related state
indicators. rts-prototype now has an authoritative stop-production command,
factory panel button, and local factory preset slots with shared hotkeys that
load through authoritative production commands, plus selected repeat-build and
production-progress state indicators, basic factory rally, and basic factory
guard. It still needs the remaining BAR-equivalent factory panel and hotkeys,
translated to authoritative-server commands.

Issue: BAR gameplay functionality tied to missing commands is absent. Implement
or explicitly redesign the systems needed by the missing command surface:
transports, capture, resurrection/wreck fields, cloak/detection, unit stances,
trajectory weapons, manual-launch weapons, factory queue
management, build-shape placement, mex upgrade workflows, spectators/replays
where applicable, and complete faction/unit/factory/building rosters. Missing
commands should not be added as dead UI; their underlying
authoritative simulation, client prediction, snapshots, and render feedback
must exist.

Issue: BAR parity must be extended through the 3D/physics-first design, not
implemented as flat 2D shortcuts. Attack lines/circles still need ballistic
previews; build grids/borders need slope, support surface, collision, and
resource-cell validation; formations need mass, acceleration, collision, and
terrain-aware spacing; transport/load/unload, resurrection, capture, and
manual launch/special-fire abilities beyond commander DGun need physical
presentation and authoritative physics interactions. Document any intentional
departure from BAR in budget_design_philosophy.html before omitting a BAR
behavior.
