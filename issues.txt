AIMING / TARGETING SYSTEM ISSUES

AIM-08 - Aiming/targeting hot paths remain TypeScript object loops instead of WASM/SoA batches.

Evidence:
- design_philosophy.html:520-570 and 614-619 call out targeting batches, per-unit loops, projectile stepping, and spatial queries as migration targets for Rust/WASM over typed memory.
- src/game/sim/combat/targetingSystem.ts still runs armed-entity traversal, disabled-weapon reset bookkeeping, and JS Turret writeback over Entity/Turret objects in TypeScript. Candidate scoring/ranking dispatches to the AIM-08.3 Rust kernel, ballistic arc solving dispatches to the AIM-08.4 Rust slab kernel, and AIM-08.5 now batches gate resolution, cooldown decrement, mode scheduling, and FSM transition writes in Rust, but the outer tick loop is still transitional.

Why it matters:
The current code has good local pooling and some WASM helpers, but the main targeting loop still scales as JS object orchestration over armed entities and candidate arrays. That is acceptable for prototype correctness, but it is not aligned with the stated 5,000-unit architecture.

Fix direction:
Keep the TypeScript behavior as the reference while designing a data-oriented targeting batch: flat turret/entity inputs, candidate slots, visibility/LOS results, target output IDs/states, and parity checks during migration. Delete the old path when parity is proven.

MIGRATION PLAN (each phase = its own PR; each lands one kernel and deletes
the TS path it replaces, never both alongside — see "Delete The Old Path").

AIM-08.5 — Port the targeting FSM
  The per-armed-entity loop (Pass 0 mount kinematics, Pass 1 existing-
  lock re-validation, Pass 2 candidate scan, fire-gate dispatch, state
  transitions, losBlockedTicks decay) becomes the top-level Rust
  kernel that consumes all of .1-.4 and writes new turret targets +
  states back into the slab. JS dispatch becomes a single
  rustTargetingTick(...) call. Delete targetingSystem.ts FSM logic.

  Current progress: per-transition slab writes are batched in Rust for
  priority-point, priority-target, existing-lock validation, fire-choice,
  and acquisition-choice. Rust now reads slab target radius/distance for
  priority/existing-lock FSM batches. Candidate scoring/selection now
  runs as a per-entity Rust batch that reads turret ranges from the slab
  and writes target/rank arrays for the batched FSM apply calls.
  Fire-choice and acquisition candidate-prep masks/seeds now also batch
  through Rust; passive mirror seed scores remain object-owned in TS.
  All four targeting passes now use Rust-internal LOS + ballistic +
  FF gates and apply the FSM transitions inside one boundary call:
    - combat_targeting_compute_and_apply_priority_point_fsm_batch
    - combat_targeting_compute_and_apply_priority_target_fsm_batch
    - combat_targeting_compute_and_apply_validate_existing_lock_fsm_batch
    - combat_targeting_compute_and_choose_best_candidates_batch
  A shared compute_turret_gates_for_aim_point Rust helper centralizes
  the LOS / ballistic / FF logic with ground-aim-fraction + under-only
  handling. Building AABB half-extents are stamped on the slab so
  aim-point resolution lives in Rust (sphere targets use entity
  center; AABB targets clamp into the box). The JS gate callback
  (passesWeaponFireGates / targetCandidatePassesFireGates) and every
  TS gate helper it relied on
  (hasWeaponLineOfSight / hasWeaponBallisticSolution /
   hasWeaponForceMaterialSightClearance /
   targetLockOnPointIsBelowWeaponMount /
   weaponNeedsBallisticSolution /
   weaponUsesUnderOnlyBallisticLock /
   weaponRequiresForceMaterialSightClearance /
   _targetingBallisticAim / UNDER_ONLY_*) have been deleted under
  Delete The Old Path. Per-turret DPS is now stamped on the slab and
  every passive-mirror computation runs through Rust helpers
  (combat_targeting_is_mirror_target_for_slot for the boolean check;
  combat_targeting_mirror_target_score_for_slot for the DPS-ranked
  score). The priority-target / existing-lock kernels compute their
  mirror_valid gate internally; combat_targeting_prepare_fire_choice_
  fsm_inputs seeds passive fire-choice ranks with the target's mirror
  DPS; and the candidate-batch kernel fills its per-candidate
  mirror_score from the slab on entry. Every TS-side
  getMirrorTargetScore / isMirrorTarget caller in targetingSystem.ts
  is gone (the JS getMirrorTargetScore export survives only for the
  damage path). Cloak observability now lives in Rust: each entity
  row carries CT_ENTITY_FLAG_CLOAKED + detector_radius +
  detection_padding, and the kernels (existing-lock, candidates,
  priority-target via combat_targeting_can_player_observe_entity)
  walk the slab for detectors instead of consuming JS-precomputed
  observable masks. Mirror-panel sightline clearance also lives in
  Rust: a per-tick MirrorPanelPool slab (unit pose + slope-aware
  pivot + panel range; per-panel arm length / offsets / base+top Y /
  half-width) feeds a Rust port of pointSegmentDistanceSq3 +
  rayTiltedRectIntersectionT + findClosestPanelHit that
  compute_turret_gates_for_aim_point consults inline. JS no longer
  fills the per-(turret, candidate) mirror_panel_clear mask, and the
  legacy hasForceMirrorPanelClearance / hasForceMaterialSightClearance
  helpers have been deleted. The per-entity boundary-crossing count
  for auto-mode is now down to 2 Rust calls
  (combat_targeting_existing_lock_and_auto_scan_tick merges the Pass
  1 validation with the auto-scan radius computation;
  combat_targeting_auto_mode_spatial_candidate_tick keeps the
  spatial-grid enemy query and candidate SoA stamping inside Rust,
  then collapses prep + choose-best + apply for fire-choice and
  acquisition into a single boundary call, with scratch buffers living
  in Rust).
  Priority-point / priority-target paths already ship as one call
  each. Pass 0 mount kinematics now runs in Rust through
  combat_targeting_update_mount_kinematics: the slab carries
  per-entity ground Z / rotation / surface normal / suspension offsets
  plus per-turret local mount and previous world-pos tick, then the
  Rust pass writes current world mount position/velocity back into the
  slab. TS only writes that slab result back to Turret.worldPos /
  worldVelocity while rendering, firing, and snapshot encode still read
  JS Turret objects. Per-turret ballistic gate config (projectile
  speed, arc preference, max flight time, ground-aim fraction, and
  under-only lock state) is now stamped on the combat-targeting slab,
  so targetingSystem.ts no longer rebuilds or passes per-entity
  projectile config arrays into the priority, existing-lock, or
  auto-mode Rust kernels. Auto-mode entities now queue into contiguous
  world-order runs and flush through
  combat_targeting_update_mount_kinematics_batch plus
  combat_targeting_auto_mode_spatial_candidate_tick_batch; the latter
  runs existing-lock validation, auto-scan, Rust spatial candidate
  queries, and fire/acquisition FSM application for each queued entity
  without returning to TypeScript between units.
  Priority-point, priority-target, fire-disabled clear, and auto-mode
  rows now also share a mixed world-order Rust FSM dispatch
  (combat_targeting_tick_batch), so TypeScript queues a single
  per-tick mode batch after its remaining object-owned command prep.
  Per-turret target aim-point resolution now lives in
  Rust too: the slab carries a lockOnToTurret flag, Rust ports
  pickTargetAimTurret / body-AABB clamping / turret-mount resolution,
  and combat_targeting_tick_batch no longer accepts TS-built aim
  arrays for priority-target or existing-lock gates. The mixed
  scheduler itself now lives in Rust through
  combat_targeting_schedule_and_tick_batch: the kernel reads stamped
  entity flags, hold-fire state, priority point/target commands,
  priority-target observability, enabled-weapon state, and probe gates,
  updates mount kinematics for processed rows, and writes the actual
  processed mode back for the transitional JS writeback pass.
  Cooldown timers now live on the combat-targeting slab too: JS stamps
  per-turret cooldown / burst-cooldown values, the scheduled Rust batch
  decrements them before probe skip decisions, and the transitional
  slab->JS writeback copies the timers back even for skipped rows.
  The scheduled Rust batch also resolves source entity IDs to targeting
  slab slots internally, so targetingSystem.ts no longer calls
  spatialGrid.getSlot while walking armed entities.
  Disabled-weapon slab reset now runs in Rust too:
  combat_targeting_reset_disabled_weapons_for_entity zeros target /
  state / cooldown / aim error / LOS bookkeeping for every disabled
  turret on entity_ready entities, inside the scheduler. JS Turret
  writeback then finishes the disabled-weapon reset for fields that
  never cross the WASM boundary (angular/pitch velocity +
  acceleration, burst.remaining, forceField.transition/range) via the
  shared weaponSystemDisabled helper in targetingInputStamping.ts. The
  legacy resetDisabledWeapon / weaponSystemDisabled helpers and the
  per-entity stampCombatTargetingEntity call were deleted under Delete
  The Old Path.
  Per-entity targeting inputs (priorityTargetId, priorityTargetPoint,
  nextCombatProbeTick) now live on the combat-targeting slab too: the
  input stamping pass writes them via the extended
  combat_targeting_set_entity boundary call, and
  combat_targeting_schedule_and_tick_batch reads them by slot instead
  of accepting priority/probe scratch arrays from JS. The scheduler
  signature lost six per-entity arrays (priority target IDs / point
  presence flags / xyz coordinates / scheduled probe ticks) in the
  same change. The TypeScript prep loop in updateTargetingAndFiringState
  is now just an armed-entity walk that pushes sourceIds into one
  Int32Array; clearCombatActivityFlags and the fire-disabled JS
  priority-command cleanup moved into the writeback path (SKIP and
  CLEAR_LOCKS branches respectively) so they fire after the kernel
  decides what each row actually ran as. CT_TARGETING_TICK_MODE_*
  constants are exported from sim-wasm/init.ts so the writeback
  dispatch reads the same byte values the Rust kernel writes.

  Done when: targetingSystem.ts only contains the slab-rebuild
  stamping pass (or has moved to its own file); parity shim runs
  end-to-end clean across the supported scenarios.

LOCK-ON-03 — Stamp lock-on exclusion data into the combat-targeting slab.

Evidence:
- AIM-08.5 has moved targeting decisions into Rust/SoA slabs. Any policy
  that remains only on JS Turret objects cannot participate in the
  scheduled Rust targeting batch without per-candidate JS callbacks.

Why it matters:
The lock-on policy must be enforced in the hot targeting path, including
priority targets, existing-lock validation, auto acquisition, fire-choice,
and mirror scoring. Reintroducing JS-side filters would undo the current
targeting migration.

Fix direction:
Encode level-0 relationship and entity-family exclusions as compact
per-turret bitmasks in the combat-targeting slab. Encode level-1 named
exclusions through stable integer blueprint ids or precompiled lookup
sets that Rust can query without string comparisons. Stamp target entity
family and blueprint id on entity rows, and stamp target turret blueprint
id on turret-target rows or equivalent candidate metadata.

Done when: Rust targeting kernels can evaluate all exclusion sets from
slab/typed-memory data only, with no JS callback and no string matching
inside per-candidate loops.

LOCK-ON-04 — Apply exclusion filters consistently across targeting modes.

Evidence:
- Targeting has several entry paths: priority-point, priority-target,
  existing-lock validation, auto acquisition, fire-choice promotion, and
  passive mirror target scoring.

Why it matters:
If exclusions only apply to auto acquisition, commands or stale locks can
violate blueprint policy. A fighting turret should not keep or accept a
friendly lock just because it arrived through a priority command, and a
construction turret should not accidentally reacquire enemies through the
auto path.

Fix direction:
Add one shared Rust helper that answers whether a turret slot may lock a
candidate target/candidate turret under the authored exclusions. Use that
helper before range, LOS, ballistic, mirror, and scoring work wherever the
candidate can be rejected cheaply. Existing locks and priority targets
must be dropped or ignored when the helper returns false.

Done when: every targeting mode calls the same eligibility helper, and
there is no separate hand-rolled friend/enemy or entity-family filtering
left in TypeScript or Rust call sites.

LOCK-ON-05 — Expand candidate collection beyond enemy-only assumptions.

Evidence:
- Construction turrets need to lock friendly entities, while combat
  turrets normally lock enemy entities. A broad default also allows
  future turrets that deliberately consider both sides.

Why it matters:
An exclusion helper is not enough if the candidate source only queries
enemy entities. Friendly-lock turrets need friendly candidates to enter
the targeting batch before exclusions and scoring can operate.

Fix direction:
Teach the candidate collection path to gather friendly candidates, enemy
candidates, or both based on each turret's relationship exclusions. Keep
the broadphase efficient: avoid scanning both ownership sets for turrets
that exclude one side, and batch/query by relationship when possible.
Make sure priority-target validation can accept friendly targets when the
turret policy allows them.

Done when: construction-style turrets can acquire friendly targets from
the normal targeting path, combat turrets still avoid friendly targets,
and mixed-policy turrets can consider both without a separate code path.

LOCK-ON-06 — Represent turret targets distinctly enough for exclusions.

Evidence:
- The design treats buildings, units, and turrets as separate level-0
  target families. It also allows named turret exclusions through
  exclude_lock_on_level_1_turrets.

Why it matters:
If the targeting slab only sees entity ids, a turret mounted on a unit or
building is hard to filter separately from its host. Mirror turrets need
to lock onto enemy turret targets while ignoring the host body as a unit
or building target.

Fix direction:
Audit the current lockOnToTurret / target aim-point flow and decide
whether candidate identity must include a target turret index in addition
to target entity id. If needed, add target-turret metadata to candidate
rows, FSM state, snapshots, and aim-point resolution. Preserve existing
entity-id target behavior for body locks.

Done when: a turret-family exclusion targets turret locks, not host
entity locks by accident, and level-1 turret blueprint exclusions can
reject an individual mounted turret without rejecting the whole host
unless that is explicitly authored.

LOCK-ON-07 — Add parity/scenario coverage for lock-on exclusions.

Evidence:
- The exclusion rules are compositional and easy to regress: relationship
  filters, entity-family filters, named filters, priority targets, stale
  locks, and mirror-only behavior all interact.

Why it matters:
These rules live in the hot targeting path and will be hard to validate
by visual playtesting alone. Regressions could look like turrets not
firing, construction beams grabbing enemies, mirrors reflecting the wrong
weapon, or commands overriding blueprint policy.

Fix direction:
Add focused tests or deterministic harness scenarios for:
  - no exclusions can lock friendly/enemy buildings, units, and turrets;
  - combat turret excludes friendly_entities;
  - construction turret excludes enemy_entities;
  - mirror turret excludes friendly_entities, buildings, and units;
  - level-1 building/unit/turret named exclusions reject only those names;
  - priority target and existing-lock validation respect the same rules.

Done when: the scenarios pass through the same targeting path used in a
real battle, and failures identify which exclusion axis regressed.

AIM-08.6 — Port the writeback + tear down the parity harness
  After .5, snapshot serialization and rendering still read JS
  Entity.combat.turrets[]. Drop a thin writeback pass that copies
  slab → JS Turret each tick (for snapshot encoder consumption), and
  remove the parity harness from .0. The slab is now the source of
  truth.

  Done when: the TS targeting pass is gone from the working tree, the
  combat slabs are the per-tick authority, and the parity harness has
  been deleted in the same PR as its last consumer.
