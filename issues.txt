[BA-PERF-002] Remote direct snapshot publishing still falls back to DTO-backed non-entity sections

Severity: High

Evidence:
- src/game/server/ServerSnapshotDirectWirePreencoder.ts now lets preencode-wire listeners emit entity, minimap, economy, and spray rows directly into the Rust/WASM snapshot encoder without first populating their NetworkServerSnapshot DTO arrays/maps, but it still delegates resource movement, audio, projectile, scan-pulse, and grid sections to existing DTO serializers.
- src/game/server/ServerSnapshotPublisher.ts still falls back to serializeGameState() when the direct path is unavailable, when WASM is not initialized, or when the entity rows require raw fallback cases such as inactive turrets, interrupted build DTOs, or factory routes.
- The direct payload reports materializationKind='direct' plus encoderKind/rawEntityCount/rawTopLevelKeys diagnostics, but rawTopLevelKeys still records DTO-backed sections that the Rust writer cannot yet own directly.

Risk:
Remote snapshot publishing no longer has to materialize entity DTO arrays on the successful direct path, but non-entity sections can still allocate DTO graphs before the Rust/WASM or typed-row encoder emits bytes. Existing pooling and row sources reduce allocation and packing cost, but the hot path remains partially listener-oriented TypeScript orchestration instead of a complete wire-first writer. budget_design_philosophy.html says snapshots should move toward compact binary/typed-buffer output and that TypeScript should orchestrate while Rust/WASM computes hot loops.

Required fix:
Extend the remote direct preencode writer so projectile events, resource movements, audio events, scan pulses, static terrain/buildability, removals, game state, and server metadata are emitted from WorldState/visibility/baseline decisions into Rust/WASM or owned typed buffers without DTO arrays. Keep serializeGameState() as the local/in-memory and parity fallback until the direct writer covers all full snapshot and delta cases. Use the payload materializationKind/encoderKind/rawEntityCount/rawTopLevelKeys diagnostics to retire raw fallback sections before deleting the DTO path.

[BA-ARCH-001] Large hot-path files make lifecycle, pooling, and Rust/WASM migration hard to audit

Severity: Medium

Evidence:
- src/game/sim/Simulation.ts is 2535 lines.
- src/game/render3d/Input3DManager.ts is 2073 lines.
- src/game/scenes/RtsScene3D.ts is 1644 lines.
- src/game/render3d/Render3DEntities.ts is 1431 lines.
- src/game/render3d/BuildingAnimationController3D.ts is 1356 lines.
- src/game/sim/WorldState.ts is 1344 lines.
- src/game/server/GameServer.ts is 1081 lines.
- budget_design_philosophy.html explicitly says large files should be split into coherent helpers when touched.

Risk:
Cleanup ownership, retained pools, hot loops, and authoritative/client boundaries are spread across large files. That raises the chance of missed teardown, duplicate runtime paths, accidental per-frame allocation, and stalled migration of TypeScript hot loops into Rust/WASM.

Required fix:
When any of these files are touched for the issues above, split along existing ownership boundaries instead of adding more logic in place. Prioritize lifecycle/teardown helpers, snapshot intake/publisher modules, unit mesh lifecycle vs pose/locomotion updates, input command builders, and simulation subsystems that are candidates for Rust/WASM.
