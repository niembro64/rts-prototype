RENDERING PIPELINE AUDIT - 2026-06-04

1. Unit drawing does not obey the client bar all/camera/pad render scope.

Evidence:
- Render3DEntities.updateUnits() reads clientViewState.getUnits() and updates
  every unit mesh, locomotion rig, turret target, fading mesh, detail instance,
  path preview, shield state, and selection-related visual state at render
  cadence.
- The file explicitly documents that pose, locomotion, and turret updates ignore
  the render scope so screen-edge units do not pop.
- The client bar already has a RENDER option for ALL/WIN/PAD. The authored
  config stores those as all/window/padded, which correspond to the intended
  all/camera/pad draw choices.
- src/game/network/ClientViewState.ts has collectActiveUnitRenderEntities(), but
  the renderer does not use it.

Problem:
This contradicts budget_design_philosophy.html: off-screen units should skip
frontend locomotion/pose work for free, and TypeScript should not own per-unit
hot loops at battle scale. The current path scales with every client-visible
unit regardless of whether the client bar is set to draw all units, only the
camera footprint, or the camera footprint plus pad.

Required fix:
Use the client bar render option as the source of truth for what gets drawn:
- all: draw/update all client-visible render entities.
- camera: draw/update only entities inside the current camera footprint.
- pad: draw/update entities inside the camera footprint plus the configured
  padding margin.
Feed expensive unit visual work from that mode-selected render packet instead
of getUnits(). The packet should still include explicit visual exceptions when
needed for selected, hovered, active-prediction, dying/fading, recently fired,
or relevant HUD/selection state. The packet should come from a spatially indexed
or WASM/typed-buffer-backed source, not a fresh full-world JavaScript scan. Keep
gameplay prediction independent of the draw scope.

2. Scoped render phase still performs whole-world entity scans.

Evidence:
- src/game/scenes/helpers/RtsScene3DRenderPhase.ts prepareEntityLists() loops
  over clientViewState.getUnitsAndBuildings() in scoped modes and filters each
  entity with entityInRenderScope().
- The same render phase separately scans projectiles and other render caches to
  build HUD, shield, ground-print, and nameplate packets.
- src/game/render3d/BuildingEntityRenderer3D.ts still updates all buildings each
  frame before scoped render-phase filtering.

Problem:
The render phase has typed packet buffers, but packet population still does
JavaScript full-list iteration each frame. Scoped render modes reduce draw work
after the scan rather than avoiding the scan.

Required fix:
Back scoped render packets with spatially culled render lists. Keep the existing
packet objects and scratch arrays, but fill them from visible-cell/entity
indices plus explicit exceptions for selected, hovered, and recently relevant
entities. Building rendering should get the same treatment instead of updating
all buildings every frame.

3. The renderer is still centered on Three.js instead of the target raw
   WebGPU/WGSL architecture.

Evidence:
- src/game/render3d/ThreeApp.ts owns the main renderer and the render tree is
  composed from many Three.js-specific renderers, meshes, materials, and
  instanced helpers.
- budget_design_philosophy.html says Three.js is legacy scaffolding and the
  long-term renderer should use owned buffers, explicit WGSL, and predictable
  batched data flow.

Problem:
The current architecture makes it easy to add more object-oriented Three.js
render paths, but the project target is data-oriented rendering. Each new
Three.js path increases migration cost and can hide JS allocation or buffer
upload costs behind library abstractions.

Required fix:
Stop adding new high-count visuals as bespoke Three.js object graphs. Identify
and migrate the highest-count paths first, such as unit detail instances, legs,
projectiles, terrain/fog/shield overlays, HUD/nameplate packets, and support
surface debug visualization. Replacement paths should use owned typed buffers,
explicit update ranges, and delete the old Three.js path once the new path is
active.

4. Auxiliary WebGL renderers are outside the main renderer singleton budget.

Evidence:
- src/game/lifecycle/sessionSingleton.ts limits active main GameInstance and
  renderer ownership.
- Loading previews, commander/avatar previews, and the world-direction HUD create
  their own WebGLRenderer instances outside that singleton.

Problem:
These contexts appear to have teardown paths, so this is not currently a proven
leak. It is still an unbudgeted GPU-memory/context-count risk on low-end
devices, especially during loading or UI-heavy screens where preview renderers
and the main renderer may overlap.

Required fix:
Add renderer/context-count telemetry and an explicit auxiliary-renderer budget.
Prefer shared rendering, canvas-2D, or a disabled/low-memory fallback for UI
previews and HUDs when the main game renderer is active.
