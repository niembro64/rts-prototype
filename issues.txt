7.0 [MAJOR, migration surface] The last of the projectile/damage/death
migration: move line/swept/area candidate PACKING and death-explosion CHAINING
into Rust. NOT blocked on the full ECS migration — the per-entity data already
exists in the CombatTargetingPool slab and can be reused. Gated only on a few
concrete coherence/plumbing items (below); land it carefully because it is core
hit-detection.

Verified against code 2026-06-02 — all the pure-computation kernels already live
in Rust/WASM: projectile integration, homing/terrain-follow thrust, reflector
intersections + reflected response, swept body/building/projectile/turret hit
selection (projectile_hitbox_sweep_batch), terminal consequence classification
(projectile_terminal_consequence_batch), terminal effect planning
(projectile_terminal_effect_plan_batch), submunition launch-velocity scatter
(projectile_submunition_launch_velocity_batch), line/swept segment hits
(damage_segment_hits_batch), splash sphere/AABB overlap + slice-cone + knockback
direction (damage_area_overlap_batch), direct-hit/splash HP/kill output math
(damage_apply_batch), and death-cleanup HP/materialization classification +
compact removal diffs (death_cleanup_diff_batch). All damage HP write-back
(beam/line, swept, area, death-explosion) shares the one batched
damage_apply_batch path (queueDamageToEntityBatch + flushDamageBatch) — there is
no second single-entity apply path to migrate.

What remains in TypeScript is mostly DIFF APPLICATION, which the C1 contract
keeps in TS on purpose: applying terminal-effect flags (despawn / hasExploded /
audio events), materializing submunitions via world.createProjectile, writing
returned HP onto JS entities, emitting death/audio events, and removing JS
entities from WorldState.

REQUIRED FIX: move line/swept/area candidate PACKING and death-explosion CHAINING
into Rust. The data is already mirrored, so reuse it — do NOT build a parallel
slab. The CombatTargetingPool slab (lib.rs ~14943, stamped by
targetingInputStamping.ts) already holds, per unit/building/tower/projectile:
position, owner/team, current HP, hitbox radius, building AABB half-extents,
entity family, and alive flag — and already computes turret world-mount positions
+ turret hitbox radii. So damage packing iterates broadphase candidate slots and
reads the slab instead of re-reading JS entities.

LANDED (the pattern to follow for the rest): damage_area_candidates_batch — a
slab-reading drop-in for damage_area_overlap_batch, Rust-unit-tested for
byte-identical output — now backs applyAreaDamage's UNIT and BUILDING passes.
They pack the combat slot (spatialGrid.getSlot, which IS the slab slot) instead
of geometry, behind a DEV-only equality gate that console.errors the first
per-row divergence vs the legacy array path (tree-shaken from prod). The slot
mapping needs no new lookup — getSlot already returns the slab slot.

Remaining:
  1. PROJECTILE COHERENCE. The slab is stamped ONCE per tick in updateCombat,
     before physics; unit/building rows are coherent at the damage phases, but
     PROJECTILE rows are STALE — projectiles integrate in updateProjectiles AFTER
     the stamp. Re-stamp projectile rows into the slab right after
     spatialGrid.updateProjectiles, before checkProjectileCollisions, then move
     the projectile area pass + line/swept projectile targets onto the kernel.
  2. COLLISION RADIUS STAMP. entity_radius_collision was added to the slab for the
     kernel's SHOT branch but is not stamped yet. Stamp
     shotProfile.runtime.radius.collision for SHOT rows (units/buildings keep
     hitbox / targetRadius) so the SHOT branch matches what TS packs today.
  3. SEGMENT KERNEL. Add damage_segment_candidates_batch (slab-reading drop-in for
     damage_segment_hits_batch) and move applyLineDamage + applySweptDamage packing
     onto it, same DEV gate. Beam/swept hit UNIT/BUILDING (coherent now) and
     PROJECTILE (needs step 1). Also migrate the area unit TURRET sub-hitbox
     fallback (the slab already has turret_mount_x/y/z + turret_radius_hitbox).
  4. SLOT FAST-PATH (perf). TS still loops candidates to call getSlot per hit. For
     the real win, have the broadphase hand slot-ids straight to the kernels so TS
     does zero per-candidate work.
  5. CHAINING. The slab is write-once per tick, so it cannot supply post-damage HP
     mid-chain. Keep kill accumulation on the live damage_apply_batch result (as
     today); move only the per-blast candidate packing onto the kernels above.

VERIFY CAREFULLY: this is core hit-detection. A subtle packing/coherence bug
shifts splash radii or projectile intercepts in ways a visual spot-check won't
catch. Gate the cutover on before/after equality (same kill sets, same hit sets)
per damage type, not a blind switch.
