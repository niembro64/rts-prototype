[BA-NET-001] Non-snapshot PeerJS sends bypass data-channel backpressure

Severity: High

Evidence:
- src/game/network/NetworkSnapshotTransport.ts:341-364 checks RTCDataChannel.bufferedAmount and drops snapshots above SNAPSHOT_BACKPRESSURE_DROP_BYTES.
- src/game/network/NetworkManager.ts:94-117 routes command and heartbeat sends through safeSend().
- src/game/network/NetworkManager.ts:928-951 safeSend() only checks conn.open and try/catches conn.send(message); it does not inspect bufferedAmount or coalesce/drop noncritical messages.
- src/game/network/NetworkManager.ts:586-617, 665, 738, 833, 868, 877, 904, 910, 982, 990, and 1031 send roster/lobby/heartbeat/command/state messages through this path.

Risk:
Snapshot traffic is bounded, but roster updates, lobby messages, heartbeat traffic, resync requests, and command messages can still pile up behind a saturated RTCDataChannel. budget_design_philosophy.html requires backpressure to drop or coarsen before queues grow without bound. Commands are authoritative input, so blindly dropping them would be wrong; they need a separate bounded/rate-limited policy.

Required fix:
Centralize send budgeting for all PeerJS messages. Coalesce or drop stale noncritical messages such as heartbeat, playerInfo, and lobbySettings. Rate-limit or reject commands explicitly instead of letting the browser send queue grow. Add bufferedAmount telemetry by message class.

[BA-PERF-002] Snapshot publishing still builds JS DTO object graphs before binary packing

Severity: High

Evidence:
- src/game/server/ServerSnapshotPublisher.ts:216-303 serializes a NetworkServerSnapshot object per listener via serializeGameState().
- src/game/network/stateSerializer.ts:265-380 walks WorldState and fills JS DTO arrays/maps before the transport encodes them.
- src/game/network/NetworkSnapshotTransport.ts:74-76 then calls encodeNetworkSnapshot(state) to convert that JS graph into a wire buffer.

Risk:
The snapshot path still pays JS object graph construction plus binary packing. Existing pooling helps, but the hot path remains TypeScript-owned and listener-oriented. budget_design_philosophy.html says snapshots should move toward compact binary/typed-buffer output and that TypeScript should orchestrate while Rust/WASM computes hot loops.

Required fix:
Continue migrating snapshot deltas/keyframes toward a Rust/WASM or typed-buffer writer that emits binary rows directly from SoA/baseline state. Keep the existing serializer only as a parity fallback until the direct writer covers full snapshots, deltas, visibility filtering, projectile events, minimap, and metadata.

[BA-PERF-003] Scoped rendering disposes off-scope meshes and rebuilds them when the camera pans back

Severity: Medium

Evidence:
- src/game/render3d/Render3DEntities.ts:829-839 enables pruneUnits whenever scopedRender is true.
- src/game/render3d/Render3DEntities.ts:905-920 builds a unit mesh when no mesh is present.
- src/game/render3d/Render3DEntities.ts:1293-1303 destroys unseen unit meshes in scopedRender mode.
- src/game/render3d/BuildingEntityRenderer3D.ts:470-482 destroys unseen building meshes in scopedRender mode.
- src/game/render3d/BuildingEntityRenderer3D.ts:545-560 rebuilds a building mesh when it re-enters scope.

Risk:
The renderer correctly avoids updating off-screen units, but it does so by tearing down and later recreating Three.js object graphs as the camera moves. Large pan/zoom movements across a battlefield can create GC churn and GPU resource churn. The philosophy doc asks for renderer-owned buffers, instancing, and work scaling with visible units; it does not require disposal/reallocation for every scoped visibility transition.

Required fix:
Replace immediate scoped dispose/rebuild with dormant mesh pools, hidden/deactivated records, or an LRU retention budget. Dispose on entity death, actual view removal, style/geometry invalidation, session teardown, or when an explicit memory budget is exceeded. Add telemetry for scoped mesh destroys/rebuilds per second.

[BA-ARCH-001] Large hot-path files make lifecycle, pooling, and Rust/WASM migration hard to audit

Severity: Medium

Evidence:
- src/game/sim/Simulation.ts is 2535 lines.
- src/game/render3d/Input3DManager.ts is 2073 lines.
- src/game/scenes/RtsScene3D.ts is 1644 lines.
- src/game/render3d/Render3DEntities.ts is 1431 lines.
- src/game/render3d/BuildingAnimationController3D.ts is 1356 lines.
- src/game/sim/WorldState.ts is 1344 lines.
- src/game/server/GameServer.ts is 1081 lines.
- budget_design_philosophy.html explicitly says large files should be split into coherent helpers when touched.

Risk:
Cleanup ownership, retained pools, hot loops, and authoritative/client boundaries are spread across large files. That raises the chance of missed teardown, duplicate runtime paths, accidental per-frame allocation, and stalled migration of TypeScript hot loops into Rust/WASM.

Required fix:
When any of these files are touched for the issues above, split along existing ownership boundaries instead of adding more logic in place. Prioritize lifecycle/teardown helpers, snapshot intake/publisher modules, unit mesh lifecycle vs pose/locomotion updates, input command builders, and simulation subsystems that are candidates for Rust/WASM.
