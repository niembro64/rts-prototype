RENDERING PIPELINE AUDIT - 2026-06-04

1. Some high-count renderer paths still lean on object-style Three.js APIs
   instead of owned buffer contracts.

Evidence:
- src/game/render3d/ThreeApp.ts owns the main renderer and the render tree is
  composed from many Three.js-specific renderers, meshes, materials, and
  instanced helpers.
- budget_design_philosophy.html now allows Three.js as the renderer backend,
  but still requires high-count paths to use owned typed buffers, explicit
  update ranges, custom shader code, predictable batching, and measurable
  draw-call behavior.
- Recent projectile work moved one hot path from setMatrixAt/setColorAt calls
  toward direct typed-buffer writes, but similar patterns remain in unit
  detail instances, legs, terrain/fog/shield overlays, HUD/nameplate packets,
  and support surface debug visualization.

Problem:
The current architecture can still hide allocation, culling, material, or
buffer-upload costs behind object-oriented Three.js helper APIs. Three.js is
allowed, but high-count paths must expose the data flow clearly enough to
audit and eventually move to lower-level GPU APIs if needed.

Required fix:
Stop adding new high-count visuals as bespoke Three.js object graphs. Continue
migrating the highest-count paths to owned typed buffers, explicit update
ranges, stable render packets, and custom shader contracts. Delete older
per-object or duplicate paths once the owned-buffer path is active.
