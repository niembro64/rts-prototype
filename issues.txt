ATOMIC BLUEPRINT / RUNTIME BOUNDARY AUDIT (NEW, found 2026-05-06)

Goal for this section: units, buildings, turrets, locomotion, and shots should
each have one canonical definition path. Runtime systems should consume built
profiles/components, not reinterpret blueprint pieces independently.

8. Render3DEntities is a major god file and owns too many atomic concepts.
   References:
   - src/game/render3d/Render3DEntities.ts:1
   - src/game/render3d/Render3DEntities.ts:2835
   - src/game/render3d/BuildingEntityRenderer3D.ts:1
   - file size: 2835 lines

   This file currently owns or coordinates unit meshes, building meshes, turret
   attachment transforms, projectile visuals, commander-specific mesh assembly,
   LOD shells, instanced pools, and cleanup. Projectile rendering,
   selection/radius/range overlays, and construction emitter/factory visual
   control have been split out. Entity mesh state, shared render tier helpers,
   commander visual kit assembly, animated building control, and building mesh
   creation have also been split out. Building mesh creation and per-frame
   building transform/detail/turret updates now live in
   `BuildingEntityRenderer3D`, but the orchestration file is still too large to keep
   blueprint/rendering boundaries clean.

   Fix direction: split into small modules: `UnitEntityRenderer`,
   `BuildingEntityRenderer`, and `TurretAttachmentRenderer`. Leave
   `Render3DEntities` as an orchestration facade. Next useful cut: move
   unit instanced pool allocation/flush into a dedicated renderer.

10. ClientViewState still mixes snapshot application, prediction stepping, and projectile behavior.
    References:
    - src/game/network/ClientViewState.ts:1
    - src/game/network/ClientViewState.ts:476
    - src/game/network/ClientViewState.ts:1159
    - src/game/network/ClientViewState.ts:1354
    - src/game/network/ClientProjectilePrediction.ts:1
    - src/game/network/ClientUnitPrediction.ts:1
    - file size: 1848 lines

    The client network state object now handles entity cloning, delta/full-snap
    application, beam prediction, and prediction orchestration. Build-state
    application, projectile spawn queuing, selection bookkeeping, prediction
    LOD cadence/storage, traveling projectile prediction, and unit visual /
    turret / force-field prediction have been split out, but unrelated behavior
    still shares one large mutable class.

    Fix direction: split into `SnapshotApplier`, `ClientPredictionStepper`, and
    `ProjectilePredictionStore`. Keep `ClientViewState` as the container/facade.

11. GameServer still mixes bootstrap, tick, physics forces, snapshots, and commands.
    References:
    - src/game/server/GameServer.ts:1
    - src/game/server/GameServer.ts:433
    - src/game/server/GameServer.ts:554
    - src/game/server/GameServer.ts:618
    - src/game/server/GameServer.ts:1028
    - src/game/server/ServerDebugGridPublisher.ts:1
    - src/game/server/ServerSnapshotMetaBuilder.ts:1
    - file size: 1530 lines

    The host server class owns map setup, spawn setup, tick scheduling,
    simulation stepping, force application, physics sync, snapshot publication,
    and command entrypoints. Spatial-grid debug snapshot publishing has been
    split into `ServerDebugGridPublisher`, and server snapshot metadata/time
    formatting has been split into `ServerSnapshotMetaBuilder`, but the
    remaining server class is still too broad. This makes server performance
    fixes harder to isolate and test.

    Fix direction: extract `ServerBootstrap`, `ServerTickLoop`,
    `UnitForceSystem`, and `ServerSnapshotPublisher`.


13. Debris rendering re-interprets unit/turret/shot blueprints instead of reusing render fragments.
    References:
    - src/game/render3d/Debris3D.ts:28
    - src/game/render3d/Debris3D.ts:537
    - src/game/render3d/Debris3D.ts:664
    - src/game/render3d/Debris3D.ts:710
    - file size: 1172 lines

    Death debris reaches back into unit, turret, and shot blueprints to decide
    which pieces exist. This is a second renderer-specific interpretation of the
    same visual structure used by live entities.

    Fix direction: live unit/building/turret render factories should emit a
    small canonical `RenderFragmentBlueprint`/`DebrisFragmentProfile`; debris
    should consume that instead of reading all gameplay blueprints directly.

14. Locomotion3D mixes mesh construction, animation, terrain sampling, and rig state.
    References:
    - src/game/render3d/Locomotion3D.ts:1
    - src/game/render3d/Locomotion3D.ts:334
    - src/game/render3d/Locomotion3D.ts:711
    - src/game/render3d/Locomotion3D.ts:1041
    - file size: 1137 lines

    Legs, treads, wheels, IK-ish foot placement, terrain sampling, debug rest
    spheres, and instanced leg pools all share one file. That makes it too easy
    for locomotion rendering to drift from locomotion physics/blueprints.

    Fix direction: split by responsibility: `LegRig3D`, `TreadRig3D`,
    `WheelRig3D`, `LocomotionTerrainSampler`, and `LocomotionAnimationState`.

16. Type files are too broad for the atomic boundaries we want.
    References:
    - src/types/sim.ts:1
    - src/types/blueprints.ts:1
    - file sizes: 852 and 541 lines

    `sim.ts` and `blueprints.ts` both mix several domains: entities, combat,
    shots, construction, economy, locomotion, buildings, and units. Large shared
    type files make cross-domain imports feel cheap and hide coupling.
    `ResourceCost`, core building identifiers, entity ids, command/waypoint
    types, combat range types, and runtime locomotion type have been split into
    `economyTypes`, `buildingTypes`, `entityTypes`, `commandTypes`,
    `combatTypes`, and `locomotionTypes`, with barrel exports preserved for
    compatibility.

    Fix direction: split into domain files (`shotTypes`, `constructionTypes`)
    with barrel exports for backwards compatibility during migration.
