BUDGET ANNIHILATION — ISSUES

This is "issues.txt" for this game "budget-annihilation" (i.e. "rts-prototype").

Your task is to fix the issues documented here.

Errors, Problems, Inconsistencies, TODOs, etc are listed here. As we fix
something we should completely delete it from this file. You should also delete
an item if you know for a fact it has been addressed. Remember that our code may
have changed since the issue was written; check the code and if the issue is no
longer relevant, delete it.

Do not document "PROGRESS" in this file, just document or update issues that need to be resolved. If you see any "progress" information in this file, just remove it, or convert it into something actionable. Just documenting progress is not helpful - update the issue with the current state and the required fixes for it.

Please always commit and push before and after any code changes we make.

BAR COMMAND AND UI PARITY GAPS

Baseline: budget_design_philosophy.html says player-facing behavior defaults to
Beyond-All-Reason unless explicitly overridden. BAR references for this backlog:
https://www.beyondallreason.info/commands-20 and
https://github.com/beyond-all-reason/Beyond-All-Reason/tree/master/luaui/configs/hotkeys.

Issue: the shared command-hotkey registry and in-game preset switcher only cover
commands that already exist in rts-prototype, with prototype, BAR-grid, and
BAR-legacy presets for that subset. Finish the BAR-compatible keymap and command
UI layer by importing the complete BAR grid/legacy bindings, adding runtime
multi-key sequence dispatch, exposing a custom binding editor, surfacing
conflict handling to the player, and adding command buttons for missing BAR
commands so visible hotkeys match the active keymap across the whole command
surface.

Issue: the selection command surface is still short of BAR. Select all owned
units, select all matching owned entities, select idle builders, select waiting
units, select same type only, select commander focus/add, and mixed-selection
narrowing, select all matching units in view, select all units from previous
selection, and select only mobile units have command-card or hotkey affordances.
Add select idle transports, split army, deselect-from-box, invert/loop
selection, and the BAR select-box modifier semantics for
Shift/Ctrl/Alt/Space/Z.

Issue: control groups still lack several BAR behaviors. Group slots 0-9,
Ctrl/Cmd+Shift group add, Ctrl/Cmd+Alt group select-toggle, and Ctrl/Cmd+`
remove-selected-from-groups, and double-tap group focus exist. Implement
auto-group creation/removal, auto-group preset save/load/show, factory preset
integration, and any required spectator/team selection behavior that maps to
rts-prototype's lobby and player-switching model.

Issue: movement and formation commands are incomplete. Keep the existing
right-click move/fight/patrol and line-drag behavior, but add BAR's explicit
Move command, Move and assume formation, Move in formation, formation-preserving
movement semantics, slow/fast movement modifiers where applicable, and visual
formation previews. Translate these to the 3D physics model so formation targets
respect mass, collision, terrain height, slopes, water, and pathability.

Issue: battle and targeting commands are incomplete. rts-prototype has visual
buttons/hotkeys for Attack, fixed-radius Attack Area, Attack Ground, Fight,
Guard, Patrol, and target set/cancel for combat units and towers. Add BAR
Attack Circle, Attack Line, Target Set No Ground, user-shaped attack
areas/lines/circles, and command-card access for any remaining battle commands
that become supported by simulation.

Issue: construction commands are missing most BAR build ergonomics. Implement
the BAR grid build menu categories, grid-menu cell hotkeys, legacy quick-build
Z/X/C/V cycling, construction shortcuts, build line, build grid, build border,
build split, build spacing increase/decrease, build facing/rotation controls,
area metal extractor placement, upgrade T1 mex area, upgrade T1 -> T2, and
visual previews for every build shape. The current allowed-building strip,
number keys, single-placement ghost, and Shift queueing do not satisfy BAR
fallback behavior.

Issue: queue editing and command insertion are incomplete. BAR supports normal
queueing, queue add in front, queue to next, commandinsert/prepend-between,
skip current command, cancel last command, factory guard, stop production, and
queued variants of several orders. rts-prototype only exposes Shift queue,
remove-last-queued-order, clear-queued-orders, wait queueing, and basic factory
rally. Implement the missing queue semantics with server authorization and
clear visual queue editing feedback.

Issue: repair and reclaim are incomplete. Fixed-radius repair area and
fixed-radius reclaim area via the Reclaim command exist. Add BAR-style area
reclaim by right-drag/circle, user-sized repair and reclaim areas,
reclaim-selected behavior where appropriate, and reclaim field highlighting.

Issue: behavior/state toggles are incomplete. BAR exposes cloak/want-cloak,
gather wait, on/off, stateful on/off, repeat, stateful repeat, stance/fire
state, positioning/move state, trajectory toggle, stop, wait, factory guard,
and related stateful command buttons. rts-prototype currently has only a simple
fire enabled toggle, producer-building on/off, wait, stop, and self-destruct.
Implement the missing states in simulation, snapshots, command authorization,
selection UI, hotkeys, and visible state indicators.

Issue: special abilities are incomplete. Add BAR Capture, DGUN, Load Transport,
Unload Transport, Resurrect, Resurrect Area, Self Destruct, manual fire, manual
launch, and any unit-specific special buttons. rts-prototype has commander
DGun and self-destruct, but no capture, resurrection, transports, load/unload
logic, manual launch weapons, or unit-specific ability command-card support.

Issue: factory and production UI are incomplete. BAR has factory production
controls beyond selecting one repeat-build unit: stop production, repeat/queue
state, factory queue mode/quotas, factory presets save/load/show, production
queue editing, rally/route display, factory guard, and complete unit build
menus. rts-prototype needs a BAR-equivalent factory panel and hotkeys, translated
to authoritative-server commands and 3D rally/path previews.

Issue: map, communication, and game UI commands are missing. Implement BAR
commands and buttons for chat, draw on map, draw label, erase drawings, ping to
map, map overview switch, map details info, show map elevation, show map metal,
show unit pathing, LOS/radar toggles, options menu, pause, game-speed adjust,
volume controls, and any equivalent lobby/spectator-safe variants.
rts-prototype has a minimap, resource bar, debug/control bars,
fullscreen toggle, screenshot capture, audio mute/sound category toggles,
ping/scan commands, a go-to-last-ping camera control, and a show/hide UI chrome
toggle, but not the BAR game UI command surface.

Issue: camera controls are missing BAR parity. FOV controls, selected-unit
follow tracking, and arrow/numpad camera movement exist. Add camera mode
switching (overhead/TA/Spring-style equivalents as appropriate for this 3D
prototype), camera flip, camera anchors set/focus, overview/minimap camera
switching, and picture-in-picture behavior or a documented 3D-focused
replacement. Existing wheel zoom, middle-drag pan/orbit, touch controls,
minimap click-to-move camera, and world-direction HUD are not enough.

Issue: BAR gameplay functionality tied to missing commands is absent. Implement
or explicitly redesign the systems needed by the missing command surface:
transports, capture, resurrection/wreck fields, cloak/detection, radar/LOS UI
states, unit stances, trajectory weapons, manual-launch weapons, factory queue
management, build-shape placement, mex upgrade workflows, unit stats/details,
chat/map drawing, pause/speed controls, spectators/replays where applicable,
and complete faction/unit/factory/building rosters. Missing commands should not
be added as dead UI; their underlying authoritative simulation, client
prediction, snapshots, and render feedback must exist.

Issue: BAR parity must be extended through the 3D/physics-first design, not
implemented as flat 2D shortcuts. Attack areas/lines/circles need 3D ground
height and ballistic previews; build lines/grids/borders need slope, support
surface, collision, and resource-cell validation; formations need mass,
acceleration, collision, and terrain-aware spacing; transport/load/unload,
reclaim, resurrection, capture, and manual fire need physical presentation and
authoritative physics interactions. Document any intentional departure from BAR
in budget_design_philosophy.html before omitting a BAR behavior.
