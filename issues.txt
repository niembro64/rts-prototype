Audited the whole repo against budget_design_philosophy.html + the BAR repo
(../Beyond-All-Reason) across 11 domains; every finding below was independently

optimization, memory-leak, missing-functionality, BAR-gap, SSoT/dead-path.

--- CORRECTNESS / DETERMINISM ---

[1] Guided shots run their own replacement-target acquisition after launch
    Category: correctness (determinism-relevant) | Severity: HIGH
    Evidence: src/game/sim/combat/projectileSystem.ts:1644-1651 calls
      findReplacementHomingTarget whenever the inherited homing target is dead/
      gone, then commits it into proj.homingTargetId at :1654.
      findReplacementHomingTarget (projectileSystem.ts:390-426) picks the nearest
      live enemy via spatialGrid.queryEnemyEntitiesInRadius with NO inclusion-
      object filtering. isLiveHomingTarget (combatUtils.ts:85-94) flips to false
      the instant the inherited target's hp<=0, so re-acquisition fires
      immediately on target death.
    Design: "Lock-on policy lives on turrets, not guided shots"
      (budget_design_philosophy.html:1989-1992): a guided shot "must not run its
      own target acquisition pass, scan for replacements, or choose a new victim
      after launch. If the inherited target no longer exists, it loses guidance
      and continues on its current flight path ... until it hits something or
      expires."
    Problem: A missile fired by a units-only turret can re-home onto a building or
      another projectile, and it changes who gets hit — a behavior bug in lockstep
      combat, not cosmetic. This is the single authoritative server projectile
      loop (NOT a determinism mirror), so the fix is safe.
    Fix: Delete the replacement-acquisition fallback for guided shots. When the
      inherited homingTargetId is no longer live, set proj.homingTargetId =
      NO_ENTITY_ID, stop applying steering thrust (fly ballistic). Remove
      findReplacementHomingTarget from the per-tick homing path (it is also a
      per-projectile spatial query running every tick for every orphaned missile).

[2] D-gun / instant ability energy spend never surfaces as a pylon ball
    Category: correctness (economy invariant) / missing-functionality | Sev: MED
    Evidence: src/game/sim/economy.ts:211 spendInstant hard-codes
      amountPerSecond:0; resourceMovement.ts:119 keeps the real value only in
      `amount`; stateSerializerResourceMovements.ts:97 drops any movement with
      amountPerSecond<=0 SERVER-SIDE before it is ever serialized; a secondary

    Design: "Resource Movement Flows Through Pylons" / "Hard invariant: resources

    Fix: Carry the burst as a real rate — pass amountPerSecond = amount/dtSec (or
      a fixed short visual rate) from spendInstant so the client emits a brief
      outbound energy pylon burst on the commander; OR add an explicit one-shot
      impulse field to ResourceMovement the renderer treats as a short spray.

[3] Beam-vertex EMA channel split: position rides the BEAM knob, velocity rides

    Category: correctness (prediction channels) | Severity: MEDIUM
    Evidence: src/game/network/ClientPredictionStepper.ts:361-363 builds movPos/

--- OPTIMIZATION (5,000-unit scale target) ---

[4] Per-tick support-surface sampling is brute-force O(awake x dynamicBodies) +
    2 x O(awake x staticBodies) with no spatial index
    Category: optimization | Severity: HIGH
    Evidence: src/game/server/PhysicsEngine3D.ts integrate() (:856) runs both
      sampleIntegrationSupportSurfaces (:1114) and refreshStepSupportIgnoredStatic
      Slots (:1141) every step. findDynamicSupportSurface (:1022) is a full
      this.dynamicBodies scan; findStaticSupportSurface (:982) is a full
      this.staticBodies scan called TWICE per awake body (via findSupportSurface
      :1062-1063 and again at :965). dynamicBodies/staticBodies are plain arrays
      (:395-396). Neighbor-contact resolvers already use the WASM cell-walk
      broadphase (:1150) — support sampling alone does not.
    Design: Physics Model / "Full 3D, not decorative altitude" — units stand on
      buildings and on each other, so support sampling must scale.
    Fix: Index static cuboids + discTop dynamic bodies in a spatial grid (reuse
      the existing broadphase / spatial_grid.rs cell-walk) and query only cells
      overlapping each body's footprint. Merge the two static scans so static
      support is found once per body per tick, not twice. TS-side path; no
      determinism mirror touched.

[5] Client projectile visual prediction crosses the WASM boundary once per
    projectile per frame and allocates a fresh closure per projectile per frame
    Category: optimization | Severity: MEDIUM
    Evidence: src/game/network/ClientPredictionStepper.ts:477-524 loops per
      projectile and allocates a new `getEntity: (id) => entities.get(id)` arrow
      at :507 (the only getEntity in the file — no reusable bound field).
      ClientProjectilePrediction.ts:470-471 (batchCount = hasTarget?2:1) + :510
      issue one sim.projectileIntegrateStepBatch PER projectile; :424 confirms no
      early-out. Contrast the batched unit path
      ClientUnitPrediction.ts:698 applyClientUnitVisualPredictionBatch (one WASM
      call for all entities). Driven per frame via RtsScene3D.ts:702 ->
      RtsScene3DPredictionPhase.ts:36.
    Design: Browser Performance Rules ("no per-unit/per-frame allocation"); "TS
      Coordinates, WASM Computes" ("hot data should cross the JS/WASM boundary as
      rarely as the design allows").
    Fix: Pack all active projectiles (+ optional correction targets) into the
      existing module-scratch typed arrays in one pass, then issue a SINGLE
      projectileIntegrateStepBatch for the frame (mirror the unit path). Hoist the
      getEntity lambda to a reusable bound field. Client-only visual path, distinct
      from the authoritative projectileSystem mirror.

[6] EntityCacheManager does a full copy-all + O(n log n) sort + ~25-list rebucket
    on every HP delta in a presentation snapshot, not just on structural changes
    Category: optimization | Severity: MEDIUM
    Evidence: src/game/sim/EntityCacheManager.ts:139-141 (clear sortedEntities,
      push all, .sort by id) and :111-275 (clear ~25 lists and re-bucket every
      entity). Invalidation: ClientViewState.ts:817-818 (cache.invalidate()) is
      triggered whenever snapClientNonVisualState is true, and
      ClientSnapshotApplier.ts:58-62/:136-140 set cacheDirty on ANY hp delta.
    Design: Scale Target — "Spatial structure beats broad scans"; full scans "are
      not the normal steady state for gameplay ticks." Entities are keyed by

    Fix: Maintain id order incrementally (insert at the sorted position on add,
      remove on delete) instead of copy-all + Array.sort on every invalidation; or
      patch only the buckets for entities that actually changed rather than
      clearing and rebuilding all ~25 lists.

[16] I think we need to abstract locomotion better. We need to have separate "friction" and "traction" for our locomotion against all three media: "ground", "air", "water". So basically every single unit should have "locomotionGroundTraction","locomotionGroundFriction", "locomotionAirTraction", "locomotionGroundFriction", "locomotionWaterTraction", "locomotionWaterFriction". All units should also have "locomotionGroundForce", "locomotionAirForce", "locomotionWaterForce", which means that they can push themselves if they are touching the ground, in the air, or under water, a specific amoount. The ground under water should count as ground. Use proper physics for these kinds of things, and also any entity should have only one of the air/water tractions or frictions applied if they are above or below the water. Think this through. Basically in this way we should be able to let units swim, or sink to the bottom of the water and walk or even walk on the top of the water, with specific values of these things. Also we will need a separate "swimHeightUpwardForce" (and all of the other random stuff similar to the flying and hover locomotions). To be honest we should fully abstract all of the locomotion stuff such taht any unit basically has all of the physics as possibilities, we can have zero for any of the values to make it act like a ground unit or an air unit. So basically we want to both add all physics for all units for "ground", "air", and "water", but we also want to fully abstract all of our units locomotion to have values for all possible locomotions - in this way we can mix and match and create strange combinations of locomotions that could walk, then fly, then swim for example. Let's fully abstract it and create an elegant exhaustive physics system for them. Let's actually add this requirement to our "budget_design_philosophy.html" document, find the appropriate places in that document to update with this new fully abstracted physics and locomotion systems.

FULLY-ABSTRACTED TURRET SYSTEM: SPAWN + RESOURCE-PYLON TURRETS (2026-06-25, user request)
--------------------------------------------------------------------------------
Replaces the single "construction turret" with a uniform turret taxonomy. Three
emitter families: COMBAT (emits shot/ray, existing); SPAWN turrets (create an
entity + fire an init laser); RESOURCE-PYLON turrets (move one resource in one
role). New turret kinds:
  turretSpawnBuildingsAndTowers, turretSpawnUnits,
  turretResourcePylonConstructionMetal, turretResourcePylonConstructionEnergy,
  turretResourcePylonExtractionMetal,   turretResourcePylonExtractionEnergy
Re-equip: builders (commander / construction drone / construction hovercraft /
fabricator) = a spawn turret + construction metal+energy pylons; queens =
turretSpawnUnits (instant + cooldown + launch); metal extractor = extraction-metal
pylon; solar/wind/geo = extraction-energy pylon. Construction pylons spin and
spray resource balls at the build target (auto-aimed, so they point down for
flying builders). Determinism: economy stays in energyDistribution; run
deterministic:audit each phase; keep encoder<->contract-test mirrors in sync.
