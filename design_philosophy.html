Design Philosophy: Two Simulations, One Authority, Full Physics

This prototype is a browser RTS inspired by the command scale and battlefield clarity of Beyond All Reason, but its technical identity is stricter: every meaningful physical interaction should reduce to mass, force, acceleration, velocity, position, collision, and correction.

Authority stays singular Client commands go directly to the authoritative simulation. The server validates and executes them. Clients never negotiate gameplay truth with each other.
Clients stay continuous The client runs a predicted local shadow on its own cadence, and the renderer samples that shadow every frame while server snapshots correct it only a few times per second.
Physics is the language Movement, recoil, collision, hover, beams, blast pressure, and terrain contact are force and mass problems before they are gameplay effects.

North Star

We are not building a lockstep RTS where every browser must simulate the same authoritative future from the same command stream. We are also not building a thin renderer that waits for the host to stream every position at display rate. The model is simpler and more scalable: run two simulations, let one own truth, and use the other as a responsive local shadow.

The authoritative simulation owns the real world: commands, ownership, economy, construction, targeting, damage, projectile hits, physics bodies, collision response, fog of war, victory, and persistent state. The client simulation owns presentation: local command feedback, prediction, smoothing, camera-visible detail, audio scheduling, and render-time interpolation.

The preferred implementation is shared code, not shared authority. Physical kernels and battlefield contracts should be described once and reused by both the authoritative sim and the client prediction sim where that reuse buys correctness: body integration, forces, projectile motion, terrain sampling, turret pose, hover, and collision math. That does not mean every browser must run every authoritative system. Economy, construction truth, target ownership, fog filtering, damage resolution, entity creation and removal, and victory stay host-owned unless a specific prediction problem proves otherwise.

The target networking feel is: commands arrive at the authoritative sim immediately, while the authoritative sim only needs to update the client sim about 5 to 10 times per second in normal play. Higher snapshot rates are a diagnostic or low-unit-count luxury, not the architecture.

Engine Ownership: No Libraries, In-House Rust

The end-state engine is something we own end-to-end. The simulation is Rust compiled to WebAssembly, living in rts-sim-wasm/. The renderer is raw WebGPU with WGSL shaders, driven directly. Networking glue, input plumbing, asset pipeline, audio mixer, math kernels, ECS, physics, collision, pathfinding, fog of war, snapshot serializer, prediction integrator — all written by us, for this game. The defining trait of this project is that we are building the engine, not assembling one.

The libraries currently in package.json (three, vue, peerjs, @msgpack/msgpack, @tonejs/midi, @tauri-apps/*) are scaffolding. They got the prototype walking; they are not the destination. The Rust sim crate and the in-tree TypeScript/shader code are the destination. Every change should leave the codebase with either the same library surface or less; never more.

  • The default answer to “should we add a library for this?” is no. Write it ourselves, preferably in Rust, ideally in the sim crate.
  • The default answer to “can we keep this library?” is only until we have time to replace it. The renderer migration off Three.js to raw WebGPU is the largest example; smaller libraries should follow the same path.
  • Adding a new npm or cargo dependency requires a deliberate, named justification — not a reflex. Vendoring the few lines we actually need is almost always preferable to taking a whole library on.
  • Libraries that have already been removed (Phaser, Pixi, Matter.js, standalone matter-js) are gone for good. Do not reintroduce them under any architectural argument. They were tried and failed our scaling rules.
  • “In-house” means the code is in our repo, in a language we ship from source, that we are willing to debug at any layer. Wrapping a library does not count as in-house ownership.
  • The Rust sim crate is the preferred home for new logic. New TypeScript modules are acceptable for orchestration, but new simulation, physics, math, or per-unit-loop code should land in rts-sim-wasm/ from the start, not get written in TS and ported later.

This is the active commitment behind Languages and Frameworks, TypeScript Coordinates, WASM Computes, and Delete The Old Path. Those sections describe what the end-state engine looks like and how migrations land; this section states why we are doing it: nobody else’s engine is going to give us thousands of physically-simulated units in a browser, and every library we keep is a ceiling someone else set for us.

Network Model

Commands flow upward

A client issues a command and sends it directly to the authoritative host. The host checks player authority, filters illegal unit IDs, clamps unsafe parameters, and enqueues the command into the server command queue. The command is not broadcast as peer truth. Other clients learn the result through snapshots and events emitted by the authoritative sim.

  • Selection can remain local because it is UI state.
  • Move, attack, guard, build, repair, reclaim, scan, production, and server-control commands must be authorized by the host.
  • Command latency should be hidden with local intent previews, not by letting clients create real gameplay state.
  • Every command path should be stable enough to work for local host play, remote WebRTC play, and a future dedicated worker or server.

Snapshots flow downward

Server snapshots are corrections, not per-frame animation frames. They should contain changed authoritative targets, creation and removal events, projectile events, audio events, fog and minimap deltas, and occasional keyframes. The client stores those values as server targets, predicts forward locally, and drifts toward the newest authoritative target.

  • Normal target cadence is 5 to 10 snapshots per second.
  • Server tick rate and snapshot rate are independent.
  • Full keyframes are rare recovery points. Delta snapshots are the default.
  • Area-of-interest and fog filtering are server responsibilities.
  • Backpressure should drop or coarsen snapshots before it lets the browser build an unbounded send queue.

Delta thresholds match channel semantics

A FULLSNAP is a recovery baseline. A DIFFSNAP is a minimal update from the last value actually sent to that recipient for that channel. Delta decisions must not compare only against the previous full snapshot, and they must not compare against an unsent local tick value. The baseline advances independently per field when that field is emitted.

Position-like channels use absolute error because their meaning is spatial: movement position uses world-unit distance, and rotation position uses radians or quaternion angular distance. Derivative channels use relative error because their meaning is rate: movement velocity and rotation velocity should ask whether the new velocity is meaningfully different by percentage from the last velocity sent, not whether it moved by one fixed unit. The same rule applies to any future acceleration-like derivative channel.

Derivative comparisons must treat zero as a semantic edge. If the last sent velocity was non-zero and the current authoritative velocity is zero, the relative change is effectively unbounded and the zero must be sent. Decaying velocities should keep producing deltas as they change by meaningful percentages, so the client eventually learns that motion has stopped instead of integrating an old non-zero derivative forever.

Client prediction is an explicit contract

The client view should continue to run smoothly whether snapshots arrive at 60 Hz, 10 Hz, 5 Hz, or briefly not at all. Prediction should use the same motion math as the server wherever possible. The current code already points this way with shared TypeScript/WASM unit motion, server targets, drift modes, and prediction LOD. New systems should preserve that shape.

Client Simulation Model

Shared core, bounded client

There should be one headless simulation core for physical behavior, but two different runtime responsibilities. The server runs the complete authoritative game simulation. The client runs a bounded prediction simulation for the parts the player sees and feels between snapshots. This client sim may be a real fixed-step loop, and it may eventually live in a worker, but it must remain a prediction layer. It is corrected by authoritative state and never becomes the source of gameplay truth.

  • Share physical kernels for body motion, projectile stepping, recoil, hover, turret pose, terrain sampling, and collision helpers.
  • Keep authoritative-only systems on the host: command validation, economy, construction completion, targeting truth, damage, fog filtering, unit creation and death, and victory.
  • Predict visible state locally: movement, velocity, turret rotation, beam path presentation, projectile travel, force-field presentation, and local command intent.
  • Use snapshots as correction targets. The client stores the most recent server target per entity. Position channels — movement position and rotation position — always apply correction by snapping or EMA-blending toward it; velocity channels — movement velocity and rotation velocity — can additionally ignore the snapshot derivative when that is useful for prediction testing. Acceleration is never shipped: the authoritative force solver lives on the host, and the client’s prediction integrates position from the last-seen velocity rather than re-deriving server-side forces from a stale snapshot.

Prediction is velocity-based

The client prediction integrator advances position from the last-seen velocity each frame. It does not re-integrate acceleration. The authoritative simulation owns force inputs (movement thrust, turret spring torque, gravity, ground spring, recoil, impulse), and those forces produce velocities and positions the snapshot eventually carries. The client never sees the acceleration vector itself, so it cannot stale-integrate it across a snapshot gap and overshoot when the authoritative force flips sign before the next correction arrives.

The client’s PREDICT mode therefore exposes just two settings: POS snaps straight to the snapshot position (no integration); VEL integrates position from velocity each frame. There is no ACC option — and consequently no movementAccel or angularAcceleration on the wire — because instantaneous second derivatives are not stable to integrate under arbitrary client dt and the EMA-toward-velocity already smooths the approach to a freshly-arrived target.

EMA channels are independent

Each prediction channel has its own EMA control: movement position, movement velocity, rotation position, and rotation velocity. Position controls expose snap, fast, medium, and slow; velocity controls also expose ignore. The client always stores the most recent snapshot value per channel so the chosen mode can be applied afresh every client tick.

  • ignore — the stored snapshot value exists but the client never applies it to the rendered entity for that channel. Prediction (if any) continues from the last applied value forever.
  • snap — each client tick the rendered value is replaced with the latest stored snapshot value for that channel. No smoothing.
  • fast / medium / slow — each tick the rendered value blends toward the latest stored snapshot value using a frame-rate independent EMA whose half-life is determined by the mode.

Splitting the EMA per channel means a player can keep movement snappy while letting rotation drift gently, or freeze velocity entirely while position catches up. The mode is a render-time decision, not a snapshot-decode decision: the target store always accepts the newest authoritative value, and the per-channel mode controls only how the rendered state walks toward it.

What each EMA channel applies to

The four EMA knobs are not abstract presentation sliders; each one binds to a specific named set of authoritative state fields, and the rendered value of every one of those fields walks toward its server target through the chosen mode every client tick. The whole point of having one knob per channel — rather than per-field tuning buried in code — is that the same smoothing behavior is applied uniformly to every field that belongs to that channel. A channel that is only applied to some of its fields is a bug, not a freedom: changing the rotation position mode must move every rotation-position field in lockstep, or the knob is lying about what it controls.

  • Movement position — the world-space 3D position of every visually rendered moving thing. That means the unit body center (entity.transform.x/y/z), the projectile body center (entity.transform.x/y/z on non-line shots), and every per-vertex point along a line-shot beam (proj.points[i].x/y/z). Anything the player sees as "where this thing is right now" rides on this channel.
  • Movement velocity — the 3D linear velocity of every moving thing whose authoritative velocity travels on the wire. That means the unit linear velocity (entity.unit.velocityX/Y/Z), the projectile linear velocity (proj.velocityX/Y/Z on non-line shots), and the per-vertex tangential velocity on a line-shot beam (proj.points[i].vx/vy/vz). These are the values the client integrates forward between snapshots under VEL prediction, so the velocity EMA controls how fast that integration realigns with the server's freshest velocity.
  • Rotation position — the orientation of every rotating thing on the wire, in both scalar and full 3-DOF form. That means the unit body yaw scalar (entity.transform.rotation), the unit hover-style body orientation quaternion (entity.unit.orientation.{x,y,z,w}), and each turret's yaw and pitch (weapon.rotation, weapon.pitch). The force-field range visualizer also rides this channel because it is a slow visual progress that pairs with the turret's rotation correction.
  • Rotation velocity — the angular rate of every rotating thing whose authoritative angular velocity travels on the wire. That means the unit hover-style angular velocity vector (entity.unit.angularVelocity3.{x,y,z}), each turret's yaw rate (weapon.angularVelocity), and each turret's pitch rate (weapon.pitchVelocity). These are the derivatives the client integrates while drifting toward the rotation-position target.

Fields that do not appear in any of the lists above are not on the wire and are therefore not subject to snapshot drift. Static building position, decorative locomotion-rig state (wheel spin phase, tread cleat phase, leg plant pose, hover fan smoke), visual-only chassis embellishments (airborne bank roll), and discrete open/close pose transitions (solar petal fold, extractor blade fold, wind-turbine close pose) live entirely client-side and get their smoothness from local controllers, not from any of the four channels. See Locomotion Visuals Are Frontend and Airborne Banking Is Visual.

Decorative client-only animations that visually represent the same kind of motion as a wire channel may opt in to the same EMA mode for consistency — the wind-turbine fan yaw reads rotation-position mode, and the wind-turbine blade speed, construction-tower orbit spin, extractor rotor spin, radar head, and radar sweep read rotation-velocity mode. These are courtesy bindings, not snapshot-drift corrections: there is no server target to drift toward, so the mode supplies a half-life and the local animator EMA-blends toward a locally derived target value.

Discrete open/close pose transitions do not opt in to rotation position. Their state is a local pose switch between authored endpoints, not a continuously rotating wire field, so they keep fixed local controller alphas unless a future gameplay reason puts an actual rotation target on the wire.

Rendering samples prediction

Rendering should not own simulation cadence. requestAnimationFrame is a display scheduler, not a simulation clock. The client prediction loop can run at a fixed or semi-fixed timestep, while the renderer samples the latest predicted state every frame and performs render-time interpolation. If the browser renders at 144 Hz, 60 Hz, 37 Hz, or is briefly tab-throttled, the prediction contract should stay coherent.

This split keeps the browser responsive and keeps future options open. The prediction loop can move toward Rust/WASM and a worker without rewriting the renderer.

Physics Model

The game should be more physics based than its inspiration. BAR is the ergonomic and strategic reference point; this project should make the physical world more fundamental. The rule of thumb is F = M * A. If a gameplay feature moves, pushes, resists, falls, recoils, hovers, blocks, crushes, reflects, or explodes, it should be described through forces acting on masses and integrated into motion.

Full 3D, not decorative altitude

The sim uses a real 3D state: horizontal x/y, vertical z, velocity, acceleration, gravity, surface normals, sphere and cuboid bodies, terrain contact, building collision, unit collision, projectile collision, and hover lift. The renderer can simplify distant visuals, but the authoritative sim should treat altitude and contact as real gameplay inputs.

Kinematic calculations must read entity position, velocity, and acceleration through the explicit accessors getEntityPosition3d, getEntityVelocity3d, and getEntityAcceleration3d. Callers pass zero-valued components when an entity has no velocity or acceleration, and targeting/ballistic solvers must receive raw kinematic values plus the universal gravity constant instead of entity IDs or optional lookup inputs.

Weapon range is a 3D envelope, not a ground-plane circle. Targeting acquire/release hysteresis, line-shot beam length, and broadphase candidate filters all use true 3D distance from the turret mount. An aircraft directly overhead at the weapon’s max radius is reachable; the same target at the same horizontal distance but at twice the altitude is not. Range overlays on the ground show the sphere’s ground projection (sqrt(R² − h²) where h is mount height above terrain), so the ring honestly represents where ground targets can be hit.

Mass must matter

  • Unit propulsion is a force profile applied to a mass, not a direct position edit.
  • Recoil applies counter-force to the firing body.
  • Explosions, beams, collisions, impact impulses, and force fields apply forces that heavier units resist differently.
  • Hover lift balances gravity through force and damping.
  • Ground contact is spring and friction behavior against terrain, not a clamp disguised as physics.

Locomotion is arrival control, not waypoint seeking

Movement should not be a magnetic pull toward the next waypoint. A position-only seek controller looks acceptable on slow, high-friction units, but it breaks down as soon as a body has meaningful momentum. Hovercraft expose the failure most clearly: the unit keeps accelerating toward a point even when its current velocity already guarantees overshoot, then crosses the point, gets pulled back, and oscillates like an undamped spring.

The locomotion question is not “which direction is the waypoint?” It is “what acceleration do I need, under this unit’s force limits, to arrive at the desired state?” That desired state includes both position and velocity. A final command point usually wants near-zero velocity. An intermediate waypoint may want a through-velocity aligned with the next path segment, so the unit flows through the path instead of stopping at every corner.

Every locomotion profile should therefore compute force from both position error and velocity error. If a unit is close to its intended spot but still moving quickly, the correct thrust is often opposite its current velocity, even if the target remains slightly ahead. This is controlled arrival: a damped physical controller that spends stopping distance before it runs out, saturates against the unit’s available thrust, and prevents waypoint overshoot from turning into visible oscillation.

  • Pathfinding chooses where the body should go; locomotion chooses how the body physically arrives there.
  • Waypoints are desired motion states, not magnets. They carry an intended position and, when useful, an intended pass-through velocity.
  • Final command points should use braking-aware arrival so the body settles without orbiting, bouncing, or springing back across the destination.
  • Hovercraft, ground units, and air units use the same high-level arrival controller; they differ only by mass, thrust, friction, damping, turn authority, contact model, and vertical constraints.
  • Hovercraft should remain slippery because their physical limits say so, not because the controller ignores velocity until after overshoot has happened.

Gravity is universal; thrust answers it

Every body with mass feels gravity every tick, without exception. The authoritative sim should never special-case “this projectile ignores gravity” as a free pass. If a rocket, missile, or other guided shot needs to fly a clean line to its target, it does so by producing thrust that cancels the gravitational pull, not by opting out of the force.

This applies inside the real-time homing equation itself. A guided projectile’s steering update should compute the lateral correction toward the predicted intercept and, in the same step, add a vertical thrust component sized to counteract g · m on this tick. The visible flight path can look indistinguishable from “gravity off,” but the physics stays whole: the body has mass, the world pulls it down, the engine pushes back, and any failure of the engine — out of fuel, seeker lost, lifespan expired — drops the body the way a spent missile drops in the real world.

  • Gravity is applied to every mass body every tick. No ignoresGravity shortcut, no projectile-class exemption, no “flat trajectory” flag that bypasses integration.
  • Guided projectiles add a counter-gravity thrust term inside the same homing/steering update that produces lateral correction toward the target. Guidance and gravity-cancellation are one acceleration vector, not two systems.
  • The counter-gravity term is bounded by the projectile’s available thrust. A weak or under-powered rocket should sag, not teleport — the homing math saturates, it does not lie.
  • Spent or failed guidance returns the body to ballistic motion automatically, because gravity was never turned off; only the thrust opposing it stopped.
  • Hover and any other “defies gravity” behavior follows the same shape: a force that fights g · m, not a conditional that skips it.

Discrete gameplay still enters through physical effects

Commands, economy, construction, fog, build queues, and ownership are intentionally discrete systems. They do not need fake physics. But when those systems touch the battlefield, their visible result should enter the world as bodies, forces, energy, mass, constraints, or collisions. A factory can make a unit shell discretely; once it exists, gravity, terrain, collision, and force response should own its motion.

Scale Target

The design target is thousands of active units in a browser, with a path toward larger battle scenes. That target changes the default answer to every implementation question. Expensive correctness that works for 200 units but collapses at 5,000 units is not correct for this project.

Hot paths are data-oriented

  • Prefer stable IDs, packed numeric fields, typed arrays, and struct-of-arrays storage for dense simulation data.
  • Batch numeric work into WASM kernels when it can run over typed arrays without heavy object marshalling.
  • Keep TypeScript as orchestration, UI, Three.js integration, networking glue, and gameplay control flow.
  • Pool snapshot DTOs, event objects, scratch arrays, and temporary buffers.
  • Do not allocate in per-unit, per-projectile, per-turret, per-cell, or per-frame loops unless the allocation is proven irrelevant.

Spatial structure beats broad scans

Targeting, collision, capture, fog, projectile queries, and selection should use spatial indices, dirty sets, visibility masks, and area-of-interest bounds. Full-world scans are acceptable for bootstrapping, debugging, and rare keyframes. They are not the normal steady state for gameplay ticks.

LOD is allowed, but it must be honest

There are two separate concerns: server simulation authority and client visual detail. Server simulation, snapshots, and client prediction run at their normal cadence without camera-distance throttles. Client render LOD may reduce labels, effects, and mesh richness, but it must not change gameplay authority or unit/turret pose update frequency.

Browser Performance Rules

Browser performance is not an afterthought. The browser is the platform. Code should be written as if V8, SpiderMonkey, WebKit, WebAssembly, WebGL, garbage collection, and WebRTC buffering are part of the game engine.

  • Use plain counted loops in hot paths. Avoid map, filter, Object.entries, spread copies, and iterator churn inside tick loops.
  • Reuse arrays by setting length = 0. Grow buffers on demand and avoid shrinking steady-state buffers.
  • Use bit flags and append-only numeric wire enums for snapshot fields and blueprint IDs.
  • Serialize snapshots as compact binary payloads. Do not rely on deep JSON object trees for high-frequency state.
  • Separate debug overlays from gameplay state and throttle them aggressively.
  • Measure TPS, CPU budget, render time, prediction time, snapshot rate, and backpressure continuously.
  • Prefer instancing, shared geometry, shared materials, LOD grids, and GPU-friendly batches over per-unit scene graph complexity.

TypeScript Coordinates, WASM Computes

The absolute performance ceiling for browser 3D is set by how little JavaScript runs in the per-frame hot path. TypeScript stays in the project because the browser’s native dialect is JS — boot, asset loading, input, UI, networking glue, calling into WASM — but it should never be the layer that integrates physics, walks per-unit update loops, builds render lists for thousands of objects, or carries large serialization work. Those belong in WebAssembly, compiled from Rust, running over typed linear memory with SIMD-friendly, struct-of-arrays layouts. The renderer that consumes the engine’s output belongs on WebGPU with WGSL shaders, so the GPU can be driven directly with instancing, compute, and large buffers instead of one draw call per object.

This is the same commitment that Languages and Frameworks makes for language choice and Config Is Data, Not Code makes for content authoring, expressed as a runtime architecture: the layers are stacked so that hot data crosses the JS/WASM boundary as rarely as possible, and the JS side never owns the inner loop. The end-state shape is a TypeScript shell wrapped around a Rust + WASM simulation that talks to a WebGPU renderer through GPU-resident buffers, with Web Workers and SharedArrayBuffer separating cadences where independent stepping or true cross-thread shared memory pays for itself.

Layer responsibilities

  • TypeScript shell — application boot, asset loading, canvas creation, WebGPU initialization, input collection, UI / menus / dev tools, WebSocket and WebRTC setup, and the bridge that calls into WASM. This layer coordinates; it does not compute.
  • WebAssembly engine (Rust) — ECS, physics, collision, pathfinding, visibility and fog, animation state, projectile simulation, spatial queries, culling, and render-packet generation. Everything that runs every tick or every unit lives here.
  • WebGPU + WGSL — GPU buffers, pipelines, compute shaders, instanced rendering, post-processing. Where the pixel work happens and where mass parallelism is recovered.
  • Web Workers + SharedArrayBuffer — when simulation and render need to step on independent cadences, or when a kernel can be parallelized across cores against shared linear memory without copying.

What TypeScript must never own in hot paths

Each item below is a per-unit, per-projectile, or per-frame loop that compounds at scale. Running any of them from JavaScript at thousands of units gives up the performance ceiling, regardless of how careful the TypeScript is.

  • Per-unit update loops.
  • Physics integration and collision detection.
  • Projectile simulation and spatial partitioning.
  • Pathfinding and fog-of-war calculation.
  • Animation blending for thousands of units.
  • Render-list generation for thousands of objects.
  • Large serialization loops and large per-frame allocations.

What TypeScript should keep doing

  • Bootstrapping, developer tools, debug overlays.
  • Settings, lobby, and the Vue / UI layer.
  • Asset loading and input event collection.
  • WebGPU initialization and pipeline assembly.
  • WebSocket / WebRTC setup and the network glue.
  • Calling exported WASM functions and marshalling the few values that genuinely need to cross the boundary.

The slow architecture vs the fast one

The slow browser game architecture is the one most browser code drifts into: thousands of JS objects, each carrying methods, each updated every frame, each allocating temporaries, each issuing its own draw call. That shape pays the cost of language overhead, garbage collection, scene-graph traversal, and one-draw-per-thing at every layer simultaneously.

The fast architecture is the inversion: flat typed memory, batched updates, almost no allocations, very few crossings of the JS/WASM boundary, large GPU buffers, instanced rendering, compute shaders where the work parallelizes, and as few draw calls as the scene can be expressed in. The rule of thumb is: if you can make the loop run over a slab of bytes instead of a list of objects, you should — and that loop almost certainly belongs in WASM or on the GPU, not in TypeScript.

Languages and Frameworks

Peak performance wins over framework convenience. For simulation and rendering, the default choice should be the fastest browser-viable language and API we can own directly. Frameworks are allowed only when they do not sit inside hot loops and do not prevent us from controlling memory layout, batching, scheduling, serialization, GPU buffers, or shader code.

Simulation language

  • Use Rust compiled to WebAssembly for dense numeric simulation kernels.
  • Move physics, projectile stepping, collision kernels, pathfinding kernels, terrain sampling, targeting batches, visibility packing, and snapshot-diff kernels toward WASM when the data can be batched.
  • Use struct-of-arrays storage in WASM linear memory for bodies, projectiles, turrets, spatial indices, and other high-count state.
  • Keep TypeScript as orchestration: UI, commands, networking glue, scene lifecycle, and code that would pay too much JS/WASM boundary cost. Configuration and blueprints are data, not TypeScript — see Config Is Data, Not Code.
  • Do not use a general-purpose JavaScript physics engine, ECS framework, pathfinding framework, or game framework in the authoritative sim hot path.

Rendering language and API

  • Use GPU-native code for the expensive work: custom GLSL for WebGL2 or WGSL for WebGPU.
  • Prefer raw WebGPU or raw WebGL2 for long-term high-count rendering paths where framework overhead blocks scale.
  • Three.js is treated as legacy scaffolding, not endorsed architecture. It can remain in place while it is still productive, but every bottlenecked renderer path should be replaced with owned buffers, owned shaders, and direct instancing — the end state is raw WebGPU + WGSL with no rendering library in the tree. See Engine Ownership.
  • Vue is UI only. Vue must not own simulation cadence, render-loop state, per-unit updates, or per-frame battlefield data.
  • Do not introduce rendering frameworks that hide draw-call count, buffer ownership, material compilation, culling, or shader control from us.

Raw by default

The preferred implementation is owned code in Rust/WASM, TypeScript, and shader code, not a stack of general-purpose frameworks. If a library saves time but costs measurable simulation or rendering throughput, it should be treated as temporary scaffolding. The end-state engine should be raw enough that we know where every per-frame allocation, buffer upload, kernel call, draw call, and branch-heavy world scan comes from. When scaffolding is replaced by owned code, the scaffolding leaves the tree in the same change — see Delete The Old Path.

Rendering Philosophy

The renderer should make huge battles legible without forcing the simulation to become fake. Rich nearby units can have turrets, legs, treads, chassis tilt, beams, labels, decals, and effects. Distant units should collapse to cheaper instanced representations. The object LOD grid is the right shape: classify a region once, then let all renderable objects in that region inherit the tier.

Rendering should never ask the authoritative sim to send more data just because a visual is expensive. If a visual cannot be supported at scale, it should degrade on the client. The sim remains physical; the rendering becomes cheaper.

Locomotion Visuals Are Frontend

Authority owns chassis position. The renderer owns where the parts of the chassis appear in the world. The sim publishes one point — the unit’s body center, at its authoritative altitude. From there the rig is responsible for putting every wheel, cleat, foot, or fan exactly where the renderer says it should be, and that “should be” must respect terrain as a hard floor.

This is the right boundary for three reasons. One: the renderer already has the terrain heightmap loaded for shading and contact shadows; sampling it under a wheel or foot is cheap. Two: per-part terrain queries scale linearly with visible units, not authoritative units — off-screen and LOD-culled units skip the work for free. Three: it removes a whole class of net-bug, because a chassis can never wedge a wheel under a hill that exists only on the client’s mesh. The visual rig is downstream of sim authority, never upstream.

The four locomotions

  • Wheels — a chassis driven through discrete rigid contact patches at fixed body-local positions. Visually: cylindrical tires, each spinning around its own axle, rolling phase derived from how far that local contact point traveled in the world this frame. Each tire owes the ground a contact patch — the bottom of the cylinder is the contract.
  • Treads — two infinite belts that share chassis velocity but read it as continuous tangential motion. Visually: scrolling cleat rails along the belt path; the belt itself never moves relative to the chassis. The contract is the entire underside of each ribbon sitting on terrain.
  • Legs — discrete step events: a foot plants, the chassis advances past it, the foot lifts and swings to a new plant. IK solves hip→knee→foot for the rendered pose. The contract is per-foot, not per-chassis — feet must touch terrain on plant and swing through air during transit.
  • Hover — a chassis that never touches anything; a duct array balances gravity with downward thrust against the air column underneath. Visually: spinning fans, downward smoke columns, chassis riding a target altitude. The contract is anti-contact: the gap is the feature, and that gap must be measurable per frame so smoke length and altitude shading look right.

Always animate; terrain only clamps position

Every rig integrates the same four canonical visual state channels every frame — the same channels the client-prediction layer uses for entity motion, named the same way and reasoned about the same way:

  • Movement position — where the part sits in chassis-local Y (wheel lift, tread side lift). EMA-converges toward the floor-clamp target every frame.
  • Movement velocity — rate of change of movement position; integrated implicitly from the position EMA.
  • Rotation position — rolling angle (wheel rotation around its axle) or belt phase (where cleats sit on the loop). Integrated from rotation velocity.
  • Rotation velocity — angular velocity for wheels, linear belt velocity for treads. EMA-couples toward the body-motion-derived target every frame.

The rig does not ask "is this part touching ground?" The velocity channels always couple to body motion; the position channels always couple to the floor clamp. A tire over a dip spins at the body-motion rate and floats slightly above terrain (target lift = 0). A tire cresting a hill spins at the same rate and rides up the surface (target lift > 0). When the body stops moving, the velocity channels EMA toward zero on the same tau they always use — cleats coast to rest over a beat, wheels spin down naturally. Simpler than switching coupling regimes, and visually indistinguishable in normal play.

Per-part position, not per-chassis

Floor clamping is per primitive. A four-wheeled vehicle straddling a crest has two wheels rising over the terrain and two floating in air over the dip; the rig clamps each tire independently. Treads sample three points along each side ribbon and lift the side to clear the max. Legs sample under each foot. Hover doesn't clamp parts but does sample a per-frame clearance scalar for fan smoke and altitude shading. The rig knows what its primitives are; uniformity is in the shape of the contract, not in the bits.

Visual contact is sticky

The renderer must not reuse the physics ground-contact epsilon as the sole answer for locomotion animation. The sim epsilon is intentionally tiny because it gates spring, friction, sleep, thrust, and collision behavior. Visual rigs need a separate, frontend-only contact band with hysteresis: acquire ground when the chassis is near the terrain, keep the grounded pose while the body remains within a larger release band or while planted feet are still reachable, and only transition to airborne posing once the visual contradiction is unmistakable.

For legs, the stable unit of truth is the foot, not the chassis. A planted foot keeps its world position until a deliberate step swing begins. Brief chassis bob, snapshot drift, suspension state, or a one-frame miss in strict body contact must not reset the gait or re-seat feet at their rest points. When a legged unit really leaves the ground, feet move smoothly into an airborne pose; when it returns, each foot swings to a terrain plant instead of being initialized in place.

Floor clamping is a hard invariant, not a tunable

  • The clamp is non-optional. Every rig writes renderedY = max(naturalY, terrainY + clearance) for every part it owns, then EMA-converges the part's movement-position channel toward that value. No flag, no debug toggle, no edge case where parts are allowed to drop under the surface.
  • Floating above terrain is acceptable; tunneling through it is not. When the sim chassis is over a dip, individual wheels are allowed to read as “slightly floating” rather than pretending to droop on suspension we do not have.
  • The clamp uses the same smoothed surface normal the chassis tilt consumes (updateUnitGroundNormal). One canonical value, no drift between “how the body leans” and “where the wheels sit.”
  • Hover gets a soft floor too: the rendered rig group is clamped at least ε above terrain, so a bad snapshot can never park fans inside dirt. The authoritative chassis altitude is untouched — the renderer just refuses to draw a contradiction.

Nothing about locomotion visuals crosses the wire

Every input the rig needs — chassis transform, terrain heightmap, body velocity, surface normal — the client already has. Computing this on the server and shipping it would burn bandwidth on per-frame visual state that only matters within the next render call. Compute it locally, render it, throw it away. Putting any of it on the wire is what got us into the jump-state-on-the-wire mess in the first place. Networked suspension/contact fields may influence body presentation, but they must not be the authoritative source for whether a foot remains visually planted.

Airborne Banking Is Visual, Mounts Stay On The Roll Axis

Hover and flying units should look like they bank into their turns — the chassis rolls about its forward axis as lateral velocity builds, then unwinds when the unit straightens out. Banking is the strongest visual cue that an airborne unit is physically committed to a turn, and the renderer is the only place that needs to know about it. The authoritative sim continues to track yaw (and yaw alone) for hover/flying chassis; pitch and roll are derived from local body velocity at render time and never touch sim state, the wire, or any combat math.

This is the same boundary as Locomotion Visuals Are Frontend: the sim publishes a chassis pose the renderer is allowed to enrich, and the rig owns every visual detail that doesn’t change targeting truth. Banking is the airborne analogue of wheel spin and tread phase — expressive on the screen, irrelevant to the simulation.

Why this is a hard rule and not a convenience

The sim’s turret-world-mount math is yaw plus surface-normal tilt — not a full 3-DOF orientation quaternion (getTurretWorldMount). If the renderer applies pitch/roll to the chassis while the sim computes mount positions with yaw only, the visible barrel and the sim’s projectile-spawn point quietly diverge. The cure is geometric: every turret mount on a hover or flying unit must sit on the chassis’ body-forward axis, i.e. mount.y === 0 and mount.z === 0. A pure roll about that axis leaves the mount unchanged in world space, so the sim’s yaw-only mount math and the renderer’s rolled chassis agree pixel-perfect.

  • The blueprint loader enforces mount.y === 0 and mount.z === 0 for every turret on a hover or flying unit. A non-centerline airborne mount is a startup error, not a runtime drift.
  • Hover/flying turrets do not use zResolver; the centerline rule would be violated the moment the resolver assigned a body-top z.
  • The free axis is mount.x. An airborne turret can sit forward or aft of the chassis center along the body’s forward axis without breaking the invariant.
  • Ground-locomotion turrets keep authoring freedom on all three axes — they don’t bank, and the slope-tilt path already runs through getTurretWorldMount.

What the sim writes, what the renderer composes

  • The sim keeps a yaw-only spring on hover/flying orientation. entity.unit.orientation ends each tick as a quaternion equivalent to setQuatFromYaw(yaw) — target pitch and target roll are pinned to zero, the yaw spring still carries rotational momentum so heading turns feel weighty.
  • The renderer derives a body-lateral acceleration — the centripetal term v_forward · ω_z — from unit.velocityX/Y projected through the current yaw and unit.angularVelocity3.z (the yaw rate, on the wire and predicted at render rate). It scales that into a target bank angle, EMA-smooths it per frame, and composes the roll about the body-forward axis on top of the chassis yaw quat. A velocity-derived bank would decay to flat mid-turn because the yaw spring continuously aligns body forward with the thrust direction, collapsing body-lateral velocity; the centripetal signal stays high for as long as the unit is actually turning.
  • The visual bank is bounded (a hard clamp keeps the chassis from doing barrel rolls under collision spikes) and decays toward zero whenever centripetal acceleration is small. It intentionally uses local render-side smoothing rather than the rotation-position channel. No sim state ever sees the value.
  • If a future visual wants pitch (nose-down when accelerating, nose-up when braking), it lives in the same place: derived from velocity at render time, never written back to the sim.

Beam And Rocket Turrets Render As Heads, Not Barrels

Beam-family turrets (beamTurret, miniBeam, megaBeamTurret, towerBeamTurret) and rocket-family turrets (salvoRocketTurret, fastRocketTurret) render as a single sphere — the turret head — with no barrel attached. The head shows the unit’s player color while idle or tracking and shifts halfway toward white the instant the turret engages a target. The color flip is the lock-on cue, and keeping it blended with the player color (rather than flattening to pure white) means every engaged turret still reads as “that player” in a crowded fight.

Marking a turret blueprint with headOnly: true is the contract. The renderer builds no barrel meshes for it, skips applyTurretAimPose3D on its root group every tick, and the snapshot serializer pins the wire’s rotation / angular-velocity / pitch / pitch-velocity slots to zero. The simulation still tracks the live aim values internally so beams trace and rockets launch in the correct direction — those values just never leave the host.

Why this is its own boundary

Aim motion is the busiest channel a turret pushes through the delta serializer. A continuously tracking turret dirties its host entity every frame on rotation / pitch / angular-velocity thresholds, which forces the entire entity row back onto the wire. For weapons where the renderer never uses aim motion — beams define their direction by the start/end points stamped on the beam projectile; verticalLauncher rockets ignore turret yaw entirely — that traffic is pure overhead. Zeroing the four angular fields collapses those transitions out of the diff path so the entity only dirties on target-acquisition or state changes, which is what the visual actually depends on.

What the renderer still reads

  • The turret’s engaged/idle state, sent as the usual FSM bit on every snapshot. The head color writer blends the player primary halfway toward white iff turret.state === 'engaged'.
  • The turret’s mount point, identical to every other turret — the head sphere sits where it always sat.
  • Beam endpoints and projectile spawns, which carry their own world-space geometry on the wire. None of them need turret yaw/pitch on the client to be reconstructed.

Where the rule does not apply

Mirror turrets and the disruptor cannon stay full-barrel: the mirror panel has to rotate to bisect its targets, and the disruptor is an authored melee gun whose visible barrel is half the read. Construction emitters, force-field turrets, and visual-only hardpoints continue to use their existing dedicated rigs — headOnly is a strict subset of the “straight-line projectile or beam, head sphere is the whole visual” category, not a catch-all.

BAR Inspiration, Project Identity

Beyond All Reason is the comparison point for command ergonomics, readable large battles, queue-friendly controls, minimap interaction, and the feeling of commanding armies instead of individual pieces. This project should feel familiar to BAR players in its command vocabulary and battlefield expectations.

The project identity is different: the browser is a hard constraint, the network model is authoritative snapshots plus client prediction, and the world should be even more physically grounded. BAR is a benchmark, not a source of code or assets.

Atomic Gameplay Pieces

Gameplay should be built from small, atomic pieces that do one job and compose cleanly: locomotion, unit, turret, shot, and building definitions. These pieces should be swappable without hidden coupling. A unit should be able to change locomotion profiles, mount different turret sets, fire different shot profiles, or appear from different factories because each piece carries a clear contract instead of leaking responsibility into its neighbors.

  • locomotion owns how a body produces and applies movement forces.
  • unit owns the entity's body, mass, health, command capabilities, build cost, and composition slots.
  • turret owns lock-on eligibility, aiming, tracking, cooldown, mount behavior, and which shot it emits.
  • shot owns projectile, beam, laser, force-field, inherited homing target, recoil, damage, impulse, and collision behavior.
  • building owns static footprint, construction, production, extraction, storage, or other building-specific behavior.

Lock-on policy lives on turrets, not rockets

A turret is the only piece that acquires a lock-on target. A rocket can inherit the exact entity the spawning turret was locked onto at launch, and it can home toward that inherited target while the target remains alive in the sim. The rocket must not run its own target acquisition pass, scan for replacements, or choose a new victim after launch. If the inherited target no longer exists, the rocket loses guidance and continues on its current flight path under normal projectile physics until it hits something or expires.

Host intent is not turret lock-on

Units and buildings can carry a host-level target chosen by the player, AI, or command system. That target expresses intent: attack this enemy, guard this ally, repair this unit, assist this building, or otherwise keep the host’s current order anchored to an entity. It is not the same thing as a turret’s current lock-on target. A host target is an input each mounted turret may consult; it must not blindly overwrite every turret target on the host.

Each turret decides whether the host target is meaningful for that turret. The turret first checks its own lock-on exclusion policy, then the normal validity gates such as aliveness, visibility, range, line of sight, and shot compatibility. If the host target passes, the turret may inherit it as a priority lock. If the host target fails, that turret rejects the inheritance and runs its own targeting flow. The host target remains intact either way.

This separation is what lets mixed-role hosts work. A unit guarding a friendly entity can keep that friendly entity as its host target; a repair or construction turret may inherit it, while a fighting turret that excludes friendly_entities ignores it and searches for enemies normally. A mirror turret can reject the same host target unless it is an allowed enemy turret. Commands belong to the host; lock-on belongs to the turret.

Turret lock-on eligibility is authored as exclusion data on the turret blueprint. Exclusions are subtractive: by default, a turret with no lock-on exclusions is eligible to lock onto any living entity the game exposes to targeting, including friendly or enemy buildings, units, and turrets. Each turret then removes the relationships, categories, or names it is not allowed to lock onto. This keeps target policy in data while preserving one shared targeting kernel.

  • exclude_lock_on_level_0_friends_and_enemies removes whole ownership relationships before scoring. Valid high-level entries are friendly_entities and enemy_entities.
  • exclude_lock_on_level_0_entities removes whole target categories before scoring. Valid high-level entries are buildings, units, and turrets.
  • exclude_lock_on_level_1_buildings removes individual building blueprint names from the remaining building candidates.
  • exclude_lock_on_level_1_units removes individual unit blueprint names from the remaining unit candidates.
  • exclude_lock_on_level_1_turrets removes individual turret blueprint names from the remaining turret candidates.

The hierarchy should be evaluated in order: ownership, aliveness, and visibility decide whether something can be considered; relationship exclusions remove friendly or enemy candidates; entity-family exclusions remove buildings, units, or turrets; level-1 exclusions remove specific authored names inside a surviving family; range, line-of-sight, and scoring then pick the best lock from the survivors. A construction turret that should lock only onto friendly entities excludes enemy_entities. A fighting turret that should only attack hostile targets excludes friendly_entities. A mirror turret that should lock only onto enemy turrets excludes friendly_entities, buildings, and units at level 0. A normal weapon turret that should ignore turret mounts excludes turrets at level 0. Specialist weapons can then use the level-1 lists to skip named buildings, units, or turret blueprints without changing the shared target acquisition code.

We do not want mixed or overlapping roles. A turret should not secretly define locomotion. A shot should not secretly define a unit. A building should not smuggle combat behavior except by mounting turrets. A unit should not duplicate shot physics. Cross-piece behavior should be expressed through explicit references and narrow data contracts, so designers can recombine pieces freely and the simulation can keep hot paths simple. These pieces are authored as data, not code — see Config Is Data, Not Code.

Materials Are Independent Of Shape

A surface material is a contract about how a surface interacts with the world: reflectivity, line-of-sight occlusion, projectile response, hit sound, shader treatment, particle reactions. Two pieces that mount the same material must look identical and behave identically. The only thing they are allowed to differ on is geometry — where the surface sits and what shape it takes.

The canonical example is the mirror surface. The mirror turret’s deployed mirror panels and the force-field generator’s sphere are the exact same material: same reflectivity, same line-of-sight occlusion rule, same projectile reflection mechanic, same shader, same visual treatment. The turret carries the material as flat panels; the generator carries it as a sphere. Beyond that geometric difference, a player should be unable to tell them apart by behavior or by eye, and there should be no code path that treats them differently.

  • A material is defined once, with all of its physical and visual properties — reflectivity, refraction, occlusion, hit response, sound profile, shader inputs, particle reactions.
  • Shapes carry materials. A flat panel and a sphere can both be made of mirror; a cuboid block and a frozen pond can both be made of ice. Geometry and material are orthogonal axes of composition.
  • When a projectile encounters a surface, the material decides what happens to the projectile; the shape decides where the encounter is and what the surface normal looks like. The two questions are answered by different systems.
  • Line-of-sight queries treat every shape carrying the same material the same way. If the mirror panel reflects a sight ray, the mirror sphere reflects it under the same rule. The intersection math reads from the shape; the policy reads from the material.
  • Visual identity is part of the contract. Same shader, same material parameters, same reflection probes, same hit reactions. The player should see immediately that the flat panel and the sphere are made of the same stuff.

This is Single Source Of Truth applied to physics surfaces. If a feature ever seems to require the mirror to behave differently in the turret case than in the force-field case, the right answer is almost always two materials with different parameter values — not a branch inside one material that switches on which piece owns the surface. Branches inside material behavior are how reflectivity drifts between two surfaces that the player still perceives as identical.

Config Is Data, Not Code

Blueprints, tuning tables, and authored content belong in language-neutral data files — JSON today, possibly a more compact schema-validated format later — not in TypeScript modules. As the simulation moves toward Rust and WebAssembly, every piece of TypeScript-only data is a wall between the current code and the end-state engine. A blueprint authored as a .ts file forces TypeScript to stay in the simulation loop forever, because Rust cannot import it. The same blueprint authored as .json is read by both, owned by neither, and survives the language migration intact.

This is the data-side counterpart to Languages and Frameworks. That section commits the hot kernels to Rust/WASM. This section commits to making sure the data those kernels need is not trapped inside a TypeScript module that prevents the migration from ever completing. Configuration in code looks convenient on the way in; it becomes a permanent dependency on the language you were trying to leave.

What belongs in JSON

  • Unit, turret, shot, locomotion, and building blueprints — every field on every atomic gameplay piece.
  • Tuning tables: damage curves, hover constants, friction values, mass tables, range tables, cost tables, build-time tables, recoil profiles.
  • Maps, scenarios, starting layouts, preset compositions, AI tuning, and any other authored content.
  • Wire codes, enum value-to-name maps, and cross-language identifiers that both TS and Rust must agree on. One file, two consumers, zero drift.

How to express behavior as data

Blueprints must not contain TypeScript functions, closures, or computed references. If a behavior requires a decision — damage falloff over range, turret aim style, shot trajectory shape, hover response curve — it is encoded as a discriminated union of data shapes that the simulation interprets, never as a callback smuggled into a blueprint. The blueprint says { "shape": "parabolic", "peak": 40, "gravity": 9.8 }; the simulation knows how to read that shape. This keeps every blueprint serializable, diffable, hot-reloadable, and parseable by Rust without a JavaScript runtime in the room.

Schema is the contract

  • One schema per atomic piece, declared in one canonical place, lives next to the data it describes.
  • TypeScript types and Rust structs are generated from the schema. They are not hand-maintained in two languages — that would just relocate the duplication problem from data to types.
  • Loaders in both TypeScript and Rust validate against the schema at startup. A blueprint that fails validation is a build error or a startup failure, not a silent runtime divergence.
  • Adding a field is a schema change. Both code generators run. Both languages see the field at the same time. There is no “I added it on the TS side and Rust never saw it” class of drift.
  • Defaults live in the schema, not in the loader. Every consumer reads the same defaulted shape.

Migration direction

Every blueprint file moved from .ts to .json is a step toward removing TypeScript from the simulation entirely. The shape stays the same; the syntax changes; the callers stop importing a module and start loading a file. As blueprints become data, the Rust/WASM simulation gains the ability to consume them directly, and the TypeScript blueprint modules disappear under Delete The Old Path. The end state is a simulation written entirely in Rust, loading entirely from JSON, with TypeScript surviving only as UI, networking glue, and orchestration — exactly the role Languages and Frameworks reserves for it.

Single Source Of Truth

No gameplay rule, command builder, validation path, physics formula, serializer shape, blueprint contract, UI label, or tuning table should be duplicated. If two systems need the same atomic action, that action belongs in one small module with one clear owner and every caller imports it. Copying logic between client, server, renderer, lobby, tests, and debug tools creates drift, and drift is a correctness bug.

  • Extract shared helpers before copying code into a second place.
  • Keep constants, enums, labels, thresholds, and wire codes in one canonical module.
  • Put command construction, authorization, and sanitization behind shared functions instead of reimplementing branches in components.
  • When duplication is temporary during a migration, delete the old path in the same change that proves the new one. See Delete The Old Path.

Delete The Old Path

Migration is not done when the new path works. Migration is done when the old path is gone. As we remove libraries, port hot kernels from TypeScript to Rust/WASM, replace framework helpers with owned code, and rebuild renderer paths on raw GPU APIs, the old implementation must be removed in the same change that lands the new one. Two implementations of the same behavior is two sources of truth, two places to fix bugs, two places to read when something looks wrong six months later, and a near certainty that one of them will drift.

This is the active counterpart to Single Source Of Truth. SSoT is the static rule: do not write the same logic twice. Delete The Old Path is the dynamic rule that keeps SSoT true while the codebase is in motion. Every time we replace a system, the replacement is incomplete until the predecessor file is deleted, its imports are gone, its config flags are gone, its wire codes are gone, and its tests are gone or repointed.

Why this matters here specifically

  • We are actively moving from JavaScript object loops and JavaScript physics math toward Rust/WASM kernels over typed-array linear memory. Leaving the JS implementation alongside the WASM kernel guarantees one becomes the de-facto path and the other rots, undetected, until something invokes it and produces a result that disagrees with the live path.
  • We are moving from Three.js convenience helpers toward owned buffers, owned shaders, and direct instancing. A renderer with both shapes present makes it impossible to reason about draw-call count, allocation behavior, material compilation, or culling — the two paths interfere even when only one is supposed to be active.
  • Pooled DTOs, prediction state, snapshot buffers, and entity-state shapes are sensitive to dead code paths that still write to them. A “removed” system that still touches pooled fields is a class of zombie-write bug that does not surface in normal tests.
  • Future readers — including future-us — should never have to ask “is this the current path or the old one?”. The codebase should answer that question by only containing the current path.

Rules during a migration

  • The change that introduces the replacement is the change that deletes the predecessor. Not the next change. Not a follow-up cleanup ticket.
  • If parity proof genuinely needs more than one change, the predecessor lives behind a single named flag, with a single owner, and a recorded removal target. There is no “we’ll come back to it.”
  • Do not keep “just in case” copies of the old implementation in legacy/, old/, _unused/, or sibling files with a .old suffix. Git history is the just-in-case copy.
  • Delete the imports, the dead branches, the feature flag, the legacy wire codes, the unused config knobs, and the tests that exercise only the dead path. A half-deleted system is worse than an undeleted one because it looks intentional.
  • If a library is being replaced, remove the dependency from package.json, Cargo.toml, and the lockfiles — not just the call sites. A still-installed library will be re-adopted by the next person reaching for the easy answer.
  • Renaming files is not a migration. If the new path lives next to the old path with a slightly different name and both still compile, no migration has happened — only confusion has been added.

What “removed” means

Removed means the code is not in the working tree, not behind a flag, not in a legacy/ folder, not in a sibling file with a .old suffix, and not in commented-out blocks. Git history is the archive; the working tree is the current truth. When someone greps for the name of an old function and finds zero matches, the migration is done. When they find one match in the active path and one match in a stale path, the migration is still in progress, and the next person to touch either file should finish it.

Explicit Absence

We are moving away from optional properties and optional chaining in our own simulation, server, prediction, and entity-state code. Absence should be declared, not inferred. A field should always exist with a typed value, and the absence of a value should be expressed as null (or a numeric sentinel in hot paths), so every site that reads it must make an explicit decision about what to do when the value is missing.

The problem we are solving is "seemingly ready when actually not." An optional field collapses three different failure modes into the same silent no-op: not yet initialized, semantically inapplicable, and previously torn down. An entity.turret?.fire() chain swallows all three. The unit just does not shoot, no stack trace, no symptom anyone can grep for. In an online multiplayer RTS with pooled state and delta-encoded snapshots, that class of bug compounds.

Why this matters more in this codebase

  • Pooled objects (snapshot DTOs, prediction state, audio events, scratch buffers) carry zombie fields between users when those fields are T? rather than T | null with an explicit reset on release.
  • V8 deoptimizes when objects have shape variance. field: T | null = null keeps the hidden class uniform and the JIT happy, which is measurable in 5,000-unit per-tick loops.
  • Snapshot serialization already encodes presence explicitly on the wire. Aligning the in-memory model with that contract eliminates a category of "I sent it but the receiver never had the property" drift bugs.
  • Determinism requires that absence is a decision, not a coincidence. ?. hides the decision; if (x === null) forces it into the code under review.

Scope of the rule

  • Inside our own simulation, server, prediction, networking, and entity-state modules: no ?: on properties, no ?. on access. Use T | null with explicit guards.
  • At third-party boundaries (Three.js, browser APIs, WebRTC): pragmatic optional chaining is acceptable. Wrap the boundary; keep our domain strict.
  • Function parameters with defaults (fn(x: number = 0)) are a different concept and remain allowed.
  • "Field is semantically inapplicable to this kind of entity" is not an optional field. It is a discriminated union. Express it that way.
  • In numeric hot loops, prefer a sentinel value (targetId = -1, tetherId = -1) over number | null. It avoids both the boxed null check and the shape variance, and it composes with WASM SoA storage.

Migration direction

This is a direction, not a hard cutover. We are not freezing the codebase to flip a lint switch. The rule we live by is simple: every file we touch should leave with fewer optional properties and fewer optional-chaining accesses than it had on the way in. Convert the field, fix the callers in that file, and move on. Over time the migration completes itself, the lint rule follows once the burden is small, and we stop shipping bugs that hide behind silent absence.

What New Code Should Optimize For

  • Keep the authoritative sim headless and portable. It should not depend on DOM, Vue, or Three.js.
  • Describe shared physical behavior once, then reuse it from both server authority and client prediction.
  • Keep the client prediction sim bounded to responsiveness and presentation unless a full-system prediction need is proven.
  • Keep rendering decoupled from simulation cadence; requestAnimationFrame samples state, it does not define truth.
  • Keep commands small, direct, validated, and host-owned.
  • Keep snapshots sparse, recipient-aware, and low cadence by default.
  • Keep client prediction physically compatible with the server, especially for motion and projectiles. Ship velocity, not acceleration: the server owns force inputs, the wire ships their integrated result, and the client extrapolates from velocity only.
  • Expose the per-channel EMA controls (movement position, movement velocity, rotation position, rotation velocity). Position channels use snap/fast/medium/slow; velocity channels additionally allow ignore so playtesters can tune each channel independently without rebuilds.
  • Make mass and force first-class fields in gameplay designs, not decorative metadata.
  • Drive locomotion with velocity-aware arrival control, not position-only waypoint seeking. Units should brake against their current velocity when stopping distance runs out, and ground, hovercraft, and air profiles should share this controller while differing through physical limits.
  • Apply gravity to every mass body every tick; guided projectiles cancel it inside their homing equation by adding a counter-gravity thrust term, not by setting an ignoresGravity flag that skips integration.
  • Keep locomotion visuals entirely client-side: the sim publishes one chassis point, the rig places every wheel, cleat, foot, or fan in the world. Each rig clamps every part it owns to max(naturalY, terrainY + clearance) as a hard invariant — tunneling is never allowed, floating over dips is.
  • Carry per-part visual state in the same four channels client prediction uses — movement position, movement velocity, rotation position, rotation velocity — and EMA each channel toward its target every frame. Animation always plays off body motion; the rig never branches on whether a part is touching ground. The floor clamp drives the position channel; the body-motion signed distance drives the velocity channel; rotation integrates from rotation velocity. Hover replaces all of this with a per-frame clearance scalar.
  • Keep locomotion, unit, turret, shot, and building definitions atomic, swappable, and free of overlapping responsibility.
  • Keep host command targets separate from turret lock-on targets. A unit or building may carry player intent such as attack, guard, repair, or assist; each mounted turret may inherit that target only if its own exclusion policy and targeting gates accept it.
  • Keep lock-on acquisition in turret blueprints and targeting kernels. Rockets inherit the spawning turret’s lock at launch, but they never scan for replacement targets; turret eligibility is broad by default and narrowed through level-0 friend/enemy plus entity-family exclusions, then level-1 named blueprint exclusions.
  • Treat surface materials as shape-independent contracts: a material defines reflectivity, occlusion, hit response, and shader treatment once, and every shape that mounts it must look and behave identically. The mirror turret’s flat mirror panels and the force-field sphere are the exact same material — they differ only in geometry.
  • Keep every atomic action and rule in one source-of-truth module, then import it everywhere it is needed.
  • When a replacement lands — a WASM kernel for a JS loop, owned shaders for a framework helper, a raw renderer path for a Three.js abstraction, a removed library — delete the predecessor in the same change. Migration is not done until the old path is gone from the tree.
  • Prefer explicit null (or numeric sentinels) over optional properties and optional chaining in our own code; treat every file you touch as an opportunity to migrate fields off ?: and ?..
  • Use Rust/WASM, TypeScript orchestration, and custom GPU shader code; keep frameworks out of simulation and rendering hot paths.
  • Stack the runtime so TypeScript coordinates and WASM computes: the TS shell handles boot, UI, IO, and WASM calls; the Rust/WASM engine owns every per-unit and per-tick loop; WebGPU/WGSL owns the GPU buffers, instancing, and compute work. Hot data should cross the JS/WASM boundary as rarely as the design allows.
  • Author blueprints, tuning tables, maps, and other content as JSON data validated by a shared schema, not as TypeScript modules. Data the Rust/WASM simulation cannot read is a wall that prevents the TS-to-Rust migration from ever completing.
  • Move dense numeric kernels toward WASM when batching can beat JS object loops.
  • Design every feature with 5,000 units in mind before celebrating it at 50 units.
Document purpose: align implementation decisions around a browser-native, massive-scale, force-and-mass RTS where the authoritative sim corrects a continuous client sim instead of streaming every frame.
