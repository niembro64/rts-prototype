Migration: Authoritative-Host-With-Client-Prediction → Deterministic Lockstep

A plan to refactor this project's networking model from a server-authoritative snapshot architecture (think Overwatch / Rocket League) to a deterministic peer-to-peer lockstep architecture (think Age of Empires / StarCraft / Beyond All Reason).

Architecture: before vs. after

AspectCurrent — "Authoritative Host with Client Prediction"Target — "Deterministic Peer-to-Peer Lockstep"
Who simulates? One host runs the authoritative sim (GameServer.ts, 60 Hz). Clients run a predictive copy of the same Rust sim and reconcile against snapshots. Every peer runs the identical simulation. No peer is "authoritative" — they're all simulating the same inputs in the same order and must agree bit-for-bit.
What goes on the wire? Packed snapshots (entities, projectiles, audio events, minimap, visibility, economy, …) at ~10 Hz from host to each client. Commands flow client → host. Huge codec surface in src/game/network/. Only player commands (build, move, attack, …) tagged with the tick on which they execute. Bandwidth drops by 1–2 orders of magnitude.
Latency model Local input is instant via prediction; rollbacks/reconciliation hide host RTT. Cheap to feel responsive. Commands are scheduled for tick N + k where k is an input-delay window (≈ slowest RTT in half-ticks). Everyone waits for everyone before advancing.
Replays / spectate Would require recording snapshots → large files. Not currently implemented end-to-end. Replays are the command log + initial seed. Megabytes for hour-long games. Spectators just run the sim from inputs.
Cheating surface Host is authoritative; clients see only what the host reveals (fog of war filtered server-side). Every peer has the full game state in memory. Maphacks are easy. Acceptable for friends-only / community RTS; not for ranked competitive without a relay-and-filter server.
Failure modes Host disconnect = lobby dies (unless host-migration is built). Client desync is rare since host is the ground truth. Any peer can drop and others continue. The new boss is desync: any divergence between peers (a stray HashMap iteration, an f32::sin that differs by 1 ULP) corrupts the game silently.
Transport WebRTC DataChannels via peerjs; mesh of host↔client connections; large binary snapshots. WebRTC DataChannels, full mesh among peers (or one peer elected as command-relay). Tiny messages, high frequency.
Tab backgrounding Background clients lag but the authoritative host keeps the world advancing. One backgrounded peer halts the entire game until they catch up. Must run the sim in a Web Worker (workers throttle far less than the main thread).

Why this is feasible in a browser

  • WASM gives deterministic floats. The WebAssembly spec mandates bit-exact IEEE-754 for basic ops across all conforming engines. The classic "lockstep desync from cross-platform float drift" problem that Spring fights with native code is largely free here.
  • The sim is already one crate. rts-sim-wasm/ is already used by both host and client prediction. The "single sim core that runs identically everywhere" prerequisite for lockstep is done.
  • WebRTC DataChannels are roughly UDP-equivalent and already wired up via NetworkManager.ts.
Determinism traps that remain even in WASM:
  • f32::sin, cos, tan, exp, ln, pow are not bit-spec'd by WASM. They compile to host libm calls that vary between Chrome (V8 / glibc) and Firefox (SpiderMonkey / fdlibm). Audit rts-sim-wasm/src/lib.rs and replace any transcendentals on the hot path with a deterministic implementation (small Taylor expansions, table lookup, or the libm crate which ships a portable softfloat implementation).
  • HashMap / HashSet iteration order is non-deterministic. Use BTreeMap, BTreeSet, or IndexMap for any collection whose iteration affects sim outcomes (and most do — spawn order, projectile resolution order, …).
  • SystemTime, Instant, thread_rng() — none of these may leak into the sim. RNG must be a seeded, explicit, in-state SmallRng (or similar) that the sim itself advances.
  • Floating-point reductions whose order is implementation-defined (e.g. SIMD horizontal sums, rayon parallel folds) — keep the sim single-threaded.

Migration phases

Phase 0 — Determinism audit (do this before any networking change)

Goal: prove the existing Rust sim, when fed the same inputs and seed, produces a bit-identical world hash on every browser. Until this holds, lockstep is impossible.

  • Add a fn world_hash(&self) -> u64 to the sim — a stable hash over all sim state (positions, hp, build state, RNG state). Order it deterministically.
  • Add a "replay from inputs" test harness: record a real game's commands + seed; replay them headlessly; assert the per-tick hash is constant across runs.
  • Run the same replay in Chrome, Firefox, Safari (via Playwright). Any divergence → grep the diff to find the offending op (almost always a transcendental or an unordered iteration).
  • Purge transcendentals or swap to libm.
  • Replace any sim-affecting HashMap with BTreeMap / IndexMap.
  • Grep for SystemTime, Instant, thread_rng, random() in sim crate — none allowed.

Exit criterion: 10-minute replay produces identical per-tick hashes across all target browsers, run 20× back-to-back.

Phase 1 — Move the sim into a Web Worker

Goal: prevent tab-backgrounding from gating the entire match. Also a prerequisite for clean tick gating.

  • Create src/game/sim-wasm/simWorker.ts. Load the WASM module inside the worker.
  • Main thread sends commands; worker sends back a snapshot of view-only state for the renderer per tick (this is local, not network — full state can cross the postMessage boundary cheaply via transferables).
  • This is largely orthogonal to networking; ship it on its own and validate the renderer still works.

Exit criterion: background a tab for 30 s, foreground it — sim caught up smoothly, no main-thread starvation.

Phase 2 — Introduce a command-scheduling layer

Goal: stop executing commands "now" and start executing them at a designated future tick. This is the philosophical core of lockstep and can be introduced before the network change — running locally, it should be invisible.

  • Define ScheduledCommand { tick: u64, player_id: u8, cmd: Command }.
  • The sim's tick loop pulls all commands tagged with the current tick from a sorted ring buffer and applies them in a deterministic order (sort by (tick, player_id, seq)).
  • Local UI no longer mutates sim state; it enqueues a command for tick current + INPUT_DELAY (start with INPUT_DELAY = 6, i.e. 100 ms at 60 Hz).
  • This will feel laggy in single-player. That's correct — it's the price of the architecture, and you want to feel it now so you can tune it.

Exit criterion: single-player still plays correctly. All commands round-trip through the scheduler. The renderer is happy reading a slightly-delayed sim state.

Phase 3 — Two-peer lockstep over the existing WebRTC transport

Goal: replace the host-authoritative loop with a peer-symmetric loop for a 1v1 only. Strip nothing yet — run lockstep alongside the existing snapshot code under a feature flag.

  • Add NetworkCommandRelay: every tick, each peer broadcasts { tick: N + INPUT_DELAY, commands: [...] }, including an empty array if they did nothing.
  • The sim only advances tick N once all peers' command packets for N have arrived. This is the "lockstep barrier".
  • Every K ticks (e.g. K = 60), each peer broadcasts { tick: N, world_hash: u64 }. If hashes diverge: log everything, pause, surface a desync error. Do not try to recover automatically yet.
  • Keep GameServer.ts alive but disabled when the lockstep flag is set. Do not delete the snapshot codecs yet.

Exit criterion: 1v1 LAN game runs for 20 minutes with no desyncs. World hashes match every tick on both peers.

Phase 4 — Adaptive input delay and slowest-peer handling

Goal: make lockstep feel acceptable on real networks.

  • Measure RTT to every peer continuously (already partially done by NetworkHeartbeatTracker.ts).
  • Set INPUT_DELAY = ceil(maxRTT / tickInterval) + slack. Renegotiate on big swings (Spring does this via "CommandFrame" adjustments).
  • When the barrier stalls waiting on a peer, show "Waiting for <player>…" overlay. (Spring's classic UX.)
  • Pre-emptively send empty command packets even on idle ticks so the barrier never starves.

Exit criterion: playable feel on 80 ms RTT. Acceptable feel on 200 ms RTT. Graceful degradation past that.

Phase 5 — N-peer mesh and command-relay election

Goal: scale beyond 1v1.

  • Option A (simple): full WebRTC mesh, every peer sends commands to every other peer.
  • Option B (more BAR-like): elect one peer as command relay. Everyone sends commands to the relay; the relay re-broadcasts the combined per-tick command bundle. Halves bandwidth and removes asymmetric arrival ordering.
  • Recommend Option B once you go past 4 players — it's how Spring/Recoil's autohost works.

Exit criterion: 4v4 game runs cleanly.

Phase 6 — Replay capture and load

Goal: monetize the architecture's biggest free win.

  • The "wire format" of a game is the replay format. Persist the initial seed + game settings + the full ordered command stream to a file. Should be < 5 MB for an hour-long 8-player game.
  • "Play replay" = boot the sim with the seed and feed the command stream through the scheduler as if it came from the network.
  • Add scrubbing by allowing the sim to re-simulate from start to any chosen tick (since the sim is fast and deterministic, this is cheap up to a few minutes; beyond that, capture periodic state snapshots as resume points).

Exit criterion: any completed match can be loaded and watched. Bonus: spectate live mid-game by replaying-then-tailing.

Phase 7 — Decommission the snapshot path

Goal: delete the old code with confidence.

  • Remove the feature flag — lockstep is the only path.
  • Delete GameServer.ts, ServerSnapshotPublisher.ts, ServerTickLoop.ts, all snapshot codecs in src/game/network/ that no longer serve a purpose (snapshot*.ts, stateSerializer*.ts, the Client*Prediction* reconciliation layer).
  • Keep: NetworkManager.ts, NetworkHeartbeatTracker.ts, NetworkLobbyRoster.ts, NetworkRoomCode.ts — these are still useful. NetworkCommandTransport.ts becomes the only message type, hugely simplified.
  • ClientViewState becomes "a view onto the local sim", not "a reconciled view onto a remote sim". Selection/UI state stays local as it already does.

Exit criterion: src/game/network/ shrinks by at least 60%. git log --stat shows the deletion.

Risks and explicit non-goals

  • Risk: an undetected desync source survives Phase 0. Mitigation: per-tick hash check in dev builds for the entire migration; never ship without it.
  • Risk: input-delay feels worse than the current prediction model. This is real and unavoidable; the win is elsewhere (replays, unit counts, bandwidth, simplicity). Tune INPUT_DELAY aggressively and consider a small client-side "command hint" animation (e.g. show the move-order flag immediately, even though the unit only starts moving 6 ticks later — Spring does this).
  • Non-goal: rollback netcode (GGPO-style). That's a different architecture again — predict-and-rollback — and would require the sim to support cheap state save/restore. Out of scope.
  • Non-goal: anti-cheat. Lockstep is for friendly play. Competitive ranked play would need a server-side observer that also runs the sim and validates command legality — a much later concern.

What "done" looks like

An 8-player game plays smoothly on residential connections. Every peer holds the full game state. Disconnecting any non-relay peer doesn't end the match. A replay of any game is a small file that re-plays bit-identically across browsers and across versions of the sim that haven't broken determinism. The networking directory is half its current size, and the host/client mental model is gone.
