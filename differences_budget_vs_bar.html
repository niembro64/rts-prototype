Budget Annihilation vs. Beyond-All-Reason

Authoritative simulation, server–client relationship, and command/update flow — a side-by-side architectural comparison.

Budget Annihilation is a from-scratch browser-native RTS written in TypeScript with hot-path simulation kernels in Rust/WASM, using WebRTC for transport. Beyond-All-Reason is a community RTS built on top of the Recoil engine (a maintained fork of the original Spring RTS engine), descended from a lineage of LAN-era real-time strategy games.

The two engines sit at almost opposite ends of the design space for networked RTS. They both want the same end-user experience — thousands of units, dozens of players, no rubber-banding — but they reach for radically different machinery to get there. The headline difference is that Budget Annihilation transmits state (the server is the only thing that “really” simulates, and snapshots of its truth flow out to viewers), while Beyond-All-Reason transmits commands (every machine simulates the same game in lockstep, and only player inputs cross the wire).

That single distinction cascades into nearly every other architectural choice, from where the code lives to what kinds of cheats are possible to whether late-joining a game is even meaningful.

1. Authoritative Simulation

Both engines have a notion of an “authoritative” simulation, but the word means different things in each. In Budget Annihilation, “authoritative” means a particular process owns the canonical world and everyone else mirrors what it tells them. In Beyond-All-Reason, it means a set of deterministic rules that every machine runs in parallel, identically.

Single authoritative process, snapshot-streamed

One GameServer instance owns the world. It runs at a configurable tick rate (default 32 Hz, range 1–512 Hz) driven by a setInterval loop in ServerTickLoop. Each tick:

  1. Update the simulation (commands, AI, construction, economy).
  2. Step the standalone Matter.js / Rust physics.
  3. Sync physics back into entity state.
  4. Update capture/territory.

Hot-path numerics live in a Rust/WASM module (rts-sim-wasm): body integration, sphere–sphere and sphere–cuboid contact resolution, voxel-hash spatial grid, projectile motion, terrain sampling, A* pathfinding. Higher-level rules (combat resolution, factory queues, build orders, energy distribution) stay in TypeScript on the server.

Determinism in Budget Annihilation is a property rather than a contract. The Rust physics kernels run on both server and client, so the client can extrapolate motion bit-for-bit between snapshots — but the simulation does not have to stay in sync, because the next snapshot will correct any drift.

Replicated lockstep, every client simulates

There is no “the server is the world.” Instead, every connected machine runs the entire game simulation at a fixed 30 Hz game-frame rate, and the engine guarantees that they all reach the same state on the same frame. The server's job is to coordinate, not to compute the world.

All state lives in two layers:

  • Engine core (C++): unit positions, pathing, physics, LOS/radar, resource flow, AI — everything that demands fast, fully deterministic execution.
  • Synced Lua gadgets (luarules/gadgets/): higher-level gameplay overlays. Each gadget's GameFrame(n) callin fires once per simulation tick, with read/write access to engine state via the Spring.* API.

Determinism here is the load-bearing contract. If one machine's f64 sum lands a ULP off, or its random seed drifts, the game suffers a sync error and effectively ends. The codebase carries hardened infrastructure for this: dbg_synctest.lua spawns every unit category and compares per-frame sync-hash artifacts; /luarules desync intentionally injects divergence to test detection paths.

The asymmetry

In Budget Annihilation, a deterministic kernel is a performance tool: it lets the client predict between sparse server updates. In Beyond-All-Reason, a deterministic kernel is an existential tool: the game cannot continue without it. The same property serves opposite roles.

2. Server-Client Relationship

Beyond different sync models, the two engines also disagree about what a “server” even is. Budget Annihilation treats the server as a privileged simulator with clients as snapshot-consuming viewers. Beyond-All-Reason treats the server as a network coordinator and command relay; the simulators are all the clients themselves.

Host-on-client with a clean abstraction layer

The codebase splits orchestration cleanly through the GameConnection interface:

  • LocalGameConnection — same process; the host's renderer talks to its own GameServer in-memory. Snapshots are cloned (via ReusableNetworkSnapshotCloner) to avoid aliasing, but never serialized.
  • RemoteGameConnection — goes over WebRTC. Snapshots are msgpack-encoded (with an optional Rust-accelerated entity codec) and shipped over PeerJS DataChannels. Commands flow back the same way.

In practice the host's browser tab is the “server”: GameCanvas.vue owns the server lifecycle. The RtsScene3D renderer is uniform — it always paints ClientViewState, regardless of whether the local user is host or guest. There is no networkRole branching in render code.

Server-side authorization runs on every inbound command in GameServer: ownership is verified, admin-only commands are restricted to player 0, mismatched orders are silently dropped before they ever enter the command queue. Backpressure-aware snapshot dropping protects the data channel when a client falls behind.

Chobby launches a relay; clients self-simulate

The runtime topology is split across multiple processes:

  • Chobby — the BAR lobby/launcher (a separate project). Handles accounts, matchmaking, mod/map distribution, and game start.
  • Spring/Recoil binary — launched per match in “server” mode by Chobby (via a generated script.txt). Other players' engines connect to it.
  • Client engines — each player's own Spring/Recoil binary, running the same simulation in lockstep with the others.

The server's actual responsibility is narrower than in Budget Annihilation: it receives commands from all clients, broadcasts them with a tick-aligned schedule, and validates checksums for desync detection. It does not own the game state in any privileged sense — it runs its own deterministic copy alongside everyone else's.

Communication back to Chobby (game results, scenario completion) happens via Spring.SendLuaMenuMsg(...) from in-game Lua, e.g. "ScenarioGameEnd ...". The lobby is the launcher and the post-match home; it is not part of the runtime simulation loop.

Mid-game host migration is not really a thing. If the server drops, the match is over. Late-joining a running game is similarly weak: the client would need to download a partial replay and fast-forward to live, which is rarely supported in practice.

3. Command and Update Flow

This is where the two engines look most different in day-to-day code. Following a single right-click move order from the player's mouse to the unit changing direction on every screen is a useful way to see the shape of each system.

Budget Annihilation: command in, state out

[Client] mouse click → InputManager (uses InputContext: getTick(), activePlayerId) → MoveCommand object built locally → selection state updates immediately on local ClientViewState (cosmetic) → GameConnection.sendCommand(cmd) → LocalGameConnection: enqueue directly in server's CommandQueue → RemoteGameConnection: JSON over WebRTC DataChannel [Server (host's tab)] GameServer.receiveCommand(cmd, playerId) → authorize: ownership check, admin-only gate, drop if invalid → enqueue into per-tick CommandQueue → next tick: Simulation.update() pops & applies the command entity unit.actions updated physics integrates next step → ServerSnapshotPublisher.emit() (gated by maxSnapshotIntervalMs, default 8 Hz) → serializeGameState() builds NetworkServerSnapshot → per-listener delta vs. keyframe decision → msgpack (or Rust wire codec) over DataChannel [Client] RemoteGameConnection.onSnapshot(bytes) → decode → ClientSnapshotApplier.apply(snapshot) → snap non-visual fields (hp, build progress, turret state, actions...) → set server-target pos/vel; ClientUnitPrediction lerps render position → UIUpdateManager refreshes selection panel, top bar, minimap → RtsScene renders next frame from ClientViewState (no sim, no branching on role)

Notice how the wire only carries two payload categories — commands in one direction, snapshots in the other — and how the client's job is reduced to “believe what the server says, with prediction filling the gap.”

A snapshot is a structured object: per-entity position/rotation, optional unit block (hp, velocity, action queue, action-hash, turret states, suspension, jump, combat mode), optional building block (hp, turrets, factory build queue, solar open flag), plus global slots for economy, audio events, projectiles, spray targets, capture state, game phase, and server meta (TPS, CPU load). Position is quantized to 0.01 world units; velocity to 0.1; rotation to 0.001 rad. Anything that hasn't changed beyond its quantum threshold is omitted from the delta's bitmask, so most ticks of most entities never make it onto the wire.

Keyframes interleave with deltas at a configurable ratio. Joining a game is just “wait for the next keyframe”: the host marks the new listener as not-ready, emits startup keyframes regardless of normal scheduling until the client confirms it's caught up, then transitions to normal delta cadence.

Beyond-All-Reason: command in, command broadcast, state never moves

[Client A: player issues an order] mouse click → unsynced widget (luaui/Widgets/*.lua) → Spring.GiveOrderToUnit(unitID, CMD.MOVE, params, opts) (or Spring.SendLuaRulesMsg for custom gameplay messages) [Engine, unsynced → network] → engine queues the command in the outbound buffer for the current frame → engine sends the command to the Spring server [Server] → aggregates commands from all players for frame N → broadcasts the frame's command set to every client (including itself) → collects per-frame sync checksums; mismatches surface a desync [Every Client: identical processing on frame N+1] Engine processes the broadcast command set → CMD.MOVE applied to unit's engine-level command queue → synced gadgets' GameFrame(n) callin runs (luarules/gadgets/*.lua, if gadgetHandler:IsSyncedCode() then ...) can modify state via Spring.*, dispatch SendToUnsynced("...", ...) → engine integrates physics, pathing, combat, resource production deterministically [Every Client: render & UI] → unsynced widgets receive SYNCED-table updates and SendToUnsynced messages → widgets re-render command queues, health bars, minimap, etc. → user sees their unit move at the same frame as every other player

The wire here carries effectively only commands (plus periodic sync-check hashes and lobby/chat traffic). Crucially, the state changes that follow a command are computed independently on every client's simulator. The synced gadget that runs on player A's machine runs an identical copy on player B's machine, with identical inputs and identical RNG seed, and produces identical output. Bandwidth is constant in match size: a hundred players issuing a hundred commands per second costs the same as a hundred players issuing a hundred commands per second, regardless of whether there are 200 units or 20,000 in play.

The synced/unsynced split is the architectural lynchpin and has no Budget-Annihilation analogue:

  • Synced gadgets (luarules/gadgets/): part of the deterministic simulation. Run on every machine. Can modify game state. Detected by the presence of SendToUnsynced in scope.
  • Unsynced gadgets (same files, after the else): client-side only. See a read-only SYNCED table snapshotted per frame. Cannot mutate the game.
  • Widgets (luaui/Widgets/): always unsynced. Pure UI. Can issue orders (which route through the engine and re-enter the synced layer everywhere) but cannot directly change state.

Communication crosses the synced/unsynced gap via SendToUnsynced("name", ...) downward and Spring.SendLuaRulesMsg(...) + gadget:RecvLuaMsg(msg, playerID) upward. The downward path is purely informational (UI hints); the upward path becomes a regular command and is replayed identically on every client.

In Budget Annihilation the analogous separation is between ClientViewState (everything the renderer can touch) and WorldState (everything the server owns). But it's not nearly as load-bearing: the client view is a read-only mirror of recent server truth, not a parallel-simulated copy. There is no determinism contract between them.

4. Trade-offs and Implications

Bandwidth profile

Budget Annihilation's wire cost scales with the size of the visible world and the rate of change in it. Quantization, delta encoding, and per-recipient fog-of-war shrouding all exist to keep this in check. In games with hundreds of buzzing units, snapshot size dominates network cost. Beyond-All-Reason's cost scales with player command rate — clicks per minute, essentially — and is independent of unit count. This is why Spring-lineage RTSes were viable on modem-era links with 2000-unit armies and why “huge battle” gameplay is so natural to them.

Latency and felt responsiveness

Budget Annihilation hides latency aggressively: selection feedback is local, the client predicts unit motion using the same Rust physics that the server runs, and a drift-correction lerp smooths over snapshot arrivals. The user sees an immediate, plausibly-correct world that is occasionally nudged toward server truth.

Beyond-All-Reason cannot hide command latency in the same way. By construction, a command issued on frame N executes on frame N+k (where k is the simulation's networking buffer). There is no local execution before the network round-trip, because that would break determinism. Spring-lineage RTSes traditionally accept this delay as the cost of doing business; the genre's pacing is slow enough that 100–200 ms of input lag is rarely fatal.

Cheating and trust

Server-authoritative architectures let Budget Annihilation enforce ownership and validity per command: a client cannot tell the server to move a unit it doesn't own, because the server simply drops the order. The client never sees state it shouldn't see — fog of war is computed per-recipient before serialization. This is the standard modern-multiplayer trust model.

Beyond-All-Reason inherits Spring's “trusted client” assumption: every machine has a copy of every unit's position because every machine is simulating. Map hacks (revealing the fog) are technically just a tweak to the local UI. The deterministic property partly compensates: a tampered client's simulation will desync, the sync-check hashes will catch it, and the offending player will be flagged. But the protection is statistical and post-hoc rather than preventive.

Late join and spectators

Adding a viewer in Budget Annihilation is trivial — the server registers a new snapshot listener, sends a keyframe, opens the “ready” gate, and the new client is fully caught up. Spectator support and post-hoc replay are essentially the same feature with different storage. Lockstep is the opposite: to spectate a running game you must download the entire command history so far and fast-forward your local simulator to the current frame. It can be done, but it's expensive and brittle.

Replay storage

In Beyond-All-Reason, the command stream is the replay. The file is small, perfectly faithful, and trivially re-derivable by re-running the simulation. In Budget Annihilation, replays would have to be a sequence of snapshots (or a command log plus initial state, but only if the server's simulation is exactly replayable — currently it isn't a hard guarantee). Snapshots are larger, but they don't depend on the simulation being deterministic in the future.

Game-logic location and iteration speed

Budget Annihilation centralizes game rules in TypeScript on the server. Adding a new ability or unit type means changing one file path that runs in exactly one place. There is no “does this run on every machine identically?” question to answer.

Beyond-All-Reason's synced gadget model is more powerful and more demanding: any contributor can add a new mechanic by dropping a Lua file into luarules/gadgets/, and it will replicate naturally. But that contributor must satisfy the determinism contract — no os.time(), no native floating-point shortcuts, no calls into the unsynced API from synced code — or risk introducing a desync that can be hard to track down.

Where each architecture earns its keep

The snapshot model wins when…

  • You want low input latency and modern feel.
  • You want anti-cheat and per-player information hiding (fog of war that's actually fog).
  • You want trivial spectator support and easy late join.
  • You want game logic to live in one place, debuggable like any normal server app.
  • You're running on a web platform (or any environment without bit-exact float guarantees across machines).
  • You're comfortable spending bandwidth to buy these things.

The lockstep model wins when…

  • Unit counts are enormous — thousands of independent agents.
  • Bandwidth is the binding constraint (LAN, modem, mobile).
  • The genre's pacing tolerates fixed input delay.
  • You want replays for free, with perfect fidelity, at no extra runtime cost.
  • You're willing to take on the engineering discipline of a hardened deterministic simulation.
  • The community can plausibly extend game rules via a modding API (Lua) without the server team mediating.

5. Where Budget Annihilation Sits Architecturally

Despite being thematically inspired by Beyond-All-Reason (and Total Annihilation/Supreme Commander before it), Budget Annihilation's networking has more in common with modern action games — Overwatch, Valorant, Halo — than with its RTS forebears. The defining choices:

  • Authoritative server with client-side prediction. Classic Quake-lineage approach, adapted for an RTS's wider entity counts via aggressive quantization and per-recipient delta tracking.
  • Decoupled tick and snapshot rates. Simulation runs at 32 Hz; snapshots emit at 8 Hz. Most engines tie these together; the decoupling here lets the sim get the precision it needs without paying for it on the wire.
  • Determinism as an optimization, not a contract. Rust/WASM kernels are shared between server and client so the client can predict accurately, but the simulation is still fundamentally one-way: server speaks, clients listen and smooth.
  • Web-native transport. WebRTC DataChannels via PeerJS, with a clean abstraction (GameConnection) so the same code runs against an in-memory local server or a remote peer.
  • Uniform client. RtsScene doesn't care whether you're the host or a guest. The same rendering and input pipeline reads ClientViewState in both cases; only the GameConnection implementation differs.

Beyond-All-Reason represents the alternative path: take the deterministic-simulation contract seriously, push every drop of complexity onto the engine to make that contract hold, and reap the bandwidth and replay benefits. Both are good answers. They're just answers to subtly different questions.
