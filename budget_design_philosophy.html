Design Philosophy: Shared Simulation, Explicit Authority, Full Physics

This prototype is a browser RTS inspired by the command scale and battlefield clarity of Beyond All Reason, but its technical identity is stricter: every meaningful physical interaction should reduce to mass, force, acceleration, velocity, position, collision, and correction.

Authority is explicit Budget Annihilation has two implemented backend architectures, selected through architecture.json. In authoritative-server, client commands go to the host simulation, which validates and executes them. In deterministic-lockstep, the command-frame stream is authoritative while every browser runs the same server simulation locally.
Clients stay continuous The client runs a predicted local shadow on its own cadence, and the renderer samples that shadow every frame while server snapshots correct it only a few times per second.
Physics is the language Movement, recoil, collision, hover, beams, blast pressure, and terrain contact are force and mass problems before they are gameplay effects.

Design Inheritance

Budget Annihilation, the rts-prototype repo, is our browser-native extension of Beyond All Reason. This document describes the places where the prototype deliberately differs: authoritative server networking, client prediction, force-and-mass physical systems, browser constraints, selectable deterministic lockstep, owned simulation code, and the atomic gameplay pieces this project adds or reshapes.

When this document does not describe a gameplay or player-facing behavior, the default is Beyond All Reason as implemented in the Beyond-All-Reason repo. That fallback covers command ergonomics, battlefield expectations, economy feel, construction and production behavior, unit control language, selection behavior, minimap/radar/sight expectations, and any other ordinary RTS behavior this document leaves unstated.

  • Rules written in this document override the BAR fallback.
  • Intentional departures from BAR should be written into this document.
  • BAR is the behavioral fallback, not a requirement to copy BAR code, Lua structure, assets, Recoil/Spring engine internals, or network assumptions.
  • When BAR behavior depends on Recoil/Spring details, translate the player-facing behavior into Budget Annihilation's browser, physics-first architecture. Authoritative-server and deterministic-lockstep are both implemented backend modes; architecture.json selects which one a match boots with.

North Star

Budget Annihilation now has two implemented backend architectures that can both be selected today through architecture.json. In authoritative-server, one server simulation owns gameplay truth and clients render from snapshots plus bounded prediction. In deterministic-lockstep, every browser runs the same server simulation locally and peers exchange ordered command frames; snapshots become local presentation data instead of multiplayer truth. We are still not building a thin renderer that waits for the host to stream every position at display rate.

In authoritative-server, the host simulation owns the real world: commands, ownership, economy, construction, targeting, damage, projectile hits, physics bodies, collision response, fog of war, victory, and persistent state. The client simulation owns presentation: local command feedback, prediction, smoothing, camera-visible presentation, audio scheduling, and render-time interpolation. In deterministic-lockstep, that same server simulation runs locally in each browser, and the ordered command-frame stream owns multiplayer truth.

That split is architectural, not permission for accidental duplicate game loops. A browser runtime may own only one active gameplay-truth simulation at a time. Booting a second sim in the same browser is a hard error, whether it comes from a duplicate worker, a second game session, or a stale teardown path. Throw immediately rather than letting two tick loops share input, networking, render state, or authority.

The preferred implementation is shared code, with authority chosen by architecture. Physical kernels and battlefield contracts should be described once and reused by both the authoritative sim and the client prediction sim where that reuse buys correctness: body integration, forces, projectile motion, terrain sampling, turret pose, hover, and collision math. That does not mean authoritative-server clients must run every authoritative system. Economy, construction truth, target ownership, fog filtering, damage resolution, entity creation and removal, and victory stay host-owned in authoritative-server. In deterministic-lockstep, those systems run in every browser as the local copy of the same server simulation.

The target networking feel is: commands arrive at the authoritative sim immediately, while the authoritative sim only needs to update the client sim about 5 to 10 times per second in normal authoritative-server play. Higher snapshot rates are a diagnostic or low-unit-count luxury, not the architecture. In deterministic-lockstep, command frames and checksums are the network truth; local snapshots are for rendering and diagnostics.

Engine Ownership: In-House Simulation, Practical Renderer

The end-state engine is something we own end-to-end. The simulation is Rust compiled to WebAssembly, living in rts-sim-wasm/. The renderer may be built on Three.js, raw WebGL2, or raw WebGPU where each makes sense, but the high-count paths must still be ours: owned typed buffers, explicit update ranges, deliberate shader code, predictable batching, and measured draw-call behavior. Networking glue, input plumbing, asset pipeline, audio mixer, math kernels, ECS, physics, collision, pathfinding, fog of war, snapshot serializer, prediction integrator — all written by us, for this game. The defining trait of this project is that we are building the engine, not assembling one.

A major goal of this project is to move as much code as possible, whenever possible, out of JavaScript/TypeScript runtime ownership. If a simulation or data-transform system can reasonably live in Rust/WebAssembly, raw GPU buffers, or shader code, that is where it belongs. JavaScript and TypeScript are coordination layers, not the place we leave game logic, physical kernels, serialization loops, or high-count data transforms once a Rust/WASM or GPU-buffer path is viable.

The libraries currently in package.json (three, vue, peerjs, @msgpack/msgpack, @tonejs/midi, @tauri-apps/*) are tools with different budgets, not all equal sins. Three.js is acceptable as the active renderer shell while it lets us ship faster without hiding the high-count data flow. Vue remains UI-only. Networking, audio, and desktop wrappers are allowed where they are boundary glue. The Rust sim crate, owned GPU/shader data paths, and a thin TypeScript orchestration shell are the destination. Every dependency still needs a reason, and every hot path still needs an owned, inspectable contract.

  • The default answer to “should we add a library for this?” is no. Write it ourselves, preferably in Rust, ideally in the sim crate.
  • The default answer to “can we keep this library?” is only if it preserves ownership of the hot path. Three.js may stay when it is a renderer shell over our buffers, shaders, culling, and batching; it should not become an excuse for per-object scene-graph work at battlefield scale.
  • Adding a new npm or cargo dependency requires a deliberate, named justification — not a reflex. Vendoring the few lines we actually need is almost always preferable to taking a whole library on.
  • Libraries that have already been removed (Phaser, Pixi, Matter.js, standalone matter-js) are gone for good. Do not reintroduce them under any architectural argument. They were tried and failed our scaling rules.
  • “In-house” means the behavior, data layout, and performance contract are in our repo, in code we are willing to debug at any layer. A Three.js draw wrapper can still be in-house when the buffers, shaders, update ranges, and culling policy are ours.
  • Whenever a Rust/WASM implementation can replace JavaScript or TypeScript functionality, the Rust/WASM version is the active target and the JS/TS version becomes removal work, not a fallback architecture.
  • The Rust sim crate is the preferred home for new logic. New TypeScript modules are acceptable for orchestration, but new simulation, physics, math, or per-unit-loop code should land in rts-sim-wasm/ from the start, not get written in TS and ported later.

This is the active commitment behind Languages and Frameworks, TypeScript Coordinates, WASM Computes, and Delete The Old Path. Those sections describe what the end-state engine looks like and how migrations land; this section states why we are doing it: nobody else’s engine is going to give us thousands of physically-simulated units in a browser by accident. A library is acceptable only when it helps us reach that target while leaving the hot data flow legible and under our control.

Network Model

Architecture selection is explicit

The active backend is selected before match startup by src/architecture.json. authoritative-server preserves the original host-owned command/snapshot model. deterministic-lockstep runs the same server simulation locally in every browser and makes ordered command frames the multiplayer truth. Both are implemented and currently usable; the architecture is a startup boundary, so changing the config does not hot-flip a running match.

  • authoritative-server: host/server owns gameplay truth; clients send commands up and receive snapshots down.
  • deterministic-lockstep: the ordered command-frame stream owns gameplay truth; every peer steps the same local server simulation and compares checksums.
  • Mid-match migration is a checkpoint-driven operation, not a config reload. A failed migration must roll back rather than leave peers running different truths.

Commands flow upward

A client issues a command and sends it directly to the authoritative host. The host checks player authority, filters illegal unit IDs, clamps unsafe parameters, and enqueues the command into the server command queue. In authoritative-server, the command is not broadcast as peer truth; other clients learn the result through snapshots and events emitted by the authoritative sim. In deterministic-lockstep, gameplay commands are sanitized, assigned future execute frames, ordered canonically, and distributed as reliable command frames. The host may coordinate/relay those frames, but it does not own gameplay truth in that mode.

  • Selection can remain local because it is UI state.
  • Move, attack, guard, build, repair, reclaim, scan, production, and server-control commands must be authorized by the host.
  • Command latency should be hidden with local intent previews, not by letting clients create real gameplay state.
  • Every command path should be stable enough to work for local host play, remote WebRTC play, and a future dedicated worker or server.

Snapshots flow downward

In authoritative-server, server snapshots are corrections, not per-frame animation frames. They should contain changed authoritative targets, creation and removal events, projectile events, audio events, fog and minimap deltas, and occasional keyframes. The client stores those values as server targets, predicts forward locally, and drifts toward the newest authoritative target.

  • Normal target cadence is 5 to 10 snapshots per second.
  • Server tick rate and snapshot rate are independent.
  • Full keyframes are rare recovery points. Delta snapshots are the default.
  • Area-of-interest and fog filtering are server responsibilities.
  • Backpressure should drop or coarsen snapshots before it lets the browser build an unbounded send queue.

Lockstep snapshots are presentation-only

In deterministic-lockstep, each browser generates local snapshots from its own local server simulation so the existing renderer can keep using the snapshot intake path. Remote snapshots are not gameplay truth. Dropping a lockstep snapshot may affect render smoothness or diagnostics, but it must never change canonical simulation state. Command frames and checksums are never coalesced by snapshot backpressure.

  • Every lockstep frame uses a fixed 1000 / 60 ms simulation step.
  • Missing command frames stall instead of causing peers to guess different futures.
  • Checksums are compared by frame. A mismatch pauses/desyncs the match and produces an exportable report with initialization hash, state-hash sections, peer ids, and recent command frames.
  • First-release lockstep support is intentionally narrow: real battles only, no AI players, no late join, no reconnect, no automatic resync, no host migration, and no spectators unless they are roster seats from frame 0.

Fog behavior and fog visuals are separate controls

The DEMO BATTLE and REAL BATTLE bars own whether fog of war is functionally active. That battle-level FOG OF WAR control governs authoritative player vision, radar coverage, recipient-specific snapshot filtering, and whether enemy units outside vision exist in the client view at all.

The PLAYER CLIENT bar owns only fog presentation. Its DEBUG: controls may show or hide the sight boundary, the live dark shade over currently unseen terrain, and the soft fog cloud puffs. Fog visual colors and shade alpha belong in colorsConfig.json; fog cloud behavior and geometry belong in fogConfig.json. Those controls must not change gameplay truth, server filtering, radar semantics, or what entity data is delivered to the player.

There is no persistent black explored/unexplored shroud. We should not track or render “ever seen” terrain history as a separate black map state unless it is deliberately reintroduced as a gameplay feature with its own design justification. The current visual model for unseen areas is live-only: optional shade and optional clouds, both disabled automatically when battle-level fog is off.

Sight, radar, and contacts are separate information tiers

The visibility model needs three named areas, because each answers a different gameplay question. Area of unit sight is the footprint of one full-vision source: a scout, commander, building mast, temporary scan pulse, or future sensor mount. It is useful for debug, overlays, source authoring, and local reasoning, but it is not the player’s whole knowledge of the map. Area of total player sight is the union of all full-vision sources owned by the player and their allies. It grants full entity identity, private strategic detail allowed by ownership rules (commanded intent only, never motion or render state — see Visible units render and update identically below), normal terrain/force-material sight checks, visual events, and the main snapshot entity row. Area of total player radar coverage is the radar-level aggregate: all total player sight areas plus radar-only sensor areas. Full sight is a stronger information tier, so it necessarily also supplies radar-level location. Radar-only coverage grants positional contact only, not full identity, health, orders, animation state, or full snapshot presence.

A radar-only enemy should be represented to the player as a contact, not as the real entity. The small sphere/blip idea is the right shape: a generic radar contact at a quantized or otherwise coarse position, with no unit blueprint, no exact body dimensions, and no private state. The authoritative sim may internally associate that contact with a real entity so projectiles can collide with the real world, but the client should only receive the contact tier it earned. Losing full sight while retaining radar should degrade the player’s information; it should not silently keep streaming a full unit under another visual.

Long-range weapons should consume player/team intelligence, not pretend their own barrel can see forever. A target can be fired on when the weapon’s range, line-of-fire policy, ballistic solver, and required intelligence tier all pass. Full sight gives exact aim and normal target selection. Radar-only contact can support weapons explicitly authored for blind or remote fire, such as artillery, missiles, or other indirect systems, but direct beams and precision line weapons should usually require full sight and unobstructed line of sight. The ballistic solver answers “can this shot physically reach that point?” It does not by itself answer “does this player have enough information to choose that point?”

This intentionally pushes scouting and radar into army composition. A long-range turret with short local sight should be powerful only when supported by forward units, scan pulses, radar coverage, or allied sight. Current prototype behavior that derives full vision from every turret’s weapon range is a convenience shortcut, not the long-term rule; fire envelope and sight envelope must be authored separately.

  • Full sight may reveal and identify, and always counts as radar-level location; radar-only coverage may locate; detector coverage may uncloak but does not locate by itself.
  • Radar-only contacts are eligible inputs to weapons that explicitly allow radar fire. They are not full entities in the recipient’s main snapshot.
  • Weapon range, tracking range, unit sight, radar range, and detector range are separate envelopes. Sharing a number between them should be an authored choice, not a default coupling.
  • Team sight is a player-level aggregate. A turret firing using another unit’s sight is using team intelligence, not gaining that other unit’s local sight radius.

Visible units render and update identically

Ownership decides what strategic information a player learns about a unit; it must never decide how a unit the player can already fully see moves on the wire or draws on screen. Once full sight places a foreign unit in the recipient’s main snapshot, every field that drives its motion and presentation travels and is corrected exactly as it would for the recipient’s own units: world position, body rotation, hover orientation quaternion, linear velocity, angular velocity, surface normal, turret yaw and pitch, and HP. There is no owner-conditional zeroing, omission, or coarsening of motion state. An enemy aircraft banks into its turns, dead-reckons between snapshots, and smooths under the same prediction channels as a friendly one, because the renderer and the client predictor receive the same inputs for both. Banking is the canonical example: it is the centripetal product v_forward · ω_z, so withholding an enemy’s linear velocity silently zeroed its bank while ours rolled — a rendering difference produced purely by an ownership gate on a motion field, which is exactly what this rule forbids.

What ownership still gates is private strategic detail, not presentation: a unit’s order queue and move/attack/guard intent, the specific entity IDs its turrets are tracking, its current build or repair target, and factory rally points. A player watches an enemy unit turn, fire, take damage, and bank in full physical fidelity; they do not see what it has been told to do. That line — physical presentation is shared, commanded intent is private — is the rule. A motion or render field hidden by ownership is a bug; a strategic-intent field leaked across ownership is also a bug.

This changes nothing about fog of war. Sight and radar tiers still decide whether a unit appears — as a full entity, a radar contact, or not at all — and a unit lost to fog still vanishes from the snapshot entirely. This rule governs only how a unit is treated once it has earned full visibility: at that point its physical state is owner-agnostic.

Visibility computation must stay aggregate and indexed

Sight and radar cannot be implemented as “each unit checks every other unit” work. The authoritative sim should build player/team coverage from sensor sources, cache it at a deliberate cadence, and let targeting and snapshot serialization query that aggregate. The current direction is already right: spatial hashes, recipient/team visibility objects, memoized entity visibility, scheduled targeting probes, and Rust/WASM slabs. New sight/radar work should extend that shape.

The scalable model is a small set of source pools and contact masks per player or team: full-sight sources, radar sources, detector sources, and temporary scan sources. Coarse cells answer “could this point be covered?” cheaply; precise distance and line-of-sight checks run only on candidates that survived the broadphase. Targeting should query enemy candidates from the existing spatial index by outer weapon envelope, stamp or derive each candidate’s visibility tier once per source entity/team, and reuse that answer across the source’s turrets. Snapshot serialization should reuse the same aggregate view instead of recomputing a separate truth.

  • Rebuild or incrementally refresh coverage when sources move, finish construction, die, cloak/uncloak, or change teams; do not run full source-vs-target scans every frame.
  • Use coarse bitsets or cell buckets for radar/full-sight membership, then exact radius and terrain/force-material LOS only when a candidate needs it.
  • Keep radar cheaper than sight: radar coverage should be mostly geometric unless a deliberate radar-jamming or radar-occlusion feature is introduced.
  • Run expensive acquisition only for turrets that need work. Existing target hysteresis, drop grace, probe scheduling, and broadphase candidate buffers are the right pattern.

Delta thresholds match channel semantics

A FULLSNAP is a recovery baseline. A DIFFSNAP is a minimal update from the last value actually sent to that recipient for that channel. Delta decisions must not compare only against the previous full snapshot, and they must not compare against an unsent local tick value. The baseline advances independently per field when that field is emitted.

Position-like channels use absolute error because their meaning is spatial: movement position uses world-unit distance, and rotation position uses radians or quaternion angular distance. Derivative channels use relative error because their meaning is rate: movement velocity and rotation velocity should ask whether the new velocity is meaningfully different by percentage from the last velocity sent, not whether it moved by one fixed unit. The same rule applies to any future acceleration-like derivative channel.

Derivative comparisons must treat zero as a semantic edge. If the last sent velocity was non-zero and the current authoritative velocity is zero, the relative change is effectively unbounded and the zero must be sent. Decaying velocities should keep producing deltas as they change by meaningful percentages, so the client eventually learns that motion has stopped instead of integrating an old non-zero derivative forever.

Client prediction is an explicit contract

The client view should continue to run smoothly whether snapshots arrive at 60 Hz, 10 Hz, 5 Hz, or briefly not at all. Prediction should use the same motion math as the server wherever possible. The current code already points this way with shared TypeScript/WASM unit motion, server targets, and drift modes. New systems should preserve that shape.

Client Simulation Model

Shared core, bounded client

There should be one headless simulation core for physical behavior, but two different runtime responsibilities in authoritative-server. The server runs the complete authoritative game simulation. The client runs a bounded prediction simulation for the parts the player sees and feels between snapshots. This client sim may be a real fixed-step loop, and it may eventually live in a worker, but it must remain a prediction layer in authoritative-server. It is corrected by authoritative state and never becomes the source of gameplay truth there.

In deterministic-lockstep, the browser also runs the full server simulation locally. That is not a second gameplay implementation: it is the same server simulation stack — WorldState, Simulation, physics, command execution, economy, construction, combat, fog truth, projectiles, and victory — driven by command frames. The renderer still consumes local snapshots from that local server simulation, so the client render path does not need a duplicate gameplay model.

  • Share physical kernels for body motion, projectile stepping, recoil, hover, turret pose, terrain sampling, and collision helpers.
  • In authoritative-server, keep authoritative-only systems on the host: command validation, economy, construction completion, targeting truth, damage, fog filtering, unit creation and death, and victory.
  • In deterministic-lockstep, run those same authoritative systems locally on every peer and keep them deterministic through fixed frames, deterministic math, canonical command ordering, and checksum diagnostics.
  • Predict visible state locally: movement, velocity, turret rotation, beam path presentation, projectile travel, and local command intent.
  • Use snapshots as correction targets. The client stores the most recent server target per entity. Position channels — movement position and rotation position — always apply correction by snapping or EMA-blending toward it; velocity channels — movement velocity and rotation velocity — can additionally ignore the snapshot derivative when that is useful for prediction testing. Acceleration is never shipped: the authoritative force solver lives on the host, and the client’s prediction integrates position from the last-seen velocity rather than re-deriving server-side forces from a stale snapshot.

Prediction is velocity-based

The client prediction integrator advances position from the last-seen velocity each frame. It does not re-integrate acceleration. The authoritative simulation owns force inputs (movement thrust, turret spring torque, gravity, ground spring, recoil, impulse), and those forces produce velocities and positions the snapshot eventually carries. The client never sees the acceleration vector itself, so it cannot stale-integrate it across a snapshot gap and overshoot when the authoritative force flips sign before the next correction arrives.

The client’s PREDICT mode therefore exposes just two settings: POS snaps straight to the snapshot position (no integration); VEL integrates position from velocity each frame. There is no ACC option — and consequently no movementAccel or angularAcceleration on the wire — because instantaneous second derivatives are not stable to integrate under arbitrary client dt and the EMA-toward-velocity already smooths the approach to a freshly-arrived target.

EMA channels are independent

Each prediction channel has its own EMA control: movement position, movement velocity, rotation position, and rotation velocity. Position controls expose snap, fast, medium, and slow; velocity controls also expose ignore. The client always stores the most recent snapshot value per channel so the chosen mode can be applied afresh every client tick.

  • ignore — the stored snapshot value exists but the client never applies it to the rendered entity for that channel. Prediction (if any) continues from the last applied value forever.
  • snap — each client tick the rendered value is replaced with the latest stored snapshot value for that channel. No smoothing.
  • fast / medium / slow — each tick the rendered value blends toward the latest stored snapshot value using a frame-rate independent EMA whose half-life is determined by the mode.

Splitting the EMA per channel means a player can keep movement snappy while letting rotation drift gently, or freeze velocity entirely while position catches up. The mode is a render-time decision, not a snapshot-decode decision: the target store always accepts the newest authoritative value, and the per-channel mode controls only how the rendered state walks toward it.

What each EMA channel applies to

The four EMA knobs are not abstract presentation sliders; each one binds to a specific named set of authoritative state fields, and the rendered value of every one of those fields walks toward its server target through the chosen mode every client tick. The whole point of having one knob per channel — rather than per-field tuning buried in code — is that the same smoothing behavior is applied uniformly to every field that belongs to that channel. A channel that is only applied to some of its fields is a bug, not a freedom: changing the rotation position mode must move every rotation-position field in lockstep, or the knob is lying about what it controls.

  • Movement position — the world-space 3D position of every visually rendered moving thing whose state travels on the wire as a continuous stream. That means the unit body center (entity.transform.x/y/z) and every per-vertex point along a beam (proj.points[i].x/y/z). Anything the player sees as "where this thing is right now" for a continuously corrected entity rides on this channel. Travelling projectiles (the shot kind, as opposed to ray emissions) are not on this channel — their position is event-driven, not continuously corrected; see Ballistic projectiles are event-driven below. The carve-out is rocket-like homing shots, whose event-installed correction targets blend on this channel (see Homing rockets blend between two real trajectories).
  • Movement velocity — the 3D linear velocity of every continuously-corrected moving thing whose authoritative velocity travels on the wire. That means the unit linear velocity (entity.unit.velocityX/Y/Z) and the per-vertex tangential velocity on a beam (proj.points[i].vx/vy/vz). These are the values the client integrates forward between snapshots under VEL prediction for unconstrained motion. Reflected beam geometry is stricter: once a beam has hit a shield, the reflector vertex and all later vertices are authoritative trace constraints unless the client is also running the same local reflection trace. Their velocities remain on the wire for correction and diagnostics, but the presentation layer must not independently dead-reckon those vertices through shield planes. Travelling-projectile velocity (proj.velocityX/Y/Z) is event-snapped, not on this channel — except rocket-like homing shots, whose correction targets blend their velocity here too (see Homing rockets blend between two real trajectories).
  • Rotation position — the orientation of every rotating thing on the wire, in both scalar and full 3-DOF form. That means the unit body yaw scalar (entity.transform.rotation), the unit hover-style body orientation quaternion (entity.unit.orientation.{x,y,z,w}), and each turret's yaw and pitch (weapon.rotation, weapon.pitch).
  • Rotation velocity — the angular rate of every rotating thing whose authoritative angular velocity travels on the wire. That means the unit hover-style angular velocity vector (entity.unit.angularVelocity3.{x,y,z}), each turret's yaw rate (weapon.angularVelocity), and each turret's pitch rate (weapon.pitchVelocity). These are the derivatives the client integrates while drifting toward the rotation-position target.

Fields that do not appear in any of the lists above are not on the wire and are therefore not subject to snapshot drift. Static building position, decorative locomotion-rig state (wheel spin phase, tread cleat phase, leg plant pose, hover fan smoke), visual-only chassis embellishments (airborne bank roll), and discrete open/close pose transitions (solar petal fold, extractor blade fold, wind-turbine close pose) live entirely client-side and get their smoothness from local controllers, not from any of the four channels. See Locomotion Visuals Are Frontend and Airborne Banking Is Visual.

Decorative client-only animations that visually represent the same kind of motion as a wire channel may opt in to the same EMA mode for consistency — the wind-turbine fan yaw reads rotation-position mode, and the wind-turbine blade speed, construction-tower orbit spin, extractor rotor spin, radar head, and radar sweep read rotation-velocity mode. These are courtesy bindings, not snapshot-drift corrections: there is no server target to drift toward, so the mode supplies a half-life and the local animator EMA-blends toward a locally derived target value.

Discrete open/close pose transitions do not opt in to rotation position. Their state is a local pose switch between authored endpoints, not a continuously rotating wire field, so they keep fixed local controller alphas unless a future gameplay reason puts an actual rotation target on the wire.

Ballistic projectiles are event-driven

Travelling projectiles (the shot kind: plasma, rockets, D-gun waves) are deliberately excluded from the continuous per-snapshot correction stream the movement EMA channels exist for. Their trajectory between discrete events is fully deterministic from starting conditions plus the universal gravity constant plus per-shot thrust (homing, terrain-follow), and the client and server run the same kernel against the same inputs. There is no continuous server/client divergence for the EMA to correct. (Rocket-like homing shots re-enter the movement channels at their discrete correction events — see Homing rockets blend between two real trajectories below — but no projectile is continuously corrected per snapshot.)

Accordingly, the wire format carries no per-snapshot rolling state for projectiles. Every server-to-client message about a travelling projectile is a discrete authoritative event: a spawn (initial position, velocity, type), a homing course correction (a velocityUpdate with the new post-event pos + velocity), or a despawn. Between events, the client dead-reckons under gravity + per-shot thrust with no correction loop.

For non-homing shots — plasma bolts, D-gun waves — when a velocityUpdate arrives, the client snaps entity.transform.x/y/z and proj.velocityX/Y/Z directly to the event's values and never populates a drift target. EMA-blending toward a step function would produce a smoothing ramp through coordinate space that does not correspond to any real trajectory — visible as wiggle in the curved-cone tail after a course correction, because the tail stamps freeze a record of wherever the predicted head actually was. Snapping makes the discontinuity instantaneous: subsequent dead-reckon stamps land on the true post-event trajectory and the tail after a course correction is as clean as the tail behind a fresh shot.

Homing rockets blend between two real trajectories

Rocket-like homing shots are the one exception to the pure snap, and the exception preserves the rule’s reasoning instead of ignoring it. A homing rocket’s course corrections arrive frequently enough that hard snaps read as teleporting rather than steering. So a rocket’s velocityUpdate installs a correction target instead of overwriting the rendered state, and every frame the client advances both the rendered rocket and that target with the same gravity + homing + terrain-follow kernel, EMA-blending the rendered state toward the target on the movement position and movement velocity channels. Because the target itself dead-reckons under the deterministic kernel, the blend is between two real trajectories — never a ramp toward a stale step function, which is exactly what the snap rule forbids — and the target is released once the rendered state settles onto it. Plasma and D-gun shots keep the pure snap: their corrections are rare and discontinuous, which is the case snapping exists for.

This generalises: any field whose authoritative evolution between events is fully reproducible from a known starting state under a deterministic kernel should be snapped on the events and dead-reckoned between them, not low-pass filtered toward a sparse target. EMA is the right answer for genuine continuous mismatch (unit physics, contacts, action latency); it is the wrong answer for known step functions in deterministic state.

Rendering samples prediction

Rendering should not own simulation cadence. requestAnimationFrame is a display scheduler, not a simulation clock. The client prediction loop can run at a fixed or semi-fixed timestep, while the renderer samples the latest predicted state every frame and performs render-time interpolation. If the browser renders at 144 Hz, 60 Hz, 37 Hz, or is briefly tab-throttled, the prediction contract should stay coherent.

This split keeps the browser responsive and keeps future options open. The prediction loop can move toward Rust/WASM and a worker without rewriting the renderer.

Physics Model

The game should be more physics based than its inspiration. BAR is the ergonomic and strategic reference point; this project should make the physical world more fundamental. The rule of thumb is F = M * A. If a gameplay feature moves, pushes, resists, falls, recoils, hovers, blocks, crushes, or explodes, it should be described through forces acting on masses and integrated into motion.

Full 3D, not decorative altitude

The sim uses a real 3D state: horizontal x/y, vertical z, velocity, acceleration, gravity, surface normals, sphere and cuboid bodies, terrain contact, building collision, unit collision, projectile collision, and hover lift. The renderer presents that state, but the authoritative sim should treat altitude and contact as real gameplay inputs.

Kinematic calculations must read entity position, velocity, and acceleration through the explicit accessors getEntityPosition3d, getEntityVelocity3d, and getEntityAcceleration3d. Callers pass zero-valued components when an entity has no velocity or acceleration, and targeting/ballistic solvers must receive raw kinematic values plus the universal gravity constant instead of entity IDs or optional lookup inputs.

Turret range is a vertical cylinder, not a sphere. Every turret range radius — fire acquire/release, minimum fire preference, tracking acquire/release, and any other turret-authored “in range” shell — uses horizontal distance as the radius. For a range R, the cylinder is centered on the turret mount in x/y, reaches up to mount.z + R, and extends downward without a lower cap. A target is inside the range shell when its body intersects that horizontal radius and it is not above the shell’s top cap. Height below the turret does not consume range budget, so cliffs, aircraft, and drop shots can target downhill or underneath cleanly. Height above the turret remains bounded by the same authored number that defines horizontal reach.

Ballistic and line-of-fire feasibility remain separate gates after the cylinder range check. The range cylinder answers whether a turret may consider, track, or engage a target; the ballistic solver still answers whether the authored projectile can actually reach the chosen aim point from the turret mount. Range overlays on the ground show the cylinder’s horizontal radius directly.

Mass must matter

  • Unit propulsion is a force profile applied to a mass, not a direct position edit.
  • Recoil applies counter-force to the firing body.
  • Explosions, beams, collisions, and impact impulses apply forces that heavier units resist differently.
  • Hover lift balances gravity through force and damping.
  • Ground contact is spring and friction behavior against terrain, not a clamp disguised as physics.

Locomotion is arrival control, not waypoint seeking

Movement should not be a magnetic pull toward the next waypoint. A position-only seek controller looks acceptable on slow, high-friction units, but it breaks down as soon as a body has meaningful momentum. Hovercraft expose the failure most clearly: the unit keeps accelerating toward a point even when its current velocity already guarantees overshoot, then crosses the point, gets pulled back, and oscillates like an undamped spring.

The locomotion question is not “which direction is the waypoint?” It is “what acceleration do I need, under this unit’s force limits, to arrive at the desired state?” That desired state includes both position and velocity. A final command point usually wants near-zero velocity. An intermediate waypoint may want a through-velocity aligned with the next path segment, so the unit flows through the path instead of stopping at every corner.

Every locomotion profile should therefore compute force from both position error and velocity error. If a unit is close to its intended spot but still moving quickly, the correct thrust is often opposite its current velocity, even if the target remains slightly ahead. This is controlled arrival: a damped physical controller that spends stopping distance before it runs out, saturates against the unit’s available thrust, and prevents waypoint overshoot from turning into visible oscillation.

  • Pathfinding chooses where the body should go; locomotion chooses how the body physically arrives there.
  • Waypoints are desired motion states, not magnets. They carry an intended position and, when useful, an intended pass-through velocity.
  • Final command points should use braking-aware arrival so the body settles without orbiting, bouncing, or springing back across the destination.
  • Hovercraft, ground units, and air units use the same high-level arrival controller; they differ only by mass, thrust, friction, damping, turn authority, contact model, and vertical constraints.
  • Hovercraft should remain slippery because their physical limits say so, not because the controller ignores velocity until after overshoot has happened.

Waypoints are intent; pathfinding points are disposable

A waypoint is a player-, AI-, factory-, or fabricator-authored command point. It belongs to the command queue, can be shown in the UI, can be edited, inherited, serialized, and reasoned about as tactical intent. A pathfinding point is different: it is a private movement-plan sample generated by the pathfinder for the currently active leg. Pathfinding points are not commands, not rally markers, not factory defaults, and not something a newly produced unit should inherit from the unit or fabricator that supplied its waypoints.

Factory and fabricator inheritance therefore copies only high-level waypoints. When a unit receives those waypoints, it does not receive pre-expanded path points. It plans the route from its actual current position to the first waypoint only when it starts moving, then plans from waypoint to waypoint as each leg becomes active. If it is pushed off route, stalls, a building changes the path mask, terrain changes, or a dynamic target moves enough to make the current leg stale, the current pathfinding points are discarded and recomputed. The durable waypoint list remains intact.

  • Do not store pathfinder-expanded intermediate points in the same array as authored command waypoints.
  • Do not serialize pathfinding points as gameplay intent; serialize waypoints and let the authoritative sim regenerate movement plans.
  • Factory rally routes and fabricator-provided routes are high-level waypoint lists. Produced units resolve those lists against their own locomotion profile and current world state.
  • Pathfinding is leg-local and demand-driven: plan the active leg, cache it while valid, and throw it away on completion, obstruction, stale progress, terrain/building-version change, or target drift.
  • Patrol and fight loops re-enter the planner per leg rather than baking one permanent expanded loop. The path should adapt to the battlefield, while the waypoint loop remains the command.

Gravity is universal; thrust answers it

Every body with mass feels gravity every tick, without exception. The authoritative sim should never special-case “this projectile ignores gravity” as a free pass. If a rocket, missile, or other guided shot needs to fly a clean line to its target, it does so by producing thrust that cancels the gravitational pull, not by opting out of the force.

This applies inside the real-time homing equation itself. A guided projectile’s steering update should compute the lateral correction toward the predicted intercept and, in the same step, add a vertical thrust component sized to counteract g · m on this tick. The visible flight path can look indistinguishable from “gravity off,” but the physics stays whole: the body has mass, the world pulls it down, the engine pushes back, and any failure of the engine — out of fuel, seeker lost, lifespan expired — drops the body the way a spent missile drops in the real world.

  • Gravity is applied to every mass body every tick. No ignoresGravity shortcut, no projectile-class exemption, no “flat trajectory” flag that bypasses integration.
  • Guided projectiles add a counter-gravity thrust term inside the same homing/steering update that produces lateral correction toward the target. Guidance and gravity-cancellation are one acceleration vector, not two systems.
  • The counter-gravity term is bounded by the projectile’s available thrust. A weak or under-powered rocket should sag, not teleport — the homing math saturates, it does not lie.
  • Spent or failed guidance returns the body to ballistic motion automatically, because gravity was never turned off; only the thrust opposing it stopped.
  • Hover and any other “defies gravity” behavior follows the same shape: a force that fights g · m, not a conditional that skips it.

Discrete gameplay still enters through physical effects

Commands, economy, construction, fog, build queues, and ownership are intentionally discrete systems. They do not need fake physics. But when those systems touch the battlefield, their visible result should enter the world as bodies, forces, energy, mass, constraints, or collisions. A factory can make a unit shell discretely; once it exists, gravity, terrain, collision, and force response should own its motion.

Scale Target

The design target is thousands of active units in a browser, with a path toward larger battle scenes. That target changes the default answer to every implementation question. Expensive correctness that works for 200 units but collapses at 5,000 units is not correct for this project.

Hot paths are data-oriented

  • Prefer stable IDs, packed numeric fields, typed arrays, and struct-of-arrays storage for dense simulation data.
  • Batch numeric work into WASM kernels when it can run over typed arrays without heavy object marshalling.
  • Keep TypeScript as orchestration, UI, temporary renderer integration, networking glue, and thin browser API plumbing. Gameplay control flow should move to Rust/WASM whenever it can do so without paying more boundary cost than it saves.
  • Pool snapshot DTOs, event objects, scratch arrays, and temporary buffers.
  • Do not allocate in per-unit, per-projectile, per-turret, per-cell, or per-frame loops unless the allocation is proven irrelevant.

Spatial structure beats broad scans

Targeting, collision, capture, fog, projectile queries, and selection should use spatial indices, dirty sets, visibility masks, and area-of-interest bounds. Full-world scans are acceptable for bootstrapping, debugging, and rare keyframes. They are not the normal steady state for gameplay ticks.

Clean structure comes before visual shortcuts

There are two separate concerns: server simulation authority and client presentation. Server simulation, snapshots, and client prediction must keep explicit ownership and stable cadence. Client rendering should be organized around clear data flow, predictable batches, and readable assets before adding extra representation modes.

Browser Performance Rules

Browser performance is not an afterthought. The browser is the platform. Code should be written as if V8, SpiderMonkey, WebKit, WebAssembly, WebGL, garbage collection, and WebRTC buffering are part of the game engine.

  • Use plain counted loops in hot paths. Avoid map, filter, Object.entries, spread copies, and iterator churn inside tick loops.
  • Reuse arrays by setting length = 0. Grow buffers on demand and avoid shrinking steady-state buffers.
  • Use bit flags and append-only numeric wire enums for snapshot fields and blueprint IDs.
  • Serialize snapshots as compact binary payloads. Do not rely on deep JSON object trees for high-frequency state.
  • Separate debug overlays from gameplay state and throttle them aggressively.
  • Measure TPS, CPU budget, render time, prediction time, snapshot rate, and backpressure continuously.
  • Prefer instancing, shared geometry, shared materials, spatial grouping, and GPU-friendly batches over per-unit scene graph complexity.

TypeScript Coordinates, WASM Computes

The absolute performance ceiling for browser 3D is set by how little JavaScript runs in the per-frame hot path. TypeScript stays in the project because the browser’s native dialect is JS — boot, asset loading, input, UI, networking glue, calling into WASM — but it should never be the layer that integrates physics, walks per-unit update loops, builds render lists for thousands of objects, or carries large serialization work. Those belong in WebAssembly, compiled from Rust, running over typed linear memory with SIMD-friendly, struct-of-arrays layouts. The renderer that consumes the engine’s output should be data-oriented regardless of backend: it may draw through Three.js, raw WebGL2, or raw WebGPU, but high-count visuals must flow through owned typed buffers, explicit update ranges, instancing, and custom shader code instead of one draw call or one object graph per thing.

This is the same commitment that Languages and Frameworks makes for language choice and Config Is Data, Not Code makes for content authoring, expressed as a runtime architecture: the layers are stacked so that hot data crosses the JS/WASM boundary as rarely as possible, and the JS side never owns the inner loop. The end-state shape is a TypeScript shell wrapped around a Rust + WASM simulation that talks to a data-oriented renderer through stable render packets and GPU-resident buffers, with Web Workers and SharedArrayBuffer separating cadences where independent stepping or true cross-thread shared memory pays for itself.

Layer responsibilities

  • TypeScript shell — application boot, asset loading, canvas creation, renderer initialization, input collection, UI / menus / dev tools, WebSocket and WebRTC setup, and the bridge that calls into WASM. This layer coordinates; it does not compute.
  • WebAssembly engine (Rust) — ECS, physics, collision, pathfinding, visibility and fog, animation state, projectile simulation, spatial queries, culling, and render-packet generation. Everything that runs every tick or every unit lives here.
  • Renderer backend — currently allowed to use Three.js, raw WebGL2, or raw WebGPU. The backend owns GPU buffers, shader programs, instanced rendering, post-processing, and optional compute work; the architectural requirement is explicit data flow and batch control, not a specific library ban.
  • Web Workers + SharedArrayBuffer — when simulation and render need to step on independent cadences, or when a kernel can be parallelized across cores against shared linear memory without copying.

What TypeScript must never own in hot paths

Each item below is a per-unit, per-projectile, or per-frame loop that compounds at scale. Running any of them from JavaScript at thousands of units gives up the performance ceiling, regardless of how careful the TypeScript is.

  • Per-unit update loops.
  • Physics integration and collision detection.
  • Projectile simulation and spatial partitioning.
  • Pathfinding and fog-of-war calculation.
  • Animation blending for thousands of units.
  • Render-list generation for thousands of objects.
  • Large serialization loops and large per-frame allocations.

What TypeScript should keep doing

  • Bootstrapping, developer tools, debug overlays.
  • Settings, lobby, and the Vue / UI layer.
  • Asset loading and input event collection.
  • Renderer initialization, asset upload, shader/pipeline assembly, and boundary glue.
  • WebSocket / WebRTC setup and the network glue.
  • Calling exported WASM functions and marshalling the few values that genuinely need to cross the boundary.

The slow architecture vs the fast one

The slow browser game architecture is the one most browser code drifts into: thousands of JS objects, each carrying methods, each updated every frame, each allocating temporaries, each issuing its own draw call. That shape pays the cost of language overhead, garbage collection, scene-graph traversal, and one-draw-per-thing at every layer simultaneously.

The fast architecture is the inversion: flat typed memory, batched updates, almost no allocations, very few crossings of the JS/WASM boundary, large GPU buffers, instanced rendering, compute shaders where the work parallelizes, and as few draw calls as the scene can be expressed in. The rule of thumb is: if you can make the loop run over a slab of bytes instead of a list of objects, you should — and that loop almost certainly belongs in WASM or on the GPU, not in TypeScript.

Languages and Frameworks

Peak performance wins over framework convenience. For simulation and rendering, the default choice should be the fastest browser-viable language and API we can own directly. Frameworks are allowed only when they do not sit inside hot loops and do not prevent us from controlling memory layout, batching, scheduling, serialization, GPU buffers, or shader code.

Simulation language

  • Use Rust compiled to WebAssembly for dense numeric simulation kernels.
  • Move physics, projectile stepping, collision kernels, pathfinding kernels, terrain sampling, targeting batches, visibility packing, and snapshot-diff kernels toward WASM when the data can be batched.
  • Use struct-of-arrays storage in WASM linear memory for bodies, projectiles, turrets, spatial indices, and other high-count state.
  • Keep TypeScript as orchestration: UI, commands, networking glue, scene lifecycle, and code that would pay too much JS/WASM boundary cost. Configuration and blueprints are data, not TypeScript — see Config Is Data, Not Code.
  • Do not use a general-purpose JavaScript physics engine, ECS framework, pathfinding framework, or game framework in the authoritative sim hot path.

Rendering language and API

  • Use GPU-native code for the expensive work: custom GLSL for WebGL2 or WGSL for WebGPU.
  • Three.js is an allowed renderer backend. Use it when it provides useful camera, scene, geometry, material, or asset-loading leverage without taking ownership away from our high-count buffers, culling, shader code, update ranges, and draw budgets.
  • Prefer raw WebGPU or raw WebGL2 only for renderer paths where Three.js overhead measurably blocks scale or prevents the buffer/shader contract we need.
  • Vue is UI only. Vue must not own simulation cadence, render-loop state, per-unit updates, or per-frame battlefield data.
  • Do not introduce rendering frameworks that hide draw-call count, buffer ownership, material compilation, culling, or shader control from us.

Owned by default

The preferred implementation is owned code in Rust/WASM, TypeScript, and shader code, with libraries acting as boundaries or backends rather than owners of the game’s hot behavior. If a library saves time but costs measurable simulation or rendering throughput, that specific path should be rebuilt around owned buffers, owned shaders, and explicit batching. The end-state engine should be transparent enough that we know where every per-frame allocation, buffer upload, kernel call, draw call, and branch-heavy world scan comes from. When a scaffolded path is replaced by an owned one, the scaffolding leaves the tree in the same change — see Delete The Old Path.

Rendering Philosophy

The renderer should make huge battles legible without forcing the simulation to become fake. Units should have consistent turrets, legs, treads, chassis tilt, beams, labels, decals, and effects because those are the shapes players use to understand the fight. The renderer should organize those shapes into clean render queues, shared resources, and predictable spatial groups before adding new rendering modes.

Rendering should never ask the authoritative sim to send more data just because a visual is expensive. If a visual cannot be supported at scale, simplify the visual contract or the renderer's structure first. The sim remains physical; rendering stays downstream of that physical truth.

Camera Anchoring Is Two Orthogonal Axes

The camera is pure client presentation. It lives in the renderer, samples the client terrain mesh, and never crosses the wire — like locomotion visuals it is downstream of sim authority and feeds nothing back upstream. Because it is frontend, how it anchors and how it treats terrain are presentation choices, and presentation choices in this codebase are data, exhaustively enumerated, not policy hardcoded inside a picker.

Every camera operation that pivots or pins on the world — wheel zoom-in, wheel zoom-out, orbit rotate, and drag pan — needs exactly one anchor point in 3D space. That point is the answer to two independent questions, and each question is its own config axis. Collapsing them into a single mode enum, or hardcoding either one, is the drift this rule exists to prevent.

Axis one — where on the screen (anchorPointCameraScreen)

  • cursor — the spot directly under the mouse cursor.
  • screen-center — the spot at the middle of the canvas.

Axis two — which surface in the world (anchorPointCameraTerrain)

The screen spot defines a ray into the world; this axis decides which surface that ray resolves against.

  • plane-2d — the original flat ground plane (building-zero). Elevation is ignored; the anchor is the horizontal point only, exactly as a flat 2D camera would read it.
  • terrain-3d — the actual rendered terrain heightfield, including the real basin floor below water. The anchor follows hills and valleys exactly as drawn.
  • terrain-3d-water — the terrain heightfield clamped up to water level, max(terrainHeight, WATER_LEVEL). Over a lake the anchor sits on the water surface the player sees, not the submerged bed — the same surface units walk on.

The two axes are orthogonal: two screen choices times three terrain choices is six anchor behaviors, and all six must be reachable by setting the two configs — never by adding a seventh special-case branch. A gesture’s anchor is fully described by its (screen, terrain) pair, and one shared resolver takes that pair and returns the point. Each gesture names its own pair, so a reversal or a tumble can anchor differently from a dolly.

Minimap camera and PIP are replaced by 3D overview controls

BAR-style minimap camera switching and picture-in-picture are not the canonical Budget Annihilation camera model. The supported 3D-first replacement is: minimap click-to-center for fast relocation, camera anchors for saved tactical views, selected-unit follow for unit-centric tracking, and the map overview switch for whole-map inspection. Those controls preserve one primary 3D camera and avoid a second live viewport that would duplicate terrain, fog, water, and high-count entity rendering work. If a later spectator or replay surface needs secondary views, it should be designed as a mode-specific tool, not as the default battle camera contract.

Passing through terrain is its own config, not an always-on law

Whether the camera may move through the terrain is a separate boolean, independent of where it anchors. One mode keeps the orbit camera outside the terrain by a minimum clearance: it is pushed back along the surface normal and floored above any peak beneath it, so a zoom or pan can never bury the view inside a hill. The other lets the camera pass freely through hills and below the ground, which is what a free-look or cinematic sweep wants. Both the clearance distance and the collide-versus-clip choice are data; neither is a constant welded into the controller.

Locomotion Visuals Are Frontend

Authority owns chassis position. The renderer owns where the parts of the chassis appear in the world. The sim publishes one point — the unit’s body center, at its authoritative altitude. From there the rig is responsible for putting every wheel, cleat, foot, or fan exactly where the renderer says it should be, and that “should be” must respect terrain as a hard floor.

This is the right boundary for three reasons. One: the renderer already has the terrain heightmap loaded for shading and contact shadows; sampling it under a wheel or foot is cheap. Two: per-part terrain queries scale linearly with visible units, not authoritative units — off-screen units skip the work for free. Three: it removes a whole class of net-bug, because a chassis can never wedge a wheel under a hill that exists only on the client’s mesh. The visual rig is downstream of sim authority, never upstream.

The four locomotions

  • Wheels — a chassis driven through discrete rigid contact patches at fixed body-local positions. Visually: cylindrical tires, each spinning around its own axle, rolling phase derived from how far that local contact point traveled in the world this frame. Each tire owes the ground a contact patch — the bottom of the cylinder is the contract.
  • Treads — two infinite belts that share chassis velocity but read it as continuous tangential motion. Visually: scrolling cleat rails along the belt path; the belt itself never moves relative to the chassis. The contract is the entire underside of each ribbon sitting on terrain.
  • Legs — discrete step events: a foot plants, the chassis advances past it, the foot lifts and swings to a new plant. IK solves hip→knee→foot for the rendered pose. The contract is per-foot, not per-chassis — feet must touch terrain on plant and swing through air during transit.
  • Hover — a chassis that never touches anything; a duct array balances gravity with downward thrust against the air column underneath. Visually: spinning fans, downward smoke columns, chassis riding a target altitude. The contract is anti-contact: the gap is the feature, and that gap must be measurable per frame so smoke length and altitude shading look right.

Always animate; terrain only clamps position

Every rig integrates the same four canonical visual state channels every frame — the same channels the client-prediction layer uses for entity motion, named the same way and reasoned about the same way:

  • Movement position — where the part sits in chassis-local Y (wheel lift, tread side lift). EMA-converges toward the floor-clamp target every frame.
  • Movement velocity — rate of change of movement position; integrated implicitly from the position EMA.
  • Rotation position — rolling angle (wheel rotation around its axle) or belt phase (where cleats sit on the loop). Integrated from rotation velocity.
  • Rotation velocity — angular velocity for wheels, linear belt velocity for treads. EMA-couples toward the body-motion-derived target every frame.

The rig does not ask "is this part touching ground?" The velocity channels always couple to body motion; the position channels always couple to the floor clamp. A tire over a dip spins at the body-motion rate and floats slightly above terrain (target lift = 0). A tire cresting a hill spins at the same rate and rides up the surface (target lift > 0). When the body stops moving, the velocity channels EMA toward zero on the same tau they always use — cleats coast to rest over a beat, wheels spin down naturally. Simpler than switching coupling regimes, and visually indistinguishable in normal play.

Per-part position, not per-chassis

Floor clamping is per primitive. A four-wheeled vehicle straddling a crest has two wheels rising over the terrain and two floating in air over the dip; the rig clamps each tire independently. Treads sample three points along each side ribbon and lift the side to clear the max. Legs sample under each foot. Hover doesn't clamp parts but does sample a per-frame clearance scalar for fan smoke and altitude shading. The rig knows what its primitives are; uniformity is in the shape of the contract, not in the bits.

Visual contact is sticky

The renderer must not reuse the physics ground-contact epsilon as the sole answer for locomotion animation. The sim epsilon is intentionally tiny because it gates spring, friction, sleep, thrust, and collision behavior. Visual rigs need a separate, frontend-only contact band with hysteresis: acquire ground when the chassis is near the terrain, keep the grounded pose while the body remains within a larger release band or while planted feet are still reachable, and only transition to airborne posing once the visual contradiction is unmistakable.

For legs, the stable unit of truth is the foot, not the chassis. A planted foot keeps its world position until a deliberate step swing begins. Brief chassis bob, snapshot drift, suspension state, or a one-frame miss in strict body contact must not reset the gait or re-seat feet at their rest points. When a legged unit really leaves the ground, feet move smoothly into an airborne pose; when it returns, each foot swings to a terrain plant instead of being initialized in place.

Floor clamping is a hard invariant, not a tunable

  • The clamp is non-optional. Every rig writes renderedY = max(naturalY, terrainY + clearance) for every part it owns, then EMA-converges the part's movement-position channel toward that value. No flag, no debug toggle, no edge case where parts are allowed to drop under the surface.
  • Floating above terrain is acceptable; tunneling through it is not. When the sim chassis is over a dip, individual wheels are allowed to read as “slightly floating” rather than pretending to droop on suspension we do not have.
  • The clamp uses the same smoothed surface normal the chassis tilt consumes (updateUnitGroundNormal). One canonical value, no drift between “how the body leans” and “where the wheels sit.”
  • Hover gets a soft floor too: the rendered rig group is clamped at least ε above terrain, so a bad snapshot can never park fans inside dirt. The authoritative chassis altitude is untouched — the renderer just refuses to draw a contradiction.

Nothing about locomotion visuals crosses the wire

Every input the rig needs — chassis transform, terrain heightmap, body velocity, surface normal — the client already has. Computing this on the server and shipping it would burn bandwidth on per-frame visual state that only matters within the next render call. Compute it locally, render it, throw it away. Putting any of it on the wire is what got us into the jump-state-on-the-wire mess in the first place. Networked suspension/contact fields may influence body presentation, but they must not be the authoritative source for whether a foot remains visually planted.

Airborne Banking Is Visual, Mounts Stay On The Roll Axis

Hover and flying units should look like they bank into their turns — the chassis rolls about its forward axis as lateral velocity builds, then unwinds when the unit straightens out. Banking is the strongest visual cue that an airborne unit is physically committed to a turn, and the renderer is the only place that needs to know about it. The authoritative sim continues to track yaw (and yaw alone) for hover/flying chassis; pitch and roll are derived from local body velocity at render time and never touch sim state, the wire, or any combat math.

This is the same boundary as Locomotion Visuals Are Frontend: the sim publishes a chassis pose the renderer is allowed to enrich, and the rig owns every visual expression that doesn’t change targeting truth. Banking is the airborne analogue of wheel spin and tread phase — expressive on the screen, irrelevant to the simulation.

Why this is a hard rule and not a convenience

The sim’s turret-world-mount math is yaw plus surface-normal tilt — not a full 3-DOF orientation quaternion (getTurretWorldMount). If the renderer applies pitch/roll to the chassis while the sim computes mount positions with yaw only, the visible barrel and the sim’s projectile-spawn point quietly diverge. The cure is geometric: every turret mount on a hover or flying unit must sit on the chassis’ body-forward axis, i.e. mount.y === 0 and mount.z === 0. A pure roll about that axis leaves the mount unchanged in world space, so the sim’s yaw-only mount math and the renderer’s rolled chassis agree pixel-perfect.

  • The blueprint loader enforces mount.y === 0 and mount.z === 0 for every turret on a hover or flying unit. A non-centerline airborne mount is a startup error, not a runtime drift.
  • Hover/flying turrets do not use zResolver; the centerline rule would be violated the moment the resolver assigned a body-top z.
  • The free axis is mount.x. An airborne turret can sit forward or aft of the chassis center along the body’s forward axis without breaking the invariant.
  • Ground-locomotion turrets keep authoring freedom on all three axes — they don’t bank, and the slope-tilt path already runs through getTurretWorldMount.

What the sim writes, what the renderer composes

  • The sim keeps a yaw-only spring on hover/flying orientation. entity.unit.orientation ends each tick as a quaternion equivalent to setQuatFromYaw(yaw) — target pitch and target roll are pinned to zero, the yaw spring still carries rotational momentum so heading turns feel weighty.
  • The renderer derives a body-lateral acceleration — the centripetal term v_forward · ω_z — from unit.velocityX/Y projected through the current yaw and unit.angularVelocity3.z (the yaw rate, on the wire and predicted at render rate). It scales that into a target bank angle, EMA-smooths it per frame, and composes the roll about the body-forward axis on top of the chassis yaw quat. A velocity-derived bank would decay to flat mid-turn because the yaw spring continuously aligns body forward with the thrust direction, collapsing body-lateral velocity; the centripetal signal stays high for as long as the unit is actually turning.
  • The visual bank is bounded (a hard clamp keeps the chassis from doing barrel rolls under collision spikes) and decays toward zero whenever centripetal acceleration is small. It intentionally uses local render-side smoothing rather than the rotation-position channel. No sim state ever sees the value.
  • If a future visual wants pitch (nose-down when accelerating, nose-up when braking), it lives in the same place: derived from velocity at render time, never written back to the sim.

Beam Turrets Track Their Beam; Rocket Turrets Spin Gatlings

Every combat turret renders a barrel. What differs is where the barrel points. Most turrets — guns, cannons, mortars, gatlings, rockets, the disruptor — orient their barrel from the authoritative turret aim (weapon.rotation / weapon.pitch) that travels on the wire and corrects through the rotation position channel like everything else. Beam-family turrets are the one exception: their barrel and head are aimed by the last beam they fired, not by turret aim. This replaces the older rule where beam and rocket turrets drew as a bare head sphere with no barrel.

Rocket turrets are spinning gatlings

The three rocket turrets (turretRocketSlow, turretRocketFast, turretAntiAir) each carry a three-barrel gatling cluster (simpleMultiBarrel) that spins up while the turret is engaged — the same rig the Hippo’s turretGatling uses, as the visual reference. They aim like any other shot turret, the cluster tracking its target in yaw and pitch. The vertical launcher is the exception its own way: the sim pins its pose straight up and lets the gatling spin run purely on engagement state, so the barrels stay skyward and still wind up when it fires. Because a rocket turret’s aim is now a real visual, it rides the wire again — these are ordinary aiming turrets, no longer head-only.

Beam turrets point where their beam last fired

Beam-family turrets (turretBeam, turretBeamMini, turretBeamMega, turretBeamLong) render their tapered emitter barrel, but the renderer never reads turret aim for them. Instead the beam is the aim: each frame, just before the turret-pose passes run, the entity renderer records the firing direction of every active beam — the first segment of the beam polyline, running from the turret mount toward the target — keyed by the emitting host (recorded under every candidate id the beam carries, since composite/parented hosts can differ). The turret-pose pass reads that direction the same frame and orients the barrel and head along it. When the beam stops, the cache keeps the last direction, so the barrel freezes pointing wherever it last fired rather than snapping back to forward. The cone barrel extends out to the beam's start-point orb, so the beam reads as leaving the barrel tip. A beam turret otherwise colours exactly like every other turret — a plain team-color head and white barrel, with no engage-time colour flip.

The headOnly contract is wire-only, not “no barrel”

Marking a turret headOnly: true now means exactly one thing: its aim motion is pinned to zero on the wire (the snapshot serializer drops the rotation / angular-velocity / pitch / pitch-velocity slots). It no longer implies “no barrel,” and it no longer changes the turret’s colour — a beam turret renders with the same team-color head and white barrel as every other turret, with no engage-time flip. Whether a head-only turret draws a barrel is derived, not a separate flag: turretBarrelFollowsBeam(config) is exactly headOnly && the turret emits a ray. A head-only ray turret renders a beam-directed barrel (built and posed from the last beam, per above); a head-only turret that emits nothing aimable would still draw as a bare head. The simulation keeps tracking live aim internally so the beam still leaves in the right direction — that value just never reaches the wire, because the beam polyline already carries the geometry the barrel reconstructs from.

Why aim stays off the wire for beams

Aim motion is the busiest channel a turret pushes through the delta serializer. A continuously tracking turret dirties its host entity every frame on rotation / pitch / angular-velocity thresholds, which forces the entire entity row back onto the wire. For beams the renderer never uses that traffic, because the beam polyline already defines the direction and the barrel is posed from it, so zeroing the four angular fields collapses those transitions out of the diff path and the entity only dirties on target-acquisition or state changes. Rocket turrets accept that aim traffic deliberately, because for them the tracking, spinning barrel cluster is the read.

Where the rule does not apply

The disruptor cannon stays an ordinary full-barrel aiming turret: it is a melee gun whose visible barrel is part of the read. Construction emitters and visual-only hardpoints continue to use their existing dedicated rigs. Beam-directed barrels are specifically the beam (ray) turrets; everything else aims from the wire.

BAR Inspiration, Project Identity

Beyond All Reason is the comparison point for command ergonomics, readable large battles, queue-friendly controls, minimap interaction, and the feeling of commanding armies instead of individual pieces. This project should feel familiar to BAR players in its command vocabulary and battlefield expectations.

The project identity is different: the browser is a hard constraint, the network model is authoritative snapshots plus client prediction, and the world should be even more physically grounded. BAR is the gameplay fallback and ergonomics reference; this project owns its implementation, data contracts, and browser runtime.

One BAR behavior we deliberately imitate is the separation between player-visible command waypoints and private pathfinder output. In BAR, queued factory rallies, move/fight/patrol orders, and dragged waypoint markers are durable command intent; the Spring/Recoil engine expands active movement into path waypoints internally and repaths on a throttled cadence when progress or world state demands it. This project should keep that same conceptual boundary even though our pathfinder is owned by our Rust/WASM sim rather than by BAR's engine.

Commands require real systems

BAR command vocabulary is the fallback only when the command has an authoritative Budget Annihilation system behind it. A command-card button, hotkey, cursor, or preview is not allowed to exist as dead UI: the simulation, server authorization, snapshots, client feedback, and physical presentation must all have a coherent contract first.

This means transports, capture, resurrection, cloak/detection, manual launch, and unit-specific special-fire commands stay absent until their authoritative mechanics exist. Metal-extractor upgrades follow the same rule: a T1-to-T2 or area-upgrade command requires an authored upgraded extractor blueprint, resource/economy behavior, construction lifecycle, replacement rules, and render profile before the command is exposed. The prototype should prefer no button over a button that appears to promise unsupported BAR behavior.

Atomic Gameplay Pieces

Gameplay should be built from small, atomic pieces that do one job and compose cleanly: unit, tower, turret, shot, ray, and building definitions. These pieces should be swappable without hidden coupling. A unit should be able to vary its locomotion profile, mount different turret sets, fire different shot profiles, or appear from different factories because each piece carries a clear contract instead of leaking responsibility into its neighbors. The two things a turret can emit — shot and ray — are separate kinds, each in its own blueprint family, because only a shot is a body.

  • unit owns a mobile entity's body, mass, health, command capabilities, build cost, its own locomotion — the movement profile and rig (wheels, treads, legs, or hover) that produces and applies its movement forces, authored inline as part of the unit rather than as a separate piece — mounted turrets and the per-mount host-directed/fully-autonomous tag that decides which of them follow the host's lock-on, a host-level lock-on target governed by the unit's own lock-on inclusion object, and composition slots. A unit that has a turret contract is built and exists with its authored turrets present; there is no finished weapon host with empty turret mounts.
  • tower owns an immobile entity's body, mass, health, mounted turrets and the per-mount host-directed/fully-autonomous tag that decides which of them follow the host's lock-on, and a host-level lock-on target governed by the tower's own lock-on inclusion object. A tower has the same turret-and-lock-on contract as a unit, but no locomotion. Anything static that takes action against an entity ID through turrets is a tower — including fabricators (whose construction turret targets a build site) and defensive shooting towers. A tower is never a bare body waiting for a turret; its turret set is part of the host from creation through death.
  • turret is a standardized mounted emitter: it owns lock-on eligibility through its own lock-on inclusion object, aiming, tracking, cooldown, firing pattern (burst and spread), mount behavior, and a single by-id reference to the one emission it produces, declared by two fields — emissionKind (shot or ray) and emissionBlueprintId (the id of the emission blueprint in the namespace that kind selects) — naming exactly one shot or ray. It is addressable and lockable through its own EntityId, but it is not a separable body: no independent cost, no independent mass, no independent health, no death explosion, and no detached life after the host. It carries no host-specific data and no inline emission payload — the emission is named by id, never copied into the turret — so the same turret blueprint drops onto any host’s mount unchanged. It does not own whether it is host-directed or fully-autonomous; that role is set per mount by the host, so one turret blueprint can inherit a host target on one mount and run autonomously on another.
  • shot is a travelling munition a turret fires: a plasma bolt or rocket that leaves the barrel and persists as a physical body with its own EntityId, mass, health, and a death explosion. It owns its travel behavior, inherited homing target, recoil, damage, impulse, lifetime, submunition payload, and collision behavior, and — per Shots are destructible bodies too — detonates the instant its tiny health reaches zero. plasma and rocket are its variants. Shots are authored in their own blueprint family (shots.json).
  • ray is a sustained line emission — a continuous beam or a timed laser — traced from the turret to its target, recomputed every tick from the turret’s aim and bound to the turret’s lifetime. A ray is not a body: it has no mass, no health, and no EntityId of its own, and is attributed through its emitting turret (see Universal Runtime Entity IDs). It owns its line behavior, damage-per-second, recoil, and duration. beam (continuous) and laser (timed) are its variants. Rays are authored in their own blueprint family (rays.json).
  • building owns static footprint, construction, production, extraction, storage, or other static infrastructure that does not mount turrets and does not carry a host-level lock-on. Anything that needs a turret or lock-on is a tower, not a building.

Keeping shot and ray as two separate kinds — not variants of one umbrella — matters because only a shot is a body. A shot has mass, health, and a death explosion and obeys the entity base ledger; a ray is a sustained emission with none of those — it exists only while its turret sustains it and is attributed to it, never minting an id of its own. That split is what lets “every shot is a destructible body” be literally true without forcing a nonsensical health or mass onto a ray of light. Each lives in its own blueprint file (shots.json, rays.json) so the two contracts cannot bleed into one another.

Locomotion is part of the unit, not a separate piece

Locomotion is not a standalone atomic piece. A unit’s movement — its profile (mass, thrust, friction, damping, turn authority, contact model, vertical constraints) and its rig (wheels, treads, legs, or hover) — is authored inline on the unit blueprint, in the same definition as the body it carries. There is no separate locomotion blueprint family and no separate locomotion entity. The decision is deliberate: a drivetrain bolted to a chassis is not a thing the game ever needs to point at, cost, or kill on its own, so making it a first-class entity bought addressability nobody used and cost real complexity everywhere. Concretely, locomotion is folded into the unit on every axis the base ledger touches:

  • No separate id. Locomotion is not an addressable entity. It mints no EntityId, appears in no host hierarchy as a child, and is never a parent_id / root_host_id link. The unit is the body and its drivetrain together.
  • No separate cost. Movement carries no metal or energy line item of its own. A unit’s build cost already includes its locomotion; the drivetrain is paid for as part of the body, not billed and built as a distinct sub-part.
  • No separate health. Locomotion shares the unit body’s health. There is no “kill the drivetrain to immobilize the tank” sub-target — the unit moves until the unit dies.
  • Dies with the unit. When the body’s health reaches zero the whole unit comes apart at once; its locomotion explodes alongside it rather than persisting as loose, still-addressable debris.
  • Not lockable. A turret cannot lock onto a unit’s locomotion, because there is no locomotion entity to target. locomotions is not a lock-on family and there is no per-name locomotion narrowing — see Lock-on eligibility.

What survives unchanged is locomotion as a behavior and as a visual: the unit still answers the arrival-control question (see Locomotion is arrival control, not waypoint seeking) and still drives a wheel/tread/leg/hover rig (see Locomotion Visuals Are Frontend). Those are properties of the unit, computed from the unit’s own state — not the contract of a separate entity that has to be spawned, costed, healed, and torn off beside it.

Every body shares one base ledger

Before a physical body is a unit, a tower, a shot, or a building, it carries the same small base ledger. The kind-specific fields above sit on top of this floor; they never replace it. Turrets are deliberately different: they are addressable mounted emitters, not separable bodies, so they keep an id, mount metadata, lock-on state, and aim state, but they do not carry independent cost, mass, health, or death explosions. (ray is also absent: it is a sustained emission attributed to its emitter, not an addressable body.) The body ledger is:

  • id — the universal runtime EntityId (see Universal Runtime Entity IDs). Every entity is addressable.
  • cost — a { metal, energy } pair. Everything that can be produced has a price in both resources. A unit or tower’s cost includes its authored mounted turret set; the turrets themselves do not carry separate cost lines and are not built or paid for independently.
  • mass — every entity is a body the physics solver can push, and F = M · A needs an M for all of them — a travelling shot included.
  • health — every body has hit points and can die. A unit, tower, building, and travelling shot all take damage and all reach zero the same way. What happens at zero is kind-specific — a shot detonates, a host body dies and takes its mounted turrets with it — but turrets themselves have no independent health pool. (A unit’s locomotion has no health of its own either; it is part of the body and dies with it — see Locomotion is part of the unit.)
  • death explosion — a { radius, force, damage } blast the entity detonates the moment its health reaches zero. Every entity explodes when it dies (see Every death is an explosion); within the radius the blast applies an outward force and damage to everything caught in it. A turret does not detonate separately; it disappears with the host body that carries it.
  • geometry — three radii, one per physical role: a radiusVisual (drawn size), a radiusHitbox (the damage-receiving hurtbox), and a radiusCollision (body-versus-body no-overlap contact). See Entity geometry is three radii.

Treating cost, mass, and health as a shared body base — not as per-kind bolt-ons — is what lets the rest of this document make blanket statements about bodies. Lock-on is broader than damage ownership: a turret can still be locked onto because it is an addressable mount with aim and emitter state, but damage to that mount resolves through the host body unless a future mechanic explicitly says otherwise.

Entity geometry is three radii

Geometry is part of the base ledger too, and it is exactly three radii — one per physical role — carried by every entity of every kind. Keeping them as three named, single-purpose radii, rather than one “size” doing triple duty, is what lets a body be drawn large, hit on a tighter core, and kept apart at a wider stand-off without any of the three lying about the others:

  • radiusVisual — the drawn size. Presentation only: how big the body reads on screen. It has no say in damage or physics.
  • radiusHitbox — the damage-receiving volume (the hurtbox). A shot, beam, or explosion that reaches inside this radius deals damage to the entity; everything that tests “did this hit?” tests against it.
  • radiusCollision — the body-versus-body contact radius: how close two solid bodies may get before they resist sharing space. This is the radius that keeps units out of each other and that a body collides on. It is hard contact, not a force at a distance.

The same three names mean the same three things on a unit, a tower, a building, and a shot — one vocabulary, not a per-kind grab-bag of ad-hoc radius fields, so the collision and damage kernels read one contract for every body they touch. A turret may expose an addressable aim/selection volume for lock-on and line targeting, but that volume is a mount target on the host, not an independent physics body with its own health. The death explosion’s own radius (inside its { radius, force, damage } field) is a fourth, separate thing — it is the blast the body throws on death, not a property of the standing body — and must not be folded into these three.

radiusCollision is deliberately named for hard contact so that the word push stays reserved for a future push field capability: an optional, authored, sustained force that shoves other bodies at a distance — a heavy unit’s bow-wave, or a launched piece plowing through a crowd. That is a force-over-distance, not a no-overlap radius; when it is added it will be its own capability layered on top of radiusCollision, never a rename of it.

Turrets are inseparable mounted emitters

A turret is addressable because targeting, source provenance, aim state, and renderer inspection need a stable thing to point at. It is not a separable body. A host does not carry decorative turrets with their own cost, mass, health, or death explosions; it carries mounted emitters that live and die with the host body. Destroying a host destroys the host’s authored turret set. There is no weaponless finished unit or tower, no turret-only wreck that keeps fighting after detachment, and no intermediate body-with-empty-hardpoints construction state.

Per-turret lock-on remains useful, but it means “aim at this mounted emitter” or “use this emitter’s targeting policy,” not “remove this turret’s private health pool.” A disarming or precision mechanic can still target a turret ID to aim at the mount, interrupt firing, or apply a host-level effect, but the health, cost, and mass accounting stays on the host body.

Every piece is an agent; the player commands only hosts

Every host body is an agent — it has its own health, its own death, and, where its kind allows, its own behavior — but agency and command are different things. The player issues orders to exactly three kinds: units, towers, and buildings. Those are the hosts. Everything mounted below a host is driven by that host, never by the player’s cursor directly. Control runs down a chain:

  • The player controls units, towers, and buildings — the orders on each type’s action panel (move, fight, build, set lock-on, ON/OFF, and so on).
  • A unit moves itself and controls its own turrets: the player’s order to the unit becomes thrust through its own (intrinsic) locomotion and a host lock-on offered to the turrets.
  • A tower controls its own turrets the same way, but does not move (it has no locomotion).
  • A turret controls the one shot it emits. Whether the turret takes its aim from its host or finds its own is the host-directed / fully-autonomous distinction — see Host-directed turrets carry the host lock-on. Either way the turret is commanded by its host, not by the player.
  • A shot, once fired, runs itself — ballistic or homing toward its inherited target, detonating on contact or at zero health. Nothing commands it after launch.

So the player never reaches past a host to puppet a turret or a shot directly. You command the unit; the unit commands its parts; a turret is either host-controlled (a host-directed mount carrying the host’s lock-on) or independent (a fully-autonomous mount running its own acquisition), but the line of control always runs through the host that carries it.

Because turrets are inseparable mounted emitters, the chain does not survive without its host. A turret has targeting agency only while the host exists and carries it. When the host dies, its mounted turrets die with it as part of the same assembly; there is no detached turret agent and no hostless turret control hierarchy.

Every death is an explosion

Death is never a quiet despawn. Every body carries a death explosion in its base ledger — a radius, a force, and a damage amount — and when its health reaches zero it detonates that blast into the world. The explosion is a physical event first (per the Physics Model): within the radius it applies an outward force that heavier bodies resist more, and it applies damage to every entity caught in it, friend or foe. A shot’s detonation is the smallest instance of exactly this rule; a commander’s is the largest. Turrets are not bodies, so they do not have separate death explosions.

The defaults matter because they make a killed host read as one physical event. A unit’s or tower’s mounted turrets are part of the host assembly, so a dying host does not leave intact guns behind and does not launch autonomous turret pieces into the battle. Its drivetrain and mounted emitters end with the body. The host explosion is still free to damage other nearby entities; it simply does not manage its own turrets through separate health pools.

Because the blast is authored data, not a fixed rule, a blueprint can deliberately break the default. A unit built to be a bomb can carry an oversized radius, a vicious force, and heavy damage so that its death shreds everything nearby — a designed suicide weapon rather than an accident. A fragile scout can carry almost nothing. Either way the contract is identical: health hits zero, the body explodes, and the radius spends force and damage on whatever is standing in it.

Shots are destructible bodies too

A shot is an entity, so a shot has health — a deliberately tiny amount. A shot dies the instant its health reaches zero, and dying means it detonates: it runs its normal explosion / splash / submunition path exactly as if it had reached its target, just wherever it happened to be. Two things drive a shot’s health to zero:

  • Contact — touching the ground, another shot, or any other solid body forces the shot’s health straight to zero and detonates it on the spot. This is the ordinary “it hit something” case expressed through the health field rather than a separate collision flag.
  • Incoming fire — because the health is small but real, a weapon can whittle it down before contact. A point-defense turret that locks onto enemy shots (its inclusion object includes only enemy_entities and shots) chips that tiny health to zero and pops the round in mid-air, short of its target.

This unifies “the projectile hit the dirt,” “two rounds collided,” and “a flak turret shot the missile down” into one rule: a shot is a body with health, and zero health detonates it. Travelling shots remain event-driven on the wire — spawn, course-correction, despawn — per Ballistic projectiles are event-driven; the detonation is just one more discrete event. This applies to shot emissions only: a ray is not a body, carries no health, and is never “shot down” — it simply stops when its turret stops sustaining it.

Shot arming is not source exclusion. A newly fired shot begins unarmed and becomes active only after its hitbox has cleared both the originating host’s collision body and the emitting turret’s collision body. While unarmed, the shot does not test entity hitbox contacts. Once armed, it collides with every solid entity hitbox, including the turret or host that fired it. Source provenance remains for attribution and replay; it is not a collision immunity list.

Materialization: build-in and death-out are one reversible opacity channel

Every host body and every mounted turret visual carries a per-host materialization opacity that runs 0 → 1 as it is built and 1 → 0 as it dies: 0 is fully transparent, 1 is fully opaque. This is the same “construction mirrors destruction” pipeline run in two directions — one channel, not two effects. It applies to the four assembled host kinds (buildings, towers, units, and their mounted turret visuals); a shot is exempt (see below).

Build-in. A host assembly fades from transparent to opaque over the host build fraction. The body and authored turret visuals materialize together because turrets are not separate construction pieces (see Hosts build as complete assemblies). A host counts as complete once the host body is complete; its mounted turrets are present as part of that product, even if their firing logic remains disabled until completion.

Death-out. The same channel runs backward, but death is still an explosion first — never a quiet fade-to-nothing (per Every death is an explosion). The opacity ramp is the aftermath layer, not a replacement for the blast: the dying body shell stays where it fell and de-rezzes from opaque to transparent in place, while the death blast’s force and the material explosion fling visual debris outward. Killing the body does not gently dissolve it; it detonates, and the shell fades as the debris flies. Mounted turrets are part of that host visual breakup, not live pieces.

The material explosion uses the host’s actual pieces, not generic shapes. Every atomic visual part as rendered — chassis edges, turret heads, barrels, treads, wheels, legs — can be thrown as matched debris. Those debris pieces are presentation artifacts, not live turret agents. The explosion applies gameplay force and damage to external bodies in its radius; the host’s own mounted turret visuals simply participate in the material breakup and then fade.

Simulation owns the truth; the renderer owns the look. The damage, the radial force, and the destruction of the host body are deterministic and authoritative — force changes future positions, so all clients must agree. The opacity ramp, the tumbling debris, and the fireball are client-only cosmetics driven off the death event; non-deterministic randomness is fine there because nothing downstream of the sim reads them (cross-ref Locomotion Visuals Are Frontend). The fade is rendered as a screen-door (dithered) dissolve in the opaque pass, not alpha blending, so a multi-part body still writes depth and self-occludes through the transition — no see-through-to-the-back-faces artifact and no transparent-queue sort cost at scale.

Shots are exempt from build-in. A shot is spawned whole from its turret, not assembled piece by piece, so it has no materialization ramp. Its “death” is its detonation (per Shots are destructible bodies too) — a small fire blast and, at most, a single shard of debris.

Sensors are capabilities, not free turret range

Sight, radar, and cloak detection should be authored as sensor capability sources, not as automatic properties of every combat turret. A sensor may be mounted on a unit, building, or a turret-like hardpoint when it needs a visible dish, mast, sweep, or directional rig, but that does not mean every turret config should inherit sight and radar fields. The common combat turret contract should stay focused: mount pose, yaw/pitch dynamics, lock-on policy, acquire/release/fire envelopes, cooldown, line-of-sight gate, ballistic/fire feasibility, host-directed policy, and the shot or utility effect it emits.

If a weapon needs unusual intelligence behavior, express that as a narrow targeting requirement on the turret, not as a hidden sight radius. Examples: requires full sight, can fire on radar contact, can maintain lock briefly after sight loss, can fire at a commanded ground point, or can only attack targets it personally sees. Those requirements are gates over the player/team sensor aggregate. They do not create new vision by themselves.

This keeps the composition clean. A radar station can be a building with a radar sensor and a decorative rotating rig — no turrets, no lock-on, so it is not a tower. A scout can be a unit with a strong full-sight sensor and weak weapons. An artillery platform can be a unit with a long fire envelope, short local sight, and permission to fire on radar contacts. A command unit can carry both full sight and detector coverage without pretending its gun barrels are what make it observant.

Lock-on policy lives on turrets, not rockets

A turret is the only piece that acquires a lock-on target. A rocket can inherit the exact entity the spawning turret was locked onto at launch, and it can home toward that inherited target while the target remains alive in the sim. The rocket must not run its own target acquisition pass, scan for replacements, or choose a new victim after launch. If the inherited target no longer exists, the rocket loses guidance and continues on its current flight path under normal projectile physics until it hits something or expires.

Host intent is not turret lock-on

Units and towers can carry a host-level target chosen by the player, AI, or command system. That target expresses intent: attack this enemy, guard this ally, repair this unit, assist this tower, or otherwise keep the host’s current order anchored to an entity. The host may lock onto any entity that has an ID — a unit, a tower, a building, or a turret — but only the kinds and relationships its own lock-on inclusion object admits. That host-level inclusion object is the same additive schema a turret carries; it simply lives on the unit or tower blueprint and governs the host lock-on rather than a single weapon. The host target is not the same thing as a turret’s current lock-on target. It is an input each mounted turret may consult; it must not blindly overwrite every turret target on the host. Buildings carry no host target, and no host-level inclusion object, because buildings do not mount turrets — that is the boundary that separates the two static types.

Two inclusion objects therefore apply in sequence. The host’s own inclusion object decides what may become the host target; each mounted turret then decides whether that host target is meaningful for it. A target survives only if both objects admit it. The turret first checks its own lock-on inclusion policy, then the normal validity gates such as aliveness, visibility, range, line of sight, and shot compatibility. If the host target passes, the turret may inherit it as a priority lock. If the host target fails, that turret rejects the inheritance and runs its own targeting flow. The host target remains intact either way.

This separation is what lets mixed-role hosts work. A unit guarding a friendly entity can keep that friendly entity as its host target; a repair or construction turret may inherit it, while a fighting turret that includes only enemy_entities ignores it and searches for enemies normally. Commands belong to the host; lock-on belongs to the turret.

Host-directed turrets carry the host lock-on; fully-autonomous turrets ignore it

Every turret mount on a unit or tower is tagged as either host-directed (inherits the host lock-on) or fully-autonomous (runs its own targeting). When the host is “locked on” to another entity — whether through a player command, an AI order, or any other host-level target intent — a host-directed turret’s lock-on id is overwritten by the host’s lock-on id (provided that target is admitted by the turret’s own inclusion object and validity gates), so it acts in concert with its unit or tower. Fully-autonomous turrets never have their lock-on overwritten by the host; they completely ignore the host’s lock-on and run their own independent targeting flow as if the host carried no target at all. The rule applies identically to units and towers: every host-directed mount adopts the lock-on, and every fully-autonomous mount ignores it. Buildings carry no turrets and therefore no host-directed/autonomous distinction.

This designation is not a property of the turret blueprint. It is authored on the mount entry in the unit or tower blueprint and definitions — the host’s own definition decides, slot by slot, which of its mounts are host-controlled and which are fully-autonomous. The same turret blueprint can therefore be host-directed on one host and fully-autonomous on another: a cannon may inherit the host target on a battle tank yet ride as a fully-autonomous weapon on a larger hull. The turret blueprint supplies the weapon — mount pose, aim dynamics, cooldown, lock-on inclusion object, and the shot it emits; the host blueprint supplies the role each mount plays in the host’s command flow. Keeping the host-directed/fully-autonomous tag on the mount rather than the turret is what lets one turret definition serve both roles without duplicating the blueprint.

A host needs host-directed mounts only for the orders that should overwrite turret acquisition. An attack order can drive several attack weapons, one attack weapon, or none; mounts left fully-autonomous keep their own target selection regardless of the host order.

This is an explicit per-mount rule, not a convention. The blueprint loader walks every unit and tower blueprint and verifies that every mounted turret declares a boolean hostDirected choice. Zero host-directed mounts means that all mounted turrets of that role are autonomous. Multiple host-directed mounts are legal: each one inherits the host target independently and then applies its own turret-level inclusion and validity gates. The loader must throw at startup only when the choice is missing or not a boolean. Hosts that carry no turret of a given kind simply have no weapon for that role. Building blueprints are skipped by this loader pass because buildings do not mount turrets at all.

A host-directed turret is not only host-directed. When the host has no current lock-on, the host-directed turret falls back to its own autonomous targeting flow exactly like a fully-autonomous turret would: it applies its inclusion policy, scans for candidates, scores them, and engages. The host lock-on is a priority input layered on top of normal targeting, not a replacement for it. The only difference between the two tags is whether the host’s declared target is even considered.

This keeps the host’s declared focus — the thing the player pointed at — anchored to the weapons that define the unit’s role, while letting auxiliary turrets (anti-air defense, point-defense, secondary cannons) keep doing their own job without being yanked off it whenever the host acquires a new lock. A tank with a host-directed main gun and a fully-autonomous anti-air turret will point its main gun at the commanded enemy ground unit while the anti-air turret keeps shooting passing aircraft, regardless of what the host is locked onto. Host-directed turrets still apply their own inclusion policy and validity gates on top of the host target; fully-autonomous turrets skip the inheritance step entirely.

Fight-move and patrol halt behavior is authored separately from host-directed targeting. Unit turret mounts can set requiredEngagedForFightStop; every non-visual turret with that mount flag must be engaged before the unit stops. Host-directed turrets that are not marked required can inherit the host target without pinning the movement leg, and autonomous turrets can participate in fight-stop gating if their mount opts in.

Lock-on eligibility is authored as an additive inclusion object, and three kinds of locker each carry one: a turret (governing that weapon’s lock-on), a unit, and a tower (each governing its host-level lock-on). The default is identical for all three, and it is deliberately as narrow as it can be: a locker with no lock-on inclusions can lock onto nothing. Lock-on is off by default and additive from there: a turret never silently inherits a family it might hit; it locks only onto the kinds, relationships, and names it explicitly opts in to. A fully-permissive locker that has added every relationship and every family can lock onto any living entity in the game that has an ID, of any kind — any friendly or enemy unit, tower, building, turret-mount, or travelling shot — but that breadth is authored, never assumed. Buildings lock onto nothing, so they carry no inclusion object. The same eligibility rule applies whether the lock-on is a unit’s or tower’s host-level target set by player command, a turret’s inheritance of that host target, or a turret’s autonomous acquisition: one additive schema, one shared targeting kernel across units, towers, and free-acquiring turrets.

  • includeLockOnLevel0FriendsAndEnemies admits whole ownership relationships before scoring. Valid high-level entries are friendly_entities and enemy_entities; a relationship that is not listed is never lockable.
  • includeLockOnLevel0Entities admits whole target categories before scoring. Valid high-level entries are units, towers, buildings, turrets, and shots — every addressable entity kind, so any kind can be added and, by omission, any kind is excluded by default. (There is no locomotions entry: a unit’s locomotion is part of its body, not a targetable entity — see Locomotion is part of the unit.)
  • includeLockOnLevel1Units narrows an already-included unit family to only the named unit blueprints; an empty list applies no name restriction.
  • includeLockOnLevel1Towers narrows an already-included tower family to only the named tower blueprints.
  • includeLockOnLevel1Buildings narrows an already-included building family to only the named building blueprints.
  • includeLockOnLevel1Turrets narrows an already-included turret family to only the named turret blueprints.
  • includeLockOnLevel1Shots narrows an already-included shot family to only the named shot blueprints — e.g. a point-defense turret that should pop only heavy rockets, not every plasma bolt.
  • lockOnRequiresTargetLockedOntoSelf is a four-state enum, separate from the family lists, deciding how an otherwise-admitted enemy is weighed by whether it is itself locked onto this locker. "ignore" (the default) does not care; "require" admits a candidate only while it is itself locked onto this turret or its host, and drops it the instant it re-targets; "preferReacquire" ranks such threats above every other candidate and rescans when the current target stops reciprocating; "preferHold" applies the same threat ranking during normal scans but does not rescan solely because the current target stops reciprocating. Every shipped blueprint stays on "ignore"; the non-default modes exist for future retaliation-style specialist weapons. See Reciprocal lock-on: only hit who is hitting you.

The hierarchy should be evaluated in order: ownership, aliveness, and visibility decide whether something can be considered; the relationship inclusions admit friendly or enemy candidates; the entity-family inclusions admit whole kinds — units, towers, buildings, turrets, or shots; level-1 inclusions, when present, narrow a surviving family to specific authored names; range, line-of-sight, and scoring then pick the best lock from the survivors. A construction turret that should lock only onto friendly entities includes friendly_entities. A fighting turret that should only attack hostile targets includes enemy_entities. A normal weapon turret that should ignore turret mounts simply leaves turrets off its family list. Because lock-on is off by default, targeting a specific piece is just a narrow inclusion object: a point-defense turret that should shoot down incoming munitions and nothing else includes only shots; a disarmer that should strip weapons includes only turrets. Most ordinary weapons, in turn, include the whole-body families — units, towers, buildings, and turrets — and leave shots off so they spend their fire on whole bodies. Specialist weapons can then use the level-1 lists to restrict a family to named unit, tower, building, turret, or shot blueprints without changing the shared target acquisition code. A unit or tower runs its own host-level inclusion object through this identical hierarchy before its host lock-on is set; each mounted turret then runs its own object again when it decides whether to inherit that host target.

We do not want mixed or overlapping roles. A turret should not secretly define locomotion. A shot should not secretly define a unit. A building should not smuggle combat behavior — if it needs a turret or a lock-on, it is a tower, not a building. A unit should not duplicate shot physics. Cross-piece behavior should be expressed through explicit references and narrow data contracts, so designers can recombine pieces freely and the simulation can keep hot paths simple. These pieces are authored as data, not code — see Config Is Data, Not Code.

Reciprocal lock-on: only hit who is hitting you

The inclusion object carries one mode beyond the family lists: lockOnRequiresTargetLockedOntoSelf, a four-state enum — "ignore", "require", "preferReacquire", or "preferHold" — that decides how a locker weighs an otherwise-admitted enemy by whether that enemy is itself locked onto it. "ignore" is the default and the value every ordinary turret, unit, and tower uses: reciprocal lock has no bearing on candidacy or scoring, and the locker does not care whether its prey is shooting back. "require" admits a candidate only while that enemy is itself locked onto this turret or its host; an enemy not aiming at us is simply not a candidate, no matter how close or how otherwise valid it is. The two preference modes treat enemies locked onto us as a strictly higher priority tier without making them mandatory: if any are aiming at us during a scan, the locker engages those alone, and only when none are does it fall back to the ordinary candidate pool. Today no blueprint leaves the default — every turret, unit, and tower stays on "ignore"; the non-default modes are reserved for future retaliation-style specialist weapons.

"require" and "preferReacquire" are both evaluated in real time, not just at acquisition. The locker re-checks against the reciprocal condition exactly as it re-checks range, line-of-sight, and aliveness. Under "require" the instant the enemy re-targets — locks onto a different body, drops its lock, or dies — the condition fails, the locker loses that lock the same tick, and re-acquisition runs over whatever else is currently aiming at it. A "require" locker therefore tracks whoever is shooting at it right now and follows the threat as it changes; with nothing aimed at its host it has zero candidates and sits idle, which is the intended counter-fire behavior — it punishes focused fire and is trivially sidestepped by not shooting at it.

Under "preferReacquire" the same re-evaluation re-sorts the two tiers instead of emptying the pool: the instant an enemy starts firing on the locker it jumps to the threat tier and is engaged ahead of any opportunistic target, and the instant the current target aims away the locker scans for a better reciprocal target while still falling back to the general pool rather than going idle. Under "preferHold", reciprocal threats still form the strict higher tier whenever a normal acquisition or scan occurs, but an existing valid target is not disturbed solely because it stopped reciprocating. Within either tier the ordinary scoring — including current-target stickiness — still picks the single best lock, so a preference weapon does not thrash between equally-ranked threats. The tiering is strict by default — any threat outranks any non-threat — with a softer scoring-bonus variant available if a weapon should let an overwhelmingly better opportunistic target win. A reciprocal preference weapon retaliates first and suppresses second.

“Locked onto us” means locked onto either this turret or its host (self_turret or self_host) and is read from the lock-on state committed at the end of the previous tick, so the check is order-independent and cannot form an intra-tick cycle when two reciprocal lockers face each other; “the same tick” above is therefore the next resolution tick, imperceptible at simulation rate. The matched state is “currently holds a lock onto us”; reacting only to enemies actively firing rather than merely aimed would be a separate future refinement, not part of this enum. The targeting kernel reads prior committed target ids from candidate turrets and checks whether any point at this turret or host, avoiding same-tick retarget cycles without relying on a speculative inverse set. Any value but "ignore" is meaningful only alongside enemy_entities: a "require" locker that admitted no enemy relationship would be inert and a reciprocal preference one would collapse to ordinary opportunistic targeting — both authoring smells the loader can reject.

If a specialist turret ever needs a second, explicit targeting contract layered on top of the ordinary inclusion object — behavior the additive lists cannot express, such as ranking threats by sustained DPS or aiming by a derived geometric rule — it gets a validated secondaryLockOnProfiles entry with its own regression tests, not a special case buried in the shared targeting kernel. Large targeting regularization passes must preserve any such profile, its validation, and its tests instead of simplifying the turret into an ordinary scanner.

Universal Runtime Entity IDs

Blueprint IDs identify authored definitions. Runtime entity IDs identify live instances. They are different contracts, and the game needs both. Every addressable thing in the simulation gets a runtime EntityId, and the test is addressability, not category: if anything can point at it, remember it, lock onto it, damage it, or explain that it came from it, it is not anonymous. That covers units, towers, and buildings (top-level hosts), every mounted turret as an addressable emitter, every travelling munition — a plasma bolt or rocket that has left the barrel and now persists, spawns submunitions, and can be hit — and every build site. It does not automatically mint an entity for every emission. A ray projected by a live turret is a sustained emission of that turret, recomputed each tick from the turret’s aim and bound to its lifetime; it is attributed through the emitting turret’s id and the emission’s source record, not by allocating a separate id. (A shot is the opposite case — a travelling munition that does carry its own id.) Transient detail — a ray’s polyline vertices, resource balls in flight — stays anonymous for the same reason. Give one of these its own id only when the design genuinely needs to point at that specific instance; the default is to attribute it to its emitter.

Opaque handle, explicit metadata

EntityId is an opaque runtime handle, preferably generation-checked. It is not the place to encode player, team, host, blueprint kind, or storage layout as authoritative meaning. Those values are mutable, contextual, or implementation details; baking them into the numeric ID creates stale IDs when a unit is captured, a team relationship changes, a storage slot is reused, or a projectile outlives the turret that spawned it. The ID resolves through the entity registry into a dense metadata row. The metadata row carries the meaning. 

EntityMeta {
    id,
    kind,              // unit, tower, building, turret, shot, ...
    blueprint_kind,
    blueprint_id,
    owner_player_id,
    team_id,
    parent_id,         // immediate containment parent, if any
    root_host_id,      // top-level unit/tower/building, or self
    mount_index,       // turret/pylon slot when mounted
    storage_pool,
    storage_slot,
    generation,
    alive,
    targetable,
}

Systems may cache storage slots for speed, but the cache is never the identity. Commands, selections, lock-ons, projectile provenance, damage attribution, replay events, resource movements, debug tools, and audio events all speak in EntityId values and resolve them through the same registry.

The host hierarchy is complete

Every entity stores both its own EntityId and the host ids it belongs to, so the chain from a shot back to the unit that fired it is always explicit data, never an inference. The rule is uniform across all five kinds:

  • Unit, tower, building — top-level hosts. Each carries its own EntityId; parent_id is null and root_host_id points at itself. A unit’s locomotion is part of that unit body, not a child entity — it carries no id of its own.
  • Turret — carries its own EntityId, its immediate parent_id, its root_host_id (the unit or tower it is bolted to), and its mount index. It is addressable and lockable, but its lifetime, resources, and damage are owned by the host; it cannot detach into a hostless entity.
  • Shot — a travelling munition (plasma, rocket) carries its own EntityId as a real body. A ray is a sustained line emission with no body of its own: it is bound to its emitter and attributed through it rather than minting an id. Either way the emission’s source record names both the turret id that fired it and the host id that turret was mounted on (see Shots carry immutable source provenance).

A tower-mounted construction emitter, for example, stores both the turret slot it occupies and the tower that carries it. This hierarchy is not inferred from blueprint order, array index, mesh ownership, or a renderer object tree. It is simulation data, validated at spawn, and available to every hot-path query that needs it.

The terms are deliberately narrow. parent_id means immediate containment. root_host_id means the top-level unit, tower, or building that owns the mounted assembly. owner_player_id and team_id describe gameplay ownership. They are stored beside the ID, not inside it.

A build site is the product, carrying its own id from frame one

A unit or building under construction is not a placeholder that later hands off to the “real” entity. It is the real entity, alive from the first frame in an under-construction state, carrying the EntityId it will keep for its whole life. Health, completion fraction, and buildability ramp up; the id never changes. A construction turret locks onto that id like any other target, the player can select it and queue commands on it, and those selections and orders survive completion because nothing is swapped underneath them. There is no separate “build shell” id pointing at a “product” id — one continuous identity from nanoframe to finished structure. That is what keeps construction, damage, capture, lock-on, and command all speaking about the same entity, with no re-id handoff bug at the instant it completes.

Hosts build as complete assemblies

Construction no longer builds turret mounts as separate paid or damaged pieces. A unit or tower under construction is still the product from frame one (per A build site is the product), but its authored turret set is part of that host assembly for the whole construction span. The host’s body health and build progress ramp together; its turrets may remain visually dim, disabled, or non-firing until the host reaches the authored readiness threshold, but they are not separate construction targets with their own cost or health.

A finished unit or tower must therefore have its authored turret set present. Interrupting construction leaves an incomplete host, not a coherent finished body with empty turret mounts. Buildings remain the turretless host kind; anything static that needs a turret is a tower and appears with that turret as part of the same product.

Shots carry immutable source provenance

A shot is not contained by its turret after launch, but it must retain exactly where it came from. Every shot, beam, laser, projectile, submunition, and spawned child effect carries a source record: 

ShotSource {
    source_turret_id,
    source_host_id,
    source_root_id,
    source_player_id,
    source_team_id,
    source_turret_blueprint_id,
    source_shot_blueprint_id,
    spawn_tick,
    parent_shot_id,
}

Source provenance is immutable for the life of the shot. If the source turret dies, the host is captured, or the owning player is eliminated while the projectile is in flight, the projectile keeps its launch attribution. Damage, friendly-fire decisions, replays, submunition inheritance, audio, and debug traces read the source record; lock-on and collision read the projectile's current entity row.

Lock-on resolves entity refs

A turret lock, host target, command target, homing target, assist target, repair target, and build target all store an entity reference, not an array index. The reference is resolved through the registry and generation-checked before use. If the row is gone or the generation no longer matches, the reference is stale and the source drops or reacquires according to its normal rule. Turrets, units, and towers can therefore lock onto any targetable EntityId in the world of any kind — a whole unit or tower, a single mounted turret, a build site, or a travelling shot — that their own inclusion object admits, while the targeting kernel still gets fast access to kind, blueprint, owner, team, hierarchy, position, radius, and storage row.

This is a major design requirement, not a debugging convenience. A universal runtime ID system is the spine that lets targeting, selection, commands, resource flows, projectile inheritance, kill attribution, save/replay, desync diffing, and renderer inspection all talk about the same world without inventing local ID schemes.

Towers Are Static Hosts That Lock On And Fire

A tower is the immobile peer of a unit. Like a unit, a tower has a body, mass, health, a host-level lock-on field, and an authored set of mounted turrets that fire shots. Unlike a unit, a tower has no locomotion: it is bolted to a footprint on the terrain and never asks the pathfinder for help. Like a unit, a tower’s host-directed turrets consume the host’s lock-on; fully-autonomous turrets ignore it. Unlike a building, a tower is allowed to take action against entities — which is precisely what mounted turrets do. A tower is never a turretless finished body; the mounted emitter set is part of the tower product.

Towers exist as a separate entity type, not as a flag on buildings, because the simulation actually treats them differently. A tower carries combat state (turrets, lock-on, inclusion masks, acquisition probes), and the targeting kernel must be able to enumerate tower candidates without conflating them with pure infrastructure. Buildings (solar, wind, extractor, radar, converter) are deliberately on the other side of that line: no turrets, no lock-on, no participation in the combat acquisition loop. Erasing that boundary — for example, by giving the fabricator a “construction turret” while still calling it a building — leaks combat plumbing into every code path that handles plain buildings.

Anything that locks onto an entity ID is a tower

The single test for tower-vs-building is mechanical: does the static thing carry a host-level lock-on, fire shots through a turret, or otherwise act on a target by entity ID? If yes, it is a tower. Concretely, this covers:

  • Fabricator — mounts a construction turret that locks onto a build site (an entity in progress) and emits energy + metal at it. The build site is addressed by its entity ID. Therefore the fabricator is a tower, not a building. The fact that the construction turret is visual-only for damage purposes does not change the contract — it still has a target, inclusion policy, and host-directed relationship to its host’s player intent.
  • Shooting towers (beam tower, cannon tower, and any future long-range static gun) — mount combat turrets that lock onto enemy entities. These are the textbook case.
  • Future static repair pads and anti-air emplacements — if they target entities through a turret, they are towers from the day they are added. Do not invent a new flag on the building schema; add the blueprint to the tower catalog.

Lock-on selection: anything with an ID is a candidate

A tower’s host-level lock-on can name an entity in the world that has an ID — another tower, a unit, a mounted turret, even (in principle) a building — but only the kinds and relationships the tower’s own lock-on inclusion object admits. Whether the surviving lock-on is then honored by a mounted turret is decided by that turret’s inclusion policy, exactly the same hierarchy that applies to a unit’s host-directed turrets:

  • Ownership inclusions (friendly_entities / enemy_entities) admit only the listed relationship first.
  • Entity-family inclusions (units, towers, buildings, turrets) admit only the listed kinds next.
  • Level-1 named inclusions, when present, narrow a surviving family to specific blueprints.
  • Range, line-of-sight, and shot compatibility gate the remaining survivors.

A shooting tower that includes enemy_entities and leaves buildings out of its family list will refuse to inherit a friendly building as a host target even if the player clicked one, and will fall through to autonomous acquisition. A fabricator’s construction turret that includes friendly_entities and leaves turrets out will refuse to aim at a mounted turret ID. The targeting kernel is the same kernel units use; the only thing the tower adds is the static-host shape.

Host-directed turrets inherit the lock-on

A tower with multiple mounted turrets follows the same host-directed contract as a unit, and authors it the same way: the hostDirected tag lives on the mount entry in the tower blueprint, not on the turret blueprint. Every mount marked hostDirected: true inherits the host’s lock-on, subject to that turret’s own inclusion object and targeting gates. Mounts marked hostDirected: false ignore the host lock-on entirely and run their own acquisition. The blueprint loader validates only that each mount declares the boolean choice.

What towers are not

  • Towers are not buildings with a turret tacked on. The blueprint goes in the tower catalog (e.g. towers.json), not the building catalog. The Rust EntityType enum carries Tower as a peer of Unit and Building, not as a building subtype.
  • Towers are not units. They have no locomotion field and no movement commands. Pathfinding, arrival control, hover, banking, and locomotion visuals are all skipped for tower bodies.
  • Towers do not get the building ON/OFF production toggle. ON/OFF is a producer-economy concept tied to solar/wind/extractor/etc.; a tower’s analogous toggle is Fire at Will / Hold Fire, applied to its mounted turrets through the same fire-control state units already expose.
  • Towers do not host static infrastructure behavior. A tower does not produce energy, store metal, or run radar coverage from its own definition. If a designed piece needs both turret-acting behavior and infrastructure behavior, the right answer is two pieces (a tower + a building) co-located on a footprint, not a smuggled hybrid.

Selection Menus Are Uniform Per Entity Type

The action panel that appears when the player selects something must be small, consistent, and entity-type-aware. The selection’s entity type dictates which actions appear; the chosen blueprint never adds bespoke buttons. A unit always reads the same as every other unit, a tower always reads the same as every other tower, a building always reads the same as every other building. Anything more bespoke is clutter, and clutter on the action panel is exactly the kind of UI drift this document is meant to retire.

Mixed selections (e.g. units + towers) reduce to the intersection of the per-type action sets, plus an explicit “select-only-X” affordance to drop down to a single type. The panel must never show an action that the active selection cannot execute.

Unit selection panel

  • Movement orders: Move, Fight, Patrol, Attack-Area, Attack-Ground, Guard, Stop, Wait.
  • Queue management: Undo Q, Clear Q, Ping.
  • Fire control toggle: Fire at Will / Hold Fire.
  • Build menu (only if the unit is a builder): the build roster filtered to what this builder is allowed to construct.
  • Commander specials (only on commander units): D-Gun, Repair, Reclaim.
  • Self-destruct.

Tower selection panel

  • Set Lock-On Target: click any entity with an ID the tower’s lock-on inclusion object allows. The selection turns into the tower’s host-level lock-on; every host-directed turret inherits it, and fully-autonomous turrets ignore it.
  • Clear Lock-On: revert to autonomous acquisition.
  • Fire control toggle: Fire at Will / Hold Fire.
  • Production menu (only on fabricator-class towers): the unit-build roster filtered to what this tower is allowed to produce.
  • Self-destruct.

Building selection panel

  • ON/OFF toggle.
  • Self-destruct.

That is the full vocabulary. Adding a new action category requires a design decision about which entity type it belongs to and an update to this section. Stuffing a new button onto only some blueprints inside an existing type is what makes the panel creep over time, and is not allowed.

Producer Buildings Are ON/OFF

Solar collectors, wind turbines, and metal extractors carry a single binary state: ON or OFF. The state means exactly one thing and gates two outcomes together:

  • ON — the building can produce its resource (energy or metal) and takes normal damage.
  • OFF — the building cannot produce, and incoming damage is multiplied by 0.1 (10× more fortified).

Production and survivability are intentionally welded to the same flag so the player tradeoff is legible: keep producing and stay vulnerable, or stop producing and ride out the hit. The ON/OFF state is the single authoritative input for both the renderer’s open/closed pose and the economy’s production accounting; they must never disagree.

Materials Are Independent Of Shape

A surface material is a contract about how a surface interacts with the world: reflectivity, line-of-sight occlusion, projectile response, hit sound, shader treatment, particle reactions. Two pieces that mount the same material must look identical and behave identically. The only thing they are allowed to differ on is geometry — where the surface sits and what shape it takes.

The canonical example is ice. A frozen pond and a cuboid ice block are the exact same material: same reflectivity, same line-of-sight occlusion rule, same projectile response, same shader, same visual treatment. One piece carries the material as a flat sheet of terrain; the other carries it as a block. Beyond that geometric difference, a player should be unable to tell them apart by behavior or by eye, and there should be no code path that treats them differently.

  • A material is defined once, with all of its physical and visual properties — reflectivity, refraction, occlusion, hit response, sound profile, shader inputs, particle reactions.
  • Shapes carry materials. A cuboid block and a frozen pond can both be made of ice; a flat panel and a sphere can both be made of the same metal. Geometry and material are orthogonal axes of composition.
  • When a projectile encounters a surface, the material decides what happens to the projectile; the shape decides where the encounter is and what the surface normal looks like. The two questions are answered by different systems.
  • Line-of-sight queries treat every shape carrying the same material the same way: if one shape carrying a material occludes or reflects a sight ray, every other shape carrying that material does so under the same rule. The intersection math reads from the shape; the policy reads from the material.
  • Visual identity is part of the contract. Same shader, same material parameters, same reflection probes, same hit reactions. The player should see immediately that the pond and the block are made of the same stuff.

This is Single Source Of Truth applied to physics surfaces. If a feature ever seems to require a material to behave differently on one shape than on another, the right answer is almost always two materials with different parameter values — not a branch inside one material that switches on which piece owns the surface. Branches inside material behavior are how reflectivity drifts between two surfaces that the player still perceives as identical.

Resource Movement Flows Through Pylons

The general gameplay shape is simple: buildings build units, units build buildings, and both do that work through their turrets. The economy that fuels those turrets — production, consumption, conversion, construction payment, repair drain, future ammo costs — is its own shared abstraction: the resource pylon. Every entity that adds to a resource pool, draws from one, or moves resource between pools carries one pylon per affected resource, and that pylon is the single source of truth for both the simulation accounting and the on-screen visualization at that point on the host.

There must be no resource movement that bypasses this contract. The sim does not have a parallel "production" path that nudges the player’s stockpile directly while construction payments go through one helper and converter ticks go through another; the renderer does not have a special-case "factory sparkle," "solar trickle," or "converter glow" that lives outside the pylon rig. One channel in the sim, one rig in the renderer, every resource movement.

Hard invariant: resources are visible only as pylon balls

Every resource flow must be represented as resource balls moving through a host-mounted resource pylon. This is not only a rendering preference; it is the contract that keeps economy accounting, host identity, target identity, and visual language aligned. If a resource is produced, consumed, converted, refunded, reclaimed, repaired, spent as upkeep, paid as construction cost, or spent as a future ammo/ability cost, the player must be able to follow that movement through the correct resource pylon on the host entity.

  • When a host uses a resource, balls are born at the host/root side, travel up the matching pylon tube, leave the pylon tip, and move toward the thing being funded, consumed, charged, repaired, built, fired, or converted.
  • When a host gets a resource, balls come from the world/source side, enter the matching pylon tip, travel down the tube, and disappear into the host/root side.
  • A host that touches both energy and metal must expose two independent pylons. A combined "economy effect," shared glow, ring, meter, sparkle, particle cloud, or building-specific animation is not allowed to carry resource-flow meaning unless it is built from those pylon balls.
  • Any code path that changes a stockpile without publishing the corresponding pylon flow is a correctness bug. Any renderer path that visualizes resource movement without consuming the shared pylon-flow data is also a correctness bug, even if the accounting is correct.

What a pylon is

  • A pylon is tagged with exactly one resource type: energy or metal. A host that touches both resources mounts one pylon per resource — not one combined pylon with a polymorphic type. Per-resource pylons keep the player’s read of "what is flowing where" decomposable.
  • A pylon’s flow is communicated entirely by its pylon resource balls: discrete coloured spheres that visibly flow either INTO the pylon head or OUT of it. Their density — how many are in flight per second — encodes the magnitude of the resource rate, and their direction encodes which way the resource is moving. There is no separate bar or readout beside them; the moving balls are the whole indicator.
  • A pylon’s direction encodes the local physical flow of the resource at the pylon’s tip — the head, where the pylon meets the world. Inbound balls mean the world is feeding the tip and the resource then rides down the tube into the pool — a solar collector pulling energy from the sun, an extractor pulling metal from the deposit beneath it, a converter’s metal tip catching the metal its energy tip just made. Outbound balls mean the resource first rises up the tube from the pool and the tip then projects it into the world — a construction turret sending energy and metal to a build site, a converter’s energy tip ejecting the energy it is consuming. Down the tube is always gaining and up the tube is always spending, so the net effect on the player’s stockpile is recovered from the host’s role plus this single direction; it is not encoded by some second orthogonal field on the pylon.
  • A pylon that has a discrete target — a construction-emitter pylon spraying a build site — aims its ball flow at that target. A pylon that interacts with "the pool" or "the environment" as a whole — solar, wind, extractor, converter — uses random world-space directions for the inbound or outbound balls. The absence of a target is a real configuration of the contract, not a missing field.

A pylon is a conduit: root, tube, head

A pylon is not a point; it is a conduit with two ends and a tube between them. The root is the bottom, where the pylon taps the player’s shared resource pool. The head — the tip — is the top, where the pylon meets the physical world: the sky, the wind, a metal deposit, a build site. The tube is the visible shaft the resource balls travel along between the two. Inbound and outbound, defined above at the tip, are simply the two ends of one journey: balls travelling down the tube mean the host is gaining that resource (production, reclaim, conversion output), and balls travelling up the tube mean the host is spending it (construction, repair, conversion input, upkeep).

The tip conserves. It is a pass-through, never a spawn and never a sink: every ball that travels up the tube must leave through the tip, and every ball that enters the tip must travel down the tube. Resource balls are created and destroyed only at the two real endpoints — the root, where they fade into or out of the abstract pool, and the world point (deposit, sky, build target) on the far side of the tip. So a producer ball’s whole life is world-source → into the tip → down the tube → gone at the root, and a consumer ball’s whole life is born at the root → up the tube → out the tip → onto the target. Nothing appears or vanishes at the tip itself; if a ball reaches the tip, it has somewhere it must go. Every ball is accounted for before and after the tip.

Because the moving stream carries both direction (which way along the tube) and magnitude (how densely the balls flow), the flowing balls are the entire resource indicator: the standalone resource-usage indicators this replaces are retired, and no separate magnitude bar stands beside them. A pylon either shows its balls in motion or, when nothing is flowing, shows nothing at all.

Rate gates a ball’s birth, not its flight

The resource rate has exactly one job: it sets how often new balls are born — the spawn cadence, and therefore the density of the stream. It must never reach into the tube and retune, relocate, or destroy a ball that is already moving. A ball is committed the instant it is created. From then on it travels its whole documented life at a constant pace — born at the root → up the tube → out the tip → onto the target, or world-source → into the tip → down the tube → gone at the root — even if the rate drops to zero or spikes the instant after it spawned. A rate change retunes the cadence for the next balls; every ball already in flight is immune and finishes its journey.

This is what makes tip conservation literal and one-to-one. The tube leg and the free leg outside the tip are not two independent rate-driven animations that happen to line up at the head; they are one conserved stream rendered in two pieces, handed off ball for ball. A ball climbing the tube that reaches the tip disappears there and spawns exactly one ball that continues — the outbound free-leg ball that flies from the tip onto the build target. Symmetrically, an inbound free-leg ball that arrives at the tip from the world disappears and spawns exactly one ball that continues down the tube toward the root. The hand-off is the whole point: nothing is created or consumed at the tip, a ball merely changes which leg — and which renderer — is drawing it, and the count on each side of the tip stays balanced over time.

The failure this rule forbids is a tube whose live ball count is recomputed from the current rate every frame. Lower the rate and the surplus balls — ones that were already halfway up the shaft — simply wink out mid-tube; raise it and balls pop into existence in the middle of the bore. Both read as resource that was accounted for and then evaporated, or that appeared from nowhere — the exact opposite of every ball is accounted for before and after the tip. The free-leg balls outside the tube already obey this rule: each carries its own start point, end point, and lifetime, so a stalled or finished build never strands a ball in mid-air — it finishes its arc and only then stops being replaced. The tube balls must live by the same contract.

Density is the absolute transfer rate, scaled by one global knob

How many balls a pylon spawns per second is set by the absolute resource transfer rate — resources per second actually moving — not by the fraction of the host’s per-tick cap it happens to be using. A large factory paying hundreds of metal a second and a tiny builder paying a handful both spawn balls in direct proportion to what they move, so the factory’s tube visibly runs thick while the builder’s only trickles — even if both are nominally “at 100% of their own cap.” This is the resource-ball reading of two pieces that move the same resource at the same rate must look identical: balls measure throughput, and throughput is absolute, not relative to the mover’s size. Reading the rate off a cap fraction is exactly the drift this retires.

The mapping from transfer rate to ball cadence is a single global scalar, authored once in resourceConfig.json as ballsPerResourcePerSecond: balls per second = resourceTransferRatePerSecond × ballsPerResourcePerSecond. Raising it makes every stream on the map denser for the same economy; lowering it thins them all out. It is the one toggle for resource-ball density, and it changes only how finely a flow is sampled into discrete spheres, never what a ball means. Because it feeds a spawn cadence and never an in-flight count, turning it down mid-flow lets the balls already travelling finish and just slows the births behind them (see Rate gates a ball’s birth, not its flight). All resource-ball tuning — this scalar plus the cosmetic spray/tube constants — lives in resourceConfig.json rather than as literals scattered through the renderers (Config Is Data, Not Code).

Who carries which pylons

Every host that affects a resource pool must mount the pylons that match what it does. These are not optional decorations.

  • Solar collector — one energy pylon, inbound: balls fall in from the sun/sky direction, enter the tip, and ride down the tube into the pool; rate tracks current production.
  • Wind turbine — one energy pylon, inbound: balls stream in horizontally from the upwind direction (the same wind vector that drives turbine rate), through the tip and down the tube; rate scales with wind speed and the open/closed state.
  • Metal extractor — one metal pylon, inbound: balls rise from the metal deposit cells beneath it, into the tip, and down the tube; rate scales with deposit coverage.
  • Construction turret (on commanders, fabricators, future construction aircraft) — one energy pylon and one metal pylon, both outbound: balls rise up each tube from the pool, leave the tip, and fly onto the build target — ideally onto the host assembly currently under construction, rather than a separate turret sub-piece. Ball density on each tube matches the per-resource throughput the builder or factory is actually paying this tick.
  • Resource converter — one energy pylon and one metal pylon running as a back-to-back consumer and producer. Converting energy→metal, the energy pylon runs outbound (balls up its tube, out its tip) and the metal pylon runs inbound (balls in at its tip, down its tube), joined by a taxed arc that recolors energy into metal in flight (see The converter is two tubes and a taxed arc). There is no separate "converter art" — the converter is two pylons running in opposing modes, and the same rig reversed would be a future metal→energy converter.
  • Any new building, unit, tower, or host-mounted turret behavior that touches a resource pool gets pylons for the resources it touches. Upkeep mechanics, ammo-cost weapons that bill energy on fire, reclaim that returns metal, repair that drains energy — all of them surface through pylons. There is no "small enough to skip" exception.

The converter is two tubes and a taxed arc

A resource converter is the clearest test of the conduit model, because it is a consumer and a producer bolted back to back. Running energy→metal, its energy pylon draws balls up from the pool and ejects them out its tip in every direction (an outbound, consuming tip); its metal pylon receives balls at its tip and sends them down into the pool (an inbound, producing tip). The metal pylon therefore reads exactly like an extractor and the energy pylon exactly like a builder — same rig, opposing modes.

What ties the two tips together is a single taxed arc. Of the energy balls thrown from the energy tip, only the fraction set by the conversion ratio after tax arcs across to the metal tip; the remainder disperse into the air and are gone. The balls that cross change resource in flight, lerping from the energy color to the metal color in mid-air, and arrive at the metal tip to ride down its tube into the pool. The visible ratio of crossing-and-recoloring balls to dispersing balls is the conversion efficiency made physical — the tax is something the player can watch leak away. The arc still obeys tip conservation: a recolored ball that reaches the metal tip goes down that tube, and a ball that disperses never entered a tip to begin with.

One channel in the sim, one rig in the renderer

On the simulation side, every per-tick stockpile delta — production loops, build payments, repair drains, converter ticks, reclaim recovery, anything — flows through a single resource-movement channel whose payload identifies the resource, the magnitude per second, the direction (inbound or outbound at the source pylon), the source entity, and, when applicable, the target entity. The accounting code that reduces those movements to per-player stockpile updates is the only code that touches stockpiles; there is no helper, no shortcut, no building-type-specific bypass that adjusts a player’s pool outside that channel.

On the renderer side, every pylon is the same rig type configured by resource + direction + rate + optional target, driven by the same channel the sim publishes. The pylon mesh and the resource-ball emission pipeline are shape- and host-agnostic: a building’s pylon and a unit-mounted construction-turret pylon render through one code path. Production-rate indicators, build-emitter rigs, and converter visuals all collapse into this one rig. When a new feature is tempted to add another resource-flow indicator, the answer is to render it as a pylon, not to invent a parallel visual.

This is the resource-side counterpart to Materials Are Independent Of Shape and Single Source Of Truth. The material rule says the surface contract is constant across shapes; the pylon rule says the resource-flow contract is constant across hosts. Two pieces that move the same resource at the same rate must look identical and accumulate identically. Branches that move resources outside the pylon channel are exactly the drift surface this philosophy is trying to retire.

Config Is Data, Not Code

Blueprints, tuning tables, and authored content belong in language-neutral data files — JSON today, possibly a more compact schema-validated format later — not in TypeScript modules. As the simulation moves toward Rust and WebAssembly, every piece of TypeScript-only data is a wall between the current code and the end-state engine. A blueprint authored as a .ts file forces TypeScript to stay in the simulation loop forever, because Rust cannot import it. The same blueprint authored as .json is read by both, owned by neither, and survives the language migration intact.

This is the data-side counterpart to Languages and Frameworks. That section commits the hot kernels to Rust/WASM. This section commits to making sure the data those kernels need is not trapped inside a TypeScript module that prevents the migration from ever completing. Configuration in code looks convenient on the way in; it becomes a permanent dependency on the language you were trying to leave.

What belongs in JSON

  • Unit, turret, shot, and building blueprints — every field on every atomic gameplay piece (a unit’s locomotion among the unit’s own fields).
  • Tuning tables: damage curves, hover constants, friction values, mass tables, range tables, cost tables, build-time tables, recoil profiles.
  • Maps, scenarios, starting layouts, preset compositions, AI tuning, and any other authored content.
  • Wire codes, enum value-to-name maps, and cross-language identifiers that both TS and Rust must agree on. One file, two consumers, zero drift.

How to express behavior as data

Blueprints must not contain TypeScript functions, closures, or computed references. If a behavior requires a decision — damage falloff over range, turret aim style, shot trajectory shape, hover response curve — it is encoded as a discriminated union of data shapes that the simulation interprets, never as a callback smuggled into a blueprint. The blueprint says { "shape": "parabolic", "peak": 40, "gravity": 9.8 }; the simulation knows how to read that shape. This keeps every blueprint serializable, diffable, hot-reloadable, and parseable by Rust without a JavaScript runtime in the room.

Schema is the contract

  • One schema per atomic piece, declared in one canonical place, lives next to the data it describes.
  • TypeScript types and Rust structs are generated from the schema. They are not hand-maintained in two languages — that would just relocate the duplication problem from data to types.
  • Loaders in both TypeScript and Rust validate against the schema at startup. A blueprint that fails validation is a build error or a startup failure, not a silent runtime divergence.
  • Adding a field is a schema change. Both code generators run. Both languages see the field at the same time. There is no “I added it on the TS side and Rust never saw it” class of drift.
  • Defaults live in the schema, not in the loader. Every consumer reads the same defaulted shape.

Migration direction

Every blueprint file moved from .ts to .json is a step toward removing TypeScript from the simulation entirely. The shape stays the same; the syntax changes; the callers stop importing a module and start loading a file. As blueprints become data, the Rust/WASM simulation gains the ability to consume them directly, and the TypeScript blueprint modules disappear under Delete The Old Path. The end state is a simulation written entirely in Rust, loading entirely from JSON, with TypeScript surviving only as UI, networking glue, and orchestration — exactly the role Languages and Frameworks reserves for it.

Single Source Of Truth

No gameplay rule, command builder, validation path, physics formula, serializer shape, blueprint contract, UI label, or tuning table should be duplicated. If two systems need the same atomic action, that action belongs in one small module with one clear owner and every caller imports it. Copying logic between client, server, renderer, lobby, tests, and debug tools creates drift, and drift is a correctness bug.

File size is part of that source-of-truth discipline. Large files are not a neutral inconvenience; they are a signal that responsibilities have piled up and the code is harder to audit, test, port to Rust/WASM, or delete cleanly. When we touch a large file, we should split cohesive behavior into small helper files with named exported functions and import those functions at the call site. Do not copy a block into a second place to avoid an import. Do not leave a large file large when a clean helper extraction is available. The default shape is always small files, imported helpers, and one implementation of every behavior.

  • Extract shared helpers before copying code into a second place.
  • Split large files into small helper files with imported functions; never duplicate code to avoid a helper import.
  • Keep constants, enums, labels, thresholds, and wire codes in one canonical module.
  • Put command construction, authorization, and sanitization behind shared functions instead of reimplementing branches in components.
  • When duplication is temporary during a migration, delete the old path in the same change that proves the new one. See Delete The Old Path.

Delete The Old Path

Migration is not done when the new path works. Migration is done when the old path is gone. As we remove libraries, port hot kernels from TypeScript to Rust/WASM, replace framework helpers with owned code, and rebuild renderer paths around owned buffers or lower-level GPU APIs, the old implementation must be removed in the same change that lands the new one. Two implementations of the same behavior is two sources of truth, two places to fix bugs, two places to read when something looks wrong six months later, and a near certainty that one of them will drift.

This is the active counterpart to Single Source Of Truth. SSoT is the static rule: do not write the same logic twice. Delete The Old Path is the dynamic rule that keeps SSoT true while the codebase is in motion. Every time we replace a system, the replacement is incomplete until the predecessor file is deleted, its imports are gone, its config flags are gone, its wire codes are gone, and its tests are gone or repointed.

One language per behavior

The most common shape of this rule in practice is the JavaScript/TypeScript-to-Rust migration. The moment a Rust/WASM implementation works — or a JS/TS library has been replaced by owned code that covers its functionality — the predecessor TypeScript file, JavaScript module, or third-party library must leave the tree in the same change. Two language implementations of the same behavior is not a transition state we are allowed to rest in: exactly one of them becomes the path that actually runs, and the other rots silently while still being importable, callable, and grep-visible.

We need one version of code, not separate code between multiple languages or libraries. A physics kernel ported to Rust deletes the TypeScript physics kernel it replaced. A renderer path rebuilt around a direct typed-buffer or custom-shader contract deletes the older per-object renderer path it replaced, whether the final draw still goes through Three.js or a lower-level API. A custom data structure or algorithm that supplants a library removes that library from package.json, its lockfile entry, and every import only when the library itself is no longer part of the chosen architecture. “Working in Rust” is the trigger; there is no “we’ll keep the TS version for safety” pause, no “leave the library installed in case we need it,” no parallel-paths phase that lingers across more than one commit.

This is an immediate deletion rule, not a cleanup preference. When a Rust/WASM version is working, the old JavaScript or TypeScript implementation is no longer allowed to coexist as duplicate functionality. It must be removed before the change is considered complete, because this codebase must never contain two versions of the same behavior split between JavaScript/TypeScript and Rust/WASM.

The end-state shape is one implementation per behavior, written in the language that layer commits to: Rust/WASM for simulation, physics, and dense numeric work; TypeScript for coordination, UI, and IO; custom GLSL / WGSL / GPU-buffer code for renderer hot paths. A behavior that exists in two of those languages is a half-finished migration, not a redundancy worth preserving. The moment the replacement works, the predecessor is dead code — and dead code in the working tree is treated as a bug to be fixed in the same change, not a TODO for later.

Why this matters here specifically

  • We are actively moving from JavaScript object loops and JavaScript physics math toward Rust/WASM kernels over typed-array linear memory. Leaving the JS implementation alongside the WASM kernel guarantees one becomes the de-facto path and the other rots, undetected, until something invokes it and produces a result that disagrees with the live path.
  • We are moving from per-object Three.js convenience helpers toward owned buffers, owned shaders, and direct instancing. Three.js may remain the backend, but a renderer with two active paths for the same visual makes it impossible to reason about draw-call count, allocation behavior, material compilation, or culling — the two paths interfere even when only one is supposed to be active.
  • Pooled DTOs, prediction state, snapshot buffers, and entity-state shapes are sensitive to dead code paths that still write to them. A “removed” system that still touches pooled fields is a class of zombie-write bug that does not surface in normal tests.
  • Future readers — including future-us — should never have to ask “is this the current path or the old one?”. The codebase should answer that question by only containing the current path.

Rules during a migration

  • The change that introduces the replacement is the change that deletes the predecessor. Not the next change. Not a follow-up cleanup ticket.
  • If parity proof genuinely needs more than one change, the predecessor lives behind a single named flag, with a single owner, and a recorded removal target. There is no “we’ll come back to it.”
  • Do not keep “just in case” copies of the old implementation in legacy/, old/, _unused/, or sibling files with a .old suffix. Git history is the just-in-case copy.
  • Delete the imports, the dead branches, the feature flag, the legacy wire codes, the unused config knobs, and the tests that exercise only the dead path. A half-deleted system is worse than an undeleted one because it looks intentional.
  • If a library is being replaced, remove the dependency from package.json, Cargo.toml, and the lockfiles — not just the call sites. A still-installed library will be re-adopted by the next person reaching for the easy answer.
  • Renaming files is not a migration. If the new path lives next to the old path with a slightly different name and both still compile, no migration has happened — only confusion has been added.

What “removed” means

Removed means the code is not in the working tree, not behind a flag, not in a legacy/ folder, not in a sibling file with a .old suffix, and not in commented-out blocks. Git history is the archive; the working tree is the current truth. When someone greps for the name of an old function and finds zero matches, the migration is done. When they find one match in the active path and one match in a stale path, the migration is still in progress, and the next person to touch either file should finish it.

Explicit Absence

We are moving away from optional properties and optional chaining in our own simulation, server, prediction, and entity-state code. Absence should be declared, not inferred. A field should always exist with a typed value, and the absence of a value should be expressed as null (or a numeric sentinel in hot paths), so every site that reads it must make an explicit decision about what to do when the value is missing.

The problem we are solving is "seemingly ready when actually not." An optional field collapses three different failure modes into the same silent no-op: not yet initialized, semantically inapplicable, and previously torn down. An entity.turret?.fire() chain swallows all three. The unit just does not shoot, no stack trace, no symptom anyone can grep for. In an online multiplayer RTS with pooled state and delta-encoded snapshots, that class of bug compounds.

Why this matters more in this codebase

  • Pooled objects (snapshot DTOs, prediction state, audio events, scratch buffers) carry zombie fields between users when those fields are T? rather than T | null with an explicit reset on release.
  • V8 deoptimizes when objects have shape variance. field: T | null = null keeps the hidden class uniform and the JIT happy, which is measurable in 5,000-unit per-tick loops.
  • Snapshot serialization already encodes presence explicitly on the wire. Aligning the in-memory model with that contract eliminates a category of "I sent it but the receiver never had the property" drift bugs.
  • Determinism requires that absence is a decision, not a coincidence. ?. hides the decision; if (x === null) forces it into the code under review.

Scope of the rule

  • Inside our own simulation, server, prediction, networking, and entity-state modules: no ?: on properties, no ?. on access. Use T | null with explicit guards.
  • At third-party boundaries (Three.js, browser APIs, WebRTC): pragmatic optional chaining is acceptable. Wrap the boundary; keep our domain strict.
  • Function parameters with defaults (fn(x: number = 0)) are a different concept and remain allowed.
  • "Field is semantically inapplicable to this kind of entity" is not an optional field. It is a discriminated union. Express it that way.
  • In numeric hot loops, prefer a sentinel value (targetId = -1, tetherId = -1) over number | null. It avoids both the boxed null check and the shape variance, and it composes with WASM SoA storage.

Migration direction

This is a direction, not a hard cutover. We are not freezing the codebase to flip a lint switch. The rule we live by is simple: every file we touch should leave with fewer optional properties and fewer optional-chaining accesses than it had on the way in. Convert the field, fix the callers in that file, and move on. Over time the migration completes itself, the lint rule follows once the burden is small, and we stop shipping bugs that hide behind silent absence.

What New Code Should Optimize For

  • Keep the authoritative sim headless and portable. It should not depend on DOM, Vue, or Three.js.
  • Describe shared physical behavior once, then reuse it from both server authority and client prediction.
  • Keep the client prediction sim bounded to responsiveness and presentation unless a full-system prediction need is proven.
  • Keep rendering decoupled from simulation cadence; requestAnimationFrame samples state, it does not define truth.
  • Keep commands small, direct, validated, and host-owned.
  • Keep snapshots sparse, recipient-aware, and low cadence by default.
  • Keep client prediction physically compatible with the server, especially for motion and projectiles. Ship velocity, not acceleration: the server owns force inputs, the wire ships their integrated result, and the client extrapolates from velocity only.
  • Expose the per-channel EMA controls (movement position, movement velocity, rotation position, rotation velocity). Position channels use snap/fast/medium/slow; velocity channels additionally allow ignore so playtesters can tune each channel independently without rebuilds.
  • Make mass and force first-class fields in gameplay designs, not decorative metadata.
  • Drive locomotion with velocity-aware arrival control, not position-only waypoint seeking. Units should brake against their current velocity when stopping distance runs out, and ground, hovercraft, and air profiles should share this controller while differing through physical limits.
  • Apply gravity to every mass body every tick; guided projectiles cancel it inside their homing equation by adding a counter-gravity thrust term, not by setting an ignoresGravity flag that skips integration.
  • Keep locomotion visuals entirely client-side: the sim publishes one chassis point, the rig places every wheel, cleat, foot, or fan in the world. Each rig clamps every part it owns to max(naturalY, terrainY + clearance) as a hard invariant — tunneling is never allowed, floating over dips is.
  • Carry per-part visual state in the same four channels client prediction uses — movement position, movement velocity, rotation position, rotation velocity — and EMA each channel toward its target every frame. Animation always plays off body motion; the rig never branches on whether a part is touching ground. The floor clamp drives the position channel; the body-motion signed distance drives the velocity channel; rotation integrates from rotation velocity. Hover replaces all of this with a per-frame clearance scalar.
  • Keep the camera frontend and exhaustively enumerated: a gesture’s anchor point is the product of two orthogonal data axes — anchorPointCameraScreen (cursor / screen-center) and anchorPointCameraTerrain (plane-2d / terrain-3d / terrain-3d-water) — and whether the camera may pass through terrain is a separate boolean. All six anchor behaviors and both collision behaviors must be reachable from config, never from a hardcoded branch in the picker.
  • Keep unit, tower, turret, shot, ray, and building definitions atomic, swappable, and free of overlapping responsibility. A unit owns its locomotion inline — movement is part of the unit, not a separate piece, blueprint, or entity, and it has no id, cost, health, or lock-on eligibility of its own. unit is mobile and has turrets; tower is immobile and has turrets; building is immobile and has no turrets. Anything static that locks onto an entity ID through a turret is a tower, not a building — fabricators and shooting towers both belong on the tower side of that boundary. Make the turret a standardized, addressable mounted emitter that names exactly one emission by id and carries no inline emission payload, so it drops onto any host’s mount unchanged. A turret has no independent cost, mass, health, death explosion, construction phase, or detached life. A turret emits exactly one of two separate kinds, each in its own blueprint file: a shot (a travelling munition that is a body with mass/health/death-explosion, in shots.json) or a ray (a sustained line emission — beam or laser variants — no body, in rays.json). Only the shot is a body; rays are emitter-bound emissions with no mass, health, or id of their own.
  • Give every physical body the same base ledger before its kind-specific fields: a universal EntityId, a { metal, energy } cost, a mass, a health pool, and a { radius, force, damage } death explosion. These are a floor that unit, tower, building, and shot bodies all meet. A turret is still an addressable entity for lock-on and provenance, but its resources and lifetime belong to the host. A unit’s locomotion is the same kind of intrinsic host behavior: part of the body, sharing the unit’s id, cost, and health.
  • Make every body death an explosion. When a unit, tower, building, or shot’s health hits zero it detonates its death blast, applying outward force (heavier bodies resist more) and damage to everything in its radius, friend or foe. A shot’s detonation is the smallest case; a host death ends its mounted turrets as part of the same assembly, without separate turret health checks, detached turret pieces, or turret death explosions.
  • Name body geometry by physical role, not by ad-hoc size: every physical body carries three radii — radiusVisual (drawn size), radiusHitbox (the damage-receiving hurtbox), and radiusCollision (body-vs-body no-overlap contact) — with the same three names meaning the same things on units, towers, buildings, and shots. A mounted turret may expose an addressable aim/selection volume for lock-on and line targeting, but that volume is a mount target on the host rather than an independent body. Keep the death explosion’s own radius separate from this trio, and keep the word “push” reserved for a future sustained push-field capability rather than this hard-contact radius.
  • Keep agency and command distinct: host bodies are agents with their own health and death, but the player commands only the three host kinds (units, towers, buildings). A unit moves itself and controls its turrets; a tower controls its turrets; a turret manages aiming and firing while its host exists and is either host-directed or fully-autonomous; a fired shot runs itself. Control runs down that chain, never past a host to puppet a part directly. Turrets remain mounted on their host for their entire life.
  • Do not let turrets live, die, detach, or build separately from their host. A turret has no independent cost, mass, health, or death explosion; killing the host kills the whole mounted assembly. A unit’s locomotion likewise has no health of its own — it is part of the body, so the only way to stop a unit moving is to kill it. Construction creates the host assembly as one product with its authored turret set, not body first and turrets later.
  • Make shots destructible bodies: every shot carries a tiny health pool and detonates the instant it reaches zero, running its normal explosion/splash/submunition path. Contact with the ground, another shot, or any solid body forces that health to zero on the spot. A shot is only entity-contact-active after its hitbox clears the originating host and turret collision bodies, but once armed it collides with everything, including the turret or host that fired it. Small-but-real shot health is what lets a point-defense turret shoot incoming munitions out of the air.
  • Give every addressable live instance — unit, tower, building, mounted turret, travelling munition, and build site — a universal runtime EntityId plus the host ids it belongs to: top-level hosts point root_host_id at themselves, mounted turrets store parent_id, root_host_id, and mount index. A unit’s locomotion is not on that list — it is part of the unit body, not a separately addressable instance. Assign ids by addressability, not independent bodyhood: a mounted turret can be locked onto and can attribute emissions without carrying its own health; a ray sustained by a live turret attributes through its emitter instead of minting its own id; and a build site is the product carrying its final id from frame one rather than a shell that hands off later. Resolve that opaque ID through explicit metadata for kind, blueprint, owner player, team, parent, root host, mount index, storage row, generation, liveness, and targetability. Do not encode mutable meaning into the numeric ID itself. Shots carry immutable source provenance (source_turret_id, source_host_id, player/team, blueprint IDs, spawn tick, and parent shot) so projectiles, submunitions, damage attribution, replays, and debug traces all agree on where they came from.
  • Keep host command targets separate from turret lock-on targets. A unit or tower may carry player intent such as attack, guard, repair, or assist, and may lock onto an entity that has an ID — of any kind, including a single mounted turret or a travelling shot — but only the kinds and relationships its own host-level lock-on inclusion object admits; each mounted turret may then inherit that target only if its own inclusion object and targeting gates accept it. Turrets, units, and towers all carry the same additive lock-on inclusion object; buildings carry none, and no host target, because they mount no turrets.
  • Tag every turret mount as host-directed (inherits the host lock-on) or fully-autonomous on the unit or tower blueprint — not on the turret blueprint, so the same turret can be host-directed on one host and fully-autonomous on another. Zero, one, or many mounts can be host-directed; the blueprint loader throws only when the boolean choice is missing. When the host is locked on to another entity, each host-directed turret’s lock-on is overwritten by the host’s, subject to that turret’s inclusion object; fully-autonomous turrets are never overwritten and run their own independent targeting flow. Host-directed turrets fall back to autonomous targeting when the host has no lock. Fight-move and patrol halt logic is a separate unit-mount flag, requiredEngagedForFightStop; every marked non-visual turret must be engaged before the unit halts.
  • Make the selection action panel uniform per entity type. Units expose movement, fire control, build, commander specials, and self-destruct. Towers expose set-lock-on (any entity with an ID), fire control, fabricator production (when applicable), and self-destruct. Buildings expose ON/OFF and self-destruct. Mixed selections degrade to the intersection. Adding a new button to only some blueprints inside an existing type is not allowed.
  • Keep sight, radar, and detection as sensor capability sources aggregated per player/team. Full sight identifies and also supplies radar-level location, radar-only coverage locates, detectors uncloak, and weapon fire consumes those tiers through explicit turret targeting requirements. Weapon range must not automatically create full vision.
  • Keep lock-on acquisition in turret blueprints and targeting kernels, and author lock-on eligibility as an additive inclusion object carried by turrets, units, and towers alike. Rockets inherit the spawning turret’s lock at launch, but they never scan for replacement targets; eligibility is off by default — a locker with no inclusions locks onto nothing — and broadened only by addition: level-0 friend/enemy and entity-family inclusions (units, towers, buildings, turrets, shots), then optional level-1 named blueprint inclusions that narrow an included family. Including only one family is how a turret specializes — point-defense includes only shots, a disarmer includes only turrets. There is no locomotions family: a unit’s movement is part of its body, not a targetable entity.
  • Keep specialist targeting behavior explicit. If a turret ever needs behavior beyond additive lock-on eligibility, give it a validated secondary lock-on profile and tests rather than a special case buried in the shared targeting kernel.
  • Treat surface materials as shape-independent contracts: a material defines reflectivity, occlusion, hit response, and shader treatment once, and every shape that mounts it must look and behave identically. A frozen pond and a cuboid ice block are the exact same material — they differ only in geometry.
  • Route every resource movement — production, consumption, construction payment, conversion, repair drain, future ammo costs — through one shared pylon abstraction in both simulation and renderer. One typed channel publishes (resource, amount, direction, source, optional target); one rig renders the flow as resource balls travelling through a conduit, their density showing magnitude and their travel direction showing inbound or outbound. A pylon is a tube between a root (the pool tap) and a tip (the world); balls down the tube mean gaining, up the tube mean spending, and the tip conserves — every ball up the tube exits the tip, every ball into the tip goes down the tube, and balls are spawned or sunk only at the root and the world point, never at the tip. Producers source balls at the physical world point (deposit, sun, upwind) and run them down the tube; construction turrets run balls up the tube and out the tip onto the build target; the converter joins its two tips with a taxed arc that recolors energy into metal in flight, the unconverted fraction visibly dispersing. The flowing balls are the sole indicator; standalone usage readouts and any separate magnitude bar are retired. No parallel resource paths in the sim, no special-case “production trickle” or “converter visual” in the renderer.
  • Keep every atomic action and rule in one source-of-truth module, then import it everywhere it is needed.
  • Keep files small. If a file is large, split cohesive behavior into helper functions exported from focused helper files and import them; never duplicate code and never let file size become the reason a behavior is hard to find, test, port, or delete.
  • When a replacement lands — a WASM kernel for a JS loop, owned shaders for a framework helper, an owned-buffer renderer path for an older per-object path, a removed library — delete the predecessor in the same change. Migration is not done until the old path is gone from the tree.
  • Prefer explicit null (or numeric sentinels) over optional properties and optional chaining in our own code; treat every file you touch as an opportunity to migrate fields off ?: and ?..
  • Use Rust/WASM, TypeScript orchestration, and custom GPU shader code; keep frameworks out of simulation hot paths and keep renderer framework use behind owned buffer, shader, culling, and batching contracts.
  • Stack the runtime so TypeScript coordinates and WASM computes: the TS shell handles boot, UI, IO, and WASM calls; the Rust/WASM engine owns every per-unit and per-tick loop; the renderer backend owns GPU buffers, instancing, shader programs, and optional compute work. Hot data should cross the JS/WASM boundary as rarely as the design allows.
  • Author blueprints, tuning tables, maps, and other content as JSON data validated by a shared schema, not as TypeScript modules. Data the Rust/WASM simulation cannot read is a wall that prevents the TS-to-Rust migration from ever completing.
  • Move dense numeric kernels toward WASM when batching can beat JS object loops.
  • Design every feature with 5,000 units in mind before celebrating it at 50 units.
